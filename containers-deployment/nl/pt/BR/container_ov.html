Sobre o contêineres únicos e escaláveis

O IBM® Bluemix fornece a infraestrutura do IBM Bluemix Container Service em regiões selecionadas como um recurso para desenvolvimento de app. Com os contêineres, é possível construir o app em qualquer idioma, com quaisquer ferramentas de programação. Cada contêiner é uma plataforma de app segura e isolada. Um contêiner pode ser executado em qualquer lugar: a partir de uma estação de trabalho de desenvolvimento que esteja executando o OS X ou Windows, um servidor que esteja executando o Ubuntu, uma máquina virtual do datacenter de produção que esteja executando o Red Hat ou uma plataforma de nuvem como o Bluemix.
Atenção: Avaliações grátis para o contêineres únicos e escaláveis não estão disponíveis. Saiba mais sobre gerenciamento de cluster e migração para o Kubernetes no IBM Bluemix Container Service no Bluemix Public.

O que é um contêiner?

Contêineres são objetos de software virtuais que incluem todos os elementos que um app precisa executar. Um contêiner é construído a partir de uma imagem, que é um modelo somente leitura para criar o contêiner. Cada imagem inclui apenas o app e suas dependências, executando como um processo isolado no sistema operacional do host. Portanto, um contêiner tem os benefícios de isolamento e alocação de recurso, mas é mais móvel e eficiente do que, por exemplo, uma máquina virtual. Os contêineres ajudam a construir apps de alta qualidade de modo mais rápido.

Contêineres são ágeis

Os contêineres simplificam a administração do sistema fornecendo ambientes padronizados para equipes de desenvolvimento e produção. O tempo de execução leve do mecanismo permite aumentar e diminuir a capacidade rapidamente em resposta às mudanças em demanda. Eles ajudam a remover a complexidade do gerenciamento de diferentes plataformas de sistema operacional e infraestrutura subjacente. Os contêineres ajudam a implementar e executar qualquer app em qualquer infraestrutura, de modo rápido e confiável.

Contêineres são pequenos

É possível ajustar mais contêineres na quantia de espaço que uma máquina virtual única requer. O tamanho de um contêiner pode ser selecionado para que você use somente a quantia de espaço que o app requer.

Contêineres são móveis

Construa uma imagem para outro contêiner usando outra imagem como base. Deixe que outra pessoa execute a o trabalho em massa em uma imagem e ajuste isso para seu uso. Também é possível migrar código do app de um ambiente temporário para um ambiente de produção rapidamente. O processo de migração pode ser automatizado com ferramentas como o Delivery Pipeline ou UrbanCode Deploy.

Docker e IBM Bluemix Container Service

O IBM Bluemix Container Service é um tempo de execução do Bluemix que alavanca a tecnologia Docker. Os usuários do Docker existentes e novos podem buscar os seguintes recursos com o IBM Bluemix Container Service.

  • Incluir as imagens existentes do Docker Hub em um registro privado hospedado pela IBM.
  • Validar imagens para vulnerabilidades potenciais.
  • Configurar a interface da linha de comandos (CLI) para o IBM Bluemix Container Service para usar os comandos da CLI do Docker que você já se sente confortável em usar.
  • Consultar dados de criação de log e monitoramento para seus contêineres sem configuração adicional.

Se você tiver usado o Docker no passado, poderá achar útil as respostas a essas perguntas comuns à medida que iniciar o IBM Bluemix Container Service.

Como eu uso uma imagem do Docker Hub no IBM Bluemix Container Service?
Usando a CLI do IBM Bluemix Container Service, é possível copiar uma imagem do Docker Hub para seu registro privado do Bluemix. Depois que a imagem for copiada, será possível usá-la para criar um contêiner.
Como criar um contêiner altamente disponível?
O IBM Bluemix Container Service fornece contêineres únicos que são semelhantes aos contêineres do Docker. Além disso, o IBM Bluemix Container Service fornece grupos de contêineres escaláveis que podem ser usados para tornar seu aplicativo altamente disponível. Os grupos de contêineres fornecem um balanceador de carga integrado para assegurar que a carga de trabalho seja roteada e equilibrada entre as instâncias de contêiner. Para aumentar a disponibilidade de seu grupo, difunda suas instâncias do grupo de contêiner entre vários hosts de cálculo físico com antiafinidade e ative a recuperação automática de instâncias inoperantes. Além disso, é possível configurar políticas para escalar automaticamente as instâncias do grupo de contêiner com base no uso de CPU e memória. Para obter mais informações, consulte Executando processos altamente disponíveis como grupos de contêineres.
Como mapear portas do contêiner para uma porta do host?
O IBM Bluemix Container Service diferencia entre a rede de contêiner privada e a rede pública. Na rede de contêiner privado, todas as portas do contêiner são expostas por padrão sem configuração adicional. Contêineres únicos e grupos de contêineres que são conectados à mesma rede de contêiner privado podem se comunicar usando seus endereços IP privados designados. Nenhum mapeamento de porta é necessário.

Para tornar seu aplicativo disponível na rede pública, deve-se expor uma porta pública e ligar um endereço IP público ao contêiner único ou uma rota pública ao seu grupo de contêiner. A exposição de uma porta pública permite enviar e receber dados públicos somente na porta exposta. Todas as outras portas públicas estão fechadas.

Como tornar um contêiner publicamente disponível na Internet?
  • Para contêineres únicos, exponha uma ou mais portas e designe um endereço IP flutuante ao contêiner.
  • Para grupos de contêiner, exponha uma porta HTTP única e designe uma rota.
O que torna o período de tempo para construir um contêiner diferente entre um Docker local e um Bluemix?
É possível observar que os contêineres são construídos mais rapidamente com sua instalação do Docker local do que no Bluemix. Com a primeira implementação, a imagem do contêiner deve ser transferida por download para o registro no host. Implementações subsequentes da imagem são mais rápidas. Para obter mais informações sobre como as velocidades de implementação são afetadas pelas diferentes configurações de contêiner, consulte Tipos de contêiner.
Como criar um contêiner em uma rede privada e não expô-lo publicamente?
Ao criar um contêiner, ele é conectado à rede privada de contêiner padrão do IBM Bluemix Container Service e um endereço IP privado é designado a seu contêiner. Nessa rede privada, todas as portas do contêiner são expostas por padrão. Nenhum mapeamento de portas do host para as portas do contêiner é necessário. Para evitar que um contêiner seja acessível pela Internet, não ligue um endereço IP ou exponha uma porta quando criar o contêiner.
Como montar um volume em um contêiner?
Um contêiner único é, pelo design, de curta duração. No entanto, é possível usar um volume para persistir dados entre as reinicializações do contêiner. Ao montar um volume no Docker, o volume é montado no sistema de arquivos local. No IBM Bluemix Container Service, o acesso ao host de cálculo é restrito, portanto, não é possível montar diretórios do host em um contêiner. Em vez disso, os volumes com escopo definido na organização são usados para persistir dados entre as reinicializações do contêiner. Os volumes são hospedados em compartilhamentos de arquivo isolados que armazenam com segurança dados de app e gerenciam o acesso e a permissão aos arquivos. Para montar um volume em um contêiner, deve-se criar um volume primeiro. Ao criar um contêiner na CLI, monte um volume existente usando a opção --volume. Na GUI do Bluemix, selecione as Opções avançadas e designe um volume existente. Para obter mais informações, consulte Incluindo arquivos em volumes com a interface da linha de comandos (CLI).
Como usar o Docker Compose com o IBM Bluemix Container Service?
É possível executar comandos do Docker Compose para criar e gerenciar implementações de múltiplos contêineres com o IBM Bluemix Container Service. Para obter mais informações, consulte Tutorial: Criando uma implementação de múltiplos contêineres com o Docker Compose e o IBM Bluemix Container Service.
Como configurar o cliente Docker Compose para que aponte para o serviço IBM Containers?
Ao efetuar login na CLI do IBM Bluemix Container Service, configure seus comandos nativos do Docker de execução da CLI. Para obter mais informações, veja Efetuando login no plug-in do bx ic.
Como efetuar login em meu contêiner em execução? SSH para ele?
É possível usar bx ic exec para efetuar login em um contêiner.
Por que meus contêineres estão sendo executados no modo separado?
Por padrão, os contêineres no Bluemix são executados no modo separado, para permitir que os contêineres com processo em execução sejam executados em segundo plano sem sair. Os contêineres em execução no modo separado são o modelo padrão para um ambiente de nuvem. É possível ainda conectar ao seu contêiner executando bx ic attach [--no-stdin] [--sig-proxy] CONTAINER.

IBM Bluemix Container Service no Bluemix Dedicated e Bluemix Local

O IBM Bluemix Container Service está disponível no Bluemix Dedicated e no Bluemix Local.

Requisitos de hardware para o IBM Bluemix Container Service no Bluemix Local

Requisitos de hardware de base
A instalação base é configurada com 64 GB de memória de capacidade de cálculo do contêiner por padrão.
  1. Seu ambiente deve atender aos requisitos de hardware do Bluemix Local.
  2. Para usar o IBM Bluemix Container Service, cada configuração requer um plano de controle que atenda requisitos a seguir. Duas instâncias também são necessárias para monitoramento e criação de log, atendendo aos mesmos requisitos de hardware.
    • VMware: três servidores ESX, cada um com 256 GB de memória e 32 núcleos físicos (64 vCore com HT ativado). Nos três servidores, são necessários 4 TB de armazenamento compartilhado de cluster.
Requisitos para aumentos adicionais de hardware
É possível aumentar a capacidade de cálculo do contêiner em incrementos de 16 GB até 256 GB sem hardware adicional, incluindo até 12 VMs (máquinas virtuais) de expansão. Cada VM deve ter 20 GB de memória, 4 vCPU e 200 GB de espaço em disco. No entanto, para cada 10 VMs de expansão implementadas no cluster, uma VM adicional é necessária para manipulação de rede. Essa VM de rede requer 4 vCPU, 16 GB de memória e 250 GB de espaço em disco.

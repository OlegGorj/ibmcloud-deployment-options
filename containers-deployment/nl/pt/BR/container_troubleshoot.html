Depurando contêineres e revisando logs

Revise as opções que você tem para depurar seus contêineres e encontrar as causas-raiz das falhas.

  1. Liste seus contêineres e revise o status atual.
    • Contêiner único
      bx ic ps -a
    • Grupo de contêiner
      bx ic groups
  2. Opcional: Consulte o que cada status significa e revise as dicas para evitar que um contêiner falhe.
  3. Se seu contêiner falhou ou travou, inspecione seus contêineres para descobrir a causa-raiz da falha. A mensagem de erro pode ser encontrada na seção Falha da resposta.
    • Contêiner único
      bx ic inspect CONTAINER
    • Grupo de contêiner
      1. Inspecione o grupo de contêiner.
        bx ic group-inspect GROUP
      2. Para obter informações adicionais, liste as instâncias de um grupo de contêiner e inspecione cada instância. 
        bx ic group-instances GROUP
        bx ic inspect GROUP_INSTANCE
  4. Se o app foi encerrado com uma falha, recupere os logs do app.
    Nota: Esse comando poderá ser usado somente se o app gravar logs nos fluxos de saída padrão STDOUT e STDERR. A gravação nesses fluxos de saída padrão permite visualizar os logs do contêiner, mesmo se o contêiner for encerrado ou travar.
    • Contêiner único
      bx ic logs CONTAINER
    • Grupo de contêiner
      1. Liste as instâncias de contêiner de um grupo de contêiner.
        bx ic group-instances GROUP
      2. Recupere os logs para cada instância. 
        bx ic logs GROUP_INSTANCE
  5. Se você precisar acessar logs de um contêiner que não está mais disponível, como uma instância de um grupo de contêiner que foi recriada com a recuperação automática, use Kibana para visualizar os logs.

    https://logmet.DomainName/

    Os logs ficam disponíveis por até 7 dias no Kibana.

  6. Se os logs foram salvos em um arquivo dentro do contêiner, deve-se efetuar login no contêiner para revisá-lo.
    Importante: Você só pode efetuar login em um contêiner em execução. Se seu contêiner travou e não pode ser iniciado novamente, você talvez não consiga acessar os logs dele. Considere mudar seu app para gravar nos fluxos de saída padrão STDOUT e STDERR para que você possa acessar os arquivos de log mesmo que o contêiner não esteja mais acessível.
    1. Se seu contêiner não estiver em execução, inicie-o. 
      bx ic start CONTAINER
    2. Efetue login no contêiner em execução. 
      bx ic exec -it CONTAINER bash
    3. Navegue para seu arquivo de log e revise os logs.
  7. Se os logs foram salvos em um volume e seu contêiner não pode ser mais iniciado, crie um novo contêiner para acessar o volume.
    1. Crie um novo contêiner que seja montado no volume no qual os arquivos de log são armazenados. 
      bx ic run --name logcontainer -m 64 --volume MOUNT_PATH_TO_VOLUME registry.DomainName/ibmliberty
    2. Efetue login no contêiner em execução. 
      bx ic exec -it logcontainer bash
    3. Navegue para o caminho de montagem do volume e revise os arquivos de log.
  8. Se você precisar de suporte adicional com a depuração seus contêineres, abra um chamado de suporte do Bluemix.

Resolvendo problemas de instalação da CLI

Erro de método (rpc) durante o init

Ao tentar executar bx ic init, você obterá uma das mensagens a seguir.
  • bx ic rpc: não é possível localizar o método CliRpcCmd.IsLoggedIn
  • bx ic rpc: não é possível localizar o método CliRpcCmd.IsMinCliVersion
A versão instalada do Cloud Foundry é uma versão anterior da que é suportada para o IBM Bluemix Container Service.
  1. Execute cf -v para ver qual versão do Cloud Foundry CLI está instalada.
  2. Se a versão instalada for anterior à 6.14.0, atualize a CLI do Cloud Foundry.

Erro de autenticação de registro recebido durante o bx ic init

Ao executar bx ic init, você verá uma das mensagens de erro de autenticação a seguir.
  • Could not authenticate with IBM Containers registry at registry.DomainName
****exit status 1
****time="2015-07-31T08:58:32-04:00" level=fatal msg="Post http:///var/run/docker.sock/v1.18/auth: dial 
unix /var/run/docker.sock: no such file or directory. Are you trying to connect to a TLS-enabled daemon without TLS?”
  • ** Retrieving client certificates from IBM Containers
Authentication failure. Check your username and password
Invalid token format. Please generate new token.
Um dos pré-requisitos de bx ic não foi atendido ou o IBM Bluemix Container Service não está disponível.
  1. Verifique se o daemon do Docker está em execução. Se não estiver, inicie-o e execute bx ic init novamente.
  2. Verifique se você está conectado à CLI do Cloud Foundry executando bx login. Então, efetue login no plug-in novamente executando bx ic init.
  3. Verifique a disponibilidade do IBM Bluemix Container Service. Verifique a página de status do Bluemix para obter informações.

O sudo é necessário com os comandos bx ic

Ao executar comandos bx ic, é obrigatório usar o prefixo sudo com todos os seus comandos.
Por padrão, os comandos bx ic devem ser executados com autenticação raiz e, portanto, podem requerer o prefixo sudo, dependendo de seu sistema operacional.
Para executar os comandos docker e bx ic sem sudo, execute o comando a seguir; se você estiver no Linux, deverá efetuar logout e depois efetuar login novamente. 
sudo usermod -a -G docker <CURRENT_USER>

Resolução de problemas de cotas

Um contêiner não pode ser criado devido à cota de organização insuficiente

Ao criar um contêiner, você recebe uma mensagem de que não têm cota suficiente em seu espaço.
BXNUI0110E: não foi possível alocar recursos do IBM Containers. Cota de ORG insuficiente para alocar 
ao novo ESPAÇO. Não é possível exceder os IPs flutuantes de sua organização 'organization'. Use o IBM
    Bluemix Gerenciar organizações para 
ajustar alocação de cota.
Essa mensagem pode ser exibida quando você tenta criar contêineres em um espaço que não tem cota suficiente alocada. Por padrão, cada espaço requer 2 endereços IP e 2 GB de memória a ser alocada para esse espaço para trabalhar com o serviço de contêiner.
Para distribuir a cota de memória e IP entre múltiplos espaços:
  1. Nos detalhes de sua conta, em Gerenciar organizações, selecione uma organização.
  2. Na seção de cota, visualize os detalhes para contêineres.
  3. Clique em Editar e aloque a memória e os endereços IP nos espaços na organização.
  4. Crie um contêiner em qualquer espaço na organização.

A conta não pode ser editada, embora você tenha uma conta paga

Não é possível modificar sua cota do contêiner.
Pode haver inconsistências entre o gerenciador da organização e as informações de faturamento.
O usuário conectado deve ser o gerenciador da organização. Para verificar o proprietário, conclua estas etapas.
  1. Nos detalhes de sua conta, em Gerenciar organizações, selecione uma organização.
  2. Na lista de usuários, verifique se o gerenciador da organização é o mesmo que o ID com o qual você está efetuando login.
  3. Efetue logout da conta e efetue login novamente.
Se você concluir estas etapas e ainda não puder editar a cota do contêiner, entre em contato com o Suporte IBM.

Se você tiver problemas relacionados especificamente a essa imagem Dockerfile, use o Rastreador de problemas GitHub. Para problemas mais gerais relativos ao IBM WebSphere Application Server Liberty, é possível obter ajuda acessando a comunidade WASdev.

Não é possível puxar a imagem de seu registro de imagens privadas do Bluemix

Você criou um contêiner a partir de uma imagem de contêiner em seu registro privado do Bluemix, mas o contêiner fica preso em um estado CONSTRUINDO. Quando você executa o comando bx ic inspect, a mensagem de erro a seguir é mostrada na seção Env de resposta de sua CLI. 
"Env": [
            "metadata_exit_code=-1",
            "logging_password=",
            "space_id=0e65c436-9413-46e2-8b42-ff06aa74ac21",
            "logstash_target=logmet.opvis.bluemix.net:9091",
            "metadata_err_msg=Cannot pull image from registry: Pull failed with error message failed 
             to register layer: ApplyLayer exit status 1 stdout:  stderr: Container ID 1234567890 
             cannot be mapped to a host ID",
            "metrics_target=logmet.opvis.bluemix.net:9095",
            "metadata_running=1"
No IBM Bluemix Container Service, o Recurso de namespace do usuário do Docker é ativado no Docker Engine. O recurso de namespace do usuário foi introduzido pela primeira vez com o Docker versão 1.10 e fornece a possibilidade de mapear um usuário raiz no contêiner para um usuário não raiz fora do contêiner. Dessa maneira, as permissões de um usuário que pode executar ações raiz dentro do contêiner podem ficar restritas fora do contêiner, portanto, os arquivos no host e em outros contêineres não podem ser acessados, alterados ou removidos.
Nota: O namespace do usuário do Docker é diferente do namespace configurado no Bluemix para identificar seu registro de imagens privadas.

Ao criar um contêiner a partir de uma imagem de contêiner com o comando bx ic run, a imagem é carregada de seu registro de imagens privadas do Bluemix para o IBM Cloud Docker Engine. O Docker Engine então começa a criar as respectivas camadas de contêiner. Dependendo do que está especificado na imagem do contêiner, arquivos de app de sua máquina local podem ser incluídos no contêiner. Todos esses arquivos são criados por um ID de usuário (UID) e um ID de grupo (GID) em sua máquina local. O UID e o GID indicam a permissão do usuário na máquina local. Quando você desejar incluir esses arquivos no contêiner, os respectivos UID e GID serão mapeados para um UID e um GID dentro do contêiner. O UID e o GID mapeados podem ter permissões diferentes dentro do contêiner do que na máquina local. Por exemplo, o UID e o GID não raiz da máquina local são mapeados para um usuário raiz dentro do contêiner. Dessa maneira, o usuário pode agir como um usuário raiz dentro do contêiner e acessar, alterar e remover arquivos do contêiner. Como o contêiner está em execução em um ambiente isolado, o UID e o GID não podem acessar nenhum arquivo no host, uma vez que ele está designado a permissões não raiz fora do contêiner.

Para mapear o UID e o GID da máquina local para um dentro de um contêiner, o recurso de namespace do usuário requer que todos os arquivos sejam criados com um UID e um GID entre 0 e 65536.

O erro que é mostrado ocorre quando um arquivo é criado a partir de um UID ou GID fora desse intervalo e quando esse arquivo está incluído em uma das camadas de imagem do contêiner. Ao criar um contêiner a partir da imagem, o Docker Engine constrói cada camada da imagem. Quando um arquivo com um UID ou GID fora do intervalo é encontrado, o Docker Engine falha porque esse UID ou GID não pôde ser mapeado para um válido dentro do contêiner. O Docker Engine também falha quando um arquivo é usado em uma camada de imagem e removido na camada de imagem seguinte.

Ativar o namespace do usuário em seu Docker Engine local
Ao construir suas imagens de contêiner localmente, certifique-se de ativar o recurso de namespace do usuário em seu Docker Engine local. Dessa maneira, todos os UIDs e GIDs em sua máquina local que estão fora do intervalo de namespace do usuário válido são mapeados automaticamente para um ID válido dentro do intervalo. Teste sua imagem do contêiner localmente antes de enviá-la por push para o registro de imagens privado do Bluemix e usá-la com o IBM Bluemix Container Service. Para obter mais informações sobre como ativar o recurso de namespace do usuário, consulte Iniciando o daemon com namespaces do usuário ativados.
Comente a camada de imagem afetada e mude a propriedade dos arquivos afetados
Se você não puder localizar o arquivo que causa o problema, tente comentar as camadas de imagem em seu Dockerfile e remover o comentário uma camada após outra. Construa sua imagem e crie um contêiner para localizar a camada de imagem afetada.

Execute chown [-R] root:root <file_or_folder> para mudar a propriedade do arquivo afetado que está armazenado em sua máquina local ou inclua esse comando na camada de imagem afetada no Dockerfile.

Dockerfile de exemplo

RUN curl -fsSL "https://www.kernel.org/pub/software/scm/git/git-${GIT_VERSION}.tar.gz" \
        | tar -xzC /tmp \
  && cd /tmp/git-${GIT_VERSION} \
  && ./configure \
  && make all \
  && make install \
  && chown -R root:root /tmp

Não é possível exportar a imagem de seu registro de imagens privadas do Bluemix

Você está conectado ao IBM Bluemix Container Service e deseja puxar uma imagem de seu registro privado do Bluemix para a máquina local usando o comando bx ic pull. O comando falha indicando que bx ic pull não é suportado.
O comando bx ic pull não é suportado para puxar uma imagem de seu registro privado do Bluemix para a máquina local.
Para exportar uma imagem de seu registro privado do Bluemix, siga estas etapas.
  1. Efetue login no IBM Bluemix Container Service. Se você já estiver conectado, certifique-se de que suas variáveis de ambiente não estejam configuradas para usar os comandos nativos do Docker no IBM Bluemix Container Service.
  2. Liste suas imagens. 
    bx ic images
  3. Puxe uma imagem de seu registro privado do Bluemix para a máquina local. Use o nome ou o ID da imagem que foi retornado na etapa anterior. 
    docker pull <image_name_or_id>
  4. Verifique se a imagem foi puxada para a máquina local. 
    docker images
  5. Salve a imagem em um arquivo .tar. 
    docker export <image_name_or_id> > <exported_image_name>.tar

O namespace para uma organização não pode ser mudado

Não é possível localizar um método para mudar o namespace.
O namespace de uma organização não pode ser mudado depois que ela é criada.
É possível criar outra organização e configurar o namespace para essa nova organização com o nome desejado.

Login solicitado durante docker push

Ao tentar executar docker push para incluir uma imagem no registro de imagens privadas do Bluemix, as credenciais de login são solicitadas a você. Ao inserir as credenciais do Docker ou Bluemix, a autenticação falha.
A autenticação pode falhar devido às razões a seguir.
  • O token de acesso, que contém informações sobre autenticação do Cloud Foundry, está expirado.
  • O serviço de autorização do Bluemix não está disponível.
  • Se você estiver usando o plug-in do IBM Bluemix Container Service:
    1. Efetue login no Cloud Foundry CLI executando o comando para atualizar o token de acesso em eu computador.
      bx login [-a api.DomainName] [-sso]
      O parâmetro de conexão única --sso é necessário quando você efetua login com um ID federado. Se essa opção for usada, abra https://login.ng.bluemix.net/UAALoginServerWAR/passcode para obter uma senha descartável. Se você não incluir a opção --sso, conclua estas subetapas.
      1. Para E-mail, insira o IBMId que você usa para efetuar login no Bluemix.
      2. Para Senha, insira a senha para o IBMId que você usa para efetuar login no Bluemix. Ao inserir seu nome de usuário e senha, a organização e o espaço do Bluemix são recuperados.
      3. Insira o número que representa uma de suas organizações do Bluemix.
      4. Insira o número que representa um de seus espaços existentes do Bluemix.
    2. Inicialize a CLI do IBM Bluemix Container Service para recriar o seu token de acesso.
      bx ic init

Se o prompt ainda ocorrer depois que você efetuou login novamente, o serviço de autorização do Bluemix não está disponível. Tentar novamente mais tarde.

Resolvendo problemas de contêineres

Erro 502 Bad Gateway ao acessar um app

Você implementou um grupo de contêiner com sucesso, mas ao abrir a URL do app, o erro a seguir é exibido.
502 Bad Gateway: Registered endpoint failed to handle the request.
  1. O roteamento pode estar sendo configurado ainda.
  2. O contêiner pode não estar respondendo.
  3. Pode haver um erro com o roteamento, embora o grupo de contêiner esteja sendo executado com sucesso.
  1. Aguarde mais alguns minutos para permitir tempo suficiente para conclusão do roteamento. Pode haver uma espera de 5 minutos após a execução do processo do contêiner antes de o grupo de contêiner poder se conectar à rota especificada.
  2. Se você tiver esperado 5 minutos e suspeitar que ocorreu um erro, veja se o contêiner está respondendo ligando um endereço IP ao contêiner em execução e, em seguida, execute um comando curl para esse endereço IP e porta.
    1. Execute o comando a seguir para obter o ID de uma das instâncias de contêiner no grupo. Observe o ID do contêiner e a porta.
      bx ic ps
    2. Solicite um endereço IP executando o comando a seguir.
      bx ic ip-request
    3. Ligue o endereço IP à instância do contêiner executando o comando a seguir.
      bx ic ip-bind IP_address container_ID
    4. Em um navegador, insira a URL no aplicativo com o endereço IP e a porta. Exemplo: http://IP_address:port

      Se não é possível acessar o aplicativo, há um problema com o grupo de contêiner e o grupo deve ser recriado. Se é possível acessar o aplicativo, o contêiner está sendo executado com sucesso, mas há um erro com o roteamento. Entre em contato com o Suporte. IBM Bluemix

Instâncias do grupo de contêiner são recriadas repetidamente durante a inicialização pelo modo de recuperação automática do IBM Bluemix Container Service

Você criou um grupo de contêiner e ativou a recuperação automática. Durante a inicialização do grupo de contêiner, instâncias de grupo não responsivas são removidas e recriadas repetidamente.

O app que está em execução em seu grupo de contêiner demora muito para começar a atender na porta exposta. Se você tiver ativado a recuperação automática para seu grupo de contêiner, o IBM Bluemix Container Service verificará o funcionamento de cada instância com uma solicitação de HTTP na porta TCP exposta após a criação do grupo. Se uma instância não estiver atendendo na porta exposta em 3 minutos, a instância será marcada como não responsiva e será removida, recriada e reiniciada. Esse processo se repetirá até que o número máximo de instâncias recriadas seja atingido. Em seguida, a recuperação automática será desativada.

Para executar um app em um grupo de contêiner com a recuperação automática ativada, certifique-se de que o app seja iniciado e atenda na porta exposta em menos de 2 minutos. Dependendo do tipo de app, existem métodos diferentes para assegurar uma inicialização rápida de seu app.

O serviço do Bluemix não é ligado diretamente a um contêiner

Seu serviço do Bluemix não é ligado diretamente a um contêiner usando a ligação de serviços.
Alguns serviços do Bluemix não suportam ligação direta de serviços com um contêiner e levam a uma falha durante uma criação de contêiner. Para esses serviços, a ligação de serviços não é passada com a variável de ambiente VCAP_SERVICES. Para contornar esse problema, é possível usar a CLI para criar um app Cloud Foundry que age como ponte para enviar VCAP_SERVICES do serviço para um contêiner.
  1. Selecione um espaço no qual implementar um contêiner, tal como dev.
  2. Identifique um app existente ou crie um app Cloud Foundry ao qual ligar o serviço do Bluemix. Esse app é usado para a passagem das informações VCAP_SERVICES, por isso é referido como app ponte.
    Para criar um app Cloud Foundry:
    Dica: O app que você cria concluindo estas etapas é um item temporário para armazenar os serviços VCAP_SERVICES. Para incorrer em menos custos do que um app em execução, esse aplicativo não é iniciado ou executado. A execução do app não é necessária para fornecer os VCAP_SERVICES para o contêiner.
    1. Crie um diretório e um arquivo de texto em seu diretório de app de serviço. Exemplo: <directory>/<empty_file.txt>. Esse arquivo serve como o arquivo de origem para o app, embora não precise conter nenhum conteúdo.
      • Exemplo do Windows: C:\Users\User\containerbridge\empty_file.txt
      • Exemplo do Linux: /user_home_directory/containerbridge/empty_file.txt
      • Exemplo do S.O. X: /Users/user_home_directory/containerbridge/empty_file.txt
    2. Execute esse comando para criar um app de tempo de execução zero, sem custos. O app pode ter qualquer nome embora containerbridge seja um nome comum uma vez que o app age como uma ponte do contêiner para um serviço.

      cf push <appname> -p <directory> -i 1 -d AppDomainName -k 1M -m 64M --no-hostname --no-manifest --no-route --no-start
      Exemplo: 
      cf push containerbridge -p containerbridge -i 1 -d AppDomainName -k 1M -m 64M --no-hostname --no-manifest --no-route --no-start
  3. Se o serviço do Bluemix que você deseja ligar ainda não tiver sido incluído, inclua-o no espaço do Bluemix.
    1. Para obter uma lista dos serviços do Bluemix disponíveis, execute o comando cf marketplace. A lista resultante inclui o nome do serviço do Bluemix, seu tipo de plano e uma descrição do que é o serviço.
    2. Execute o comando create-service e inclua o nome do serviço e o plano a partir da lista cf marketplace. Além disso, inclua seu próprio nome de instância do serviço.
      cf create-service <service> <plan> <service_instance>
      Exemplo
      cf create-service cleardb spark my-clear-db
  4. Ligue o serviço do Bluemix ao app executando o comando bind-service. Repita para ligar outros serviços do Bluemix ao app ponte.
    cf bind-service <appname> <service_instance>
    Exemplo
    cf bind-service containerbridge my-clear-db
  5. Se for solicitado para remontar o app, execute o comando a seguir.
    cf restage containerbridge

Cannot create child process com o comando su

Ao utilizar o comando do Linux de usuário substituto (su) para editar os usuários de um contêiner, você recebe a mensagem su: cannot create child process: Resource temporarily unavailable, mesmo o comando su tendo funcionado no ambiente do Docker local. É possível passar por esse problema ao efetuar login em um contêiner ou quando o comando su é executado como parte de um script dentro do contêiner.

Exemplo:

bx ic exec -it my_container bash
adduser user
su - user
Resultado:
su: cannot create child process: Resource temporarily unavailable

Esse problema ocorre com frequência em contêineres construídos com imagens do Centos 7, embora também possa ocorrer com outros tipos de imagem.

É possível resolver o problema editando o Dockerfile e recriando a imagem ou efetuando login no contêiner em execução.
Para resolver o problema editando o Dockerfile:
  1. No diretório local do qual você está criando sua imagem, crie uma pasta etc, se ainda não existir uma.
    mkdir etc
  2. Navegue para o diretório etc.
    cd etc
  3. Na pasta etc, crie um diretório pam.d.
    mkdir pam.d
  4. Navegue para o diretório pam.d.
    cd pam.d
  5. No diretório pam.d, crie um arquivo chamado su e abra-o com um editor de texto.
  6. No arquivo su, copie o texto a seguir, em que todas as variáveis session estão configuradas como optional e salve as mudanças no arquivo. 
    #%PAM-1.0
auth            sufficient      pam_rootok.so
# Uncomment the following line to implicitly trust users in the "wheel" group.
#auth           sufficient      pam_wheel.so trust use_uid
# Uncomment the following line to require a user to be in the "wheel" group.
#auth           required        pam_wheel.so use_uid
auth            substack        system-auth
auth            include         postlogin
account         sufficient      pam_succeed_if.so uid = 0 use_uid quiet
account         include         system-auth
password        include         system-auth
session         optional        system-auth
session         optional        postlogin
session         optional        pam_xauth.so
  7. No Dockerfile da imagem, inclua as informações a seguir.
    RUN sed -i 's/session.*include/session        optional/' /etc/pam.d/su
  8. Recrie a imagem. Para obter mais informações, consulte Construindo uma imagem.
  9. Recrie o contêiner único ou o grupo de contêiner. Para obter mais informações, consulte Criando um contêiner único ou Criando um grupo de contêiner.
  10. Para verificar, execute o comando a seguir.
    bx ic logs container_name
    Resultado:
    %Changing password for user user_name.
passwd: todos os tokens de autenticação atualizados com sucesso.
Para resolver o problema efetuando login no contêiner em execução:
  1. Crie o contêiner.
  2. Efetue login no contêiner.
    bx ic exec -it my_container bash
  3. Navegue para o diretório etc.
    cd etc
  4. Navegue para o diretório pam.d.
    cd pam.d
  5. Abra o editor visual usando o comando vi do Linux.
    vi su
  6. No arquivo su, mude todas as variáveis de sessão para optional e salve as mudanças no arquivo. Para obter mais informações sobre como usar o editor vi, consulte Como usar o editor vi.
    #%PAM-1.0
auth            sufficient      pam_rootok.so
# Uncomment the following line to implicitly trust users in the "wheel" group.
#auth           sufficient      pam_wheel.so trust use_uid
# Uncomment the following line to require a user to be in the "wheel" group.
#auth           required        pam_wheel.so use_uid
auth            substack        system-auth
auth            include         postlogin
account         sufficient      pam_succeed_if.so uid = 0 use_uid quiet
account         include         system-auth
password        include         system-auth
session         optional        system-auth
session         optional        postlogin
session         optional        pam_xauth.so
  7. Efetue login como o usuário.
    su - user
  8. Execute o comando do Linux id para ver as informações de configuração do usuário.
    id
    Resultado:
    uid=1000(user) gid=1000(user) groups=1000(user)

O contêiner para e é encerrado após a criação

Você criou um contêiner, mas ele foi encerrado imediatamente.

Você executou um contêiner localmente e ele foi executado com sucesso, mas ao executar o contêiner no Bluemix, você vê uma das situações a seguir.
  • No painel do Bluemix, o estado do contêiner muda de Running para Stopped.
  • Na CLI, ao executar bx ic inspect container, você vê que o status do estado é Shutdown.
Você também tentou executar bx ic logs, mas não localizou nenhum problema com o contêiner.

Ao executar um contêiner no Bluemix, ele é sempre executado no modo separado, semelhante à execução de docker run -d quando um contêiner é executado localmente. Como resultado, mesmo os contêineres iniciados localmente não são executados continuamente no Bluemix.

Deve-se tornar o contêiner um aplicativo de longa execução editando o Dockerfile. Por exemplo, se você tiver incluído um comando para gerar a frase "hello world" a cada dez segundos, essa ação contínua tornará seu contêiner um aplicativo de longa execução.
ENTRYPOINT ["/bin/sh"]
CMD ["-c","while true; do echo hello world; sleep 10;done"]

O contêiner excluído continua exibido na GUI do Bluemix

Na GUI do Bluemix, você excluiu um contêiner, mas ele continua exibido na GUI do Bluemix e usa cota.

Na GUI do Bluemix, a exclusão não pôde ser concluída.

Exclua o contêiner a partir da linha de comandos.

Para contêineres únicos, siga estas etapas.
  1. Exclua o contêiner executando uma das opções da linha de comandos a seguir.
    • Plug-in do IBM Bluemix Container Service com comandos bx ic
      bx ic rm [-f] CONTAINER [CONTAINER]
    • Plug-in do IBM Bluemix Container Service com comandos docker 
      docker rm [-f] CONTAINER
  2. Verifique se o contêiner foi removido executando um dos comandos a seguir para listar todos os contêineres em seu espaço:
    • Plug-in do IBM Bluemix Container Service com comandos bx ic
      bx ic ps -a
    • Plug-in do IBM Bluemix Container Service com comandos docker 
      docker ps -a
Para grupos de contêineres, siga estas etapas.
  1. Exclua o grupo de contêiner executando o comando a seguir.
    • Plug-in do IBM Bluemix Container Service
      bx ic group-remove [-f] GROUP [GROUP]
  2. Verifique se o contêiner foi removido executando o comando para listar todos os contêineres em seu espaço.
    • Plug-in do IBM Bluemix Container Service
      bx ic groups [-q]

Se você executar os comandos e o contêiner continuar sendo exibido na GUI do Bluemix, espere para ver se ele é removido mais tarde.

O grupo de contêiner está mostrando um status HAS_INACTIVE_INSTANCES

Ao executar bx ic group-inspect, o grupo de contêiner está exibindo um status HAS_INACTIVE_INSTANCES.

Se a rede falhar ao ser configurada quando o grupo for criado ou falhar na verificação de funcionamento para uma ou mais instâncias, o estado da instância mudará para INACTIVE. Se alguma instância em um grupo estiver inativa, o estado do grupo mudará para HAS_INACTIVE_INSTANCES.

Se o problema ocorrer quando a verificação de funcionamento falhar para uma instância, o problema poderá ser resolvido sem nenhuma ação necessária. Se o problema ocorrer quando o grupo for criado pela primeira vez, será possível tentar recriar o grupo. Se o problema ocorrer novamente após a recriação, a rede poderá não ser configurável para o app quando ele for implementado como um grupo. Tente implementar o app como um contêiner único em vez disso.

Resolução de problemas de armazenamento de dados

Erro de permissão ao mudar o proprietário de um diretório de volume montado em um contêiner

Você obtém um erro de permissão quando tenta mudar a propriedade de um arquivo ou diretório para um volume montado em um contêiner.
Para o IBM Bluemix Container Service, o recurso de namespace do usuário está ativado para o Docker Engine. A raiz efetiva dentro do contêiner é mapeada para um usuário não raiz no host de cálculo. A origem do volume montado é um Network File System (NFS). Como o NFS não está ciente de namespaces do usuário, todas as verificações não de usuário são executadas para as operações em arquivos e diretórios montados no contêiner.

  1. No Dockerfile para uma imagem, inclua as linhas a seguir para criar um usuário um ID do usuário (UID) disponível. Por exemplo, é possível nomear o usuário como volumeeditor.
    RUN groupadd --gid 1010 volumeeditor
RUN useradd --uid 1010 --gid 1010 -m --shell /bin/bash volumeeditor
  2. Construa e envie por push a imagem para seu registro de imagens privadas do Bluemix.
  3. Crie um contêiner com a imagem na qual o caminho do volume está montado.
    Dica: Execute bx ic namespace-get para recuperar seu namespace e substitua <my_namespace> por suas informações de namespace.
    bx ic run --volume volume_name:/path/to/files --name fixVolumePermissions registry.DomainName/<my_namespace>/image
  4. Efetue login no contêiner.
    bx ic exec -it container_name bash
  5. Defina uma variável de ambiente denominada MOUNTPATH para o caminho de montagem.
    MOUNTPATH="/path/to/files"
  6. Inclua o usuário no grupo root.
    adduser volumeeditor root
  7. Configure permissões para que todos os membros do grupo tenham acesso de leitura e gravação à montagem do volume. 
    chmod 775 $MOUNTPATH
  8. Crie uma pasta no caminho de montagem e configure as permissões de leitura e gravação. Neste exemplo, a pasta é denominada data.
    su -c "mkdir -p $MOUNTPATH/data" volumeeditor
    su -c "chmod 700 $MOUNTPATH/data" volumeeditor
  9. Verifique se o usuário volumeeditor pode acessar os arquivos e diretórios no caminho de montagem.
    sudo -u volumeeditor ls -al $MOUNTPATH
  10. Opcional: vincule uma pasta no contêiner à pasta data no volume. Esse link associa as permissões de gravação de $MOUNTPATH/data a qualquer pasta no contêiner.
    ln -sf $MOUNTPATH/data /folder/in/container/
  11. Se as edições puderem ser feitas sem um erro de permissão, remova o usuário raiz do grupo volumeeditor.
    deluser volumeeditor root

Os logs de instâncias do grupo de contêiner são sobrescritos em um volume

Quando você criou um grupo de contêiner, você montou um volume e os logs foram gerados a partir de cada instância do grupo de contêiner para o volume. Cada instância do grupo de contêiner grava nos mesmos arquivos e os arquivos são sobrescritos por cada uma das instâncias.
Exemplo:
bx ic group-create –p 9080 –m 64 -v log_vol:/opt/ibm/logs -e LOG_LOCATIONS=/opt/ibm/logs/messages.log,/opt/ibm/logs/debug.log,/opt/ibm/logs/trace.log -–name my_container_group registry.domain_name/ibmliberty:latest
Cada instância do contêiner está gravando um arquivo no volume como se fosse o único gravador. Cada instância possui um cache local na memória. Os arquivos no volume são constantemente substituídos pelos dados em cache de cada instância do contêiner.
Ajuste o nome do arquivo de log para incluir um identificador para que os arquivos sejam diferentes para cada instância. Por exemplo, inclua a variável $HOSTNAME no nome do arquivo de log que é gerado pelo app e especificado quando o grupo de contêiner é criado.
Exemplo:
bx ic group-create –p 9080 –m 64 -v log_vol:/opt/ibm/logs -e LOG_LOCATIONS=/opt/ibm/logs/messages.$HOSTNAME.log,/opt/ibm/logs/debug.$HOSTNAME.log,/opt/ibm/logs/trace.$HOSTNAME.log -–name my_container_group registry.domain_name/ibmliberty:latest

Transferir dados de ou para um volume demora mais do que o esperado

Ler ou gravar dados em um volume demora mais do que o esperado.
No IBM Bluemix Container Service, os volumes não possuem acesso ao host local. Portanto, todos os volumes são montados em um compartilhamento de arquivo NFS. O desempenho do volume é determinado pelo tamanho e pelo IOPS (entrada/saída por segundo) de seu compartilhamento de arquivo, junto à carga de trabalho de seu aplicativo. O compartilhamento de arquivo em que seu volume está hospedado pode não ser adequado para a carga de trabalho de seus volumes e contêineres. Além disso, múltiplos volumes em um compartilhamento de arquivo podem competir pelos recursos do compartilhamento de arquivo.

Para criar um novo compartilhamento de arquivo, deve-se ter a função de gerenciador de organização. Para obter informações detalhadas, consulte Usuários e funções.

  1. Determine o ID do compartilhamento de arquivo em que seu volume está hospedado.
    bx ic volume-inspect my_volume
    {
    "fsId": "fa277ff4-8a64-435b-9b75-0f11d59a3ae4", 
    "hostPath": "/vol/fa277ff4-8a64-435b-9b75-0f11d59a3ae4/my_volume", 
    "otherSpaceVisibility": [], 
    "spaceGuid": "fa277ff4-8a64-435b-9b75-0f11d59a3ae4", 
    "volName": "my_volume"
}
  2. Visualize seu compartilhamento de arquivo para determinar seu tamanho e IOPS.
    bx ic volume-fs
    Name                                   Size   IOPS/GB   Created               State   
fa277ff4-8a64-435b-9b75-0f11d59a3ae4   20     4.00      2016-05-20 13:48:10   READY
  3. Visualize os tamanhos disponíveis em GB para os compartilhamentos de arquivos.
    bx ic volume fs-flavor
    [
    20, 
    40, 
    80, 
    100, 
    250, 
    500, 
    1000, 
    2000, 
    4000, 
    8000, 
    12000
]
  4. Determine o tamanho e o IOPS necessários para a carga de trabalho de seus apps. Os valores válidos de IOPS são 0,25, 2 ou 4 IOPS por GB. O tamanho do compartilhamento de arquivo é multiplicado pelo IOPS por GB para determinar o total de IOPS do compartilhamento de arquivo. Esta tabela inclui exemplos de como os diferentes tamanhos de compartilhamento de arquivo em 4 IOPS por GB são escalados.
    Tabela 1. Exemplos de tamanhos de compartilhamento de arquivo e IOPS por GB
    Tamanhos de compartilhamento de arquivo IOPS por GB Total de IOPS
    20 GB 4 80
    100 GB 4 400
    1.000 GB 4 4.000
    4.000 GB 4 16.000
  5. Criar um novo compartilhamento de arquivo. O primeiro número é o tamanho em GB. O segundo número é o IOPS de compartilhamento de arquivo.
    bx ic volume-fs-create my_new_fs 1000 4
  6. Verifique o novo compartilhamento de arquivo e aguarde o status ser exibido como pronto.
    bx ic volume-fs
  7. Crie seu volume no novo compartilhamento de arquivo.
    bx ic volume-create my_new_volume my_new_fs
  8. Execute um contêiner ou um grupo de contêiner a partir de sua imagem com o novo volume montado.
  9. Faça backup dos dados a partir de my_volume e copie os dados para my_new_volume. Consulte Fazendo backup e restaurando dados para obter mais informações.

Erro de análise sintática ao criar um contêiner com um volume a partir da CLI

Ao criar um único contêiner ou um grupo de contêiner a partir da CLI, você recebe uma mensagem com falha.
Exemplo
bx ic group-create --name my_container -n my_host -d AppDomainName -m 2048 --min 1 --max 2 --desired 1 -p 9080 --auto -v my_volume:/my/path registry.DomainName/ibmliberty
Resultado
docker: Error parsing reference: “my_volume:/my/path” is not a valid repository/tag.
A CLI for Cloud Foundry versão 6.17 e mais recente usa o parâmetro -v para outro propósito.
Em seu comando, mude a opção -v para a opção --volume.
Exemplo
bx ic group-create --name my_container -n my_host -d AppDomainName -m 2048 --min 1 --max 2 --desired 1 -p 9080 --auto --volume my_volume:/my/path registry.DomainName/ibmliberty

Problemas conhecidos

Saiba mais sobre os problemas conhecidos que ocorrem no IBM Containers.

Docker e IBM Bluemix Container Service

Alguns comandos do Docker não são suportados no plug-in do IBM Bluemix Container Service (bx ic)

Somente determinados comandos do Docker são suportados em (bx ic). É possível visualizar uma lista integral de comandos suportados na Referência de comando docker. Por exemplo, tag não está disponível em bx ic, mas docker tag pode ser usado localmente e, seguida, a imagem atualizada pode ser enviada por push para o registro do Bluemix.

Problemas comuns entre o Docker e o Windows

Existem problemas comuns para instalar e configurar o Docker em computadores do Windows. Para obter mais informações sobre esses problemas comuns, revise Blog: Instalando o Docker for Windows: Correções para problemas comuns e Posts do Stack Overflow identificados com boot2docker.

Imagens

A publicação e o envio de imagens grandes falham

Ao executar um comando do Docker para enviar por push ou puxar uma imagem grande do registro privado do Bluemix de sua organização, a solicitação falha. Para determinar se existe um problema com a imagem, primeiro verifique se a imagem pode ser enviada por push para DockerHub. Se não existir problema com a imagem, se possível, reduza o tamanho da imagem.

O tamanho de uma imagem no Bluemix corresponde a seu tamanho localmente

Ao enviar por push uma imagem para seu registro privado do Bluemix, o tamanho relatado para a imagem é menor do que o tamanho da mesma imagem em seu mecanismo local do Docker. A diferença entre os tamanhos não indica que ocorreu um problema quando a imagem foi enviada por push. O tamanho compactado da imagem é relatado no IBM Bluemix Container Service.

Monitorando e criando logs

Caracteres extras na saída para bx ic logs
Ao executar bx ic logs, caracteres extras, como &, são exibidos no início de cada linha. Esse problema ocorre quando os logs de docker recebidos são analisados por NGINX.
Adições do arquivo de log não podem ser especificadas com expressões

Não é possível usar filtros ou expressões curingas para especificar os arquivos de log que devem ser coletados. Por exemplo, não é possível especificar /var/log/*.log. Em vez disso, liste cada arquivo de log individualmente.

Contêineres únicos e grupos de contêiner

Conexão com falha após a criação de um contêiner

Em razão de uma limitação na arquitetura de rede do Docker do IBM Bluemix Container Service, o acesso à rede de saída pode não funcionar na inicialização inicial do contêiner. Se o contêiner tentar fazer uma conexão de rede de saída para retornar um código de erro ou para lançar uma exceção se a rede não estiver disponível, as saídas de contêiner e um estado "Travado" serão exibidos.

Para permitir que a rede seja inicializada e permitir que a implementação do contêiner continue, deve-se ter uma instrução de suspensão antes de iniciar quaisquer processos que requeiram conectividade de saída. Crie um Dockerfile customizado para agrupar quaisquer comandos de inicialização existentes do servidor no Dockerfile de base. Para obter um exemplo, consulte o aplicativo de amostra Let's Chat Bluemix.

Conectividade de entrada não é imediata

A conectividade de entrada não é imediata para grupos de contêineres e contêineres únicos, incluindo rotas e apps externos. Aguarde até que a conectividade se torne disponível, o que pode levar até 5 minutos.

Diferentes unidades padrão de medida usada em docker run -m e bx ic run -m
Quando nenhuma unidade de medida é especificada com docker run -m, bytes é usado como a unidade padrão de medida. Quando nenhuma unidade de medida é especificada com bx ic run -m, MB é usado como a unidade padrão de medida.
O limite de dados do usuário impede que um contêiner seja criado
Se você tiver muitos serviços ligados a um contêiner, ele não poderá ser criado. Cada serviço injeta dados no contêiner e é contado para um limite de 64 KB. Ao criar um contêiner que excede esse limite, você receberá uma mensagem, Falha ao CRIAR recurso... Dados muitos grandes do usuário. Se você vir essa mensagem, remova um serviço do contêiner.
A porta 25 não está disponível para exposição em um contêiner
A porta 25 é bloqueada contra uso pela SoftLayer, que faz parte da arquitetura do IBM Bluemix Container Service. Esse problema afeta o uso de alguns serviços SMTP em contêineres. Para obter mais informações sobre o motivo de bloqueio dessa porta, consulte o artigo E-mail de saída na porta 25 na documentação da SoftLayer.
O processo de construção ou rede do contêiner não é concluído
Se um contêiner permanece no estado de construção ou rede por muito tempo, pode ser necessário remover o contêiner e criar um novo contêiner a partir da mesma imagem. É possível usar os comandos da CLI bx ic a seguir:
  1. Recuperar uma lista de contêineres que estão em execução no espaço. Por padrão, somente contêineres em execução são exibidos. Para obter mais informações, revise a Referência da CLI bx ic ps.
    bx ic ps -a
  2. Remover um ou mais contêineres. Para obter mais informações, revise a Referência da CLI bx ic rm. 
    bx ic rm -f container_ID
  3. Iniciar um novo contêiner no serviço de nuvem do contêiner a partir de um nome de imagem. Para obter mais informações, revise a Referência da CLI bx ic run. 
    bx ic run
A recuperação automática não é imediata para grupos de contêiner

A recuperação automática para grupos de contêineres pode demorar mais de 15 minutos para que novos sistemas fiquem on-line. Aguarde até que a recuperação automática se torne disponível, o que pode demorar mais de 15 minutos.

O comando cf delete-orphaned-routes exclui rotas dos grupos de contêiner

Cloud Foundry não reconhece rotas que são mapeadas para grupos de contêineres da mesma forma que reconhece rotas como ligadas aos aplicativos. Não execute o comando cf delete-orphaned-routes ao usar grupos de contêineres.

Obtendo ajuda e suporte para o IBM Bluemix Container Service

Onde você inicia a resolução de problemas de um contêiner?

  • Para ver se o Bluemix está disponível, verifique a página de status do Bluemix.
  • Revise os fóruns para ver se outros usuários tiveram o mesmo problema. Ao usar os fóruns para fazer uma pergunta, marque sua pergunta para que ela seja vista pelas equipes de desenvolvimento do Bluemix.
    • Se você tiver questões técnicas sobre como desenvolver ou implementar um app com o IBM Bluemix Container Service, poste sua pergunta no Stack Overflow e marque-a com ibm-bluemix e containers.
    • Para perguntas sobre o serviço e instruções de introdução, use o fórum do IBM developerWorks dW Answers. Inclua as tags bluemix e containers.
    Consulte Obtendo ajuda para obter mais detalhes sobre o uso dos fóruns.
  • Entre em contato com o Suporte IBM. Para obter informações sobre como abrir um chamado de suporte IBM ou sobre níveis de suporte e severidades de chamado, consulte Entrando em contato com o suporte.

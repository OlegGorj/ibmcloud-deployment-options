Instalando o plug-in do IBM Bluemix Container Service (bx ic)

Para executar comandos, seu sistema operacional deve atender a estes requisitos. Esta tarefa inclui as informações para instalar esses requisitos. Versões anteriores dessas ferramentas não são suportadas para uso com o plug-in do IBM Bluemix Container Service (bx ic).

  • Docker versão 1.8 para 1.12
  • CLI do Cloud Foundry versão 6.14.0 - 6.22.0
    Nota: Os usuários do MacOS Sierra devem instalar o Cloud Foundry CLI versão 6.22.0 ou mais recente.
  • CLI do Bluemix
  • Plug-in do IBM Bluemix Container Service

Para instalar as CLIs:

  1. Instalar o Docker 1.12. Para sistemas operacionais Windows e OS X, o Docker é instalado junto com o Docker Toolbox, que inclui o Docker Machine, Docker Engine e Kitematic. Para sistemas operacionais Linux, os mecanismos de instalação variam dependendo do tipo de distribuição Linux que você usa. Consulte os links a seguir para instalar ou, se já tiver instalado o Docker, para fazer upgrade para o Docker 1.12.
    Nota: Caso já tenha instalado o Docker em seu computador e não desejar fazer upgrade da versão atual, será possível ignorar esta etapa e continuar com a instalação do Cloud Foundry CLI.
    • OS X

      https://github.com/docker/toolbox/releases/download/v1.12/DockerToolbox-1.12.pkg

    • Windows

      https://github.com/docker/toolbox/releases/download/v1.12/DockerToolbox-1.12.exe

    • Linux

      Consulte a Documentação de instalação do Docker para localizar instruções sobre como instalar o Docker versão 1.12 na distribuição Linux usada.

  2. Inicie o Docker.
    • Para o OS X e o Windows, clique duas vezes no ícone Terminal de iniciação rápida do Docker. O daemon do Docker é iniciado e as variáveis de ambiente são configuradas para você.
    • Para o Linux, em um utilitário de linha de comandos, verifique se o daemon do Docker está em execução. Além disso, configure o daemon do Docker para reiniciar automaticamente. Para obter mais informações, consulte a documentação do Docker para seu sistema operacional.
  3. Verifique se sua instalação do Docker está configurada corretamente executando docker run hello-word em seu terminal. 
    docker run hello-world
    O Docker procura a imagem hello-world localmente da qual executar um contêiner, portanto, se esta for a primeira vez que o comando é executado, você receberá a mensagem a seguir. 
    Unable to find image 'hello-world:latest' locally
    A imagem é extraída do Docker Hub, um repositório público de imagens, e depois que o comando for concluído, a saída será semelhante à seguinte: 
    $ docker run hello-world

Hello from Docker.
This message shows that your installation appears to be working correctly.

To generate this message, Docker took the following steps:
 1. The Docker client contacted the Docker daemon.
 2. The Docker daemon pulled the "hello-world" image from the Docker Hub.
 3. The Docker daemon created a new container from that image which runs the
    executable that produces the output you are currently reading.
 4. The Docker daemon streamed that output to the Docker client, which sent it
    to your terminal.

To try something more ambitious, you can run an Ubuntu container with:
 $ docker run -it ubuntu bash

Share images, automate workflows, and more with a free Docker Hub account:
 https://hub.docker.com

For more examples and ideas, visit:
 https://docs.docker.com/userguide/
    Se o comando não for concluído adequadamente, execute docker version para ver se o Docker está instalado corretamente.
  4. Instale a versão CLI do Cloud Foundry versão 6.14.0 - 6.22.0 da CLI do Cloud Foundry no repositório GitHub. Deve-se instalar o Cloud Foundry CLI no local padrão de seu sistema operacional, caso contrário, a variável de ambiente PATH não corresponderá a seu diretório de instalação.
    Nota: Os usuários do MacOS Sierra devem instalar o Cloud Foundry CLI versão 6.22.0 ou mais recente.
    Para verificar se o Cloud Foundry CLI está instalado adequadamente ou, se você já o instalou, para verificar qual versão foi instalada, execute o comando a seguir:
    cf -v
  5. Instale a CLI do Bluemix.
  6. Efetue login na CLI do Bluemix. 
    bx login -a https://api.ng.bluemix.net
  7. Instale o plug-in do IBM Bluemix Container Service. 
    bx plugin install IBM-Containers -r Bluemix
    Para verificar se o plug-in está instalado adequadamente, execute o comando a seguir:
    bx plugin list

    O plug-in é exibido nos resultados como IBM-Containers.

Você está pronto para efetuar login e começar a gerenciar contêineres.

Efetuando login no plug-in da CLI do IBM Bluemix Container Service (bx ic)

Depois de instalar a CLI, efetue login para usá-la.

Antes de começar, conclua as tarefas a seguir.
  • Instale o plug-in do IBM Bluemix Container Service (bx ic).
  • Decida qual organização e espaço do Bluemix usar ao efetuar login no Cloud Foundry. Para obter mais informações sobre organizações e espaço, consulte Organizações e espaços.
  • Para obter mais informações sobre as opções que podem ser especificadas com o comando login do Cloud Foundry, consulte o comando Login na documentação do Cloud Foundry.
  1. Abra um utilitário de linha de comandos.
    • Para o OS X e o Windows, clique duas vezes no ícone Terminal de iniciação rápida do Docker. O daemon do Docker é iniciado e as variáveis de ambiente são configuradas para você.
    • Para o Linux, abra qualquer utilitário de linha de comandos.
  2. Efetue login no Bluemix por meio do Cloud Foundry CLI. Para especificar uma região específica, inclua a opção -a com o terminal de API para a região. Se for a primeira vez que estiver usando a CLI e você não incluir a opção, será solicitado que insira um terminal de API. Após o terminal de API ser especificado, não será necessário inserir um durante logins subsequentes. 
    bx login [-a api.DomainName] [-sso]
    Para efetuar login em uma região específica, inclua o terminal de API dessa região quando efetuar login no Cloud Foundry CLI.
    • SUL dos EUA
      bx login -a api.ng.bluemix.net
    • United Kingdom
      bx login -a api.eu-gb.bluemix.net
    O parâmetro de conexão única --sso é necessário quando você efetua login com um ID federado. Se essa opção for usada, abra https://login.ng.bluemix.net/UAALoginServerWAR/passcode para obter uma senha descartável. Se você não incluir a opção --sso, conclua estas subetapas.
    1. Para E-mail, insira seu IBMId para o Bluemix.
    2. Para Senha, insira a senha para o IBMId. Suas organizações e espaços Bluemix são recuperados.
    3. Insira o número que representa uma de suas organizações do Bluemix.
    4. Insira o número que representa um de seus espaços existentes do Bluemix.
  3. Configure um espaço de nomes para a sua organização. Um namespace é um nome exclusivo para identificar seu registro de imagens privado do Bluemix. Ao criar um contêiner, deve-se especificar o local de uma imagem incluindo o namespace com o nome da imagem.
    Nota: Ao criar um espaço de nomes, ele não poderá ser mudado após ser criado, apenas excluído. Se os usuários de sua organização do Bluemix já implementaram contêineres e configuraram um espaço de nomes, uma mensagem será mostrada exibindo o espaço de nomes para a sua organização. Nesse caso, é possível continuar com a próxima etapa.
    bx ic namespace-set <my_namespace>
    Dica: Observe as regras a seguir para o namespace:
    • Ele pode conter somente letras minúsculas, números ou sublinhados (_).
    • Pode ter entre 4 e 30 caracteres. Se planeja gerenciar contêineres a partir da linha de comandos, é possível que prefira ter um namespace curto que possa ser digitado rapidamente.
    • O seu nome deve ser exclusivo no Bluemix.
  4. Inicialize o IBM Bluemix Container Service. 
    bx ic init
  5. Decida se você deseja executar comandos do bx ic ou nativos do Docker. É possível selecionar uma dessas opções toda vez que você efetuar login no plug-in ou então usar o comando shellinit para sempre selecionar a Opção 2 .
    • Opção 1 (Padrão)

      Um shell simples para o IBM Bluemix Container Service agrupa o Docker. O daemon do Docker local não é afetado pelo IBM Bluemix Container Service. Os comandos bx ic são usados para gerenciar contêineres em seu espaço do Bluemix e os comandos docker são usados para gerenciar contêineres localmente. No entanto, os comandos docker são sempre necessários para desenvolvimento de imagem local.

      Por exemplo, com a Opção 1, o comando bx ic run cria um contêiner no espaço do Bluemix e docker run cria um contêiner localmente. Você poderá preferir essa opção se for inexperiente no Docker e no IBM Bluemix Container Service ou se gostar de testar os contêineres localmente.

    • Opção 2

      Redirecione o daemon do Docker local da sessão da CLI atual para o IBM Bluemix Container Service para que os comandos docker possam ser usados no lugar dos comandos bx ic quando estiverem disponíveis para gerenciar contêineres em seu espaço do Bluemix.

      Por exemplo, com a opção 2, o comando bx ic run ou docker run pode ser usado para criar um contêiner em seu espaço. Você poderá preferir essa opção se for um usuário experiente do Docker que se sente mais à vontade com os comandos docker ou se estiver adaptando scripts automatizados para trabalhar com o IBM Bluemix Container Service.

      Algumas exceções ainda requerem o uso de comandos bx ic para executar tarefas específicas do IBM Bluemix Container Service. Por exemplo, para configurar um espaço de nomes, deve-se usar o comando bx ic namespace-set. docker namespace-set não é um comando válido. Para obter uma lista de comandos docker que são suportados com essa configuração, consulte Comandos suportados do Docker para o plug-in do IBM Bluemix Container Service (bx ic).

      Para configurar essa opção, copie e cole as variáveis de ambiente que são fornecidas no terminal durante o login.
      • DOCKER_HOST configura o host do Docker para o IBM Bluemix Container Service.
      • DOCKER_CERT_PATH configura o local das chaves de autenticação para o plug-in IBM Bluemix Container Service para Cloud Foundry.
      • DOCKER_TLS_VERIFY usa TLS e verifica o servidor remoto.
      Dica: Quando você estiver concluindo as tarefas na documentação do IBM Bluemix Container Service, será possível executar comandos docker em todas as etapas marcadas com um asterisco (*).

bx ic attach [--no-stdin] [--sig-proxy] CONTAINER

Propósito
Controlar um contêiner em execução ou visualizar sua saída. Use CTRL+C para sair e parar o contêiner. Para obter mais informações, veja o comando attach na ajuda do Docker.
Parâmetros
--no-stdin
(Opcional) Não incluir a entrada padrão.
--sig-proxy
(Opcional) O padrão é true.
CONTAINER
(Necessário) Um nome ou um ID do contêiner.
Exemplo
O exemplo a seguir é uma solicitação para anexar ao contêiner my_container.
bx ic attach my_container

bx ic build -t TAG [--no-cache] [--pull] [-q] DOCKERFILE_LOCATION

Propósito
Construir uma imagem do Docker localmente ou no registro de imagens privadas do Bluemix. Para obter mais informações, consulte o comando build na ajuda do Docker.
Parâmetros
DOCKERFILE_LOCATION
(Obrigatório) O caminho para o Dockerfile e o contexto no host local.
--no-cache
(Opcional) Quando uma imagem é construída pela primeira vez, as camadas da imagem são armazenadas em cache para uma organização inteira. Ao construir as versões subsequentes da imagem, essas versões em cache da imagem são usadas para acelerar o processo de construção. Ao incluir a opção --no-cache na construção de uma nova versão da imagem, será possível assegurar que cada camada da imagem seja reconstruída também.
--pull
(Opcional) Por padrão, a imagem base só será extraída do registro se não estiver ainda no cache de imagem da sua organização. Ao incluir essa opção, será possível assegurar que a versão mais recente da imagem base seja usada para construir a nova imagem.
-q, --quiet
(Opcional) Suprimir a saída detalhada gerada pelos contêineres. O padrão é false.
-t TAG, --tag TAG
(Necessário) O caminho para seu registro de imagens privadas a ser aplicado à imagem criada.
Exemplo
O exemplo a seguir é uma solicitação para construir uma imagem nomeada myimage. O Dockerfile e outros artefatos para uso na construção estão no mesmo diretório a partir do qual o comando é executado. Como o registro e o namespace são incluídos com o nome da imagem, a imagem é construída no registro de imagens privadas do Bluemix de sua organização.
Dica: Execute bx ic namespace-get para recuperar seu namespace e substitua <my_namespace> por suas informações de namespace.
bx ic build -t registry.DomainName/<my_namespace>/myimage.

bx ic cp SRC_PATH CONTAINER:DEST_PATH | bx ic cp CONTAINER:SRC_PATH DEST_PATH

Propósito
Copiar um arquivo ou uma pasta de um contêiner único em execução ou interrompido para sua máquina local e vice-versa. Para obter mais informações, consulte o comando cp na ajuda do Docker.
Nota: Esse comando é suportado para ser usado somente com contêineres únicos. Quando se deseja copiar arquivos de ou para um grupo de contêiner, deve-se copiar os arquivos de ou para cada instância de contêiner do grupo.
Parâmetros
DEST_PATH
(Necessário) O caminho para o arquivo ou pasta em sua máquina local ou contêiner no qual você deseja copiar as informações do caminho de origem. O arquivo ou pasta deve existir em sua máquina local ou contêiner para poder iniciar a cópia. Se você quiser copiar para sua máquina local, será possível usar o caminho absoluto ou relativo. Para copiar para o contêiner único, deve-se incluir o nome do contêiner, o delimitador ":" e o caminho para o arquivo ou a pasta de destino. O caminho deve ser relativo ao diretório raiz do contêiner.
SRC_PATH
(Necessário) O caminho para o arquivo ou pasta que você deseja copiar. Se o arquivo ou pasta estiver em sua máquina local, será possível usar o caminho absoluto ou relativo. Se o arquivo ou a pasta estiver no contêiner único, o nome do contêiner, o delimitador ":" e o caminho para o arquivo ou a pasta que você deseja copiar deverão ser incluídos. O caminho deve ser relativo ao diretório raiz do contêiner.
Nota: Não é possível copiar determinados arquivos do sistema que são armazenados na pasta /dev, /etc, /proc e /sys de seu contêiner.
Exemplo
Esse comando de exemplo copia um arquivo chamado config.txt do diretório atual de sua máquina local para o diretório config de um contêiner único chamado my_container.
bx ic cp ./config.txt my_container:/config

bx ic cpi SRC DST

Propósito
Acessar uma imagem do Docker Hub ou uma imagem de seu registro local e copiá-la para o registro de imagens privadas do Bluemix.
Nota: Esse comando é uma alternativa ao uso de docker pull e docker push juntos para incluir uma imagem em seu registro de imagens do Bluemix. Se você preferir usar docker pull e docker push, consulte Enviando imagens locais por push para seu registro privado do Bluemix a partir da linha de comandos.
Parâmetros
DST
(Necessário) A URL do registro de imagens privadas do Bluemix, que inclui o namespace e o nome da imagem de destino. Uma tag para a imagem é opcional.
SRC
(Necessário) O repositório de origem e o nome da imagem.
Exemplo
bx ic cpi source_repository/source_image_name private_registry_URL/destination_image_name:tag
Dica: Execute bx ic namespace-get para recuperar seu namespace e substitua <my_namespace> por suas informações de namespace.
bx ic cpi training/sinatra registry.DomainName/<my_namespace>/mysinatra:v1

bx ic exec [-d] [-it ] [-u USER] CONTAINER CMD

Propósito
Executar um comando em um contêiner. Para obter mais informações, veja o comando exec na ajuda do Docker.
Parâmetros
CMD
(Necessário) O comando a ser executado no contêiner ou contêineres especificados.
CONTAINER
(Necessário) Um nome ou um ID do contêiner.
-d , --detach
(Opcional) Executar o comando especificado no plano de fundo.
-it
(Opcional) Modo interativo. Mantenha a exibição da entrada padrão. Digite exit para sair.
-u USER, --user USER
(Opcional) Um nome de usuário.
Exemplos
Este exemplo de comando executa um terminal bash interativo no contêiner my_container. 
bx ic exec -it my_container bash
Este exemplo de comando executa o comando de data no contêiner my_container. 
bx ic exec my_container date

bx ic group-create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session-affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

Propósito
Criar um grupo de ajuste de escala. Por padrão, os contêineres no Bluemix são executados no modo separado.
Parâmetros
--anti
(Opcional) Use antiafinidade para tornar seu grupo de contêiner mais altamente disponível. A opção --anti força cada instância de contêiner no grupo a ser colocada em um nó de cálculo físico separado, reduzindo as chances de impacto de todos os contêineres em um grupo em razão de uma falha de hardware. Você poderá não ser capaz de usar essa opção com tamanhos de grupo maiores porque cada região e organização do Bluemix têm um conjunto limitado de nós de cálculo disponíveis para implementação. Se sua implementação não for bem-sucedida, reduza o número de instâncias de contêiner no grupo ou remova a opção --anti.
--auto
(Opcional) Quando a recuperação automática é ativada para um grupo de contêiner e o grupo fica em execução durante 10 minutos, o balanceador de carga do grupo inicia a verificação regular do funcionamento de cada instância do contêiner no grupo por meio de solicitações de HTTP. Se uma instância do contêiner não responder dentro de 100 segundos, ela será marcada como inativa. As instâncias de contêiner inativas são removidas do grupo e recriadas pela recuperação automática. A recuperação automática tenta recuperar instâncias do contêiner em um grupo 3 vezes. Após a terceira tentativa, a recuperação automática não recuperará nenhuma instância de contêiner no grupo durante 60 minutos. Depois de 60 minutos, o processo de recuperação automática inicia novamente. Para obter mais informações sobre a verificação de funcionamento, veja a opção --http_monitor_enabled.
CMD
(Opcional) O comando e os argumentos são passados para o grupo de contêiner para serem executados. Esse deve ser um comando de longa execução. Não use um comando de curta duração, por exemplo, /bin/date, porque isso pode causar travamento do contêiner.
Exemplo de comandos de longa execução usando a imagem ibmnode que é fornecida pelo IBM Bluemix Container Service:
bx ic group-create --name my_container_group registry.DomainName/ibmnode ping localhost
bx ic group-create --name my_container_group registry.DomainName/ibmnode -- tail -f /dev/null
Nota: O CMD e seus argumentos devem vir no término da linha de comandos bx ic group-create.
-d DOMAIN, --domain DOMAIN
(Opcional) O domínio do sistema padrão é AppDomainName e já fornece um certificado SSL, para que seja possível acessar seus grupos de contêiner com HTTPS sem nenhuma configuração adicional. Para usar um domínio customizado, deve-se registrar o domínio customizado em um servidor DNS público, configurar o domínio customizado no Bluemix e, em seguida, mapear o domínio customizado para o domínio do sistema do Bluemix no servidor DNS público. Depois que seu domínio customizado estiver mapeado para o domínio do sistema do Bluemix, as solicitações de seu domínio customizado serão roteadas para seu aplicativo no Bluemix. Ao criar um domínio customizado, não inclua sublinhados (_) no nome do domínio. Para obter mais informações sobre domínios customizados, consulte Criando e usando um domínio customizado. Para tornar seu domínio customizado seguro, faça upload de um certificado SSL, para que seus grupos de contêiner possam ser acessados via HTTPS.

O host e o domínio combinados formam a URL de rota pública completa, como http://mycontainerhost.AppDomainName, e devem ser exclusivos no Bluemix. Quando você revisa os detalhes de um grupo de contêiner com o comando bx ic group-inspect, o host e o domínio são listados juntos como a rota.

--desired DESIRED
(Opcional) O número de instâncias no grupo. O padrão é 2. O número de instâncias do grupo de contêiner que seu app requer depende da carga de trabalho esperada e do tempo em que uma solicitação pode ser processada pelo app. Por exemplo, a conectividade de multitarefas ou de rede afeta quantas instâncias podem ser necessárias para seu app. Para determinar o número de instâncias desejadas, implemente um contêiner único no Bluemix que execute seu app e execute um teste de carga neste contêiner. Se o app puder manipular, por exemplo, 100 solicitações por segundo, mas você estiver esperando uma carga de trabalho média de 300 solicitações por segundo, o número desejado de instâncias para o grupo de contêiner será 3. Para tornar seu grupo de contêiner altamente disponível e mais resiliente à falha, considere a inclusão de instâncias extras, além do mínimo, para manipular a carga de trabalho esperada. Instâncias extras podem manipular a carga de trabalho no caso de travamento de outra instância. Para assegurar a recuperação automática de uma instância com falha sem afetar a carga de trabalho, inclua uma instância extra. Para proteção contra duas falhas simultâneas, inclua duas instâncias extras. Essa configuração é um padrão N+2, em que N é o número de instâncias para manipular as solicitações e +2 são duas instâncias extras.
Importante: Um grupo de contêiner deve incluir pelo menos 2 instâncias de contêiner. Se você incluir somente 1 instância no grupo e essa instância se tornar indisponível, seus usuários poderão experimentar tempo de inatividade para o app.

--http_monitor_enabled
(Opcional) O valor padrão é true. Quando a recuperação automática é ativada para um grupo de contêiner e o grupo fica em execução durante 10 minutos, o balanceador de carga do grupo inicia a verificação regular do funcionamento de cada instância do contêiner no grupo por meio de solicitações de HTTP. Se uma instância do contêiner não responder dentro de 100 segundos, ela será marcada como inativa. As instâncias de contêiner inativas são removidas do grupo e recriadas pela recuperação automática. A recuperação automática tenta recuperar instâncias do contêiner em um grupo 3 vezes. Após a terceira tentativa, a recuperação automática não recuperará nenhuma instância de contêiner no grupo durante 60 minutos. Depois de 60 minutos, o processo de recuperação automática inicia novamente. Alguns apps podem não requerer uma verificação de funcionamento de HTTP, como um app que usa TCP para comunicação, em vez de HTTP. Para desativar o monitor de HTTP, configure a opção como false. Exemplo: --http_monitor_enabled=false
--http_monitor_path
(Opcional) O valor padrão é /. Quando --http_monitor_enabled está ativado, é possível usar essa opção para configurar um caminho customizado para a verificação de funcionamento, como --http_monitor_path="/health/path".
--http_monitor_rc_list
(Opcional) Quando --http_monitor_enabled está ativado, é possível especificar a lista de códigos de resposta para a verificação de funcionamento. O valor padrão inclui os códigos 200, 201, 202, 204, 300, 301, 302, 401, 403 e 404. Qualquer resposta que não estiver na lista de respostas válidas mudará o estado da instância do contêiner para INACTIVE pelo balanceador de carga. Um contêiner inativo não recebe nenhuma carga e o estado do grupo de contêiner muda para HAS_INACTIVE_INSTANCES. Se nenhum dos contêineres estiver ACTIVE, o estado do grupo de contêiner mudará para NO_ACTIVE_INSTANCES. Para saber mais sobre estados, veja Estados do contêiner no IBM Bluemix Container Service.

bx ic wait CONTAINER [CONTAINER]

Propósito
Aguarde a saída de um contêiner. Durante esse tempo de espera, a linha de comandos não é retornada e você não pode inserir os comandos. Assim que ocorrer a saída ou a parada do contêiner, o código de saída será impresso na saída da CLI. Para obter mais informações, veja o comando wait na ajuda do Docker.
Parâmetros
CONTAINER
(Necessário) O nome ou o ID do contêiner.
Exemplo
O exemplo a seguir é uma solicitação para aguardar a saída de um contêiner denominado my_container. Assim que ocorre a saída, o código de saída é impresso na CLI. 
bx ic wait my_container

bx ic wait-status CONTAINER

Propósito
Aguarde até que um único contêiner ou um grupo de contêiner atinja um estado não temporário. Durante esse tempo de espera, a linha de comandos não é retornada e você não pode inserir os comandos. Assim que o contêiner atinge um estado não temporário, uma mensagem OK é exibida. Para contêineres únicos, os estados não temporários incluem Executando, Encerramento, Travado, Pausado ou Suspensos. Para grupos de contêineres, os estados não temporários incluem CREATE_COMPLETE, UPDATE_COMPLETE ou FAILED.
Parâmetros
CONTAINER
(Necessário) O nome ou o ID do contêiner ou do grupo.
Exemplo
O exemplo a seguir é uma solicitação para aguardar até que um contêiner denominado my_container atinja um estado não temporário. 
bx ic wait-status my_container

Desinstalando o plug-in do IBM Bluemix Container Service (bx ic)

Se você não precisar mais do plug-in do IBM Bluemix Container Service, será possível desinstalá-lo com um comando bx.

Nota: Para desinstalar o plug-in do IBM Bluemix Container Service Cloud Foundry, execute o comando a seguir. 
cf uninstall-plugin IBM-Containers
  1. Para desinstalar o plug-in do IBM Bluemix Container Service para a CLI do Bluemix, execute o comando a seguir. 
    bx plugin uninstall IBM-Containers
  2. Verifique se o plug-in foi desinstalado executando o comando a seguir e verificando a lista dos plug-ins que estão instalados. 
    bx plugin list

    O plug-in do IBM Bluemix Container Service não é exibido nos resultados.

単一コンテナーとスケーラブル・コンテナー について

IBM® Bluemix® は、アプリ開発用のフィーチャーとして、選ばれた地域で IBM Bluemix Container Service インフラストラクチャーを提供します。コンテナーにより、どの言語でも、どのプログラミング・ツールでも、アプリを作成できます。各コンテナーは、分離された、安全なアプリ・プラットフォームです。コンテナーは、OS X または Windows を実行している開発ワークステーション、Ubuntu を実行しているサーバー、Red Hat を実行している実動データ・センターの仮想マシン、Bluemix のようなクラウド・プラットフォームなど、任意の場所で実行できます。
重要: 単一コンテナーとスケーラブル・コンテナーはありません。クラスター管理について学び、Bluemix Public の IBM Bluemix Container Service の Kubernetes にマイグレーションします。

コンテナーとは?

コンテナーは、アプリが実行のために必要とするすべてのエレメントを含む仮想ソフトウェア・オブジェクトです。コンテナーは、コンテナー作成用の読み取り専用テンプレートであるイメージからビルドされます。 各イメージには、ホスト・オペレーティング・システムで分離したプロセスとして稼働するアプリとその従属物のみが含まれています。そのため、コンテナーは、リソースの分離と割り振りの利点を持ちながら、例えば仮想マシンなどよりも、移植可能性と効率性が高まっています。コンテナーは、高品質のアプリを素早くビルドするのに役立ちます。

コンテナーは迅速

コンテナーは、開発チームおよび実動チームのための、標準化された環境を提供することにより、システム管理を簡素化します。エンジンの軽量ランタイムにより、需要の変化に応じて、素早いスケールアップおよびスケールダウンが可能です。コンテナーは、さまざまなオペレーティング・システム・プラットフォームおよび基礎となるインフラストラクチャーを管理する複雑さをなくすのに役立ちます。コンテナーは、どのインフラストラクチャーでもすべてのアプリを素早く確実にデプロイして実行するのに役立ちます。

コンテナーは小さい

1 つの仮想マシンで必要とされる量のスペースに、複数のコンテナーを入れることができます。 コンテナーのサイズを、アプリで必要とされる量のスペースのみを使用するように選択できます。

コンテナーはポータブル

別のイメージをベースとして使用して、別のコンテナー用のイメージをビルドします。イメージの処理の大部分の作業は他の人にまかせ、そのイメージを自分の用途に合わせて微調整します。ステージング環境から実稼働環境に素早くアプリのコードをマイグレーションすることもできます。マイグレーション・プロセスは、Delivery Pipeline または UrbanCode Deploy などのツールで自動化できます。

Docker および IBM Bluemix Container Service

IBM Bluemix Container Service は、 Docker テクノロジーを利用する Bluemix ランタイムです。既存の Docker ユーザー、および Docker の新しいユーザーは、IBM Bluemix Container Service での以下の機能を期待できます。

  • IBM がホストするプライベート・レジストリーに既存の Docker Hub イメージを追加する。
  • 潜在的な脆弱性がないかイメージを検証する。
  • 既に快適に使用している Docker CLI コマンドを使用するために IBM Bluemix Container Service 用のコマンド・ライン・インターフェース (CLI) を構成する。
  • 追加の構成なしで、コンテナーのロギングとモニタリングのデータを表示する。

過去に Docker を使用した経験があれば、IBM Bluemix Container Service の使用を開始する際に、これらの一般的な質問に対する回答が役に立つかもしれません。

Docker Hub のイメージを IBM Bluemix Container Service で使用する方法を教えてください。
IBM Bluemix Container Service CLI を使用して、Bluemix プライベート・レジストリーに Docker Hub のイメージをコピーできます。 イメージがコピーされたら、それを使用してコンテナーを作成できます。
高可用性のコンテナーはどのようにして作成するのですか?
IBM Bluemix Container Service は、Docker コンテナーのような単一コンテナーを提供します。また、IBM Bluemix Container Service が提供する拡張が容易なコンテナー・グループを使用すると、アプリケーションの可用性を高めることができます。コンテナー・グループが提供する組み込みのロード・バランサーは、コンテナー・インスタンス間でワークロードを経路指定し、バランスを保ちます。グループの可用性を高めるには、アンチアフィニティーを有効にして複数の物理計算ホスト間にコンテナー・グループ・インスタンスを分散させ、正常でないインスタンスの自動リカバリーを有効にします。 また、CPU とメモリーの使用量に基づいて、コンテナー・グループ・インスタンスを自動スケーリングするポリシーを設定することもできます。 詳細については、可用性の高いプロセスをコンテナー・グループとして実行するを参照してください。
コンテナー・ポートをホスト・ポートにマップするにはどのようにすればいいですか?
IBM Bluemix Container Service では、プライベート・コンテナー・ネットワークとパブリック・ネットワークが区別されます。 プライベート・コンテナー・ネットワークでは、すべてのコンテナー・ポートが追加構成なしのデフォルトで公開されます。同じプライベート・コンテナー・ネットワークに接続されている単一のコンテナーとコンテナー・グループは、それぞれに割り当てられているプライベート IP アドレスを使用して相互に通信できます。 ポート・マッピングは不要です。

アプリをパブリック・ネットワークで使用できるようにするには、パブリック・ポートを公開し、パブリック IP アドレスを単一コンテナーにバインドするか、またはパブリック経路をコンテナー・グループにバインドする必要があります。 パブリック・ポートを公開すると、公開したポートでのみパブリック・データを送受信できるようになります。 その他のパブリック・ポートは、すべて閉じられます。

コンテナーをインターネットでパブリックに使用可能にするにはどのようにすればいいですか?
  • 単一コンテナーの場合、1 つ以上のポートを公開し、浮動 IP アドレスをそのコンテナーに割り当てます。
  • コンテナー・グループの場合、単一の HTTP ポートを公開し、経路を割り当てます。
ローカルの Docker と Bluemix とでコンテナーの作成に要する時間が違うのはなぜですか?
Bluemix でよりもローカル Docker インストール済み環境の方が、より速くコンテナーを作成できることにお気付きかもしれません。最初のデプロイメントの際に、コンテナー・イメージをホスト上のレジストリーにダウンロードする必要があります。それ以降のイメージのデプロイメントは速くなります。コンテナー構成が異なるとデプロイメント速度にどのような影響があるかについて詳しくは、コンテナー・タイプを参照してください。
プライベート・ネットワークでコンテナーを作成してパブリックに公開はしないようにするにはどうすればいいですか?
コンテナーを作成すると、そのコンテナーは IBM Bluemix Container Service のデフォルトのコンテナー・プライベート・ネットワークに接続され、プライベート IP アドレスが割り当てられます。 このプライベート・ネットワークでは、デフォルトですべてのコンテナー・ポートが公開されます。 ホストのポートをコンテナーのポートにマッピングする必要はありません。コンテナーをインターネットからアクセスできないようにするには、コンテナーの作成時に IP アドレスをバインドしないか、ポートを公開しないようにしてください。
ボリュームをコンテナーにマウントするにはどのようにすればいいですか?
単一コンテナーは、設計上、存続時間が短期です。しかし、ボリュームを使用することにより、コンテナーを再始動してもデータが永続するようにできます。Docker 内にボリュームをマウントすると、そのボリュームはローカル・ファイル・システムにマウントされます。 IBM Bluemix Container Service では、計算ホストへのアクセスが制限されるので、ホスト・ディレクトリーをコンテナーにマウントすることができません。 代わりに、組織規模のボリュームを使用して、コンテナーを再始動してもデータを保持します。ボリュームは、分離されたファイル共有上でホストされます。そこでは、安全にアプリ・データが保管され、ファイルに対するアクセスや権限が管理されます。 ボリュームをコンテナーにマウントするには、まず、ボリュームを作成する必要があります。CLI からコンテナーを作成する場合は、--volume オプションを使用して、既存のボリュームをマウントします。 Bluemix GUI から、「詳細オプション」を選択して、既存のボリュームを割り当てます。 詳細については、コマンド・ライン・インターフェース (CLI) を使用したボリュームへのファイルの追加を参照してください。
IBM Bluemix Container Service では Docker Compose をどのように使用するのですか?
Docker Compose コマンドを実行することで、IBM Bluemix Container Service でマルチコンテナー・デプロイメントを作成および管理できます。詳細については、チュートリアル: Docker Compose と IBM Bluemix Container Service を使用してマルチコンテナー・デプロイメントを作成するを参照してください。
IBM Containers サービスを指定するように Docker Compose クライアントをセットアップするにはどうすればいいですか?
IBM Bluemix Container Service CLI にログインするときに、Docker のネイティブ・コマンドを実行するように CLI を構成します。 詳しくは、『bx ic プラグインへのログイン』を参照してください。
実行中のコンテナーにログインするにはどのようにすればいいですか? SSH を使用するのですか?
bx ic exec を使用してコンテナーにログインできます。
なぜ私のコンテナーはデタッチ・モードで実行されているのですか?
実行中のプロセスを持つコンテナーが、終了せずにバックグラウンドで実行されるように、デフォルトでは、Bluemix のコンテナーはデタッチ・モードで実行されます。 デタッチ・モードで実行されるコンテナーは、クラウド環境の標準モデルです。 bx ic attach [--no-stdin] [--sig-proxy] CONTAINER を実行してコンテナーに接続することもできます。

Bluemix Dedicated および Bluemix Local における IBM Bluemix Container Service

IBM Bluemix Container Service は、Bluemix Dedicated および Bluemix Local で使用可能です。

Bluemix Local での IBM Bluemix Container Service のハードウェア要件

ハードウェアの基本要件
デフォルトでは、基本セットアップはコンテナー計算の容量として 64GB のメモリーを使用して構成されます。
  1. ご使用の環境が Bluemix Local のハードウェア要件を満たしている必要があります。
  2. IBM Bluemix Container Service を使用するには、セットアップごとに以下の要件を満たすコントロール・プレーンが必要です。モニタリング、ロギングを行うには、同じハードウェア要件を満たす、2 つのインスタンスも必要です。
    • VMware: 3 つの ESX サーバーと、各サーバーに 256GB のメモリーと 32 個の物理コア (ハイパースレッドが使用可能な 64 vCore)。3 つのサーバーにかかる、4TB のクラスター共有ストレージが必要です。
追加ハードウェアの要件
最大 12 まで拡張 VM を追加すると、ハードウェアを追加しなくても、コンテナー計算の容量を 16GB ずつ最大 256GB まで増やすことができます。各 VM には 20GB のメモリー、4 つの vCPU、200GB のディスク・スペースがなければなりません。しかし、クラスター上に 10 個の拡張 VM をデプロイするたびに、ネットワーキングの処理のために追加の VM が 1 つ必要になります。このネットワーキング VM には 4 つの vCPU、16GB のメモリー、250GB のディスク・スペースが必要です。

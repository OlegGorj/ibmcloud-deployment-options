bx ic attach [--no-stdin] [--sig-proxy] CONTAINER

目的
実行中のコンテナーを制御するか、その出力を表示します。コンテナーを終了して停止するには、CTRL+C を使用します。詳しくは、Docker ヘルプ内の attach コマンドを参照してください。
パラメーター
--no-stdin
(オプション) 標準入力を含めません。
--sig-proxy
(オプション) デフォルトは true です。
CONTAINER
(必須) コンテナーの名前または ID。
例
以下の例は、コンテナー my_container に接続する要求です。
bx ic attach my_container

bx ic build -t TAG [--no-cache] [--pull] [-q] DOCKERFILE_LOCATION

目的
Docker イメージを、ローカルに、またはプライベート Bluemix イメージ・レジストリーにビルドする。 詳しくは、Docker ヘルプ内の build コマンドを参照してください。
パラメーター
DOCKERFILE_LOCATION
(必須) ローカル・ホスト上の Dockerfile およびコンテキストへのパス。
--no-cache
(オプション) イメージが初めてビルドされる際には、組織全体のイメージのレイヤーがキャッシュに入れられます。 イメージのそれ以降のバージョンを作成するときには、ビルド・プロセスの速度向上のため、キャッシュ中にあるそのイメージのバージョンが使用されます。 イメージの新しいバージョンをビルドする際に --no-cache オプションを含めることにより、イメージの各レイヤーも確実に再ビルドされるようにすることができます。
--pull
(オプション) デフォルトでは、ベース・イメージは、まだ組織のイメージ・キャッシュ中にない場合にのみ、レジストリーからプルされます。このオプションを含めると、ベース・イメージの最新バージョンが新しいイメージのビルドに使用されるようにすることができます。
-q, --quiet
(オプション) コンテナーによって生成される詳細出力を抑止します。デフォルトは false です。
-t TAG, --tag TAG
(必須) 作成されるイメージに適用するプライベート・イメージ・レジストリーへのパス。
例
以下の例は、myimage という名前のイメージをビルドする要求です。このビルドで使用する Dockerfile および他の成果物は、コマンドが実行される同じディレクトリーにあります。 レジストリーと名前空間がイメージ名と共に組み込まれているので、イメージは組織のプライベート Bluemix イメージ・レジストリー内でビルドされます。
ヒント: bx ic namespace-get を実行して名前空間を取得し、<my_namespace> をその名前空間情報に置き換えます。
bx ic build -t registry.DomainName/<my_namespace>/myimage .

bx ic cp SRC_PATH CONTAINER:DEST_PATH | bx ic cp CONTAINER:SRC_PATH DEST_PATH

目的
ファイルまたはフォルダーを、実行中または停止中の単一コンテナーからローカル・マシンにコピーしたり、その逆を行ったりします。詳しくは、Docker ヘルプ内の cp コマンドを参照してください。
注: このコマンドは、単一コンテナーにしか使用できません。 コンテナー・グループとの間でファイルをコピーする場合は、グループに含まれている各インスタンスとの間でファイルをコピーする必要があります。
パラメーター
DEST_PATH
(必須) ソース・パスから情報をコピーする宛先である、ローカル・マシンまたはコンテナー上のファイルまたはフォルダーへのパス。 ファイルまたはフォルダーのコピーを開始する前に、それらがローカル・マシンまたはコンテナーに存在する必要があります。 ローカル・マシンにコピーする場合、絶対パスまたは相対パスのどちらでも使用できます。 単一コンテナーにコピーする場合、コンテナーの名前、区切り文字「:」、そしてコピー先のファイルまたはフォルダーへのパスを含める必要があります。パスは、コンテナーのルート・ディレクトリーからの相対パスにする必要があります。
SRC_PATH
(必須) コピーするファイルまたはフォルダーへのパス。 ファイルまたはフォルダーがローカル・マシン上にある場合、絶対パスまたは相対パスのどちらでも使用できます。 ファイルまたはフォルダーが単一コンテナーにある場合、コンテナーの名前、区切り文字「:」、そしてコピーするファイルまたはフォルダーへのパスを含める必要があります。パスは、コンテナーのルート・ディレクトリーからの相対パスにする必要があります。
注: コンテナー内の /dev、/etc、/proc および /sys フォルダーの下に格納されている一部のシステム・ファイルはコピーできません。
例
このサンプル・コマンドは、config.txt という名前のファイルをローカル・マシンの現行ディレクトリーから my_container という名前の単一コンテナーの config ディレクトリーにコピーします。
bx ic cp ./config.txt my_container:/config

bx ic cpi SRC DST

目的
Docker Hub イメージ、またはローカル・レジストリーからのイメージにアクセスし、それをプライベート Bluemix イメージ・レジストリーにコピーします。
注: このコマンドは、docker pull と docker push を一緒に使用することでイメージを Bluemix イメージ・レジストリーに追加する代わりに使用します。docker pull と docker push を使用する方法を選択する場合は、コマンド・ラインからのプライベート Bluemix レジストリーへのローカル・イメージのプッシュを参照してください。
パラメーター
DST
(必須) プライベート Bluemix イメージ・レジストリー URL (名前空間を含む) およびデスティネーション・イメージ名。イメージのタグはオプションです。
SRC
(必須) ソース・リポジトリーおよびイメージ名。
例
bx ic cpi source_repository/source_image_name private_registry_URL/destination_image_name:tag
ヒント: bx ic namespace-get を実行して名前空間を取得し、<my_namespace> をその名前空間情報に置き換えます。
bx ic cpi training/sinatra registry.DomainName/<my_namespace>/mysinatra:v1

bx ic exec [-d] [-it ] [-u USER] CONTAINER CMD

目的
コンテナー内でコマンドを実行します。詳しくは、Docker ヘルプ内の exec コマンドを参照してください。
パラメーター
CMD
(必須) 指定された 1 つ以上のコンテナー内で実行するコマンド。
CONTAINER
(必須) コンテナーの名前または ID。
-d , --detach
(オプション) 指定されたコマンドをバックグラウンドで実行します。
-it
(オプション) 対話モード。標準入力が表示されている状態を維持します。終了するには exit と入力します。
-u USER, --user USER
(オプション) ユーザー名。
例
このサンプル・コマンドは、コンテナー my_container 内でインタラクティブ bash ターミナルを実行します。
bx ic exec -it my_container bash
このサンプル・コマンドは、コンテナー my_container 内で date コマンドを実行します。
bx ic exec my_container date

bx ic group-create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session-affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

目的
スケーリング・グループを作成します。 デフォルトでは、Bluemix のコンテナーはデタッチ・モードで実行されます。
パラメーター
--anti
(オプション) アンチアフィニティーを使用して、コンテナー・グループの可用性を高めることができます。 --anti オプションは、グループ内のすべてのコンテナー・インスタンスを別々の物理計算ノード上に強制的に配置します。これにより、1 つのハードウェア障害によってグループ内のすべてのコンテナーが異常終了してしまう可能性を減らします。 大きいグループ・サイズの場合は、このオプションを使用できない可能性があります。 Bluemix のどの地域およびどの組織にも、デプロイできる計算ノードのセットには限りがあるからです。デプロイメントに失敗した場合は、グループ内のコンテナー・インスタンスの数を減らすか、--anti オプションを外してください。
--auto
(オプション) コンテナー・グループで自動リカバリーが有効になり、そのグループが 10 分間実行すると、そのグループのロード・バランサーは HTTP 要求を介して、グループ内の各コンテナー・インスタンスの正常性に関する定期的な検査を開始します。 コンテナー・インスタンスが 100 秒以内に応答しない場合、それは非アクティブとしてマークされます。 非アクティブなコンテナー・インスタンスはグループから削除されて、自動リカバリーにより再作成されます。 自動リカバリーでは、グループ内にコンテナー・インスタンスをリカバリーする試行を 3 回行います。 3 回目の試行の後、そのグループでは、自動リカバリーによるコンテナー・インスタンスのリカバリーが 60 分間行われなくなります。 60 分後に、自動リカバリーのプロセスが再開します。ヘルス・チェックについて詳しくは、--http_monitor_enabled オプションを参照してください。
CMD
(オプション) コマンドと引数が実行のためにコンテナー・グループに渡されます。このコマンドは長期実行コマンドでなければなりません。実行時間が短いコマンド (例えば /bin/date) は、コンテナーが異常終了する原因になることがあるため、使用しないでください。
IBM Bluemix Container Service によって提供される ibmnode イメージを使用した長期実行コマンドの例:
bx ic group-create --name my_container_group registry.DomainName/ibmnode ping localhost
bx ic group-create --name my_container_group registry.DomainName/ibmnode -- tail -f /dev/null
注: CMD とその引数は bx ic group-create コマンド・ラインの最後に指定する必要があります。
-d DOMAIN, --domain DOMAIN
(オプション) デフォルトのシステム・ドメインは AppDomainName であり、それにより SSL 証明書が既に提供されているため、付加的な構成なしで HTTPS によりコンテナー・グループにアクセスすることができます。 カスタム・ドメインを使用するには、パブリック DNS サーバー上でカスタム・ドメインを登録し、Bluemix でカスタム・ドメインを構成してから、カスタム・ドメインをパブリック DNS サーバー上の Bluemix システム・ドメインにマップする必要があります。 カスタム・ドメインが Bluemix システム・ドメインにマップされた後、カスタム・ドメインを求める要求は、Bluemix 内のアプリケーションにルーティングされます。 カスタム・ドメインを作成する際、ドメイン名に下線 (_) を含めないでください。 カスタム・ドメインについて詳しくは、カスタム・ドメインの作成と使用を参照してください。 カスタム・ドメインのセキュリティーを確保するには、SSL 証明書をアップロードして、HTTPS でコンテナー・グループにアクセスできるようにします。

ホストとドメインの組み合わせが、完全なパブリック経路 URL (例えば http://mycontainerhost.AppDomainName) を形成します。 これは、Bluemix 内で固有でなければなりません。 bx ic group-inspect コマンドを使用してコンテナー・グループの詳細を検討すると、ホストとドメインが経路として一緒にリストされます。

--desired DESIRED
(オプション) グループ内のインスタンスの数。 デフォルトは 2です。アプリに必要なコンテナー・グループのインスタンス数は、予想されるワークロードや、アプリが要求を処理できる時間によって異なります。 例えば、マルチタスキングやネットワーク接続によって、アプリに必要なインスタンス数は左右されます。 必要なインスタンス数を決定するには、アプリを実行するコンテナーを 1 つ Bluemix にデプロイし、そのコンテナーで負荷テストを実行します。 例えば、アプリで処理できる要求が 1 秒あたり 100 個であるのに対し、予想される平均ワークロードが 1 秒あたり 300 個だとすると、コンテナー・グループに必要なインスタンス数は 3 になります。 コンテナー・グループの可用性と耐障害性を高めるために、予想されるワークロードを処理する最低限の数のインスタンスではなく余分にインスタンスを組み込むことを検討してください。 インスタンスを余分に組み込んでおくことで、さらに別のインスタンスが異常終了した場合でもワークロードを処理できます。障害が起きたインスタンスをワークロードに影響を与えずに自動復旧するには、1 つ余分にインスタンスを組み込みます。2 つが同時に障害を発生した場合に対応できるようにするには、2 つ余分にインスタンスを組み込みます。このセットアップは N+2 パターンです。 N は要求を処理するインスタンスの数、+2 は追加の 2 つのインスタンスです。
重要: 1 つのコンテナー・グループには、少なくとも 2 つのコンテナー・インスタンスを含める必要があります。 グループにインスタンスを 1 つしか含めていないと、そのインスタンスが使用不可になった場合に、ユーザーがアプリのダウン時間を経験する可能性があります。
-e ENV, --env ENV
(オプション) 環境変数を設定します。ここで、ENV は key=value ペアです。複数のキーを別々にリストします。 引用符を含める場合は、環境変数名と値の両方を囲みます。 例: -e "key1=value1" -e "key2=value2a,value2b" -e "key3=value3"

<appname>

--http_monitor_enabled
(オプション) デフォルト値は true です。コンテナー・グループで自動リカバリーが有効になり、そのグループが 10 分間実行すると、そのグループのロード・バランサーは HTTP 要求を介して、グループ内の各コンテナー・インスタンスの正常性に関する定期的な検査を開始します。 コンテナー・インスタンスが 100 秒以内に応答しない場合、それは非アクティブとしてマークされます。 非アクティブなコンテナー・インスタンスはグループから削除されて、自動リカバリーにより再作成されます。 自動リカバリーでは、グループ内にコンテナー・インスタンスをリカバリーする試行を 3 回行います。 3 回目の試行の後、そのグループでは、自動リカバリーによるコンテナー・インスタンスのリカバリーが 60 分間行われなくなります。 60 分後に、自動リカバリーのプロセスが再開します。 HTTP ではなく TCP を通信に使用するアプリなどのように、HTTP ヘルス・チェックが不要なアプリもあります。HTTP モニターを無効にするには、このオプションを false に設定します。 例: --http_monitor_enabled=false
--http_monitor_path
(オプション) デフォルト値は / です。--http_monitor_enabled を有効にした場合は、このオプションを使用して、ヘルス・チェックのカスタム・パスを設定できます (例: --http_monitor_path="/health/path")。
--http_monitor_rc_list
(オプション) --http_monitor_enabled を有効にした場合は、ヘルス・チェックの応答コードのリストを指定できます。デフォルト値に含まれているコードは、200、201、202、204、300、301、302、401、403、404 です。有効な応答のリストに含まれていない応答が返ると、ロード・バランサーがコンテナー・インスタンスの状態を INACTIVE に変更します。 非アクティブなコンテナーはロードを受け取らないので、コンテナー・グループの状態が HAS_INACTIVE_INSTANCES に変わります。いずれのコンテナーも ACTIVE ではない場合、コンテナー・グループの状態は NO_ACTIVE_INSTANCES に変わります。状態について詳しくは、IBM Bluemix Container Service のコンテナーの状態を参照してください。
IMAGE
(必須) コンテナー・グループ中の各コンテナー・インスタンスに含めるイメージ。イメージの後にコマンドをリストできますが、イメージの後にはオプションを置かないでください。イメージ指定の前にすべてのオプションを含めるようにしてください。

組織のプライベート Bluemix イメージ・レジストリー内のイメージを使用する場合、形式 registry.DomainName/NAMESPACE/IMAGE でイメージを指定します。

IBM Bluemix Container Service によって提供されるイメージを使用している場合は、組織の名前空間を含めないでください。イメージを、registry.DomainName/IMAGEという形式で指定します。

--ip IP_ADDRESS
(オプション) 使用可能な IP アドレスがある場合は、パブリック IP アドレスをコンテナー・グループにバインドします。 -p フラグを指定して、ポートを公開する必要もあります。コンテナー・グループは、パブリック経路またはパブリック IP アドレスのどちらかを取ることができますが、両方を取ることはできません。IP アドレスをアンバインドしてその IP をスペースのクォータに戻すには、コンテナー・グループを一度削除して、IP アドレスを指定しないでそのグループを再作成します。コンテナー・グループを作成する前に IP アドレスを要求するには、bx ic ip-request を実行します。
注: パブリック IP アドレスをバインドしても、コンテナー・グループとの間で送受信するデータを暗号化する SSL 証明書は組み込まれません。ご使用のアプリに SSL 暗号化が必要である場合は、独自の SSL ソリューションを実装するか、またはパブリック IP アドレスをバインドする代わりにパブリック経路をコンテナー・グループにマップすることができます。パブリック経路には SSL 証明書が既に組み込まれているので、追加の構成を行わずに HTTPS でコンテナー・グループにアクセスできます。
-m MEMORY, --memory MEMORY
(オプション) コンテナーのメモリー限度を MB 単位で入力します。 メモリー限度は、コンテナーが実行時に計算ホスト上で取得するメモリーとディスク・スペースの最大量を定義するコンテナー・サイズの一部です。 コンテナー・サイズが割り当てられた後、その値を変更することはできません。IBM Bluemix Container Service で設定可能なサイズは、以下のとおりです。
  • ピコ (64 MB メモリー、4 GB ディスク・スペース)
  • ナノ (128 MB メモリー、8 GB ディスク・スペース)
  • マイクロ (256 MB メモリー、16 GB ディスク・スペース)
  • タイニー (512 MB メモリー、32 GB ディスク・スペース)
  • スモール (1024 MB メモリー、64 GB ディスク・スペース)
  • ミディアム (2048 MB メモリー、128 GB ディスク・スペース)
  • ラージ (4096 MB メモリー、256 GB ディスク・スペース)
  • X ラージ (8192 MB メモリー、512 GB ディスク・スペース)
  • 2X ラージ (16384 MB メモリー、1 TB ディスク・スペース)
コンテナーのサイズを設定しない場合、各コンテナー・インスタンスは 256 MB で作成されます。
重要: コンテナー・メモリーは、単位ラベルを付けずに MB で入力してください。 例えば、ピコ・コンテナーを作成する場合は、-m 64 と入力します。
--max MAX
(オプション) インスタンスの最大数。デフォルトは 2 です。
--min MIN
(オプション) インスタンスの最小数。デフォルトは 1 です。

(オプション) ホスト名 (例えば mycontainerhost)。ホスト名に下線 (_) を含めないでください。ホストとドメインの組み合わせが、完全なパブリック経路 URL (例えば http://mycontainerhost.AppDomainName) を形成します。 bx ic group-inspect コマンドを使用してコンテナー・グループの詳細を検討すると、ホストとドメインが経路として一緒にリストされます。

(必須) グループに名前を付けます。-n は推奨されません。
ヒント: コンテナー名は文字で始める必要があり、その後には、大文字、小文字、数字、ピリオド (.)、下線 (_)、およびハイフン (-) を使用できます。
-p PORT, --publish PORT

(必須) HTTP トラフィックとプライベート・ネットワーク・トラフィック用のポートを公開します。ポートを公開しない場合は、コンテナー・グループは実行されますが、パブリック・ネットワークかプライベート・ネットワーク上のグループのロード・バランサーを使用してコンテナー・グループにアクセスできません。

コンテナー・グループの場合、複数のポートを含めることはできません。複数のポートがリストされる場合、最後にリストされているポートのみが公開されます。他のポートは、コマンドに無視されます。

経路をバインドするとき、グループ内のコンテナーはグループの公開ポートで HTTP トラフィックを listen する必要があります。 HTTP 以外のポートはパブリックに公開できません。HTTPS トラフィックが公開されたポートに届くと、(Go)Router は HTTPS 終端を実行します。その後、(Go)Router とコンテナーの間のプライベート仮想ネットワークでは HTTP プロトコルが使用されます。 IP アドレスをバインドするとき、 グループ内のコンテナーは HTTP トラフィックまたは HTTPS トラフィックを listen できますが、公開可能なポートは 1 つだけです。
使用するイメージの Dockerfile 内にポートが指定されている場合、そのポートを含めてください。
ヒント:
  • IBM 認定 Liberty Server イメージ、またはこのイメージの変更版の場合、ポート 9080 を入力します。
  • IBM 認定 Node.js イメージ、またはこのイメージの変更版の場合、ポート 8000. など、任意のポートを入力します。
--session_affinity

(オプション) デフォルト値は false です。--http_monitor_enabled を true に設定した場合は、--session_affinity オプションを true に設定して、セッション Cookie の作成と管理をアプリの代わりに IBM Bluemix Container Service で行うことができます。 --http_monitor_enabled を false に設定した場合や、HTTP ではなく TCP を通信に使用する場合、このオプションは無効です。

--volume VOLUME:/DIRECTORY_PATH[:ro]
(オプション) VOLUME:/DIRECTORY_PATH[:ro] 形式で詳細を指定して、コンテナーにボリュームを接続します。
  • VOLUME: ボリューム ID または名前。
  • DIRECTORY_PATH: コンテナー内でのディレクトリーの絶対パス。
  • ro: オプション。デフォルトは読み取り/書き込みですが、ro を指定するとボリュームは読み取り専用になります。
例

目的

パラメーター
DOCKERFILE_LOCATION
(必須) ローカル・ホスト上の Dockerfile およびコンテキストへのパス。
--no-cache
(オプション) イメージが初めてビルドされる際には、組織全体のイメージのレイヤーがキャッシュに入れられます。 イメージのそれ以降のバージョンを作成するときには、ビルド・プロセスの速度向上のため、キャッシュ中にあるそのイメージのバージョンが使用されます。 イメージの新しいバージョンをビルドする際に --no-cache オプションを含めることにより、イメージの各レイヤーも確実に再ビルドされるようにすることができます。

cf ic group create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session_affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

目的
スケーリング・グループを作成します。 デフォルトでは、Bluemix のコンテナーはデタッチ・モードで実行されます。
パラメーター
--anti
(オプション) アンチアフィニティーを使用して、コンテナー・グループの可用性を高めることができます。 --anti オプションは、グループ内のすべてのコンテナー・インスタンスを別々の物理計算ノード上に強制的に配置します。これにより、1 つのハードウェア障害によってグループ内のすべてのコンテナーが異常終了してしまう可能性を減らします。 大きいグループ・サイズの場合は、このオプションを使用できない可能性があります。 Bluemix のどの地域およびどの組織にも、デプロイできる計算ノードのセットには限りがあるからです。デプロイメントに失敗した場合は、グループ内のコンテナー・インスタンスの数を減らすか、--anti オプションを外してください。
--auto
(オプション) コンテナー・グループで自動リカバリーが有効になり、そのグループが 10 分間実行すると、そのグループのロード・バランサーは HTTP 要求を介して、グループ内の各コンテナー・インスタンスの正常性に関する定期的な検査を開始します。 コンテナー・インスタンスが 100 秒以内に応答しない場合、それは非アクティブとしてマークされます。 非アクティブなコンテナー・インスタンスはグループから削除されて、自動リカバリーにより再作成されます。 自動リカバリーでは、グループ内にコンテナー・インスタンスをリカバリーする試行を 3 回行います。 3 回目の試行の後、そのグループでは、自動リカバリーによるコンテナー・インスタンスのリカバリーが 60 分間行われなくなります。 60 分後に、自動リカバリーのプロセスが再開します。ヘルス・チェックについて詳しくは、--http_monitor_enabled オプションを参照してください。
CMD
(オプション) コマンドと引数が実行のためにコンテナー・グループに渡されます。このコマンドは長期実行コマンドでなければなりません。実行時間が短いコマンド (例えば /bin/date) は、コンテナーが異常終了する原因になることがあるため、使用しないでください。
IBM Bluemix Container Service によって提供される ibmnode イメージを使用した長期実行コマンドの例:
cf ic group create --name my_container_group registry.DomainName/ibmnode ping localhost
cf ic group create --name my_container_group registry.DomainName/ibmnode -- tail -f /dev/null
注: CMD およびその引数は cf ic group create コマンド・ラインの末尾に指定する必要があります。
-d DOMAIN, --domain DOMAIN
(オプション) デフォルトのシステム・ドメインは AppDomainName であり、それにより SSL 証明書が既に提供されているため、付加的な構成なしで HTTPS によりコンテナー・グループにアクセスすることができます。 カスタム・ドメインを使用するには、パブリック DNS サーバー上でカスタム・ドメインを登録し、Bluemix でカスタム・ドメインを構成してから、カスタム・ドメインをパブリック DNS サーバー上の Bluemix システム・ドメインにマップする必要があります。 カスタム・ドメインが Bluemix システム・ドメインにマップされた後、カスタム・ドメインを求める要求は、Bluemix 内のアプリケーションにルーティングされます。 カスタム・ドメインを作成する際、ドメイン名に下線 (_) を含めないでください。 カスタム・ドメインについて詳しくは、カスタム・ドメインの作成と使用を参照してください。 カスタム・ドメインのセキュリティーを確保するには、SSL 証明書をアップロードして、HTTPS でコンテナー・グループにアクセスできるようにします。

ホストとドメインの組み合わせが、完全なパブリック経路 URL (例えば http://mycontainerhost.AppDomainName) を形成します。 これは、Bluemix 内で固有でなければなりません。 cf ic group inspect コマンドを使用してコンテナー・グループの詳細を検討すると、ホストとドメインが経路として一緒にリストされます。

--desired DESIRED
(オプション) グループ内のインスタンスの数。 デフォルトは 2です。アプリに必要なコンテナー・グループのインスタンス数は、予想されるワークロードや、アプリが要求を処理できる時間によって異なります。 例えば、マルチタスキングやネットワーク接続によって、アプリに必要なインスタンス数は左右されます。 必要なインスタンス数を決定するには、アプリを実行するコンテナーを 1 つ Bluemix にデプロイし、そのコンテナーで負荷テストを実行します。 例えば、アプリで処理できる要求が 1 秒あたり 100 個であるのに対し、予想される平均ワークロードが 1 秒あたり 300 個だとすると、コンテナー・グループに必要なインスタンス数は 3 になります。 コンテナー・グループの可用性と耐障害性を高めるために、予想されるワークロードを処理する最低限の数のインスタンスではなく余分にインスタンスを組み込むことを検討してください。 インスタンスを余分に組み込んでおくことで、さらに別のインスタンスが異常終了した場合でもワークロードを処理できます。障害が起きたインスタンスをワークロードに影響を与えずに自動復旧するには、1 つ余分にインスタンスを組み込みます。2 つが同時に障害を発生した場合に対応できるようにするには、2 つ余分にインスタンスを組み込みます。このセットアップは N+2 パターンです。 N は要求を処理するインスタンスの数、+2 は追加の 2 つのインスタンスです。
重要: 1 つのコンテナー・グループには、少なくとも 2 つのコンテナー・インスタンスを含める必要があります。 グループにインスタンスを 1 つしか含めていないと、そのインスタンスが使用不可になった場合に、ユーザーがアプリのダウン時間を経験する可能性があります。

(オプション) 環境変数を設定します。ここで、ENV は key=value ペアです。複数のキーを別々にリストします。 引用符を含める場合は、環境変数名と値の両方を囲みます。 例: -e "key1=value1" -e "key2=value2a,value2b" -e "key3=value3"

(オプション) --http_monitor_enabled を有効にした場合は、ヘルス・チェックの応答コードのリストを指定できます。デフォルト値に含まれているコードは、200、201、202、204、300、301、302、401、403、404 です。有効な応答のリストに含まれていない応答が返ると、ロード・バランサーがコンテナー・インスタンスの状態を INACTIVE に変更します。 非アクティブなコンテナーはロードを受け取らないので、コンテナー・グループの状態が HAS_INACTIVE_INSTANCES に変わります。いずれのコンテナーも ACTIVE ではない場合、コンテナー・グループの状態は NO_ACTIVE_INSTANCES に変わります。状態について詳しくは、IBM Bluemix Container Service のコンテナーの状態を参照してください。

--ip IP_ADDRESS
(オプション) 使用可能な IP アドレスがある場合は、パブリック IP アドレスをコンテナー・グループにバインドします。 -p フラグを指定して、ポートを公開する必要もあります。コンテナー・グループは、パブリック経路またはパブリック IP アドレスのどちらかを取ることができますが、両方を取ることはできません。IP アドレスをアンバインドしてその IP をスペースのクォータに戻すには、コンテナー・グループを一度削除して、IP アドレスを指定しないでそのグループを再作成します。コンテナー・グループを作成する前に IP アドレスを要求するには、cf ic ip request を実行します。
注: パブリック IP アドレスをバインドしても、コンテナー・グループとの間で送受信するデータを暗号化する SSL 証明書は組み込まれません。ご使用のアプリに SSL 暗号化が必要である場合は、独自の SSL ソリューションを実装するか、またはパブリック IP アドレスをバインドする代わりにパブリック経路をコンテナー・グループにマップすることができます。パブリック経路には SSL 証明書が既に組み込まれているので、追加の構成を行わずに HTTPS でコンテナー・グループにアクセスできます。
-m MEMORY, --memory MEMORY
(オプション) コンテナーのメモリー限度を MB 単位で入力します。 メモリー限度は、コンテナーが実行時に計算ホスト上で取得するメモリーとディスク・スペースの最大量を定義するコンテナー・サイズの一部です。 コンテナー・サイズが割り当てられた後、その値を変更することはできません。IBM Bluemix Container Service で設定可能なサイズは、以下のとおりです。
  • ピコ (64 MB メモリー、4 GB ディスク・スペース)
  • ナノ (128 MB メモリー、8 GB ディスク・スペース)
  • マイクロ (256 MB メモリー、16 GB ディスク・スペース)
  • タイニー (512 MB メモリー、32 GB ディスク・スペース)
  • スモール (1024 MB メモリー、64 GB ディスク・スペース)
  • ミディアム (2048 MB メモリー、128 GB ディスク・スペース)
  • ラージ (4096 MB メモリー、256 GB ディスク・スペース)
  • X ラージ (8192 MB メモリー、512 GB ディスク・スペース)
  • 2X ラージ (16384 MB メモリー、1 TB ディスク・スペース)
コンテナーのサイズを設定しない場合、各コンテナー・インスタンスは 256 MB で作成されます。
重要: コンテナー・メモリーは、単位ラベルを付けずに MB で入力してください。 例えば、ピコ・コンテナーを作成する場合は、-m 64 と入力します。

(オプション) インスタンスの最大数。デフォルトは 2 です。
--min MIN
(オプション) インスタンスの最小数。デフォルトは 1 です。

(オプション) ホスト名 (例えば mycontainerhost)。ホスト名に下線 (_) を含めないでください。ホストとドメインの組み合わせが、完全なパブリック経路 URL (例えば http://mycontainerhost.AppDomainName) を形成します。 cf ic group inspect コマンドを使用してコンテナー・グループの詳細を検討すると、ホストとドメインが経路として一緒にリストされます。

(必須) グループに名前を付けます。-n は推奨されません。

(必須) HTTP トラフィックとプライベート・ネットワーク・トラフィック用のポートを公開します。ポートを公開しない場合は、コンテナー・グループは実行されますが、パブリック・ネットワークかプライベート・ネットワーク上のグループのロード・バランサーを使用してコンテナー・グループにアクセスできません。

コンテナー・グループの場合、複数のポートを含めることはできません。複数のポートがリストされる場合、最後にリストされているポートのみが公開されます。他のポートは、コマンドに無視されます。

経路をバインドするとき、グループ内のコンテナーはグループの公開ポートで HTTP トラフィックを listen する必要があります。 HTTP 以外のポートはパブリックに公開できません。HTTPS トラフィックが公開されたポートに届くと、(Go)Router は HTTPS 終端を実行します。その後、(Go)Router とコンテナーの間のプライベート仮想ネットワークでは HTTP プロトコルが使用されます。IP アドレスをバインドするとき、 グループ内のコンテナーは HTTP トラフィックまたは HTTPS トラフィックを listen できますが、公開可能なポートは 1 つだけです。

ヒント:
  • IBM 認定 Liberty Server イメージ、またはこのイメージの変更版の場合、ポート 9080 を入力します。
  • IBM 認定 Node.js イメージ、またはこのイメージの変更版の場合、ポート 8000. など、任意のポートを入力します。

(オプション) デフォルト値は false です。--http_monitor_enabled を true に設定した場合は、--session_affinity オプションを true に設定して、セッション Cookie の作成と管理をアプリの代わりに IBM Bluemix Container Service で行うことができます。 --http_monitor_enabled を false に設定した場合や、HTTP ではなく TCP を通信に使用する場合、このオプションは無効です。

(オプション) VOLUME:/DIRECTORY_PATH[:ro] 形式で詳細を指定して、コンテナーにボリュームを接続します。

  • ro: オプション。デフォルトは読み取り/書き込みですが、ro を指定するとボリュームは読み取り専用になります。

パラメーター
CONTAINER
(必須) 少なくとも 1 つのコンテナーのコンテナー名または ID。 コマンドの中で、コンテナーの名前や ID をスペースで区切ってリストして指定することにより、複数のコンテナーを対象としてこのコマンドを実行することができます。
-t SECS, --time SECS
コンテナーが強制終了されるまでに待機する秒数。
例
以下の例は、my_container という名前のコンテナーを停止する要求です。
cf ic stop my_container

cf ic top CONTAINER [CONTAINER] [ps options]

目的
コンテナーで実行されているプロセスをリストします。

Linux のネイティブ top コマンド以外の cf ic top コマンドは、コンテナー内で実行されているメインプロセスに関する Linux ps コマンドの出力を表示します。結果として、メモリーと CPU の使用状況は、あるプロセスのために変化しても、CLI 出力では更新されません。

注: ネイティブ Linux top コマンドの出力を調べる必要がある場合は、コンテナー内で exec を実行してから、top コマンドを実行してください。
詳しくは、Docker ヘルプ内の top コマンドを参照してください。
パラメーター
CONTAINER
(必須) コンテナーの名前または ID。
ps のオプション
(オプション) cf ic top コマンドの出力をさらに詳しく調べたりフィルタリングしたりするには、ネイティブ Linux ps コマンドでのオプションを使用します。
例
以下の例は、my_container というコンテナー内で実行されているすべての現行プロセスの詳細情報を表示する要求です。 
cf ic top my_container -aux

cf ic wait CONTAINER [CONTAINER]

目的
コンテナーが終了するまで待機します。この待機時間中は、コマンド・ラインが戻らないため、コマンドを入力できません。 コンテナーが終了または停止すると、すぐに終了コードが CLI 出力に表示されます。 詳しくは、Docker ヘルプ内の wait コマンドを参照してください。
パラメーター
CONTAINER
(必須) コンテナーの名前または ID。
例
以下の例は、my_container というコンテナーが終了するまで待機する要求です。コンテナーが終了すると、すぐに終了コードが CLI に表示されます。

cf ic wait-status CONTAINER

目的
単一コンテナーまたはコンテナー・グループが非過渡的な状態になるまで待機します。この待機時間中は、コマンド・ラインが戻らないため、コマンドを入力できません。 コンテナーが非過渡的な状態になると、すぐに OK メッセージが表示されます。単一コンテナーの場合、非過渡的な状態には Running、Shutdown、Crashed、Paused、Suspended があります。コンテナー・グループの場合、非過渡的な状態には CREATE_COMPLETE、UPDATE_COMPLETE、FAILED があります。
パラメーター
CONTAINER
(必須) コンテナーまたはグループの名前または ID。
例
以下の例は、my_container というコンテナーが非過渡的な状態になるまで待機する要求です。 
cf ic wait-status my_container

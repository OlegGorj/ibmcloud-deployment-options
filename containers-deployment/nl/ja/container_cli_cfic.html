単一コンテナーとスケーラブル・コンテナーの管理用 IBM Bluemix Container Service プラグイン (bx ic)

Bluemix® の IBM® Bluemix Container Service プラグインを使用して、コマンド・ラインからイメージまたは単一コンテナーとスケーラブル・コンテナーを操作できます。

単一コンテナーとスケーラブル・コンテナーでネイティブ Docker CLI を使用するための IBM Bluemix Container Service プラグイン (bx ic) のセットアップ

ネイティブの Docker CLI コマンドを実行してご使用のコンテナーを管理するには、IBM Bluemix Container Service プラグイン (bx ic) をインストールします。
重要: 単一コンテナーとスケーラブル・コンテナーはありません。クラスター管理について学び、Bluemix Public の IBM Bluemix Container Service の Kubernetes にマイグレーションします。

IBM Bluemix Container Service プラグイン (bx ic) のインストール

コマンドを実行するには、ご使用のオペレーティング・システムで以下の要件が満たされている必要があります。このタスクには、これらの要件をインストールするための情報が含まれています。以下に示すバージョンのツールよりも古いものと IBM Bluemix Container Service プラグイン (bx ic) との併用はサポートされていません。

  • Docker バージョン 1.8 から 1.12
  • Cloud Foundry CLI バージョン 6.14.0 - 6.22.0
    注: MacOS Sierra ユーザーは、Cloud Foundry CLI バージョン 6.22.0 以降をインストールする必要があります。
  • Bluemix CLI
  • IBM Bluemix Container Service プラグイン

CLI をインストールするには、以下のことを行います。

  1. Docker 1.12 をインストールします。 Windows および OS X オペレーティング・システムの場合は、Docker Machine、Docker Engine、および Kitematic を含む Docker Toolbox と共に Docker がインストールされます。Linux オペレーティング・システムの場合、インストールの仕組みはご使用の Linux ディストリビューションのタイプによって異なります。Docker をインストールするには、あるいは Docker が既にインストール済みである場合 Docker 1.12 にアップグレードするには、以下のリンクを参照してください。
    注: ご使用のコンピューターに Docker をインストール済みであり、現行バージョンをアップグレードしたくない場合は、このステップをスキップし、Cloud Foundry CLI のインストールを続行することができます。
    • OS X

      https://github.com/docker/toolbox/releases/download/v1.12/DockerToolbox-1.12.pkg

    • Windows

      https://github.com/docker/toolbox/releases/download/v1.12/DockerToolbox-1.12.exe

    • Linux

      ご使用の Linux ディストリビューションに Docker バージョン 1.12 をインストールする方法については、Docker インストール資料を参照してください。

  2. Docker を開始します。
    • OS X および Windows の場合、「Docker Quickstart Terminal」アイコンをダブルクリックします。 Docker デーモンが開始され、環境変数が自動的に設定されます。
    • Linux の場合、コマンド・ライン・ユーティリティーで、Docker デーモンが実行中であることを確認します。 また、自動的に再始動するよう Docker デーモンをセットアップします。 詳しくは、ご使用のオペレーティング・システム用の Docker 資料を参照してください。
  3. 端末で docker run hello-word を実行して、Docker インストール環境が正しくセットアップされていることを確認してください。 
    docker run hello-world
    Docker は、コンテナー実行元の hello-world イメージをローカルで検索するため、このコマンドを初めて実行する場合は、以下のメッセージを受け取ります。 
    Unable to find image 'hello-world:latest' locally
    イメージのパブリック・リポジトリーである Docker Hub からイメージがプルされ、コマンド完了後、出力は以下のようになります。 
    $ docker run hello-world

    コマンドが正常に完了しない場合は、docker version を実行して、Docker が正しくインストールされているかどうか確認してください。
  4. GitHub リポジトリーから Cloud Foundry CLI バージョン Cloud Foundry CLI バージョン 6.14.0 - 6.22.0 をインストールします。 Cloud Foundry CLI は、オペレーティング・システムのデフォルト・ロケーションにインストールする必要があります。そうしないと、PATH 環境変数がインストール・ディレクトリーと一致しません。
    注: MacOS Sierra ユーザーは、Cloud Foundry CLI バージョン 6.22.0 以降をインストールする必要があります。
    Cloud Foundry CLI が正しくインストールされていることを確認するため、または既に Cloud Foundry がインストールされている場合にインストールしたバージョンを確認するためには、以下のコマンドを実行します。
    cf -v
  5. Bluemix CLI をインストールします。
  6. Bluemix CLI にログインします。 
    bx login -a https://api.ng.bluemix.net
  7. IBM Bluemix Container Service プラグインをインストールします。 
    bx plugin install IBM-Containers -r Bluemix
    プラグインが正常にインストールされていることを検証するには、以下のコマンドを実行します。 
    bx plugin list

    プラグインが IBM-Containers として結果に表示されます。

これで、ログインしてコンテナーの管理を開始する準備ができました。

IBM Bluemix Container Service CLI プラグイン (bx ic) へのログイン

CLI をインストールすると、ログインして CLI を使用できます。

始める前に、以下のタスクを実行してください。
  • IBM Bluemix Container Service (bx ic) プラグインをインストールします。
  • Cloud Foundry にログインするときに使用する Bluemix 組織とスペースを決めます。 組織とスペースについて詳しくは、組織とスペースを参照してください。
  • Cloud Foundry の login コマンドを使用して指定できるオプションについて詳しくは、Cloud Foundry 資料の login コマンドを参照してください。
  1. コマンド・ライン・ユーティリティーを開きます。
    • OS X および Windows の場合、「Docker Quickstart Terminal」アイコンをダブルクリックします。 Docker デーモンが開始され、環境変数が自動的に設定されます。
    • Linux の場合、任意のコマンド・ライン・ユーティリティーを開きます。
  2. Cloud Foundry CLI を通して Bluemix にログインします。 特定の地域を指定するには、-a オプションを、その地域の API エンドポイントと共に含めます。初めて CLI を使用するときに、このオプションを含めない場合は、API エンドポイントの入力を求めるプロンプトが出されます。API エンドポイントが指定された後、それ以降のログイン時に API エンドポイントを入力する必要はありません。 
    bx login [-a api.DomainName] [-sso]
    特定の地域にログインするには、Cloud Foundry CLI にログインするときに、その地域の API エンドポイントを含めます。
    • 米国南部
      bx login -a api.ng.bluemix.net
    • 英国
      bx login -a api.eu-gb.bluemix.net
    フェデレーテッド ID を使用してログインする場合は、シングル・サインオン・パラメーター --sso が必要です。このオプションを使用する場合は、https://login.ng.bluemix.net/UAALoginServerWAR/passcode を開いて、ワンタイム・パスコードを取得します。 --sso オプションを含めない場合は、以下のサブステップを実行します。
    1. 「E メール」に、Bluemix の IBM ID を入力します。
    2. 「パスワード」に、IBM ID のパスワードを入力します。 Bluemix の組織とスペースが取得されました。
    3. Bluemix 組織の 1 つを表す数値を入力します。
    4. 既存の Bluemix スペースの 1 つを表す数値を入力します。
  3. 組織の名前空間 を設定します。 名前空間とは、プライベートの Bluemix イメージ・レジストリーを識別するための固有の名前です。 コンテナーの作成時には、名前空間をイメージ名とともに含めることによってイメージの場所を指定する必要があります。
    注: 名前空間を作成する場合、作成した後で変更することはできません。削除のみ可能です。 Bluemix 組織内のユーザーが既にコンテナーをデプロイし、namespace を設定している場合は、その組織の名前空間を示すメッセージが表示されます。 その場合は、次のステップから開始できます。
    bx ic namespace-set <my_namespace>
    ヒント: 名前空間に関しては、以下の規則に従ってください。
    • 使用できるのは、小文字、数字、下線 (_) のみです。
    • 4 文字から 30 文字までの長さにすることができます。コマンド・ラインからコンテナーを管理する予定の場合は、素早く入力できる短い名前空間を使用することをお勧めします。
    • この名前は、Bluemix において固有でなければなりません。
  4. IBM Bluemix Container Service を初期設定します。 
    bx ic init
  5. bx ic またはネイティブ Docker コマンドのどちらを実行するかを決めます。 プラグインにログインするたびに以下のオプションのいずれかを選択するか、または、shellinit コマンドを使用して常にオプション 2 が選択されるようにできます。
    • オプション 1 (デフォルト)

      IBM Bluemix Container Service ラッパー Docker の単純なシェル。 ローカルの Docker デーモンは、IBM Bluemix Container Service の影響を受けません。 bx ic コマンドは Bluemix スペース内のコンテナーを管理する場合に使用し、docker コマンドはローカルでコンテナーを管理する場合に使用します。 いずれにしても docker コマンドは、ローカルのイメージ開発には常に必要です。

      例えば、オプション 1 を使用する場合、bx ic run コマンドは Bluemix スペース内にコンテナーを作成し、docker run はローカルにコンテナーを作成します。 このオプションは、Docker と IBM Bluemix Container Service のどちらの使用にも慣れていない場合や、コンテナーのテストをローカルで行う場合に便利です。

    • オプション 2

      現在の CLI セッションのローカルの Docker デーモンを IBM Bluemix Container Service にリダイレクトして、Bluemix スペース内でのコンテナーの管理に docker コマンドが使用可能なときは、常に bx ic コマンドの代わりに使用できるようにします。

      例えば、オプション 2 では、bx ic run コマンドと docker run コマンドのどちらを使用してもスペース内にコンテナーを作成できます。 このオプションは、Docker の使用経験があり、docker コマンドの方が慣れている場合や、IBM Bluemix Container Service の処理に自動化スクリプトを適用している場合に便利です。

      例外として、IBM Bluemix Container Service に固有なタスクを行うのに、引き続き bx ic コマンドを使用する必要があります。 例えば、名前空間を設定するには、コマンド bx ic namespace-set を使用する必要があります。 docker namespace-set は、有効なコマンドではありません。 この構成でサポートされる docker コマンドのリストについては、IBM Bluemix Container Service プラグイン (bx ic) でサポートされている Docker コマンドを参照してください。

      このオプションを構成するには、ログイン時に端末に表示される環境変数をコピーして貼り付けます。
      • DOCKER_HOST は、Docker ホストを IBM Bluemix Container Service に設定します。
      • DOCKER_CERT_PATH は、Cloud Foundry 用 IBM Bluemix Container Service プラグインの認証鍵の場所を設定します。
      • DOCKER_TLS_VERIFY は、TLS を使用し、リモート・サーバーを検証します。
      ヒント: IBM Bluemix Container Service 資料に記載されている作業を実行している場合、 アスタリスク (*) のマークがあるすべてのステップで docker コマンドを実行できます。

bx ic attach [--no-stdin] [--sig-proxy] CONTAINER

目的
実行中のコンテナーを制御するか、その出力を表示します。コンテナーを終了して停止するには、CTRL+C を使用します。詳しくは、Docker ヘルプ内の attach コマンドを参照してください。
パラメーター
--no-stdin
(オプション) 標準入力を含めません。
--sig-proxy
(オプション) デフォルトは true です。
CONTAINER
(必須) コンテナーの名前または ID。
例
以下の例は、コンテナー my_container に接続する要求です。
bx ic attach my_container

bx ic build -t TAG [--no-cache] [--pull] [-q] DOCKERFILE_LOCATION

目的
Docker イメージを、ローカルに、またはプライベート Bluemix イメージ・レジストリーにビルドする。 詳しくは、Docker ヘルプ内の build コマンドを参照してください。
パラメーター
DOCKERFILE_LOCATION
(必須) ローカル・ホスト上の Dockerfile およびコンテキストへのパス。
--no-cache
(オプション) イメージが初めてビルドされる際には、組織全体のイメージのレイヤーがキャッシュに入れられます。 イメージのそれ以降のバージョンを作成するときには、ビルド・プロセスの速度向上のため、キャッシュ中にあるそのイメージのバージョンが使用されます。 イメージの新しいバージョンをビルドする際に --no-cache オプションを含めることにより、イメージの各レイヤーも確実に再ビルドされるようにすることができます。
--pull
(オプション) デフォルトでは、ベース・イメージは、まだ組織のイメージ・キャッシュ中にない場合にのみ、レジストリーからプルされます。このオプションを含めると、ベース・イメージの最新バージョンが新しいイメージのビルドに使用されるようにすることができます。
-q, --quiet
(オプション) コンテナーによって生成される詳細出力を抑止します。デフォルトは false です。
-t TAG, --tag TAG
(必須) 作成されるイメージに適用するプライベート・イメージ・レジストリーへのパス。
例
以下の例は、myimage という名前のイメージをビルドする要求です。このビルドで使用する Dockerfile および他の成果物は、コマンドが実行される同じディレクトリーにあります。 レジストリーと名前空間がイメージ名と共に組み込まれているので、イメージは組織のプライベート Bluemix イメージ・レジストリー内でビルドされます。
ヒント: bx ic namespace-get を実行して名前空間を取得し、<my_namespace> をその名前空間情報に置き換えます。
bx ic build -t registry.DomainName/<my_namespace>/myimage .

bx ic cp SRC_PATH CONTAINER:DEST_PATH | bx ic cp CONTAINER:SRC_PATH DEST_PATH

目的
ファイルまたはフォルダーを、実行中または停止中の単一コンテナーからローカル・マシンにコピーしたり、その逆を行ったりします。詳しくは、Docker ヘルプ内の cp コマンドを参照してください。
注: このコマンドは、単一コンテナーにしか使用できません。 コンテナー・グループとの間でファイルをコピーする場合は、グループに含まれている各インスタンスとの間でファイルをコピーする必要があります。
パラメーター
DEST_PATH
(必須) ソース・パスから情報をコピーする宛先である、ローカル・マシンまたはコンテナー上のファイルまたはフォルダーへのパス。 ファイルまたはフォルダーのコピーを開始する前に、それらがローカル・マシンまたはコンテナーに存在する必要があります。 ローカル・マシンにコピーする場合、絶対パスまたは相対パスのどちらでも使用できます。 単一コンテナーにコピーする場合、コンテナーの名前、区切り文字「:」、そしてコピー先のファイルまたはフォルダーへのパスを含める必要があります。パスは、コンテナーのルート・ディレクトリーからの相対パスにする必要があります。
SRC_PATH
(必須) コピーするファイルまたはフォルダーへのパス。 ファイルまたはフォルダーがローカル・マシン上にある場合、絶対パスまたは相対パスのどちらでも使用できます。 ファイルまたはフォルダーが単一コンテナーにある場合、コンテナーの名前、区切り文字「:」、そしてコピーするファイルまたはフォルダーへのパスを含める必要があります。パスは、コンテナーのルート・ディレクトリーからの相対パスにする必要があります。
注: コンテナー内の /dev、/etc、/proc および /sys フォルダーの下に格納されている一部のシステム・ファイルはコピーできません。
例
このサンプル・コマンドは、config.txt という名前のファイルをローカル・マシンの現行ディレクトリーから my_container という名前の単一コンテナーの config ディレクトリーにコピーします。
bx ic cp ./config.txt my_container:/config

bx ic cpi SRC DST

目的
Docker Hub イメージ、またはローカル・レジストリーからのイメージにアクセスし、それをプライベート Bluemix イメージ・レジストリーにコピーします。
注: このコマンドは、docker pull と docker push を一緒に使用することでイメージを Bluemix イメージ・レジストリーに追加する代わりに使用します。docker pull と docker push を使用する方法を選択する場合は、コマンド・ラインからのプライベート Bluemix レジストリーへのローカル・イメージのプッシュを参照してください。

bx ic group-create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session-affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

目的
スケーリング・グループを作成します。 デフォルトでは、Bluemix のコンテナーはデタッチ・モードで実行されます。
パラメーター
--anti
(オプション) アンチアフィニティーを使用して、コンテナー・グループの可用性を高めることができます。 --anti オプションは、グループ内のすべてのコンテナー・インスタンスを別々の物理計算ノード上に強制的に配置します。これにより、1 つのハードウェア障害によってグループ内のすべてのコンテナーが異常終了してしまう可能性を減らします。 大きいグループ・サイズの場合は、このオプションを使用できない可能性があります。 Bluemix のどの地域およびどの組織にも、デプロイできる計算ノードのセットには限りがあるからです。デプロイメントに失敗した場合は、グループ内のコンテナー・インスタンスの数を減らすか、--anti オプションを外してください。
--auto
(オプション) コンテナー・グループで自動リカバリーが有効になり、そのグループが 10 分間実行すると、そのグループのロード・バランサーは HTTP 要求を介して、グループ内の各コンテナー・インスタンスの正常性に関する定期的な検査を開始します。 コンテナー・インスタンスが 100 秒以内に応答しない場合、それは非アクティブとしてマークされます。 非アクティブなコンテナー・インスタンスはグループから削除されて、自動リカバリーにより再作成されます。 自動リカバリーでは、グループ内にコンテナー・インスタンスをリカバリーする試行を 3 回行います。 3 回目の試行の後、そのグループでは、自動リカバリーによるコンテナー・インスタンスのリカバリーが 60 分間行われなくなります。 60 分後に、自動リカバリーのプロセスが再開します。ヘルス・チェックについて詳しくは、--http_monitor_enabled オプションを参照してください。
CMD
(オプション) コマンドと引数が実行のためにコンテナー・グループに渡されます。このコマンドは長期実行コマンドでなければなりません。実行時間が短いコマンド (例えば /bin/date) は、コンテナーが異常終了する原因になることがあるため、使用しないでください。
IBM Bluemix Container Service によって提供される ibmnode イメージを使用した長期実行コマンドの例:
bx ic group-create --name my_container_group registry.DomainName/ibmnode ping localhost
bx ic group-create --name my_container_group registry.DomainName/ibmnode -- tail -f /dev/null
注: CMD とその引数は bx ic group-create コマンド・ラインの最後に指定する必要があります。
-d DOMAIN, --domain DOMAIN
(オプション) デフォルトのシステム・ドメインは AppDomainName であり、それにより SSL 証明書が既に提供されているため、付加的な構成なしで HTTPS によりコンテナー・グループにアクセスすることができます。 カスタム・ドメインを使用するには、パブリック DNS サーバー上でカスタム・ドメインを登録し、Bluemix でカスタム・ドメインを構成してから、カスタム・ドメインをパブリック DNS サーバー上の Bluemix システム・ドメインにマップする必要があります。 カスタム・ドメインが Bluemix システム・ドメインにマップされた後、カスタム・ドメインを求める要求は、Bluemix 内のアプリケーションにルーティングされます。 カスタム・ドメインを作成する際、ドメイン名に下線 (_) を含めないでください。 カスタム・ドメインについて詳しくは、カスタム・ドメインの作成と使用を参照してください。 カスタム・ドメインのセキュリティーを確保するには、SSL 証明書をアップロードして、HTTPS でコンテナー・グループにアクセスできるようにします。

ホストとドメインの組み合わせが、完全なパブリック経路 URL (例えば http://mycontainerhost.AppDomainName) を形成します。 これは、Bluemix 内で固有でなければなりません。 bx ic group-inspect コマンドを使用してコンテナー・グループの詳細を検討すると、ホストとドメインが経路として一緒にリストされます。

--desired DESIRED
(オプション) グループ内のインスタンスの数。 デフォルトは 2です。アプリに必要なコンテナー・グループのインスタンス数は、予想されるワークロードや、アプリが要求を処理できる時間によって異なります。 例えば、マルチタスキングやネットワーク接続によって、アプリに必要なインスタンス数は左右されます。 必要なインスタンス数を決定するには、アプリを実行するコンテナーを 1 つ Bluemix にデプロイし、そのコンテナーで負荷テストを実行します。 例えば、アプリで処理できる要求が 1 秒あたり 100 個であるのに対し、予想される平均ワークロードが 1 秒あたり 300 個だとすると、コンテナー・グループに必要なインスタンス数は 3 になります。 コンテナー・グループの可用性と耐障害性を高めるために、予想されるワークロードを処理する最低限の数のインスタンスではなく余分にインスタンスを組み込むことを検討してください。 インスタンスを余分に組み込んでおくことで、さらに別のインスタンスが異常終了した場合でもワークロードを処理できます。障害が起きたインスタンスをワークロードに影響を与えずに自動復旧するには、1 つ余分にインスタンスを組み込みます。2 つが同時に障害を発生した場合に対応できるようにするには、2 つ余分にインスタンスを組み込みます。このセットアップは N+2 パターンです。 N は要求を処理するインスタンスの数、+2 は追加の 2 つのインスタンスです。
重要: 1 つのコンテナー・グループには、少なくとも 2 つのコンテナー・インスタンスを含める必要があります。 グループにインスタンスを 1 つしか含めていないと、そのインスタンスが使用不可になった場合に、ユーザーがアプリのダウン時間を経験する可能性があります。

(オプション) デフォルト値は / です。--http_monitor_enabled を有効にした場合は、このオプションを使用して、ヘルス・チェックのカスタム・パスを設定できます (例: --http_monitor_path="/health/path")。
--http_monitor_rc_list
(オプション) --http_monitor_enabled を有効にした場合は、ヘルス・チェックの応答コードのリストを指定できます。デフォルト値に含まれているコードは、200、201、202、204、300、301、302、401、403、404 です。有効な応答のリストに含まれていない応答が返ると、ロード・バランサーがコンテナー・インスタンスの状態を INACTIVE に変更します。 非アクティブなコンテナーはロードを受け取らないので、コンテナー・グループの状態が HAS_INACTIVE_INSTANCES に変わります。いずれのコンテナーも ACTIVE ではない場合、コンテナー・グループの状態は NO_ACTIVE_INSTANCES に変わります。状態について詳しくは、IBM Bluemix Container Service のコンテナーの状態を参照してください。

(オプション) 使用可能な IP アドレスがある場合は、パブリック IP アドレスをコンテナー・グループにバインドします。 -p フラグを指定して、ポートを公開する必要もあります。コンテナー・グループは、パブリック経路またはパブリック IP アドレスのどちらかを取ることができますが、両方を取ることはできません。IP アドレスをアンバインドしてその IP をスペースのクォータに戻すには、コンテナー・グループを一度削除して、IP アドレスを指定しないでそのグループを再作成します。コンテナー・グループを作成する前に IP アドレスを要求するには、bx ic ip-request を実行します。
注: パブリック IP アドレスをバインドしても、コンテナー・グループとの間で送受信するデータを暗号化する SSL 証明書は組み込まれません。ご使用のアプリに SSL 暗号化が必要である場合は、独自の SSL ソリューションを実装するか、またはパブリック IP アドレスをバインドする代わりにパブリック経路をコンテナー・グループにマップすることができます。パブリック経路には SSL 証明書が既に組み込まれているので、追加の構成を行わずに HTTPS でコンテナー・グループにアクセスできます。

(オプション) コンテナーのメモリー限度を MB 単位で入力します。 メモリー限度は、コンテナーが実行時に計算ホスト上で取得するメモリーとディスク・スペースの最大量を定義するコンテナー・サイズの一部です。 コンテナー・サイズが割り当てられた後、その値を変更することはできません。IBM Bluemix Container Service で設定可能なサイズは、以下のとおりです。

  • ナノ (128 MB メモリー、8 GB ディスク・スペース)
  • マイクロ (256 MB メモリー、16 GB ディスク・スペース)
  • タイニー (512 MB メモリー、32 GB ディスク・スペース)
  • スモール (1024 MB メモリー、64 GB ディスク・スペース)
  • ミディアム (2048 MB メモリー、128 GB ディスク・スペース)
  • ラージ (4096 MB メモリー、256 GB ディスク・スペース)

-p PORT, --publish PORT

(必須) HTTP トラフィックとプライベート・ネットワーク・トラフィック用のポートを公開します。ポートを公開しない場合は、コンテナー・グループは実行されますが、パブリック・ネットワークかプライベート・ネットワーク上のグループのロード・バランサーを使用してコンテナー・グループにアクセスできません。

コンテナー・グループの場合、複数のポートを含めることはできません。複数のポートがリストされる場合、最後にリストされているポートのみが公開されます。他のポートは、コマンドに無視されます。

経路をバインドするとき、グループ内のコンテナーはグループの公開ポートで HTTP トラフィックを listen する必要があります。 HTTP 以外のポートはパブリックに公開できません。HTTPS トラフィックが公開されたポートに届くと、(Go)Router は HTTPS 終端を実行します。その後、(Go)Router とコンテナーの間のプライベート仮想ネットワークでは HTTP プロトコルが使用されます。 IP アドレスをバインドするとき、 グループ内のコンテナーは HTTP トラフィックまたは HTTPS トラフィックを listen できますが、公開可能なポートは 1 つだけです。
使用するイメージの Dockerfile 内にポートが指定されている場合、そのポートを含めてください。
ヒント:
  • IBM 認定 Liberty Server イメージ、またはこのイメージの変更版の場合、ポート 9080 を入力します。
  • IBM 認定 Node.js イメージ、またはこのイメージの変更版の場合、ポート 8000. など、任意のポートを入力します。
--session_affinity

(オプション) デフォルト値は false です。--http_monitor_enabled を true に設定した場合は、--session_affinity オプションを true に設定して、セッション Cookie の作成と管理をアプリの代わりに IBM Bluemix Container Service で行うことができます。 --http_monitor_enabled を false に設定した場合や、HTTP ではなく TCP を通信に使用する場合、このオプションは無効です。

(オプション) VOLUME:/DIRECTORY_PATH[:ro] 形式で詳細を指定して、コンテナーにボリュームを接続します。
  • VOLUME: ボリューム ID または名前。
  • DIRECTORY_PATH: コンテナー内でのディレクトリーの絶対パス。
  • ro: オプション。デフォルトは読み取り/書き込みですが、ro を指定するとボリュームは読み取り専用になります。

注: Cloud Foundry コマンド・ツールがインストールされていて、Cloud Foundry トークンがあることを確認する必要があります。bx login および bx ic init を使用したログインが成功すると、必要なトークンが生成されます。
パラメーター
CMD
(オプション) コマンドと引数が実行のためにコンテナーに渡されます。このコマンドは長期実行コマンドでなければなりません。実行時間が短いコマンド (例えば /bin/date) は、 コンテナーが異常終了する原因になることがあるため、使用しないでください。
長期実行コマンドの例を以下に示します。 これらの例では、 IBM Bluemix Container Service レジストリー内に用意されている ibmnode イメージを使用することを想定しています。
bx ic run --name my_container registry.DomainName/ibmnode ping localhost
bx ic run --name my_container registry.DomainName/ibmnode -- tail -f /dev/null
bx ic run --name my_container registry.DomainName/ibmnode -- sh -c "while true; do date; sleep 20; done"
注: CMD とその引数は bx ic group-create コマンド・ラインの最後に指定する必要があります。
-d
(オプション) コンテナーをデタッチ・モードで実行します。bx ic run を使用する場合は、コンテナーはデフォルトでデタッチ・モードで実行されます。 デタッチ・モードでは、コンテナーのルート・プロセスが終了すると、コンテナーも終了します。 例えば、ibmnode イメージからコンテナーをデプロイした場合、長時間実行されるプロセスが存在しないため、コンテナーはビルド直後にシャットダウンします。
ヒント: bx ic run ではなく docker run を使用する場合、-d オプションは必須です。
-e ENV, --env ENV
(オプション) 任意の数の環境変数を設定します。ENV は key=value ペアです。複数のキーを別々にリストします。 引用符を含める場合は、環境変数名と値の両方を囲みます。

例

-e "key1=value1" -e "key2=value2" -e "key3=value3"

IMAGE
(必須) コンテナーに含めるイメージ。イメージの後にコマンドをリストできますが、イメージの後にはオプションを置かないでください。イメージ指定の前にすべてのオプションを含めるようにしてください。

-it
(オプション) コンテナーを対話モードで実行します。コンテナーの作成後に、標準入力の表示を維持します。終了するには exit と入力します。
--link NAME:ALIAS
(オプション) あるコンテナーが実行中の別のコンテナーと通信するようにしたい場合は、ホスト名の別名を使用してそのコンテナーを指定できます。詳細については、コマンド・ライン・インターフェース (CLI) による単一コンテナーのリンクを参照してください。
-m MEMORY, --memory MEMORY
(オプション) コンテナーのメモリー限度を MB 単位で入力します。 メモリー限度は、コンテナーが実行時に計算ホスト上で取得するメモリーとディスク・スペースの最大量を定義するコンテナー・サイズの一部です。 コンテナー・サイズが割り当てられた後、その値を変更することはできません。IBM Bluemix Container Service で設定可能なサイズは、以下のとおりです。
  • ピコ (64 MB メモリー、4 GB ディスク・スペース)
  • ナノ (128 MB メモリー、8 GB ディスク・スペース)
  • マイクロ (256 MB メモリー、16 GB ディスク・スペース)
  • タイニー (512 MB メモリー、32 GB ディスク・スペース)
  • スモール (1024 MB メモリー、64 GB ディスク・スペース)
  • ミディアム (2048 MB メモリー、128 GB ディスク・スペース)
  • ラージ (4096 MB メモリー、256 GB ディスク・スペース)
  • X ラージ (8192 MB メモリー、512 GB ディスク・スペース)
  • 2X ラージ (16384 MB メモリー、1 TB ディスク・スペース)
コンテナーのサイズを設定しない場合、各コンテナー・インスタンスは 256 MB で作成されます。
重要: コンテナー・メモリーは、単位ラベルを付けずに MB で入力してください。 例えば、ピコ・コンテナーを作成する場合は、-m 64 と入力します。
--name NAME
(必須) コンテナーに名前を付けます。
ヒント: コンテナー名は文字で始める必要があり、その後には、大文字、小文字、数字、ピリオド (.)、下線 (_)、およびハイフン (-) を使用できます。
-p PORT, --publish PORT

(オプション) アプリをインターネットからアクセス可能にするには、パブリック・ポートを公開する必要があります。 パブリック・ポートを公開するときには、公開したポートでのみパブリック・データを送受信できるパブリック・ネットワーク・セキュリティー・グループをコンテナー用に作成します。その他のすべてのパブリック・ポートは閉じられるため、それらのポートを使用してインターネットからアプリにアクセスすることはできません。複数の -p オプションを使用して、複数のポートを含めることができます。ポートはマップすることも転送することもできません。ポートを公開しない場合は、プライベート・コンテナー・ネットワークのみからアクセス可能になります。 割り当てられたプライベート IP アドレスを使用して、プライベート・ネットワーク上のコンテナーと通信できます。

使用するイメージの Dockerfile 内にポートが指定されている場合、そのポートを含めてください。
ヒント:
  • IBM 認定 Liberty Server イメージ、またはこのイメージの変更版の場合、ポート 9080 を入力します。
  • IBM 認定 Node.js イメージ、またはこのイメージの変更版の場合、ポート 8000. など、任意のポートを入力します。

使用する IP プロトコルとして UDP か TCP のいずれかを選択できます。 プロトコルを指定しない場合、自動的にポートは TCP トラフィックに対して公開されます。

例
  • -p 9080
  • -p 9080/udp
  • -p 9080/udp -o 8000/tcp

コンテナーにバインドしたい IP アドレスがスペース内に既に存在する場合、その IP アドレスをポート情報と共に指定できます。IP アドレス、ポート、または IP アドレスおよびコンテナー・ポート番号 (例えば <ip-address>:<port_on_IP>:<container-port>) を含めてください。 IP アドレスのポートは、Dockerfile で公開されたコンテナー・ポートと一致する必要があります。

例
-p 192.0.2.23:9080:9080
注: パブリック IP アドレスをバインドしても、単一のコンテナーとの間で送受信するデータを暗号化する SSL 証明書は組み込まれません。ご使用のアプリに SSL 暗号化が必要である場合は、独自の SSL ソリューションを実装するか、または単一のコンテナーの代わりにコンテナー・グループを使用することができます。 既に SSL 証明書が組み込まれているパブリック経路にコンテナー・グループがバインドされているなら、追加の構成を行わずに HTTPS でコンテナー・グループにアクセスできます。詳しくは、『Bluemix GUI によるコンテナー・グループの作成』を参照してください。
-P
(オプション) イメージの Dockerfile 内に指定されたポートを HTTP トラフィック用に自動的に公開します。
--volume VOLUME:/DIRECTORY_PATH[:ro]
(オプション) VOLUME:/DIRECTORY_PATH[:ro] 形式で詳細を指定して、コンテナーにボリュームを接続します。
  • VOLUME ボリューム ID または名前。
  • DIRECTORY_PATH: コンテナー内でのディレクトリーの絶対パス。
  • ro オプション。デフォルトは読み取り/書き込みですが、ro を指定するとボリュームは読み取り専用になります。
例
以下の例は、my_namespace/nginx イメージを使用して、名前が my_namespace でメモリー限度が 1024 MB の新規コンテナーを作成および開始する有効な要求です。ここで、my_namespace はログインしたユーザーに関連付けられた名前空間です。
bx ic run -m 1024 --name my_container registry.DomainName/my_namespace/nginx
以下の例は、 my_namespace/blog イメージを使用し、環境変数としていくつかの資格情報を渡すことによって、新規コンテナーを作成および開始する有効な要求です。 ここで、my_namespace はログインしたユーザーに関連付けられた名前空間です。
bx ic run -e USER=johnsmith -e PASS=password --name my_container registry.DomainName/my_namespace/blog
以下の例は、 my_namespace/blog イメージを使用してコンテナーにボリュームを追加する有効な要求です。 ここで、my_namespace はログインしたユーザーに関連付けられた名前空間です。

bx ic wait-status CONTAINER

目的
単一コンテナーまたはコンテナー・グループが非過渡的な状態になるまで待機します。この待機時間中は、コマンド・ラインが戻らないため、コマンドを入力できません。 コンテナーが非過渡的な状態になると、すぐに OK メッセージが表示されます。単一コンテナーの場合、非過渡的な状態には Running、Shutdown、Crashed、Paused、Suspended があります。コンテナー・グループの場合、非過渡的な状態には CREATE_COMPLETE、UPDATE_COMPLETE、FAILED があります。
パラメーター
CONTAINER
(必須) コンテナーまたはグループの名前または ID。
例

bx ic wait-status my_container

IBM Bluemix Container Service プラグイン (bx ic) のアンインストール

IBM Bluemix Container Service プラグインが不要になったら、bx コマンドでアンインストールできます。

注: IBM Bluemix Container Service Cloud Foundry プラグインをアンインストールするには、以下のコマンドを実行します。 
cf uninstall-plugin IBM-Containers
  1. Bluemix CLI の IBM Bluemix Container Service プラグインをアンインストールするには、以下のコマンドを実行します。 
    bx plugin uninstall IBM-Containers
  2. 以下のコマンドを実行し、インストールされているプラグインのリストを確認することで、プラグインがアンインストールされたことを検証します。 
    bx plugin list

    IBM Bluemix Container Service プラグインは結果に表示されません。

Vous pouvez continuer à utiliser l'interface de ligne de commande cf ic jusqu'au 1er janvier 2018.
Pour commencer à utiliser à la place le plug-in IBM Bluemix Container Service pour Bluemix, reportez-vous aux rubriques suivantes.

  • Référence de plug-in bx ic

bx ic attach [--no-stdin] [--sig-proxy] CONTENEUR

Objet
Contrôle d'un conteneur en cours d'exécution ou affichage de sa sortie. Utilisez CTRL+C pour quitter et arrêter le conteneur. Pour plus d'informations, reportez-vous à la commande attach dans l'aide de Docker.
Paramètres
--no-stdin
(Facultatif) Indique de ne pas inclure l'entrée standard.
--sig-proxy
(Facultatif) Valeur par défaut : true.
CONTENEUR
(Requis) Nom ou ID de conteneur.
Exemple
L'exemple suivant est une requête d'association au conteneur mon_conteneur.
bx ic attach mon_conteneur

bx ic build -t TAG [--no-cache] [--pull] [-q] emplacement_Dockerfile

Objet
Créez une image Docker localement ou dans le registre d'images Bluemix privé. Pour plus d'informations, reportez-vous à la commande build dans l'aide de Docker.
Paramètres
EMPLACEMENT_DOCKERFILE
(Requis) Chemin d'accès au fichier Dockerfile et au contexte sur l'hôte local.
--no-cache
(Facultatif) Lorsqu'une image est construite pour la première fois, met en cache les couches de l'image pour l'ensemble de l'organisation. Lorsque vous créez des versions ultérieures de l'image, ces versions en mémoire cache de l'image sont utilisées afin d'accélérer le processus de génération. En incluant l'option --no-cache lorsque vous construisez une nouvelle version de l'image, chaque couche de l'image est elle-même reconstruite.
--pull
(Facultatif) Par défaut, l'image de base n'est extraite du registre que si elle n'est pas encore présente dans le cache d'images de votre organisation. En incluant cette option, la version la plus récente de l'image de base est utilisée pour construire la nouvelle image.
-q, --quiet
(Facultatif) Supprimer la sortie prolixe qui est générée par les conteneurs. La valeur par défaut est false.
-t ETIQUETTE, --tag ETIQUETTE
(Requis) Chemin de votre registre d'images privé à appliquer à l'image créée.
Exemple
L'exemple suivant illustre une demande de génération d'une image nommée mon_image. Le fichier Dockerfile et d'autres artefacts à utiliser dans la génération sont situés dans le même répertoire que celui depuis lequel la commande est exécutée. Etant donné que le registre et l'espace de nom sont inclus avec le nom de l'image, l'image est générée dans le registre d'images Bluemix privé de votre organisation.
Conseil : Exécutez bx ic namespace-get pour extraire votre espace de nom et remplacez <my_namespace> par votre espace de nom. 
bx ic build -t
registry.DomainName/<mon_espace_nom>/mon_image.

bx ic cp CHEMIN_SRC CONTENEUR:CHEMIN_DESTINATION | bx ic cp CONTENEUR:CHEMIN_SRC CHEMIN_DESTINATION

Objet
Permet de copier un fichier ou un dossier depuis un conteneur unique en exécution, ou arrêté, vers votre machine locale, et vice versa. Pour plus d'informations, reportez-vous à la commande cp dans l'aide de Docker.
Remarque : Cette commande n'est prise en charge que pour les conteneurs uniques. Si vous désirez copier des fichiers depuis ou vers un groupe de conteneurs, vous devez les copier depuis ou vers chaque instance de conteneur du groupe.
Paramètres
CHEMIN_DESTINATION
(Requis) Chemin du fichier ou du dossier sur votre machine locale ou dans votre conteneur où copier les informations du chemin source. Le fichier ou le dossier doit exister sur votre machine locale avant de lancer la copie. Si vous désirez les copier sur votre machine locale, vous pouvez utiliser son chemin absolu ou relatif. Si vous désirez les copier dans votre conteneur unique, vous devez inclure le nom du conteneur, le délimiteur ":" et le chemin du fichier ou du dossier de destination. Ce chemin doit être relatif au répertoire racine du conteneur.
CHEMIN_SRC
(Requis) Chemin du fichier ou du dossier que vous désirez copier. Si le fichier ou le dossier réside sur votre machine locale, vous pouvez utiliser son chemin absolu ou relatif. Si le fichier ou le dossier est situé dans votre conteneur unique, vous devez inclure le nom du conteneur, le délimiteur ":" et le chemin du fichier ou du dossier que vous désirez copier. Le chemin doit être relatif au répertoire racine du conteneur.
Remarque : Vous ne pouvez pas copier certains fichiers système enregistrés sous les dossiers /dev, /etc, /proc et /sys de votre conteneur.
Exemple
Cet exemple de commande copie un fichier nommé config.txt depuis le répertoire actuel de votre machine locale vers le répertoire config d'un conteneur unique nommé mon_conteneur.
bx ic cp ./config.txt mon_conteneur:/config

Permet d'accéder à une image Docker Hub ou à une image de votre registre local et de la copier dans votre registre d'images Bluemix privé .
Remarque : Cette commande est une alternative à l'utilisation conjointe de docker pull et de docker push pour ajouter une image à votre registre Bluemix. Si vous préférer utiliser les commandes docker pull et docker push, voir Transfert d'images locales à votre registre Bluemix privé à l'aide de la ligne de commande.
Paramètres
DST
(Requis) URL du registre d'images Bluemix privé, qui inclut l'espace de nom et le nom de l'image de destination. L'étiquette pour l'image est facultative.
SRC
(Requis) Référentiel source et nom de l'image.
Exemple
bx ic cpi référentiel_source/nom_image_source
URL_registre_privé/nom_image_destination:indicateur

bx ic exec [-d] [-it ] [-u UTILISATEUR] CONTENEUR CMD

Objet
Exécute une commande dans un conteneur. Pour plus d'informations, reportez-vous à la commande exec dans l'aide de Docker.
Paramètres
CMD
(Requis) Commande à exécuter dans le ou les conteneurs spécifiés.
CONTENEUR
(Requis) Nom ou ID de conteneur.
-d , --detach
(Facultatif) Exécute la commande spécifiée en arrière-plan.
-it
(Facultatif) Mode interactif. L'entrée standard continue à être affichée. Entrez exit pour quitter.
-u USER, --user USER
(Facultatif) Nom d'un utilisateur.
Exemples
Cet exemple de commande exécute un terminal bash interactif dans le conteneur mon_conteneur.
bx ic exec -it mon_conteneur bash
Cet exemple de commande exécute la commande date dans le conteneur mon_conteneur. 
bx ic exec mon_conteneur date

bx ic group-create [--anti] [--auto] [-d DOMAINE] [--desired SOUHAITE] [-e ENV] [--env-file FICHIER_ENVIRONNEMENT] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip ADRESSE_IP] [-m MEMOIRE] [--max MAX] [--min MIN] [-n HOTE --name NOM [-p PORT] [--session-affinity] [--volume VOLUME:/CHEMIN_REPERTOIRE] IMAGE [CMD]

Objet
Permet de créer un groupe de mise à l'échelle. Par défaut, les conteneurs dans Bluemix s'exécutent en mode détaché.
Paramètres
--anti
(Facultatif) Utilisez l'option anti-affinité pour accroître la haute disponibilité de votre groupe de conteneurs. L'option --anti force le placement de chaque instance de conteneur de votre groupe dans un noeud de traitement physique distinct, ce qui réduit le risque d'une défaillance de tous les conteneurs d'un groupe suite à une panne matérielle. Il se peut que vous ne puissiez pas utiliser cette option avec des tailles de groupe plus volumineuses vu que chaque région et organisation Bluemix dispose d'un nombre de noeuds de traitement limité disponibles pour le déploiement. Si votre déploiement échoue, vous pouvez soit réduire le nombre d'instances de conteneur dans le groupe, soit supprimer l'option --anti.
--auto
(Facultatif) Lorsque la reprise automatique est activée pour un groupe de conteneurs et que ce groupe est en opération depuis 10 minutes, l'équilibreur de charge du groupe commence à vérifier régulièrement la santé de chaque instance de conteneur du groupe par le biais de demandes HTTP. Si une instance de conteneur ne répond pas d'ici 100 secondes, elle est marquée comme inactive. Les instances de conteneur inactives sont supprimées du groupe et recréées par le processus de reprise automatique. La reprise automatique tente de récupérer par trois fois les instances de conteneur d'un groupe. Après la troisième tentative, la reprise automatique ne récupère aucune instance de conteneur du groupe pendant 60 minutes. Au bout de 60 minutes, le processus de reprise automatique démarre à nouveau.Pour plus d'informations sur la vérification de l'état de santé, voir l'option --http_monitor_enabled.
CMD
(Facultatif) Commande et arguments à transmettre pour exécution au groupe de conteneurs. Cette commande doit être une commande à exécution longue. N'utilisez pas de commande éphémère (par exemple, /bin/date) car celle-ci pourrait entraîner une panne du conteneur.
Exemples de commandes pérennes si vous utilisez l'image ibmnode fournie par IBM Bluemix Container Service:
bx ic group-create --name mon_groupe_conteneurs registry.DomainName/ibmnode ping localhost
bx ic group-create --name mon_groupe_conteneurs registry.DomainName/ibmnode -- tail -f /dev/null
Remarque : La CMD et ses arguments doivent figurer à la fin de la ligne de commande bx ic group-create.
-d DOMAINE, --domain DOMAINE
(Facultatif) Le domaine système par défaut est AppDomainName et fournit déjà un certificat SSL, de sorte que vous pouvez accéder à vos groupes de conteneurs via HTTPS sans configuration supplémentaire. Pour utiliser un domaine personnalisé, vous devez l'enregistrer sur un serveur DNS public, configurer le domaine personnalisé dans Bluemix, puis le mapper au domaine système Bluemix sur le serveur DNS public. Une fois que votre domaine personnalisé est mappé au domaine système Bluemix, les requêtes adressées à votre domaine personnalisé sont acheminées à votre application dans Bluemix. Si vous utilisez un domaine personnalisé, n'incluez pas de trait de soulignement (_) dans le nom de domaine. Pour plus d'informations sur les domaines personnalisés, voir Création et utilisation d'un domaine personnalisé. Pour sécuriser votre domaine personnalisé, téléchargez un certificat SSL afin que votre groupe de conteneurs soit accessible via HTTPS.

L'hôte et le port combinés forment l'URL de route publique complète, par exemple http://mycontainerhost.AppDomainName, qui doit être unique dans votre environnement Bluemix. Lorsque vous examinez les informations d'un groupe de conteneurs avec la commande bx ic group-inspect, l'hôte et le domaine sont mentionnés ensemble en tant que route.

--desired SOUHAITE
(Facultatif) Nombre d'instances dans votre groupe. La valeur par défaut est 2. Le nombre d'instances de groupe de conteneurs requis par votre application dépend de la charge de travail anticipée et du délai sous lequel une demande peut être traitée par l'application. Par exemple, la capacité multi-tâches ou la connectivité réseau ont une incidence sur le nombre d'instances pouvant être requis pour votre application. Pour déterminer le nombre d'instances souhaité, déployez dans Bluemix un seul conteneur exécutant votre application et effectuez un test de charge sur ce conteneur. Si l'application peut gérer, par exemple, 100 requêtes par seconde, mais que vous anticipez une charge de travail moyenne de 300 requêtes par seconde, le nombre d'instances souhaité pour le groupe de conteneurs est de 3. Pour assurer la haute disponibilité de votre groupe de conteneurs et le rendre plus réfractaire aux défaillances, pensez à ajouter des instances supplémentaires par rapport au nombre minimal d'instances afin de traiter la charge de travail prévue. Des instances supplémentaires peuvent traiter la charge de travail en cas de panne d'une autre instance. Pour assurer la reprise automatique d'une instance ayant échoué sans affecter la charge de travail, ajoutez une instance supplémentaire. Pour protéger votre système contre deux pannes simultanées, ajoutez deux instances supplémentaires. Cette configuration correspond à un canevas N+2 où N désigne le nombre d'instances permettant de traiter les demandes et +2 indique que deux instances supplémentaires doivent être ajoutées.
Important : Un groupe de conteneurs devrait contenir au moins 2 instances de conteneur. Si vous incluez une seule instance dans le groupe et que celle-ci devient indisponible, l'utilisateur peut être confronté à un temps d'indisponibilité de l'application.
-e ENV, --env ENV
(Facultatif) Définit la variable d'environnement où ENV est une paire clé=valeur. Répertoriez plusieurs clés séparément et si vous incluez des guillemets, incluez-les autour du nom de la variable d'environnement et de la valeur. Exemple : -e "clé1=valeur1" -e "clé2=valeur2a,valeur2b" -e "clé3=valeur3"

--http_monitor_enabled
(Facultatif) La valeur par défaut est true. Lorsque la reprise automatique est activée pour un groupe de conteneurs et que ce groupe est en opération depuis 10 minutes, l'équilibreur de charge du groupe commence à vérifier régulièrement la santé de chaque instance de conteneur du groupe par le biais de demandes HTTP. Si une instance de conteneur ne répond pas d'ici 100 secondes, elle est marquée comme inactive. Les instances de conteneur inactives sont supprimées du groupe et recréées par le processus de reprise automatique. La reprise automatique tente de récupérer par trois fois les instances de conteneur d'un groupe. Après la troisième tentative, la reprise automatique ne récupère aucune instance de conteneur du groupe pendant 60 minutes. Au bout de 60 minutes, le processus de reprise automatique démarre à nouveau. Certaines applications peuvent ne pas nécessiter une vérification de l'état de santé HTTP, par exemple, une application qui utilise TCP au lieu de HTTP pour les communications. Pour désactiver le moniteur HTTP, affectez à cette option la valeur false. Exemple : --http_monitor_enabled=false
--http_monitor_path
(Facultatif) La valeur par défaut est /. Lorsque l'option --http_monitor_enabled est activée, vous pouvez l'utiliser pour définir un chemin personnalisé pour la vérification de l'état de santé, par exemple, --http_monitor_path="/health/path".
--http_monitor_rc_list
(Facultatif) Lorsque l'option --http_monitor_enabled est activée, vous pouvez spécifier la liste de codes de réponse pour la vérification de l'état de santé. La valeur par défaut inclut les codes 200, 201, 202, 204, 300, 301, 302, 401, 403, et 404. Une réponse ne figurant pas dans la liste des réponses valides change le statut de l'instance de conteneur en INACTIF par l'équilibreur de charge. Un conteneur inactif ne reçoit aucune charge et l'état du groupe de conteneurs passe à HAS_INACTIVE_INSTANCES. Si aucun des conteneurs n'affiche l'état ACTIVE, celui du groupe de conteneurs passe à NO_ACTIVE_INSTANCES. Pour plus d'informations sur les différents états, voir Etats de conteneur dans IBM Bluemix Container Service.
IMAGE
(Requis) Image à inclure dans chaque instance de conteneur du groupe de conteneurs. Vous pouvez spécifier des commandes après l'image, mais n'incluez pas d'options après celle-ci ; incluez toutes les options avant de spécifier l'image.

Si vous utilisez une image située dans le registre d'images Bluemix privé de votre organisation, spécifiez l'image sous le format suivant : registry.DomainName/ESPACE_NOM/IMAGE

Si vous utilisez une image fournie par IBM Bluemix Container Service, n'incluez pas l'espace de nom de votre organisation. Spécifiez l'image en respectant le format suivant : registry.DomainName/IMAGE

--ip IP_ADDRESS
(Facultatif) Si vous disposez d'une adresse IP, liez une adresse IP publique à votre groupe de conteneurs. Vous devez également exposer le port à l'aide de l'indicateur -p. Un groupe de conteneurs peut avoir une route publique ou une adresse IP publique, mais non pas les deux. Pour déconnecter l'adresse IP et renvoyer l'adresse IP au quota de l'espace, retirez le groupe de conteneurs et recréez le groupe sans l'adresse IP. Pour demander une adresse IP avant de créer un groupe de conteneurs, exécutez bx ic ip-request.
Remarque : La liaison d'une adresse IP publique n'inclut pas de certificat SSL qui sert à chiffrer les données envoyées vers ou reçues de votre groupe de conteneurs. Si votre application nécessite un chiffrement SSL, vous pouvez implémenter votre propre solution SSL ou mapper une route publique à votre groupe de conteneurs au lieu de lier une adresse IP publique. Les routes publiques comprennent déjà un certificat SSL, par conséquent, vous pouvez accéder à votre groupe de conteneurs avec HTTPS sans configuration supplémentaire.
-m MEMOIRE, --memory MEMOIRE
(Facultatif) Entrez une limite de mémoire, en Mo, pour votre conteneur. La limite de mémoire est un élément de la taille de conteneur qui définit la mémoire maximale et l'espace disque attribués à un conteneur sur l'hôte de traitement en phase d'exécution. Une fois qu'une taille a été affectée à un conteneur, cette valeur ne peut plus être modifiée. Les tailles disponibles dans IBM Bluemix Container Service sont les suivantes :
  • Pico (64 Mo de mémoire, 4 Go d'espace disque)
  • Nano (128 Mo de mémoire, 8 Go d'espace disque)
  • Micro (256 Mo de mémoire, 16 Go d'espace disque)
  • Très petit (512 Mo de mémoire, 32 Go d'espace disque)
  • Petit (1024 Mo de mémoire, 64 Go d'espace disque)
  • Moyen (2048 Mo mémoire, 128 Go d'espace disque)
  • Grand (4096 Mo de mémoire, 256 Go d'espace disque)
  • Très grand (8192 Mo de mémoire, 512 Go d'espace disque)
  • Très très grand (16384 Mo de mémoire, 1 To d'espace disque)
Si vous ne définissez pas une taille pour votre conteneur, chaque instance de conteneur est créée avec 256 Mo.
Important : Entrez la mémoire du conteneur en Mo, sans le libellé de l'unité. Par exemple, si vous désirez créer un conteneur de taille pico, entrez -m 64.
--max MAX
(Facultatif) Nombre maximal d'instances. La valeur par défaut est 2.
--min MIN
(Facultatif) Nombre minimal d'instances. La valeur par défaut est 1.
-n HOTE, --hostname HOTE
(Facultatif) Nom d'hôte, par exemple monhôteconteneur. N'incluez pas de traits de soulignement (_) dans le nom d'hôte. L'hôte et le port combinés forment l'URL de route publique complète. Par exemple, http://mycontainerhost.AppDomainName. Lorsque vous examinez les informations d'un groupe de conteneurs avec la commande bx ic group-inspect, l'hôte et le domaine sont mentionnés ensemble en tant que route.
--name NOM
(Requis) Affecter un nom au groupe. -n est obsolète.
Conseil : Le nom du conteneur doit commencer par une lettre et peut ensuite comprendre des majuscules, des minuscules, des chiffres, des points (.), des traits de soulignement (_) ou des traits d'union (-).
-p PORT, --publish PORT

(Obligatoire) Expose le port pour le trafic HTTP et du réseau privé. Si vous n'exposez pas de port, le groupe de conteneurs s'exécute mais n'est pas accessible via l'équilibreur de charge du groupe sur le réseau public ou privé.

Vous ne pouvez pas inclure plusieurs ports pour les groupes de conteneurs. Si plusieurs ports sont mentionnés, seul le dernier port mentionné est exposé. Les autres ports sont ignorés par la commande.

Lorsque vous liez une route, les conteneurs de votre groupe doivent être à l'écoute du trafic HTTP sur le port exposé du groupe. Les ports non HTTP ne peuvent pas être exposés publiquement. Lorsque le trafic HTTPS arrive sur le port exposé, (Go)Router exécute l'arrêt de HTTPS. Le protocole HTTP est ensuite utilisé sur le réseau privé virtuel entre (Go)Router et les conteneurs. Lorsque vous liez une adresse IP, les conteneurs de votre groupe peuvent se mettre à l'écoute du trafic HTTP ou HTTPs, mais vous ne pouvez cependant exposer qu'un seul port.
Si un port est spécifié dans le fichier Dockerfile pour l'image que vous utilisez, incluez ce port.

Exemples de commande à exécution longue, si vous utilisez l'image ibmnode fournie dans le registre IBM Bluemix Container Service :
bx ic run --name
mon_conteneur registry.DomainName/ibmnode ping localhost
bx ic
run --name mon_conteneur registry.DomainName/ibmnode -- tail
-f /dev/null
bx ic run --name mon_conteneur
registry.DomainName/ibmnode -- sh -c "while true; do date; sleep 20; done"
Remarque : La CMD et ses arguments doivent figurer à la fin de la ligne de commande bx ic group-create.
-d
(Facultatif) Exécute le conteneur en mode détaché. Les conteneurs opèrent par défaut en mode détaché lorsque vous utilisez la commande bx ic run. En mode détaché, lorsque le processus racine dans le conteneur rend la main, le conteneur rend également la main. Par exemple, lorsque vous déployez un conteneur depuis l'image ibmnode, votre conteneur s'arrête immédiatement après sa construction vu qu'aucun processus de longue durée n'existe.
Conseil : Si vous décidez d'utiliser docker run au lieu de bx ic run, l'option -d est requise.

(Facultatif) Entrez une limite de mémoire, en Mo, pour votre conteneur. Une fois qu'une taille a été affectée à un conteneur, cette valeur ne peut plus être modifiée. Les tailles disponibles dans IBM Bluemix Container Service sont les suivantes :
  • Pico (64 Mo de mémoire, 4 Go d'espace disque)
  • Nano (128 Mo de mémoire, 8 Go d'espace disque)
  • Micro (256 Mo de mémoire, 16 Go d'espace disque)
  • Très petit (512 Mo de mémoire, 32 Go d'espace disque)
  • Petit (1024 Mo de mémoire, 64 Go d'espace disque)
  • Moyen (2048 Mo mémoire, 128 Go d'espace disque)
  • Grand (4096 Mo de mémoire, 256 Go d'espace disque)
  • Très grand (8192 Mo de mémoire, 512 Go d'espace disque)
  • Très très grand (16384 Mo de mémoire, 1 To d'espace disque)

-p PORT, --publish PORT

(Facultatif) Si vous désirez que votre application soit accessible depuis Internet, vous devez exposer un port public. Lorsque vous exposez un port public, vous créez un Public Network Security Group pour votre conteneur vous autorisant à envoyer et à recevoir des données publiques uniquement sur le port exposé. Tous les autres ports publics sont fermés et ne peuvent pas être utilisés pour accéder à votre application depuis Internet. Vous pouvez inclure plusieurs ports avec des options -p diverses. Les ports ne peuvent pas être mappés ou réacheminés. Si vous n'exposez pas un port, votre conteneur est accessible uniquement depuis le réseau de conteneurs privé. Vous pouvez utiliser l'adresse IP privée affectée pour communiquer avec votre conteneur sur le réseau privé.

Si un port est spécifié dans le fichier Dockerfile pour l'image que vous utilisez, incluez ce port.
Conseil :
  • Pour l'image Liberty Server certifiée IBM ou une version modifiée de cette image, entrez le port 9080.
  • Pour l'image Node.js certifiée IBM Node.js ou une version modifiée de cette image, entrez n'importe quel port, par exemple le port 8000.

Vous pouvez choisir comme protocole IP à utiliser soit UDP, soit TCP. Si vous ne spécifiez pas de protocole, votre port est automatiquement exposé pour le trafic TCP.

Exemples
  • -p 9080
  • -p 9080/udp
  • -p 9080/udp -o 8000/tcp

Si vous disposez d'une adresse IP existante dans l'espace que vous désirez lier au conteneur, vous pouvez spécifier l'adresse IP avec les informations du port. Incluez l'adresse IP, le port, ou l'adresse IP et le numéro de port du conteneur, par exemple <adresse_ip>:<port_sur_IP>:<port_conteneur>. Le port sur l'adresse IP doit correspondre au port de conteneur exposé dans le fichier Dockerfile.

Exemple
-p 192.0.2.23:9080:9080
Remarque : La liaison d'une adresse IP publique n'inclut pas de certificat SSL qui sert à chiffrer les données envoyées vers ou reçues d'un conteneur unique. Si votre application nécessite un chiffrement SSL, vous pouvez implémenter votre propre solution SSL ou utiliser un groupe de conteneurs à la place d'un conteneur unique. Des groupes de conteneurs sont liés à une route publique qui inclut déjà un certificat SSL, par conséquent, vous pouvez accéder à votre groupe de conteneurs avec HTTPS sans configuration supplémentaire. Pour plus d'informations, voir Création d'un groupe de conteneurs avec l'interface graphique Bluemix.

cf ic group create [--anti] [--auto] [-d DOMAINE] [--desired SOUHAITE] [-e ENV] [--env-file FICHIER_ENVIRONNEMENT] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip ADDRESSE_IP] [-m MEMOIRE] [--max MAX] [--min MIN] [-n HOTE] --name NOM [-p PORT] [--session_affinity] [--volume VOLUME:/CHEMIN_REPERTOIRE] IMAGE [COMMANDE]

Objet
Permet de créer un groupe de mise à l'échelle. Par défaut, les conteneurs dans Bluemix s'exécutent en mode détaché.
Paramètres
--anti
(Facultatif) Utilisez l'option anti-affinité pour accroître la haute disponibilité de votre groupe de conteneurs. L'option --anti force le placement de chaque instance de conteneur de votre groupe dans un noeud de traitement physique distinct, ce qui réduit le risque d'une défaillance de tous les conteneurs d'un groupe suite à une panne matérielle. Il se peut que vous ne puissiez pas utiliser cette option avec des tailles de groupe plus volumineuses vu que chaque région et organisation Bluemix dispose d'un nombre de noeuds de traitement limité disponibles pour le déploiement. Si votre déploiement échoue, vous pouvez soit réduire le nombre d'instances de conteneur dans le groupe, soit supprimer l'option --anti.
--auto
(Facultatif) Lorsque la reprise automatique est activée pour un groupe de conteneurs et que ce groupe est en opération depuis 10 minutes, l'équilibreur de charge du groupe commence à vérifier régulièrement la santé de chaque instance de conteneur du groupe par le biais de demandes HTTP. Si une instance de conteneur ne répond pas d'ici 100 secondes, elle est marquée comme inactive. Les instances de conteneur inactives sont supprimées du groupe et recréées par le processus de reprise automatique. La reprise automatique tente de récupérer par trois fois les instances de conteneur d'un groupe. Après la troisième tentative, la reprise automatique ne récupère aucune instance de conteneur du groupe pendant 60 minutes. Au bout de 60 minutes, le processus de reprise automatique démarre à nouveau.Pour plus d'informations sur la vérification de l'état de santé, voir l'option --http_monitor_enabled.
CMD
(Facultatif) Commande et arguments à transmettre pour exécution au groupe de conteneurs. Cette commande doit être une commande à exécution longue. N'utilisez pas de commande éphémère (par exemple, /bin/date) car celle-ci pourrait entraîner une panne du conteneur.
Exemples de commandes pérennes si vous utilisez l'image ibmnode fournie par IBM Bluemix Container Service:

cf ic group create --name mon_groupe_conteneurs registry.DomainName/ibmnode -- tail -f /dev/null
Remarque : La CMD et ses arguments doivent figurer à la fin de la ligne de commande cf ic group create.
-d DOMAINE, --domain DOMAINE
(Facultatif) Le domaine système par défaut est AppDomainName et fournit déjà un certificat SSL, de sorte que vous pouvez accéder à vos groupes de conteneurs via HTTPS sans configuration supplémentaire. Pour utiliser un domaine personnalisé, vous devez l'enregistrer sur un serveur DNS public, configurer le domaine personnalisé dans Bluemix, puis le mapper au domaine système Bluemix sur le serveur DNS public. Une fois que votre domaine personnalisé est mappé au domaine système Bluemix, les requêtes adressées à votre domaine personnalisé sont acheminées à votre application dans Bluemix. Si vous utilisez un domaine personnalisé, n'incluez pas de trait de soulignement (_) dans le nom de domaine. Pour plus d'informations sur les domaines personnalisés, voir Création et utilisation d'un domaine personnalisé. Afin de sécuriser votre domaine, transférez un certificat SSL pour que vos groupes de conteneurs soient accessibles via HTTPS.

L'hôte et le port combinés forment l'URL de route publique complète, par exemple http://mycontainerhost.AppDomainName, qui doit être unique dans votre environnement Bluemix. Lorsque vous examinez les informations d'un groupe de conteneurs avec la commande cf ic group inspect, l'hôte et le domaine sont mentionnés ensemble en tant que route.

--desired SOUHAITE
(Facultatif) Nombre d'instances dans votre groupe. La valeur par défaut est 2. Le nombre d'instances de groupe de conteneurs requis par votre application dépend de la charge de travail anticipée et du délai sous lequel une demande peut être traitée par l'application. Par exemple, la capacité multi-tâches ou la connectivité réseau ont une incidence sur le nombre d'instances pouvant être requis pour votre application. Pour déterminer le nombre d'instances souhaité, déployez dans Bluemix un seul conteneur exécutant votre application et effectuez un test de charge sur ce conteneur. Si l'application peut gérer, par exemple, 100 requêtes par seconde, mais que vous anticipez une charge de travail moyenne de 300 requêtes par seconde, le nombre d'instances souhaité pour le groupe de conteneurs est de 3. Pour assurer la haute disponibilité de votre groupe de conteneurs et le rendre plus réfractaire aux défaillances, pensez à ajouter des instances supplémentaires par rapport au nombre minimal d'instances afin de traiter la charge de travail prévue. Des instances supplémentaires peuvent traiter la charge de travail en cas de panne d'une autre instance. Pour assurer la reprise automatique d'une instance ayant échoué sans affecter la charge de travail, ajoutez une instance supplémentaire. Pour protéger votre système contre deux pannes simultanées, ajoutez deux instances supplémentaires. Cette configuration correspond à un canevas N+2 où N désigne le nombre d'instances permettant de traiter les demandes et +2 indique que deux instances supplémentaires doivent être ajoutées.
Important : Un groupe de conteneurs devrait contenir au moins 2 instances de conteneur. Si vous incluez une seule instance dans le groupe et que celle-ci devient indisponible, l'utilisateur peut être confronté à un temps d'indisponibilité de l'application.

(Obligatoire) Expose le port pour le trafic HTTP et du réseau privé. Si vous n'exposez pas de port, le groupe de conteneurs s'exécute mais n'est pas accessible via l'équilibreur de charge du groupe sur le réseau public ou privé.

Lorsque le trafic HTTPS arrive sur le port exposé, (Go)Router exécute l'arrêt de HTTPS. Le protocole HTTP est ensuite utilisé sur le réseau privé virtuel entre (Go)Router et les conteneurs. Si un port est spécifié dans le fichier Dockerfile pour l'image que vous utilisez, incluez ce port.
Conseil :

(Facultatif) La valeur par défaut est false.

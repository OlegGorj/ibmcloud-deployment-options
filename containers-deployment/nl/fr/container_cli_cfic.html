Installation du plug-in IBM Bluemix Container Service (bx ic)

Pour que vous puissiez exécuter des commandes, votre système d'exploitation doit satisfaire les exigences ci-après. Cette tâche inclut les informations pour l'installation de ces exigences. Les versions antérieures de ces outils ne sont pas prises en charge pour une utilisation avec le plug-in IBM Bluemix Container Service (bx ic).

  • Docker version 1.8 à 1.12
  • Cloud Foundry CLI version 6.14.0 - 6.22.0
    Remarque : Les utilisateurs MacOS Sierra doivent installer l'interface de ligne de commande Cloud Foundry version 6.22.0 ou ultérieure.
  • Bluemix interface de ligne de commande
  • Plug-in IBM Bluemix Container Service

Pour installer les interfaces CLI, procédez comme suit :

  1. Installez Docker 1.12. Pour les systèmes d'exploitation Windows et OS X, Docker est installé en même temps que Docker Toolbox, lequel inclut les éléments Docker Machine, Docker Engine et Kitematic. Pour les systèmes d'exploitation Linux, le mécanisme d'installation varie selon le type de distribution Linux que vous utilisez. Reportez-vous aux liens ci-après pour installer Docker, ou, si vous l'avez déjà installé, pour effectuer une mise à niveau vers 1.12.
    Remarque : Si vous avez déjà installé Docker sur votre ordinateur et ne désirez pas mettre à niveau votre version actuelle, vous pouvez ignorer cette étape et passer à l'installation de l'interface CLI de Cloud Foundry.
    • OS X

      https://github.com/docker/toolbox/releases/download/v1.12/DockerToolbox-1.12.pkg

    • Windows

      https://github.com/docker/toolbox/releases/download/v1.12/DockerToolbox-1.12.exe

    • Linux

      Reportez-vous à la Documentation d'installation Docker pour les instructions d'installation de Docker version 1.12 sur la distribution Linux que vous utilisez.

  2. Lancez Docker.
    • Pour OS X et Windows, effectuez un double-clic sur l'icône Docker Quickstart Terminal. Le démon Docker est démarré et les variables d'environnement sont définies pour vous.
    • Pour Linux, dans un utilitaire de ligne de commande, vérifiez que le démon Docker est en opération. Configurez également le démon Docker pour son redémarrage automatique. Pour plus d'informations, voir voir la documentation Docker de votre système d'exploitation.
  3. Vérifiez que votre installation Docker est configurée correctement en exécutant la commande docker run hello-word dans votre terminal. 
    docker run hello-world
    Docker recherche l'image locale hello-world afin de lancer un conteneur depuis celle-ci. Par conséquent, si vous exécutez la commande pour la première fois, le message suivant s'affiche. 
    Image locale 'hello-world:latest' introuvable
    L'image est extraite du référentiel public d'images Docker Hub et, à l'issue de la commande, la sortie devrait être similaire à ceci : 
    $ docker run hello-world

Hello from Docker.
Ce message indique que votre installation semble fonctionner correctement.

Pour générer ce message, Docker a procédé comme suit :
 1. Le client Docker a contacté le démon Docker.
 2. Le démon Docker a extrait l'image "hello-world" depuis Docker Hub.
 3. Le démon Docker a créé depuis cette image un nouveau conteneur qui utilise l'exécutable
    ayant produit la sortie que vous consultez actuellement.
 4. Le démon Docker a transmis cette sortie au client Docker, lequel l'a envoyée
    à votre terminal.

Pour générer quelque chose de plus ambitieux, vous pouvez lancer un conteneur Ubuntu avec la commande :
 $ docker run -it ubuntu bash

Partagez des images, automatisez des flux de travaux, et exploitez d'autres possibilités avec un compte Docker Hub gratuit :
 https://hub.docker.com

Pour plus d'exemples et d'idées, visitez le site :
 https://docs.docker.com/userguide/
    Si la commande ne s'est pas terminée correctement, exécutez la commande docker version pour vérifier si Docker a été correctement installé.
  4. Installez l'interface CLI de Cloud Foundry version Cloud Foundry CLI version 6.14.0 - 6.22.0 depuis le référentiel GitHub. Vous devez installer l'interface de ligne de commande Cloud Foundry sous l'emplacement par défaut pour votre système d'exploitation sans quoi la variable d'environnement PATH ne correspondra pas à votre répertoire d'installation.
    Remarque : Les utilisateurs MacOS Sierra doivent installer l'interface de ligne de commande Cloud Foundry version 6.22.0 ou ultérieure.
    Pour vérifier que l'interface de ligne de commande Cloud Foundry a été installée correctement, ou si elle était déjà installée, pour vérifier la version installée, exécutez la commande suivante :
    cf -v
  5. Installez l'interface CLI de Bluemix.
  6. Connectez-vous à l'interface CLI d'Bluemix. 
    bx login -a https://api.ng.bluemix.net
  7. Installez le plug-in IBM Bluemix Container Service. 
    bx plugin install IBM-Containers -r Bluemix
    Pour vérifier que le plug-in a été correctement installé, exécutez la commande suivante :
    bx plugin list

    Le plug-in du registre s'affiche dans les résultats sous le nom IBM-Containers.

Vous êtes prêt à vous connecter et à commencer à gérer des conteneurs.

Connexion au plug-in de l'interface CLI d'IBM Bluemix Container Service (bx ic)

Après son installation, connectez-vous à l'interface de ligne de commande pour l'utiliser.

Avant de commencer, effectuez les tâches suivantes.
  • Installez le plug-in IBM Bluemix Container Service (bx ic).
  • Déterminez l'organisation et l'espace Bluemix à utiliser lorsque vous vous connectez à Cloud Foundry. Pour plus d'informations sur les organisations et les espaces, voir Organisations et espaces.
  • Pour plus d'informations sur les options que vous pouvez spécifier avec les commandes Cloud Foundry login, voir Commande Login dans la documentation Cloud Foundry.
  1. Ouvrez un utilitaire de ligne de commande.
    • Pour OS X et Windows, effectuez un double-clic sur l'icône Docker Quickstart Terminal. Le démon Docker est démarré et les variables d'environnement sont définies pour vous.
    • Pour Linux, ouvrez n'importe quel utilitaire de ligne de commande.
  2. Connectez-vous à Bluemix via l'interface de ligne de commande Cloud Foundry. Pour spécifier une région spécifique, incluez l'option -a en indiquant le noeud final de l'API pour la région. Si vous utilisez l'interface de ligne de commande pour la première fois et n'incluez pas cette option, vous êtes invité à indiquer un noeud final d'API. Une fois que le noeud final de l'API a été spécifié, vous n'avez plus besoin d'en entrer un lors des connexions suivantes. 
    bx login [-a api.DomainName] [-sso]
    Pour vous connecter dans une région spécifique, incluez son noeud final d'API lorsque vous vous connectez à l'interface CLI de Cloud Foundry.
    • Sud des Etats-Unis
      bx login -a api.ng.bluemix.net
    • Royaume-Uni
      bx login -a api.eu-gb.bluemix.net
    Le paramètre de connexion unique, --sso, est requis lorsque vous vous connectez avec un ID fédéré. Si cette option est utilisée, ouvrez https://login.ng.bluemix.net/UAALoginServerWAR/passcode pour obtenir un code d'accès à usage unique. Si vous omettez l'option --sso, exécutez les sous-étapes ci-après.
    1. Dans Adresse électronique, entrez votre IBMId pour Bluemix.
    2. Dans Mot de passe, entrez le mot de passe correspondant à l'IBMId. Vos organisations et espaces Bluemix sont extraits.
    3. Entrez le numéro représentant l'une de vos organisations Bluemix.
    4. Entrez le numéro correspondant à l'un de vos espaces Bluemix existants.
  3. Définissez un espace de nom pour votre organisation. Un espace de nom est un nom unique identifiant votre registre d'images Bluemix privé. Lorsque vous créez un conteneur, vous devez spécifier l'emplacement de l'image en incluant l'espace de nom avec le nom de l'image.
    Remarque : Une fois l'espace de nom créé, il ne peut plus être modifié. Vous ne pouvez que le supprimer. Si des utilisateurs de votre organisation Bluemix ont déjà déployé des conteneurs et défini un espace de nom, un message affiche l'espace de nom de votre organisation. Dans ce cas, vous pouvez passer à l'étape suivante.
    bx ic namespace-set <mon_espace_nom>
    Conseil : Respectez les règles suivantes pour l'espace de nom :
    • Il ne peut comporter que des lettres, des chiffres ou des traits de soulignement (_).
    • Il peut comporter entre 4 et 30 caractères. Si vous prévoyez de gérer des conteneurs depuis la ligne de commande, il est préférable d'utiliser un espace de nom court pouvant être saisi rapidement.
    • Son nom doit être unique dans Bluemix.
  4. Initialisez IBM Bluemix Container Service. 
    bx ic init
  5. Décidez si vous souhaitez exécuter des commandes bx ic ou Docker natives. Vous pouvez sélectionner l'une de ces options à chaque fois que vous vous connectez au plug-in ou utiliser la commande shellinit pour toujours sélectionner l'option 2 pour vous.
    • Option 1 (option par défaut)

      Shell simple pour IBM Bluemix Container Service encapsuleurs Docker. Le démon Docker local n'est pas affecté par IBM Bluemix Container Service. Les commandes bx ic sont utilisées pour gérer les conteneurs dans votre espace Bluemix et les commandes docker sont utilisées pour les gérer en local. Toutefois, les commandes docker sont toujours requises pour le développement d'image locale.

      Par exemple, avec l'option 1, la commande bx ic run crée un conteneur dans votre espace Bluemix et la commande docker run en crée un en local. Vous pourriez préférer cette option si vous débutez avec Docker et IBM Bluemix Container Service, ou si vous désirez tester vos conteneurs localement.

    • Option 2

      Redirige votre démon Docker local pour la session actuelle de l'interface de ligne de commande vers IBM Bluemix Container Service de sorte que les commandes docker puissent être utilisées à la place des commandes bx ic chaque fois qu'elles sont disponibles afin de gérer des conteneurs dans votre espace Bluemix.

      Par exemple, avec l'option 2, la commande bx ic run ou la commande docker run peuvent être utilisées pour créer un conteneur dans votre espace. Vous pourriez préférer cette option si vous êtes un utilisateur Docker expérimenté et plus à l'aise avec les commandes docker ou si vous adaptez des scripts automatisés pour leur utilisation avec IBM Bluemix Container Service.

      Certaines exceptions nécessitent néanmoins d'utiliser des commandes bx ic pour réaliser des tâches spécifiques à IBM Bluemix Container Service. Par exemple, pour définir un espace de nom, vous devez utiliser la commande bx ic namespace-set. docker namespace-set n'est pas une commande valide. Pour la liste des commandes docker prises en charge avec cette configuration, voir Commandes Docker prises en charge pour le plug-in IBM Bluemix Container Service (bx ic).

      Pour configurer cette option, copiez et collez les variables d'environnement affichées dans le terminal à la connexion.
      • DOCKER_HOST définit l'hôte Docker sur IBM Bluemix Container Service.
      • DOCKER_CERT_PATH définit l'emplacement des clés d'authentification pour le plug-in de IBM Bluemix Container Service pour Cloud Foundry.
      • DOCKER_TLS_VERIFY utilise TLS et vérifie le serveur distant.
      Conseil : Lorsque vous effectuez les tâches mentionnées dans la documentation IBM Bluemix Container Service, vous pouvez lancer des commandes docker dans toutes les étapes accompagnées d'un astérisque (*).

bx ic attach [--no-stdin] [--sig-proxy] CONTENEUR

Objet
Contrôle d'un conteneur en cours d'exécution ou affichage de sa sortie. Utilisez CTRL+C pour quitter et arrêter le conteneur. Pour plus d'informations, reportez-vous à la commande attach dans l'aide de Docker.
Paramètres
--no-stdin
(Facultatif) Indique de ne pas inclure l'entrée standard.
--sig-proxy
(Facultatif) Valeur par défaut : true.
CONTENEUR
(Requis) Nom ou ID de conteneur.
Exemple
L'exemple suivant est une requête d'association au conteneur mon_conteneur.
bx ic attach mon_conteneur

bx ic build -t TAG [--no-cache] [--pull] [-q] emplacement_Dockerfile

Objet
Créez une image Docker localement ou dans le registre d'images Bluemix privé. Pour plus d'informations, reportez-vous à la commande build dans l'aide de Docker.
Paramètres
EMPLACEMENT_DOCKERFILE
(Requis) Chemin d'accès au fichier Dockerfile et au contexte sur l'hôte local.
--no-cache
(Facultatif) Lorsqu'une image est construite pour la première fois, met en cache les couches de l'image pour l'ensemble de l'organisation. Lorsque vous créez des versions ultérieures de l'image, ces versions en mémoire cache de l'image sont utilisées afin d'accélérer le processus de génération. En incluant l'option --no-cache lorsque vous construisez une nouvelle version de l'image, chaque couche de l'image est elle-même reconstruite.
--pull
(Facultatif) Par défaut, l'image de base n'est extraite du registre que si elle n'est pas encore présente dans le cache d'images de votre organisation. En incluant cette option, la version la plus récente de l'image de base est utilisée pour construire la nouvelle image.
-q, --quiet
(Facultatif) Supprimer la sortie prolixe qui est générée par les conteneurs. La valeur par défaut est false.
-t ETIQUETTE, --tag ETIQUETTE
(Requis) Chemin de votre registre d'images privé à appliquer à l'image créée.
Exemple
L'exemple suivant illustre une demande de génération d'une image nommée mon_image. Le fichier Dockerfile et d'autres artefacts à utiliser dans la génération sont situés dans le même répertoire que celui depuis lequel la commande est exécutée. Etant donné que le registre et l'espace de nom sont inclus avec le nom de l'image, l'image est générée dans le registre d'images Bluemix privé de votre organisation.
Conseil : Exécutez bx ic namespace-get pour extraire votre espace de nom et remplacez <my_namespace> par votre espace de nom. 
bx ic build -t
registry.DomainName/<mon_espace_nom>/mon_image.

bx ic cp CHEMIN_SRC CONTENEUR:CHEMIN_DESTINATION | bx ic cp CONTENEUR:CHEMIN_SRC CHEMIN_DESTINATION

Objet
Permet de copier un fichier ou un dossier depuis un conteneur unique en exécution, ou arrêté, vers votre machine locale, et vice versa. Pour plus d'informations, reportez-vous à la commande cp dans l'aide de Docker.
Remarque : Cette commande n'est prise en charge que pour les conteneurs uniques. Si vous désirez copier des fichiers depuis ou vers un groupe de conteneurs, vous devez les copier depuis ou vers chaque instance de conteneur du groupe.
Paramètres
CHEMIN_DESTINATION
(Requis) Chemin du fichier ou du dossier sur votre machine locale ou dans votre conteneur où copier les informations du chemin source. Le fichier ou le dossier doit exister sur votre machine locale avant de lancer la copie. Si vous désirez les copier sur votre machine locale, vous pouvez utiliser son chemin absolu ou relatif. Si vous désirez les copier dans votre conteneur unique, vous devez inclure le nom du conteneur, le délimiteur ":" et le chemin du fichier ou du dossier de destination. Ce chemin doit être relatif au répertoire racine du conteneur.
CHEMIN_SRC
(Requis) Chemin du fichier ou du dossier que vous désirez copier. Si le fichier ou le dossier réside sur votre machine locale, vous pouvez utiliser son chemin absolu ou relatif. Si le fichier ou le dossier est situé dans votre conteneur unique, vous devez inclure le nom du conteneur, le délimiteur ":" et le chemin du fichier ou du dossier que vous désirez copier. Le chemin doit être relatif au répertoire racine du conteneur.
Remarque : Vous ne pouvez pas copier certains fichiers système enregistrés sous les dossiers /dev, /etc, /proc et /sys de votre conteneur.
Exemple
Cet exemple de commande copie un fichier nommé config.txt depuis le répertoire actuel de votre machine locale vers le répertoire config d'un conteneur unique nommé mon_conteneur.
bx ic cp ./config.txt mon_conteneur:/config

bx ic group-create [--anti] [--auto] [-d DOMAINE] [--desired SOUHAITE] [-e ENV] [--env-file FICHIER_ENVIRONNEMENT] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip ADRESSE_IP] [-m MEMOIRE] [--max MAX] [--min MIN] [-n HOTE --name NOM [-p PORT] [--session-affinity] [--volume VOLUME:/CHEMIN_REPERTOIRE] IMAGE [CMD]

Objet
Permet de créer un groupe de mise à l'échelle. Par défaut, les conteneurs dans Bluemix s'exécutent en mode détaché.
Paramètres
--anti
(Facultatif) Utilisez l'option anti-affinité pour accroître la haute disponibilité de votre groupe de conteneurs. L'option --anti force le placement de chaque instance de conteneur de votre groupe dans un noeud de traitement physique distinct, ce qui réduit le risque d'une défaillance de tous les conteneurs d'un groupe suite à une panne matérielle. Il se peut que vous ne puissiez pas utiliser cette option avec des tailles de groupe plus volumineuses vu que chaque région et organisation Bluemix dispose d'un nombre de noeuds de traitement limité disponibles pour le déploiement. Si votre déploiement échoue, vous pouvez soit réduire le nombre d'instances de conteneur dans le groupe, soit supprimer l'option --anti.
--auto
(Facultatif) Lorsque la reprise automatique est activée pour un groupe de conteneurs et que ce groupe est en opération depuis 10 minutes, l'équilibreur de charge du groupe commence à vérifier régulièrement la santé de chaque instance de conteneur du groupe par le biais de demandes HTTP. Si une instance de conteneur ne répond pas d'ici 100 secondes, elle est marquée comme inactive. Les instances de conteneur inactives sont supprimées du groupe et recréées par le processus de reprise automatique. La reprise automatique tente de récupérer par trois fois les instances de conteneur d'un groupe. Après la troisième tentative, la reprise automatique ne récupère aucune instance de conteneur du groupe pendant 60 minutes. Au bout de 60 minutes, le processus de reprise automatique démarre à nouveau.Pour plus d'informations sur la vérification de l'état de santé, voir l'option --http_monitor_enabled.
CMD
(Facultatif) Commande et arguments à transmettre pour exécution au groupe de conteneurs. Cette commande doit être une commande à exécution longue. N'utilisez pas de commande éphémère (par exemple, /bin/date) car celle-ci pourrait entraîner une panne du conteneur.
Exemples de commandes pérennes si vous utilisez l'image ibmnode fournie par IBM Bluemix Container Service:
bx ic group-create --name mon_groupe_conteneurs registry.DomainName/ibmnode ping localhost
bx ic group-create --name mon_groupe_conteneurs registry.DomainName/ibmnode -- tail -f /dev/null
Remarque : La CMD et ses arguments doivent figurer à la fin de la ligne de commande bx ic group-create.
-d DOMAINE, --domain DOMAINE
(Facultatif) Le domaine système par défaut est AppDomainName et fournit déjà un certificat SSL, de sorte que vous pouvez accéder à vos groupes de conteneurs via HTTPS sans configuration supplémentaire. Pour utiliser un domaine personnalisé, vous devez l'enregistrer sur un serveur DNS public, configurer le domaine personnalisé dans Bluemix, puis le mapper au domaine système Bluemix sur le serveur DNS public. --desired SOUHAITE
(Facultatif) Nombre d'instances dans votre groupe. La valeur par défaut est 2. Le nombre d'instances de groupe de conteneurs requis par votre application dépend de la charge de travail anticipée et du délai sous lequel une demande peut être traitée par l'application. Par exemple, la capacité multi-tâches ou la connectivité réseau ont une incidence sur le nombre d'instances pouvant être requis pour votre application. Pour déterminer le nombre d'instances souhaité, déployez dans Bluemix un seul conteneur exécutant votre application et effectuez un test de charge sur ce conteneur. Si l'application peut gérer, par exemple, 100 requêtes par seconde, mais que vous anticipez une charge de travail moyenne de 300 requêtes par seconde, le nombre d'instances souhaité pour le groupe de conteneurs est de 3. Pour assurer la haute disponibilité de votre groupe de conteneurs et le rendre plus réfractaire aux défaillances, pensez à ajouter des instances supplémentaires par rapport au nombre minimal d'instances afin de traiter la charge de travail prévue. Des instances supplémentaires peuvent traiter la charge de travail en cas de panne d'une autre instance. Pour assurer la reprise automatique d'une instance ayant échoué sans affecter la charge de travail, ajoutez une instance supplémentaire. Pour protéger votre système contre deux pannes simultanées, ajoutez deux instances supplémentaires. Cette configuration correspond à un canevas N+2 où N désigne le nombre d'instances permettant de traiter les demandes et +2 indique que deux instances supplémentaires doivent être ajoutées.
Important : Un groupe de conteneurs devrait contenir au moins 2 instances de conteneur. Si vous incluez une seule instance dans le groupe et que celle-ci devient indisponible, l'utilisateur peut être confronté à un temps d'indisponibilité de l'application.

--http_monitor_enabled
(Facultatif) La valeur par défaut est true. Lorsque la reprise automatique est activée pour un groupe de conteneurs et que ce groupe est en opération depuis 10 minutes, l'équilibreur de charge du groupe commence à vérifier régulièrement la santé de chaque instance de conteneur du groupe par le biais de demandes HTTP. Si une instance de conteneur ne répond pas d'ici 100 secondes, elle est marquée comme inactive. Les instances de conteneur inactives sont supprimées du groupe et recréées par le processus de reprise automatique. La reprise automatique tente de récupérer par trois fois les instances de conteneur d'un groupe. Après la troisième tentative, la reprise automatique ne récupère aucune instance de conteneur du groupe pendant 60 minutes. Au bout de 60 minutes, le processus de reprise automatique démarre à nouveau. Certaines applications peuvent ne pas nécessiter une vérification de l'état de santé HTTP, par exemple, une application qui utilise TCP au lieu de HTTP pour les communications. Pour désactiver le moniteur HTTP, affectez à cette option la valeur false. Exemple : --http_monitor_enabled=false

(Facultatif) La valeur par défaut est /. Lorsque l'option --http_monitor_enabled est activée, vous pouvez l'utiliser pour définir un chemin personnalisé pour la vérification de l'état de santé, par exemple, --http_monitor_path="/health/path".
--http_monitor_rc_list
(Facultatif) Lorsque l'option --http_monitor_enabled est activée, vous pouvez spécifier la liste de codes de réponse pour la vérification de l'état de santé. La valeur par défaut inclut les codes 200, 201, 202, 204, 300, 301, 302, 401, 403, et 404. Un conteneur inactif ne reçoit aucune charge et l'état du groupe de conteneurs passe à HAS_INACTIVE_INSTANCES. Si aucun des conteneurs n'affiche l'état ACTIVE, celui du groupe de conteneurs passe à NO_ACTIVE_INSTANCES. Pour plus d'informations sur les différents états, voir Etats de conteneur dans IBM Bluemix Container Service.

bx ic top CONTENEUR [CONTENEUR] [ps options]

Objet
Liste les processus s'exécutant dans le conteneur.

A la différence de la commande top dans Linux, la commande bx ic top affiche la sortie de la commande Linux ps pour les processus principaux s'exécutant dans un conteneur. Par conséquent, l'utilisation de la mémoire et de l'UC n'est pas mise à jour dans votre sortie CLI lorsqu'elle change pour un processus.

Remarque : Si vous avez besoin d'examiner la sortie de la commande Linux native top, utilisez l'instruction exec pour accéder à votre conteneur et exécutez la commande top.
Pour plus d'informations, reportez-vous à la commande top dans l'aide de Docker.
Paramètres
CONTENEUR
(Requis) ID ou nom du conteneur.
Options de la commande ps
(Facultatif) Utilisez les options fournies par la commande Linux native ps pour plus de détails ou pour filtrer la sortie de la commande bx ic top.

bx ic wait-status CONTENEUR

Objet
Indique d'attendre qu'un conteneur unique ou un groupe de conteneurs passe à un état non transitoire. Pendant cette attente, la ligne de commande ne renvoie pas de résultat et vous ne pouvez pas entrer de commandes. Dès que le conteneur passe à un état non transitoire, un message OK est affiché. Pour les conteneurs isolés, les états non transitoires incluent : En cours d'exécution, Arrêt, En panne, En pause ou Suspendu. Pour les groupes de conteneurs, les états non transitoires incluent CREATE_COMPLETE (Création terminée), UPDATE_COMPLETE (Mise à jour terminée) ou FAILED (Echec).
Paramètres
CONTENEUR
(Requis) Nom ou ID du conteneur ou du groupe.
Exemple
L'exemple suivant illustre une demande d'attente qu'un conteneur nommé mon_conteneur passe à un état non transitoire. 
bx ic wait-status mon_conteneur

Désinstallation du plug-in IBM Bluemix Container Service (bx ic)

Si vous n'avez plus besoin du plug-in IBM Bluemix Container Service, vous pouvez le désinstaller à l'aide de la commande bx.

Remarque : Pour désinstaller le plug-in IBM Bluemix Container Service Cloud Foundry, exécutez la commande suivante : 
cf uninstall-plugin IBM-Containers
  1. Pour désinstaller le plug-in IBM Bluemix Container Service pour l'interface de ligne de commande Bluemix, exécutez la commande suivante : 
    bx plugin uninstall IBM-Containers
  2. Vérifiez que le plug-in a bien été désinstallé en exécutant la commande suivante et n examinant la liste des plug-in installés. 
    bx plugin list

    Le plug-in IBM Bluemix Container Service n'est pas affiché dans les résultats.

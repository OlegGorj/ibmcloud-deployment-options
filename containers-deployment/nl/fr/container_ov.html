A propos de conteneurs uniques et évolutifs

IBM® Bluemix fournit l'infrastructure IBM Bluemix Container Service sous forme de fonction disponible dans certaines régions pour le développement d'applications. Les conteneurs vous permettent de construire votre application dans n'importe quel langage, avec n'importe quel outil de programmation. Chaque conteneur constitue une plateforme applicative sécurisée et isolée. Un conteneur peut s'exécuter n'importe où : sur un poste de travail de développement sous OS X ou Windows, un serveur sous Ubuntu, une machine virtuelle de centre de données de production sous Red Hat, ou une plateforme cloud comme Bluemix.
Avertissement : Des comptes d'essai gratuits ne sont pas disponibles pour conteneurs uniques et évolutifs. Découvrez la gestion de cluster et migrez vers Kubernetes dans IBM Bluemix Container Service sur Bluemix Public.

Qu'est-ce qu'un conteneur ?

Les conteneurs sont des objets logiciels virtuels qui incluent tous les éléments qu'une application a besoin d'exécuter. Un conteneur est généré à partir d'une image, c'est-à-dire un modèle en lecture seule destiné à la création du conteneur. Chaque image inclut l'application et ses dépendances, et s'exécute en tant que processus isolé sur le système d'exploitation hôte. Par conséquent, un conteneur tire parti de l'isolation et de l'allocation des ressources, mais est plus portable et efficace que, par exemple, une machine virtuelle. Les conteneurs permettent de construire des applications de grande qualité rapidement.

Les conteneurs sont agiles

Les conteneurs simplifient l'administration du système en fournissant des environnements normalisés aux équipes de développement et de production. L'empreinte allégée du contexte d'exécution du moteur permet un redimensionnement rapide (augmentation ou atténuation) en réponse à des fluctuations de la demande. Ils suppriment la complexité issue de la gestion de différentes plateformes de système d'exploitation et de l'infrastructure sous-jacente. Ils permettent de déployer et d'exécuter n'importe quelle application dans n'importe quelle infrastructure rapidement et de manière fiable.

Les conteneurs sont de taille modeste

Vous pouvez englober plus de conteneurs dans l'espace que ne le permettrait une machine virtuelle. Vous pouvez sélectionner la taille d'un conteneur afin d'utiliser seulement l'espace requis par l'application.

Les conteneurs sont portables

Vous pouvez générer une image pour un autre conteneur en utilisant comme base une autre image. Quelqu'un d'autre peut faire le gros du travail sur une image que vous pouvez personnaliser pour votre propre utilisation. Vous pouvez également migrer rapidement le code d'application depuis un environnement de transfert vers un environnement de production. Le processus de migration peut être automatisé grâce à des outils tels que le pipeline de livraison Delivery Pipeline ou UrbanCode Deploy.

Docker et IBM Bluemix Container Service

IBM Bluemix Container Service est une exécution Bluemix qui optimise la technologie Docker. Les utilisateurs Docker existants et les nouveaux utilisateurs de Docker peuvent exécuter les fonctions suivantes avec IBM Bluemix Container Service.

  • Ajouter des images Docker Hub existantes à un registre privé hébergé par IBM.
  • Valider les images pour les vulnérabilités potentielles.
  • Configurer l'interface de ligne de commande (CLI) de IBM Bluemix Container Service afin d'utiliser les commandes CLI Docker avec lesquelles vous êtes déjà familiarisé.
  • Consulter les données de journalisation et de surveillance de vos conteneurs sans configuration supplémentaire.

Si vous avez utilisé Docker dans le passé, les réponses à ces questions fréquentes peuvent vous être utiles lors de vos premier pas avec IBM Bluemix Container Service.

Comment utiliser une image issue de Docker Hub dans IBM Bluemix Container Service ?
En utilisant l'interface de ligne de commande d'IBM Bluemix Container Service, vous pouvez copier une image depuis Docker Hub sur votre registre Bluemix privé. Après la copie de l'image, vous pouvez l'utiliser pour créer un conteneur.
Comment créer un conteneur à haute disponibilité ?
IBM Bluemix Container Service fournit des conteneurs uniques qui sont similaires à des conteneurs Docker. De plus, IBM Bluemix Container Service fournit des groupes de conteneurs évolutifs qui peuvent être utilisés pour assurer la haute disponibilité de votre application. Les groupes de conteneurs fournissent un équilibreur de charge intégré pour garantir le routage et l'équilibrage de la charge de travail entre les instances de conteneur. Pour augmenter la disponibilité de votre groupe, répartissez vos instances de groupe de conteneurs sur plusieurs hôtes de traitement physiques dotés de l'option anti-affinité et activez la reprise automatique des instances défectueuses. En outre, vous pouvez définir des règles pour la mise à l'échelle des instances de groupe de conteneurs en fonction de l'utilisation de la mémoire et de l'unité centrale. Pour plus d'informations, voir Exécution de processus à haute disponibilité sous forme de groupes de conteneurs.
Comment mapper des ports de conteneur à un port d'hôte ?
IBM Bluemix Container Service fait la distinction entre le réseau de conteneurs privé et le réseau public. Sur le réseau de conteneurs privé, tous les ports de conteneur sont exposés par défaut sans configuration supplémentaire. Les conteneurs et les groupes de conteneurs connectés à un seul et même réseau de conteneurs privé peuvent communiquer à l'aide des adresses IP privées qui leur sont affectées. Aucun mappage de port n'est nécessaire.

Pour rendre une application disponible au réseau public, vous devez exposer un port public et lier soit une adresse IP publique à votre conteneur unique, soit une route publique à votre groupe de conteneurs. Lorsqu'un port public est exposé, les données publiques peuvent être envoyées et reçues uniquement sur le port exposé. Tous les autres ports publics sont fermés.

Comment rendre un conteneur disponible comme public sur Internet ?
  • Pour des conteneurs uniques, exposez un ou plusieurs ports et affectez une adresse IP flottante au conteneur.
  • Pour des groupes de conteneurs, exposez un port HTTP unique et affectez une route.
Pour quelle raison le temps nécessaire à la création d'un conteneur est différent selon qu'une installation Docker locale ou Bluemix est utilisé ?
Vous constaterez peut-être que les conteneurs sont créés plus rapidement si vous utilisez votre installation Docker locale au lieu deBluemix. Avec le premier déploiement, l'image de conteneur doit être téléchargée dans le registre sur l'hôte. Les déploiements suivants de l'image sont plus rapides. Pour savoir de quelle façon les différentes configurations de conteneur impactent les vitesses de déploiement, voir Types de conteneur.
Comment créer un conteneur sur un réseau privé et ne pas l'exposer comme public ?
Lorsque vous créez un conteneur, il est connecté au réseau privé de conteneurs par défaut IBM Bluemix Container Service et une adresse IP privée lui est affectée. Sur ce réseau privé, tous les ports de conteneur sont exposés par défaut. Le mappage des ports d'hôte à des ports de conteneur n'est pas nécessaire. Pour empêcher un conteneur d'être accessible depuis Internet, ne liez pas d'adresse IP et n'exposez pas de port lorsque vous créez le conteneur.
Comment monter un volume sur un conteneur ?
Un conteneur unique a, de par sa conception, une durée de vie brève. Toutefois, vous pouvez utiliser un volume pour rendre persistantes les données entre redémarrages du conteneur. Lorsque vous montez un volume dans Docker, le volume est monté sur votre système de fichiers local. Dans IBM Bluemix Container Service, l'accès à l'hôte de traitement est restreint, de sorte que vous ne pouvez pas monter des répertoires hôte sur un conteneur. A la place, des volumes avec portée organisation sont utilisés pour rendre persistantes les données entre les redémarrages de conteneurs. Les volumes sont hébergés sur des partages de fichiers isolés assurant un stockage sécurisé des données d'application et qui gèrent l'accès et les autorisations sur les fichiers. Pour monter un volume dans un conteneur, vous devez d'abord en créer un. Lorsque vous créez un conteneur depuis l'interface de ligne de commande, montez un volume existant à l'aide de l'option --volume. A partir de l'interface graphique Bluemix, sélectionnez Options avancées et affectez un volume existant. Pour plus d'informations, voir Ajout de fichiers à des volumes à l'aide de l'interface de ligne de commande.
Comment utiliser Docker Compose avec IBM Bluemix Container Service ?
Vous pouvez utiliser des commandes Docker Compose pour créer et gérer des déploiements multi-conteneurs avec IBM Bluemix Container Service. Pour plus d'informations, voir Tutoriel : Création d'un déploiement multi-conteneur avec Docker Compose et IBM Bluemix Container Service.
Comment configurer mon client Docker Compose pour qu'il pointe sur le service IBM Containers ?
Lorsque vous vous connectez à l'interface de ligne de commande IBM Bluemix Container Service, configurez-la pour exécuter des commandes Docker natives. Pour plus d'informations, voir Connexion au plug-in bx ic.
Comment me connecter à mon conteneur en exécution ? Dois-je y accéder via SSH ?
Vous pouvez utiliser bx ic exec pour vous connecter à un conteneur.
Pourquoi mes conteneurs s'exécutent-ils en mode détaché ?
Par défaut, les conteneurs dans Bluemix s'exécutent en mode détaché, ce qui permet aux conteneurs ayant des processus en cours d'exécution de s'exécuter en arrière-plan sans se fermer. Ce mode d'opération est le modèle pour un environnement de cloud. Vous pouvez toutefois passer au mode attaché en exécutant la commande bx ic attach [--no-stdin] [--sig-proxy] CONTENEUR.

IBM Bluemix Container Service dans Bluemix Dedicated et Bluemix Local

IBM Bluemix Container Service est disponible dans les versions Bluemix Dedicated et Bluemix Local.

Configuration matérielle requise pour IBM Bluemix Container Service dans Bluemix Local

Configuration matérielle requise pour Base
L'installation de base est configurée par défaut avec 64 Go de mémoire pour la capacité de traitement du conteneur.
  1. Votre environnement doit également être conforme à la configuration matérielle requise pour Bluemix Local .
  2. Pour utiliser IBM Bluemix Container Service, chaque configuration requiert un plan de contrôle conforme à l' exigences suivantes. Deux instances sont également requises pour la surveillance et la journalisation, en se conformant à la même configuration matérielle.
    • VMware : 3 serveurs ESX, chacun avec 256 Go de mémoire et 32 coeurs physiques (64 vCore avec Hyper-Threading activé). 4 To de stockage partagé de cluster sont requis entre les 3 serveurs.
Configuration requise pour des augmentations de matériel supplémentaires
Vous pouvez augmenter la capacité de traitement du conteneur par incréments de 16 Go jusqu'à une capacité totale de 256 Go sans matériel supplémentaire en ajoutant jusqu'à 12 machines virtuelles d'expansion. Chaque machine virtuelle doit disposer de 20 Go de mémoire, 4 UC virtuelles et 200 Go d'espace disque. Toutefois, pour chaque 10 machines virtuelles d'expansion déployées sur le cluster, 1 machine virtuelle supplémentaire est requise pour gérer l'architecture réseau. Cette machine virtuelle dédiée à l'architecture réseau requiert 4 UC virtuelles, 16 Go de mémoire et 250 Go d'espace disque.

bx ic attach [--no-stdin] [--sig-proxy] CONTAINER

Zweck
Einen aktiven Container steuern oder die zugehörige Ausgabe anzeigen. Drücken Sie die Tastenkombination STRG + C, um den Container zu verlassen und zu stoppen. Weitere Informationen finden Sie unter dem Befehl attach in der Docker-Hilfe.
Parameter
--no-stdin
(Optional) Die Standardeingabe nicht einschließen.
--sig-proxy
(Optional) Der Standardwert ist true.
CONTAINER
(Erforderlich) Ein Containername oder eine ID.
Beispiel
Bei dem folgenden Beispiel handelt es sich um eine Anforderung zum Anhängen an den Container mein_container.
bx ic attach mein_container

bx ic build -t TAG [--no-cache] [--pull] [-q] DOCKERFILE_LOCATION

Zweck
Erstellen Sie ein Docker-Image lokal oder in der privaten Bluemix-Registry. Weitere Informationen finden Sie unter dem Befehl build in der Docker-Hilfe.
Parameter
DOCKERFILE_LOCATION
(Erforderlich) Der Pfad zur Dockerfile und zu den Kontextdateien auf dem lokalen Host.
--no-cache
(Optional) Wenn ein Image zum ersten Mal erstellt wird, werden die Ebenen für das Image für die gesamte Organisation im Cache gespeichert. Wenn Sie nachfolgende Versionen des Image erstellen, werden diese im Cache gespeicherten Versionen des Image zum Beschleunigen des Erstellungsprozesses verwendet. Wenn Sie beim Erstellen einer neuen Version eines Image die Option --no-cache einschließen, können Sie sicherstellen, dass jede Ebene des Image neu erstellt wird.
--pull
(Optional) Das Basisimage wird standardmäßig nur aus der Registry extrahiert, wenn es noch nicht in den Image-Cache der Organisation gestellt wurde. Wenn Sie diese Option einschließen, können Sie sicherstellen, dass die neueste Version des Basisimage zum Erstellen des neuen Image verwendet wird.
-q, --quiet
(Optional) Die umfangreiche Ausgabe unterdrücken, die von den Containern generiert wird. Der Standardwert ist false.
-t TAG, --tag TAG
(Erforderlich) Der Pfad zu Ihrer privaten Image-Registry, der auf das zu erstellende Image angewendet wird.
Beispiel
Das folgende Beispiel ist eine Anforderung zum Erstellen eines Images mit dem Namen myimage. Die Dockerfile und andere Artefakte, die im Build verwendet werden, befinden sich in demselben Verzeichnis, in dem der Befehl ausgeführt wird. Da die Registry und der Namensbereich im Imagenamen enthalten sind, wird das Image in der privaten Bluemix-Image-Registry Ihrer Organisation erstellt.
Tipp: Führen Sie bx ic namespace-get aus, um Ihren Namensbereich abzurufen, und ersetzen Sie <mein_namensbereich> durch Ihren Namensbereich.
bx ic build -t registry.DomainName/<mein_namensbereich>/mein_image.

bx ic cp SRC_PATH CONTAINER:DEST_PATH | bx ic cp CONTAINER:SRC_PATH DEST_PATH

Zweck
Kopieren einer Datei oder eines Ordners von einem aktiven oder gestoppten Container auf Ihre lokale Maschine und umgekehrt. Weitere Informationen finden Sie unter dem Befehl cp in der Docker-Hilfe.
Anmerkung: Die Unterstützung für diesen Befehl auf die Verwendung mit einzelnen Containern beschränkt. Wenn Sie Dateien aus oder in eine Gruppe von Containern kopieren möchten, müssen Sie die Dateien jeweils aus jeder oder in jede Containerinstanz der Gruppe kopieren.
Parameter
DEST_PATH
(Erforderlich) Der Pfad zu der Datei oder dem Ordner auf Ihrer lokalen Maschine oder in Ihrem Container, wohin die Informationen aus dem Quellenpfad kopiert werden sollen. Die Datei oder der Ordner müssen auf der lokalen Maschine oder im Container vorhanden sein, bevor Sie mit dem Kopieren beginnen können. Wenn Sie auf Ihre lokale Maschine kopieren möchten, können Sie den absoluten oder den relativen Pfad verwenden. Wenn Sie in Ihren einzelnen Container kopieren möchten, müssen Sie den Namen des Containers, das Begrenzungszeichen ":" und den Pfad zu der Zieldatei bzw. zum Zielordner einschließen. Der Pfad muss relativ zum Stammverzeichnis des Containers angegeben werden.
SRC_PATH
(Erforderlich) Der Pfad zur Datei oder zu dem Ordner, den Sie kopieren möchten. Wenn die Datei oder der Ordner sich auf Ihrer lokalen Maschine befindet, können Sie den absoluten oder relativen Pfad verwenden. Wenn sich die Datei oder der Ordner in Ihrem einzelnen Container befindet, müssen Sie den Namen des Containers, das Begrenzungszeichen ":" und den Pfad zu der Datei oder dem Ordner, die bzw. den Sie kopieren möchten, einschließen. Der Pfad muss relativ zum Stammverzeichnis des Containers angegeben werden.
Anmerkung: Sie können nicht bestimmte Systemdateien kopieren, die unter dem Verzeichnis /dev, /etc, /proc und /sys Ihres Containers gespeichert sind.
Beispiel
Dieser Beispielbefehl kopiert eine Datei mit dem Namen config.txt aus dem aktuellen Verzeichnis Ihrer lokalen Maschine in das Verzeichnis config eines einzelnen Containers mit dem Namen mein_container. 
bx ic cp ./config.txt mein_container:/config

bx ic cpi SRC DST

Zweck
Auf ein Image aus Docker Hub oder ein Image aus der lokalen Registry zugreifen und in die private Bluemix-Image-Registry kopieren.
Anmerkung: Dieser Befehl ist eine Alternative zur gemeinsamen Verwendung von docker pull und docker push, um ein Image zu Ihrer Bluemix-Image-Registry hinzuzufügen. Wenn Sie die Verwendung von docker pull und docker push bevorzugen, finden Sie Informationen dazu unter Lokale Images mit Push-Operation über die Befehlszeile in die private Bluemix-Registry übertragen.
Parameter
DST
(Erforderlich) Die private Bluemix-Image-Registry-URL, die den Namensbereich sowie den Namen des Ziel-Images enthält. Ein Tag für das Image ist optional.
SRC
(Erforderlich) Das Quellenrepository und der Imagename.
Beispiel
bx ic cpi quellenrepository/name_des_quellenimages private_registry_URL/name_des_zielimages:tag
Tipp: Führen Sie bx ic namespace-get aus, um Ihren Namensbereich abzurufen, und ersetzen Sie <mein_namensbereich> durch Ihren Namensbereich.
bx ic cpi training/sinatra registry.DomainName/<mein_namensbereich>/mein_sinatra:v1

bx ic exec [-d] [-it ] [-u USER] CONTAINER CMD

Zweck
Einen Befehl in einem Container ausführen. Weitere Informationen finden Sie unter dem Befehl exec in der Docker-Hilfe.
Parameter
CMD
(Erforderlich) Der Befehl, der in den angegebenen Containern ausgeführt werden soll.
CONTAINER
(Erforderlich) Ein Containername oder eine ID.
-d , --detach
(Optional) Den angegebenen Befehl im Hintergrund ausführen.
-it
(Optional) Interaktiver Modus. Die Standardeingabe soll weiterhin angezeigt werden. Geben Sie zum Verlassen exit ein.
-u USER, --user USER
(Optional) Ein Benutzername.
Beispiele
Dieser Beispielbefehl wird in einem interaktiven Bash-Terminal im Container mein_container ausgeführt.
bx ic exec -it mein_container bash
Dieser Beispielbefehl führt den Datumsbefehl im Container mein_container aus. 
bx ic exec mein_container date

bx ic group-create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session-affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

Zweck
Erstellung einer Skalierungsgruppe. Standardmäßig werden Container in Bluemix im Modus mit aufgehobener Zuordnung ausgeführt.
Parameter
--anti
(Optional) Bewirkt die Verwendung von Anti-Affinität, um die Hochverfügbarkeit Ihrer Containergruppe zu steigern. Durch die Option --anti wird erzwungen, dass jede Containerinstanz in Ihrer Gruppe auf einem separaten physischen Rechenknoten angeordnet werden muss, was das Risiko eines Absturzes aller Container in einer Gruppe infolge eines Hardwarefehlers verringert. Diese Option kann in Verbindung mit größeren Gruppengrößen möglicherweise nicht verwendet werden, da für jede Bluemix-Region und -Organisation nur eine begrenzte Anzahl von Rechenknoten für die Bereitstellung verfügbar ist. Falls Ihre Bereitstellung nicht erfolgreich ist, reduzieren Sie entweder die Anzahl der Containerinstanzen in der Gruppe oder entfernen Sie die Option --anti.
--auto
(Optional) Wenn die automatische Wiederherstellung für eine Containergruppe aktiviert ist und die Gruppe 10 Minuten lang ausgeführt wurde, beginnt die Lastausgleichsfunktion der Gruppe mit der regelmäßigen Prüfung des Status jeder Containerinstanz in der Gruppe durch HTTP-Anforderungen. Wenn eine Containerinstanz nicht innerhalb von 100 Sekunden antwortet, wird sie als inaktiv markiert. Inaktive Containerinstanzen werden durch die automatische Wiederherstellung aus der Gruppe entfernt und erneut erstellt. Die automatische Wiederherstellung versucht dreimal, Containerinstanzen in einer Gruppe wiederherzustellen. Nach dem dritten Versuch wird die automatische Wiederherstellung für Containerinstanzen für 60 Minuten ausgesetzt. Nach 60 Minuten wird der automatische Wiederherstellungsprozess wieder gestartet.Weitere Informationen zur Statusprüfung können Sie den Erläuterungen zur Option --http_monitor_enabled entnehmen.
CMD
(Optional) Der Befehl und die Argumente werden zum Ausführen an die Containergruppe übergeben. Bei dem Befehl muss es sich um einen Befehl mit langer Laufzeit handeln. Verwenden Sie keinen Befehl mit kurzer Laufzeit, z. B. /bin/date. Dies kann dazu führen, dass der Container abstürzt.
Beispiel für Befehle mit langer Laufzeit, die das Image ibmnode verwenden, das von IBM Bluemix Container Service bereitgestellt wird: 
bx ic group-create --name meine_containergruppe registry.DomainName/ibmnode ping localhost
bx ic group-create --name meine_containergruppe registry.DomainName/ibmnode -- tail -f /dev/null
Anmerkung: CMD und die zugehörigen Argumente müssen sich am Ende der Befehlszeile bx icgroup-create befinden.
-d DOMAIN, --domain DOMAIN
(Optional) Die Standardsystemdomäne ist AppDomainName und stellt bereits ein SSL-Zertifikat bereit, sodass Sie auf die Containergruppen mit HTTPS ohne eine weitere Konfiguration zugreifen können. Wenn Sie eine angepasste Domäne verwenden möchten, müssen Sie die angepasste Domäne bei einem öffentlichen DNS-Server registrieren, die angepasste Domäne in Bluemix konfigurieren und die angepasste Domäne anschließend der Systemdomäne von Bluemix auf dem öffentlichen DNS-Server zuweisen. Nachdem die angepasste Domäne der Bluemix-Systemdomäne zugeordnet wurde, werden Anforderungen für die angepasste Domäne an die Anwendung in Bluemix weitergeleitet. Bei Erstellung einer angepassten Domäne darf der Domänenname keine Unterstreichungszeichen (_) enthalten. Weitere Informationen zu angepassten Domänen finden Sie unter Angepasste Domäne erstellen und verwenden. Wenn Sie die angepasste Domäne schützen möchten, laden Sie ein SSL-Zertifikat hoch, sodass auf die Containergruppen über HTTPS zugegriffen werden kann.

Der Host und die Domäne werden zu einer vollständigen URL für eine öffentliche Route kombiniert, zum Beispiel http://meincontainerhost.AppDomainName, die in Bluemix eindeutig sein muss. Wenn Sie die Details einer Containergruppe mit dem Befehl bx ic group-inspect überprüfen, werden der Host und die Domäne zusammen als die Route aufgelistet.

--desired DESIRED
(Optional) Die Anzahl von Instanzen in Ihrer Gruppe. Der Standardwert ist 2. Welche Anzahl von Containergruppeninstanzen Ihre App erfordert, richtet sich nach der erwarteten Arbeitslast und dem Zeitrahmen, in dem eine Anforderung von der App verarbeitet werden kann. Multitasking oder Netzkonnektivität wirken sich beispielsweise darauf aus, wie viele Instanzen für Ihre App erforderlich sind. Sie bestimmen die Anzahl der gewünschten Instanzen, indem Sie einen einzelnen Container in Bluemix bereitstellen und für diesen einen Belastungstest ausführen. Wenn die App zum Beispiel in der Lage ist, 100 Anforderungen pro Sekunde zu verarbeiten, jedoch eine durchschnittliche Arbeitslast von 300 Anforderungen pro Sekunde zu erwarten ist, so beläuft sich die gewünschte Anzahl von Instanzen für die Containergruppe auf 3. Um Ihre Containergruppe hoch verfügbar zu machen und mit erhöhter Ausfallsicherheit zu versehen, sollten Sie mehr Instanzen als das Minimum einbeziehen, damit die erwartete Arbeitslast verarbeitet werden kann. Zusätzliche Instanzen können die Arbeitslast übernehmen, falls andere Instanzen ausfallen. Um die automatische Wiederherstellung einer fehlgeschlagenen Instanz ohne Auswirkung auf die Arbeitslast sicherzustellen, beziehen Sie eine zusätzliche Instanz ein. Zum Schutz vor zwei gleichzeitigen Ausfällen beziehen Sie zwei zusätzliche Instanzen ein. Diese Konfiguration folgt dem Muster N+2, wobei N für die Anzahl der Instanzen steht, die zur Verarbeitung der Anforderungen dienen, und +2 für die beiden zusätzlichen Instanzen.
Wichtig: Eine Containergruppe sollte mindestens zwei Containerinstanzen enthalten. Wenn Sie lediglich eine Instanz in die Gruppe einbeziehen und diese Instanz irgendwann nicht verfügbar ist, treten bei den Benutzern möglicherweise Ausfallzeiten für die App auf.
-e ENV, --env ENV
(Optional) Umgebungsvariable festlegen, wobei ENV ein Name/Wert-Paar im Format Schlüssel=Wert ist. Listen Sie mehrere Schlüssel separat auf und schließen Sie bei Verwendung von Anführungszeichen sowohl den Namen der Umgebungsvariablen als auch den Wert ein. Beispiel: -e "schlüssel1=wert1" -e "schlüssel2=wert2a,wert2b" -e "schlüssel3=wert3"

--env-file ENVFILE
(Optional) Umgebungsvariablen aus einer Datei importieren, wobei ENVFILE der Pfad zu der Datei im lokalen Verzeichnis ist. Jede Zeile in der Datei stellt ein Paar im Format Schlüssel=Wert dar.

Beispiel

SCHLÜSSEL2=WERT2
--http_monitor_enabled
(Optional) Der Standardwert ist true. Wenn die automatische Wiederherstellung für eine Containergruppe aktiviert ist und die Gruppe 10 Minuten lang ausgeführt wurde, beginnt die Lastausgleichsfunktion der Gruppe mit der regelmäßigen Prüfung des Status jeder Containerinstanz in der Gruppe durch HTTP-Anforderungen. Wenn eine Containerinstanz nicht innerhalb von 100 Sekunden antwortet, wird sie als inaktiv markiert. Inaktive Containerinstanzen werden durch die automatische Wiederherstellung aus der Gruppe entfernt und erneut erstellt. Die automatische Wiederherstellung versucht dreimal, Containerinstanzen in einer Gruppe wiederherzustellen. Nach dem dritten Versuch wird die automatische Wiederherstellung für Containerinstanzen für 60 Minuten ausgesetzt. Nach 60 Minuten wird der automatische Wiederherstellungsprozess wieder gestartet. Bei einigen Apps, z. B. Apps, die TCP anstelle von HTTP für die Kommunikation verwenden, ist eine HTTP-Statusprüfung nicht erforderlich. Inaktivieren Sie die HTTP-Überwachung, indem Sie für diese Option den Wert false angeben. Beispiel: --http_monitor_enabled=false
--http_monitor_path
(Optional) Der Standardwert ist /. Ist die Option --http_monitor_enabled aktiviert, können Sie mit dieser Option einen angepassten Pfad für die Statusprüfung angeben, z. B. den Pfad --http_monitor_path="/health/path".
--http_monitor_rc_list
(Optional) Ist die Option --http_monitor_enabled aktiviert, können Sie die Liste der Antwortcodes für die Statusprüfung angeben. Der Standardwert umfasst die Codes 200, 201, 202, 204, 300, 301, 302, 401, 403 und 404. Jede Antwort, die nicht in der Liste der gültigen Antworten enthalten ist, ändert den Status der Containerinstanz durch die Lastausgleichsfunktion in INACTIVE. Ein inaktiver Container erhält keine Last und der Status der Containergruppe ändert sich in HAS_INACTIVE_INSTANCES. Wenn keiner der Container den Status ACTIVE aufweist, dann wechselt der Status der Containergruppe zu NO_ACTIVE_INSTANCES. Weitere Informationen zum Status finden Sie in Containerstatus in IBM Bluemix Container Service.
IMAGE
(Erforderlich) Das Image, das in die einzelnen Containerinstanzen in der Containergruppe aufgenommen werden soll. Sie können Befehle im Anschluss an das Image auflisten, jedoch keine Optionen. Geben Sie alle Optionen an, bevor Sie das Image angeben.

Verwenden Sie bei einem Image in der privaten Bluemix-Image-Registry der Organisation das folgende Format zum Angeben des Images: registry.DomainName/NAMESPACE/IMAGE

Geben Sie bei einem von IBM Bluemix Container Service bereitgestellten Image nicht den Namensbereich Ihrer Organisation an. Geben Sie das Image im folgenden Format an: registry.DomainName/IMAGE

--ip IP-ADRESSE
(Optional) Falls eine IP-Adresse verfügbar ist, binden Sie eine öffentliche IP-Adresse an Ihre Containergruppe. Mit dem Flag -p müssen Sie auch den Port zugänglich machen. Eine Containergruppe kann entweder eine öffentliche Route oder eine öffentliche IP-Adresse, jedoch nicht beides haben. Um die Bindung der IP-Adresse aufzuheben und die IP-Adresse an das Kontingent des Bereichs zurückzugeben, entfernen Sie die Containergruppe und erstellen anschließend die Gruppe erneut ohne die IP-Adresse. Um vor der Erstellung einer Containergruppe eine IP-Adresse anzufordern, führen Sie den Befehl bx ic ip-request aus.
Anmerkung: Das Binden einer öffentlichen IP-Adresse schließt kein SSL-Zertifikat ein, mit dem die Daten verschlüsselt werden, die an Ihre und von Ihrer Containergruppe gesendet werden. Falls Ihre App eine SSL-Verschlüsselung erfordert, können Sie entweder eine eigene SSL-Lösung implementieren oder Ihrer Containergruppe eine öffentliche Route zuordnen, statt eine öffentliche IP-Adresse zu binden. Da öffentliche Routes bereits ein SSL-Zertifikat enthalten, können Sie ohne weitere Konfiguration über HTTPS auf die Containergruppe zugreifen.
-m MEMORY, --memory MEMORY
(Optional) Geben Sie eine Speicherbegrenzung für Ihren Container in MB ein. Die Speicherbegrenzung ist Teil der Containergröße, die den Maximalwert an Hauptspeicher und Plattenspeicher definiert, den ein Container zur Laufzeit auf dem Rechenhost erhält. Eine zugewiesene Containergröße kann nicht mehr geändert werden. Folgende Größen sind in IBM Bluemix Container Service verfügbar:
  • Pico (64 MB Hauptspeicher, 4 GB Plattenspeicher)
  • Nano (128 MB Hauptspeicher, 8 GB Plattenspeicher)
  • Micro (256 MB Hauptspeicher, 16 GB Plattenspeicher)
  • Tiny (512 MB Hauptspeicher, 32 GB Plattenspeicher)
  • S (Klein) (1024 MB Hauptspeicher, 64 GB Plattenspeicher)
  • M (Mittel) (2048 MB Hauptspeicher, 128 GB Plattenspeicher)
  • L (Groß) (4096 MB Hauptspeicher, 256 GB Plattenspeicher)
  • XL (8192 MB Hauptspeicher, 512 GB Plattenspeicher)
  • XXL (16384 MB Hauptspeicher, 1 TB Plattenspeicher)
Wenn Sie keine Größe für Ihren Container festlegen, wird jede Containerinstanz mit einer Größe von 256 MB erstellt.
Wichtig: Geben Sie den Containerspeicher in MB ohne die Einheitenbezeichnung ein. Wenn Sie zum Beispiel einen Container der Größe 'Pico' erstellen möchten, geben Sie -m 64 ein.
--max MAX
(Optional) Die maximale Anzahl von Instanzen. Der Standardwert ist 2.
--min MIN
(Optional) Die minimale Anzahl von Instanzen. Der Standardwert ist 1.
-n HOST, --hostname HOST
(Optional) Der Hostname, z. B. meincontainerhost. Der Hostname darf keine Unterstreichungszeichen (_) enthalten. Der Host und die Domäne werden zu einer vollständigen URL für eine öffentliche Route kombiniert, zum Beispiel http://meincontainerhost.AppDomainName. Wenn Sie die Details einer Containergruppe mit dem Befehl bx ic group-inspect überprüfen, werden der Host und die Domäne zusammen als die Route aufgelistet.
--name NAME
(Erforderlich) Der Gruppe einen Namen zuordnen. Das Argument -n wird nicht mehr verwendet.

Zweck
Dieser Befehl listet alle aktiven und gestoppten Container des Containersystems auf. Weitere Informationen finden Sie unter dem Befehl docker-compose ps in der Docker-Hilfe.
Unterstützte Befehlsoptionen
-q

docker-compose ps

docker-compose rm [SERVICE]

Zweck
Bei Angabe eines Service entfernt dieser Befehl die Containerinstanz, die diesen Service ausführt. Wenn Sie keinen Service angeben, wird das gesamte Containersystem entfernt. In beiden Fällen ist es wichtig, dass der Service vor dem Entfernen des Containers gestoppt wird; ansonsten wird ein Fehler angezeigt. Weitere Informationen finden Sie unter dem Befehl docker-compose rm in der Docker-Hilfe.
Parameter
SERVICE
(Optional) Der Name des Container-Service, der entfernt werden soll.
Anmerkung: Der Name des Service ist in der Datei docker-compose.yml enthalten. Verwenden Sie nicht den Namen des Containers, der den Service ausführt; dieser wurde bei Ausführung des Befehls docker-compose up erstellt.
Beispiel
Bei dem folgenden Beispiel handelt es sich um eine Anforderung zum Entfernen des Containers, der den Service mit dem Namen mein_service ausführt.
docker-compose rm mein_service

docker-compose scale SERVICE=NUM

Zweck
Mit diesem Befehl kann die Anzahl der Containerinstanzen, die einen Service ausführen, erhöht oder reduziert werden. Es wird empfohlen, mehrere Containerinstanzen zu verwenden, um die Skalierbarkeit und Zuverlässigkeit des Service zu gewährleisten. Weitere Informationen finden Sie unter dem Befehl docker-compose rm in der Docker-Hilfe.
Parameter
SERVICE
(Erforderlich) Der Name des Service, für den die Anzahl der Container erhöht oder reduziert werden soll.
Anmerkung: Der Name des Service ist in der Datei docker-compose.yml enthalten. Verwenden Sie nicht den Namen des Containers, der den Service ausführt; dieser wurde bei Ausführung des Befehls docker-compose up erstellt.
NUM
(Erforderlich) Die Anzahl der Container, die Sie für den Service erstellen möchten.
Beispiel
Bei dem folgenden Beispiel handelt es sich um eine Anforderung zum Erstellen von drei Containerinstanzen für einen Service mit dem Namen mein_service.
docker-compose scale mein_service=3

docker-compose start [SERVICE]

Zweck
Bei Angabe eines Service startet dieser Befehl alle Containerinstanzen, die mit diesem Service verknüpft sind. Wenn Sie keinen Service angeben, werden alle Containerinstanzen in Ihrem Containersystem gestartet.
Anmerkung: Dieser Befehl kann nur verwendet werden, wenn bereits Containerinstanzen mit dem Befehl docker-compose up erstellt wurden.
Weitere Informationen finden Sie unter dem Befehl docker-compose start in der Docker-Hilfe.
Parameter
SERVICE
(Optional) Der Name des Service, der gestartet werden soll.
Anmerkung: Der Name des Service ist in der Datei docker-compose.yml enthalten. Verwenden Sie nicht den Namen des Containers, der den Service ausführt; dieser wurde bei Ausführung des Befehls docker-compose up erstellt.
Beispiel
Bei dem folgenden Beispiel handelt es sich um eine Anforderung zum Starten eines Service mit dem Namen mein_service.
docker-compose start mein_service

docker-compose stop [-t] [SERVICE]

Zweck
Bei Angabe eines Service stoppt dieser Befehl alle Containerinstanzen, die diesen Service ausführen. Wenn Sie keinen Service angeben, werden alle Containerinstanzen in Ihrem Containersystem gestoppt. Gestoppte Container-Services werden nicht entfernt und können erneut gestartet werden. Weitere Informationen finden Sie unter dem Befehl docker-compose stop in der Docker-Hilfe.
Parameter
SERVICE
(Optional) Der Name des aktiven Service, der gestoppt werden soll.
Anmerkung: Der Name des Service ist in der Datei docker-compose.yml enthalten. Verwenden Sie nicht den Namen des Containers, der den Service ausführt; dieser wurde bei Ausführung des Befehls docker-compose up erstellt.
Unterstützte Befehlsoptionen
-t, --timeout
(Optional) Mit der Option -t oder --timeout kann das Standardzeitlimit bei Systemabschluss geändert werden. Das Zeitlimit bei Systemabschluss ist die Zeit, die der Docker-Dämon auf die Antwort der Containerinstanz warten muss, dass diese gestoppt wird. Das Standardzeitlimit bei Systemabschluss beträgt 10 Sekunden.
Beispiel
Bei dem folgenden Beispiel handelt es sich um eine Anforderung zum Stoppen eines Service mit dem Namen mein_service.
docker-compose stop mein_service

docker-compose up [-d]

Zweck
Dieser Befehl erstellt, startet und verknüpft die Containerinstanzen für die Services, die Sie in der Datei docker-compose.yml definiert haben. Wenn die Multi-Container-Bereitstellung bereits aktiv ist, sucht dieser Befehl nach Aktualisierungen, die Sie in der Datei docker-compose.yml definiert haben, und erstellt nur die Container neu, deren Konfiguration geändert wurde. Weitere Informationen finden Sie unter dem Befehl docker-compose up in der Docker-Hilfe.
Unterstützte Befehlsoptionen
-d
(Optional) Sie können den Befehl docker-compose up entweder interaktiv ausführen, sodass Sie alle Schritte während der Containererstellung verfolgen können, oder die Option -d verwenden, um die Container im Hintergrund zu starten und auszuführen. In IBM Bluemix Container Service stellen die meisten Services und Apps Tasks mit langer Laufzeit dar. Verwenden Sie aus diesem Grund die Option -d, um zu verhindern, dass Sie in der Konsolenausgabe der Container hängen bleiben, ohne die Zuordnung aufheben zu können. Bei Verwendung der Option -d werden Containernamen nach der Erstellung weiterhin in der CLI angezeigt.
Beispiel
Bei dem folgenden Beispiel handelt es sich um eine Anforderung zum Erstellen einer Multi-Container-Bereitstellung mit den zugehörigen Abhängigkeiten. 
docker-compose up

cf ic group create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session_affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

Zweck
Erstellung einer Skalierungsgruppe. Standardmäßig werden Container in Bluemix im Modus mit aufgehobener Zuordnung ausgeführt.
Parameter
--anti
(Optional) Bewirkt die Verwendung von Anti-Affinität, um die Hochverfügbarkeit Ihrer Containergruppe zu steigern. Durch die Option --anti wird erzwungen, dass jede Containerinstanz in Ihrer Gruppe auf einem separaten physischen Rechenknoten angeordnet werden muss, was das Risiko eines Absturzes aller Container in einer Gruppe infolge eines Hardwarefehlers verringert. Diese Option kann in Verbindung mit größeren Gruppengrößen möglicherweise nicht verwendet werden, da für jede Bluemix-Region und -Organisation nur eine begrenzte Anzahl von Rechenknoten für die Bereitstellung verfügbar ist. Falls Ihre Bereitstellung nicht erfolgreich ist, reduzieren Sie entweder die Anzahl der Containerinstanzen in der Gruppe oder entfernen Sie die Option --anti.
--auto
(Optional) Wenn die automatische Wiederherstellung für eine Containergruppe aktiviert ist und die Gruppe 10 Minuten lang ausgeführt wurde, beginnt die Lastausgleichsfunktion der Gruppe mit der regelmäßigen Prüfung des Status jeder Containerinstanz in der Gruppe durch HTTP-Anforderungen. Wenn eine Containerinstanz nicht innerhalb von 100 Sekunden antwortet, wird sie als inaktiv markiert. Inaktive Containerinstanzen werden durch die automatische Wiederherstellung aus der Gruppe entfernt und erneut erstellt. Die automatische Wiederherstellung versucht dreimal, Containerinstanzen in einer Gruppe wiederherzustellen. Nach dem dritten Versuch wird die automatische Wiederherstellung für Containerinstanzen für 60 Minuten ausgesetzt. Nach 60 Minuten wird der automatische Wiederherstellungsprozess wieder gestartet.Weitere Informationen zur Statusprüfung können Sie den Erläuterungen zur Option --http_monitor_enabled entnehmen.
CMD
(Optional) Der Befehl und die Argumente werden zum Ausführen an die Containergruppe übergeben. Bei dem Befehl muss es sich um einen Befehl mit langer Laufzeit handeln. Verwenden Sie keinen Befehl mit kurzer Laufzeit, z. B. /bin/date. Dies kann dazu führen, dass der Container abstürzt.
Beispiel für Befehle mit langer Laufzeit, die das Image ibmnode verwenden, das von IBM Bluemix Container Service bereitgestellt wird: 
cf ic group create --name meine_containergruppe registry.DomainName/ibmnode ping localhost
cf ic group create --name meine_containergruppe registry.DomainName/ibmnode -- tail -f /dev/null
Anmerkung: CMD und die zugehörigen Argumente müssen sich am Ende der Befehlszeile cf ic group create befinden.
-d DOMAIN, --domain DOMAIN
(Optional) Die Standardsystemdomäne ist AppDomainName und stellt bereits ein SSL-Zertifikat bereit, sodass Sie auf die Containergruppen mit HTTPS ohne eine weitere Konfiguration zugreifen können. Wenn Sie eine angepasste Domäne verwenden möchten, müssen Sie die angepasste Domäne bei einem öffentlichen DNS-Server registrieren, die angepasste Domäne in Bluemix konfigurieren und die angepasste Domäne anschließend der Systemdomäne von Bluemix auf dem öffentlichen DNS-Server zuweisen. Nachdem die angepasste Domäne der Bluemix-Systemdomäne zugeordnet wurde, werden Anforderungen für die angepasste Domäne an die Anwendung in Bluemix weitergeleitet. Bei Erstellung einer angepassten Domäne darf der Domänenname keine Unterstreichungszeichen (_) enthalten. Weitere Informationen zu angepassten Domänen finden Sie unter Angepasste Domäne erstellen und verwenden. Wenn Sie die angepasste Domäne schützen möchten, laden Sie ein SSL-Zertifikat hoch, sodass auf die Containergruppen über HTTPS zugegriffen werden kann.

Der Host und die Domäne werden zu einer vollständigen URL für eine öffentliche Route kombiniert, zum Beispiel http://meincontainerhost.AppDomainName, die in Bluemix eindeutig sein muss. Wenn Sie die Details einer Containergruppe mit dem Befehl cf ic group inspect überprüfen, werden der Host und die Domäne zusammen als die Route aufgelistet.

--desired DESIRED
(Optional) Die Anzahl von Instanzen in Ihrer Gruppe. Der Standardwert ist 2. Welche Anzahl von Containergruppeninstanzen Ihre App erfordert, richtet sich nach der erwarteten Arbeitslast und dem Zeitrahmen, in dem eine Anforderung von der App verarbeitet werden kann. Multitasking oder Netzkonnektivität wirken sich beispielsweise darauf aus, wie viele Instanzen für Ihre App erforderlich sind. Sie bestimmen die Anzahl der gewünschten Instanzen, indem Sie einen einzelnen Container in Bluemix bereitstellen und für diesen einen Belastungstest ausführen. Wenn die App zum Beispiel in der Lage ist, 100 Anforderungen pro Sekunde zu verarbeiten, jedoch eine durchschnittliche Arbeitslast von 300 Anforderungen pro Sekunde zu erwarten ist, so beläuft sich die gewünschte Anzahl von Instanzen für die Containergruppe auf 3. Um Ihre Containergruppe hoch verfügbar zu machen und mit erhöhter Ausfallsicherheit zu versehen, sollten Sie mehr Instanzen als das Minimum einbeziehen, damit die erwartete Arbeitslast verarbeitet werden kann. Zusätzliche Instanzen können die Arbeitslast übernehmen, falls andere Instanzen ausfallen. Um die automatische Wiederherstellung einer fehlgeschlagenen Instanz ohne Auswirkung auf die Arbeitslast sicherzustellen, beziehen Sie eine zusätzliche Instanz ein. Zum Schutz vor zwei gleichzeitigen Ausfällen beziehen Sie zwei zusätzliche Instanzen ein. Diese Konfiguration folgt dem Muster N+2, wobei N für die Anzahl der Instanzen steht, die zur Verarbeitung der Anforderungen dienen, und +2 für die beiden zusätzlichen Instanzen.
Wichtig: Eine Containergruppe sollte mindestens zwei Containerinstanzen enthalten. Wenn Sie lediglich eine Instanz in die Gruppe einbeziehen und diese Instanz irgendwann nicht verfügbar ist, treten bei den Benutzern möglicherweise Ausfallzeiten für die App auf.
-e ENV, --env ENV
(Optional) Umgebungsvariable festlegen, wobei ENV ein Name/Wert-Paar im Format Schlüssel=Wert ist. Listen Sie mehrere Schlüssel separat auf und schließen Sie bei Verwendung von Anführungszeichen sowohl den Namen der Umgebungsvariablen als auch den Wert ein. Beispiel: -e "schlüssel1=wert1" -e "schlüssel2=wert2a,wert2b" -e "schlüssel3=wert3"

(Optional) Der Standardwert ist true. cf ic wait CONTAINER [CONTAINER]

Zweck
Warten, bis ein Container beendet ist. Während dieser Wartezeit gibt die Befehlszeile keine Antworten zurück und Sie können auch keine Befehle eingeben. Nach dem Beenden oder Stoppen des Containers wird der Beendigungscode in der Ausgabe der Befehlszeilenschnittstelle angezeigt. Weitere Informationen finden Sie unter dem Befehl wait in der Docker-Hilfe.
Parameter
CONTAINER
(Erforderlich) Der Containername oder die ID.
Beispiel
Bei dem folgenden Beispiel handelt es sich um eine Anforderung zum Warten auf die Beendigung eines Containers mit dem Namen mein_container. Nach dem Beenden des Containers wird der Beendigungscode in der Ausgabe der Befehlszeilenschnittstelle angezeigt. 
cf ic wait mein_container

cf ic wait-status CONTAINER

Zweck
Warten, bis ein einzelner Container oder eine Containergruppe in einen nicht-transienten Status wechselt. Während dieser Wartezeit gibt die Befehlszeile keine Antworten zurück und Sie können auch keine Befehle eingeben. Nach dem Wechsel des Containers in einen nicht-transienten Status wird eine Bestätigungsnachricht angezeigt. Bei einzelnen Containern lauten die nicht-transienten Status Running (Aktiv), Shutdown (Heruntergefahren), Crashed (Ausgefallen), Paused (Angehalten) und Suspended (Ausgesetzt). Bei Containergruppen lauten die nicht-transienten Status CREATE_COMPLETE, UPDATE_COMPLETE und FAILED.
Parameter
CONTAINER
(Erforderlich) Der Container- oder Gruppenname oder die ID.
Beispiel
Bei dem folgenden Beispiel handelt es sich um eine Anforderung zum Warten auf den Wechsel eines Containers mit dem Namen mein_container in einen nicht-transienten Status. 
cf ic wait-status mein_container

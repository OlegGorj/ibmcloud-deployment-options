Sicherheit für einzelne und skalierbare Container

IBM® Bluemix Container Service bietet eine vertrauenswürdige und sichere Cloudplattform, die Sicherheitsfunktionen von Linux und Docker optimiert. Wenn Sie Container mit IBM Bluemix Container Service erstellen und ausführen, profitiert Ihre App von der Container- und Netzisolation, von dem eingeschränkten Zugriff auf Hostdaten und -ressourcen und von einer Bereitstellung sicherer Container.
Achtung: Kostenlose Testversionen für einzelne und skalierbare Container sind nicht verfügbar. Lesen Sie die Informationen zum Cluster-Management und zur Migration auf Kubernetes in IBM Bluemix Container Service unter Bluemix Public.

Sicherheitseinstellung auf dem IBM Bluemix Container Service-Rechenhost

IBM Bluemix Container Service optimiert die Sicherheitsfunktionalität des Linux-Kernels und von Docker, um die Containerisolation sicherzustellen und um den Zugriff auf die Ressourcen des Rechenhosts einzuschränken.

Vollständig verwaltete und geschützte Docker-Dämonprozesse
Die Docker-Dämonprozesse, die auf den IBM Rechenhosts ausgeführt werden, werden ohne direkten Benutzerzugriff eingerichtet und können nur von IBM konfiguriert werden. Alle Docker-Dämonsockets werden mit TLS-Zertifikaten geschützt.
Docker-Benutzernamensbereiche aktiviert
Die Funktion für Docker-Benutzernamensbereiche wird in der Docker Engine auf den Rechenhosts aktiviert. Die Funktion für Benutzernamensbereiche isoliert Prozesse, die innerhalb eines Containers ausgeführt werden, von allen Prozessen und Containern, die auf demselben Rechenhost ausgeführt werden. Der effektive Rootbenutzer innerhalb des Containers wird zu einem Benutzer ohne Rootberechtigung auf dem Rechenhost zugeordnet. Auch wenn der Rootbenutzer über den Container hinausgehen könnte, hätte der Benutzer keine Berechtigung zur Bearbeitung von Dateien, Prozessen oder anderen Containern, die auf dem Rechenhost ausgeführt werden.
Keine privilegierten Container zugelassen
Die Option --privileged des Befehls run wird nicht unterstützt. Container haben keinen Zugriff auf Geräte und Festplatten auf dem Rechenhost.
Verwaltete Steuergruppen (cgroups) für Speicher, CPU und Festplattenspeicher
Container, die auf demselben Rechenhost ausgeführt werden, nutzen die den Speicher, die CPU und den Festplattenspeicher des Hosts gemeinsam. Um zu verhindern, dass ein Container alle verfügbaren Ressourcen verbraucht oder den Absturz von Anwendungen oder Hostsystemen verursacht, besteht eine Begrenzung der maximalen Menge an Speicher, CPU und Festplattenspeicher, die einem Container zugeordnet werden kann.

Die Menge an Speicher, CPU und Festplattenspeicher, die für einen Container während der Laufzeit verfügbar ist, wird durch die Containergröße bestimmt, die Sie bei der Containererstellung festlegen. Diese Begrenzung stellt sicher, dass jeder Container auf dem Rechenhost den Anteil an den verfügbaren Hostressourcen erhält, der ihm zusteht. Wenn sich ein Container nicht mehr regelkonform verhält, sorgt die Deckelung der Ressourcen doch für eine konsistente Leistung und ein konsistentes Verhalten für andere Container auf dem Rechenhost.

Optimierung der Kernel der Rechenhosts
Kernels der Rechenhosts begrenzen die Gesamtzahl der Threads und Prozesse, die auf dem Rechenhost von einem bestimmten Benutzer oder von einer bestimmten Gruppe ausgeführt werden können. Diese Optimierung stellt sicher, dass der Host nicht überlastet wird und jederzeit über genügend Prozessslots verfügt, bis das Maximum des Benutzers erreicht wird.
Kontinuierliche Überwachung der Rechenhosts
Alle Rechenhosts werden kontinuierlich von IBM überwacht, um Fork-Bombs und andere Denial-of-Service-Attacken auf Prozessebene steuern und abwehren zu können.
AppArmor-Profil für Docker-Container aktiviert
Jeder Rechenhost wendet das IBM spezifische AppArmor-Profil an, um die Umgebung zu schützen und um den Funktionsspielraum eines Containers auf dem Rechenhost einzuschränken. AppArmor ist ein Linux-Kernel-Sicherheitsmodul, das den Zugriff auf Ordner, Dateien, Netzdomänen und Berechtigungen zum Erstellen und Ändern von Daten steuert.
Funktion zum Anhängen lokaler Datenträger an den Host inaktiviert
IBM Bluemix Container Service bietet keine Unterstützung für das Anhängen (Mounting) lokaler Datenträger oder lokaler Verzeichnisse/Hostverzeichnisse an einen Container. Stattdessen werden organisationsbezogene Datenträger verwendet, um Daten zwischen Containerneustarts zu erhalten und um Daten für mehrere Container zur Verfügung zu stellen. Datenträger werden auf isolierten Dateifreigaben gehostet, die App-Daten sicher speichern und den Dateizugriff und Dateiberechtigungen verwalten.

Organisationsbezogene Datenträger müssen erstellt werden, bevor Sie sie an Ihren Container anhängen können. Sie können einen Datenträger über die Bluemix-GUI oder die Befehlszeilenschnittstelle erstellen.

Sicherheitseinstellungen im privaten IBM Bluemix Container Service-Containernetz

Sie können die Liste der integrierten IBM Bluemix Container Service-Funktionen anzeigen, die eine sichere Kommunikation zwischen Containern in demselben privaten Netz, zwischen Containern in verschiedenen Bluemix-Bereichen und zwischen Ihrem Bluemix-Bereich und einem fernen Netz ermöglichen.

Privates L2-isoliertes Containernetz pro Bereich
Ein privates Netz zwischen Containern schafft eine isolierte und sichere Umgebung für die einzelnen Container und Containergruppen, die in einem Bereich ausgeführt werden. Container, die mit demselben privaten Netz verbunden sind, können unter Verwendung der privaten IP-Adressen Daten an andere Container im privaten Netz senden und von anderen Containern im privaten Netz empfangen. Container sind erst öffentlich verfügbar, nachdem ein öffentlicher Port und entweder eine öffentliche IP-Adresse für einzelne Container oder eine öffentliche Route für Containergruppen gebunden wurden.

In Bluemix wird jeder Bereich mit einem privaten Containernetz bereitgestellt, in dem die IBM Bluemix Container Service-Standardnetzeinstellungen angewendet werden. Wenn Sie einen Container erstellen, wird der Container automatisch mit dem privaten Standardnetz für den Bereich verbunden und es wird eine private IP-Adresse zugeordnet.

Beibehaltung privater Container oder selektive Zugänglichkeit öffentlicher Services über Netzsicherheitsgruppen
Alle Container werden durch IBM spezifische Netzsicherheitsgruppen geschützt, die den eingehenden Netzverkehr steuern und als Software-Firewall auf der Ebene des privaten Netzgateways dienen. Daten, die über das private Netzgateway gesendet werden, werden auf ihre Konformität mit den Network Security Group-Regeln geprüft und können entweder an den Empfängercontainer übergeben oder aber blockiert werden.

Wenn Sie einen Container erstellen und ihn nicht öffentlich zugänglich machen, wird er mit der standardmäßigen privaten Netzsicherheitsgruppe konfiguriert. Diese Sicherheitsgruppe ermöglicht die Kommunikation zwischen Containern im privaten Netz unter Verwendung der privaten IP-Adressen. Standardmäßig werden alle Container-Ports im privaten Netz automatisch zugänglich gemacht. Es ist nicht erforderlich, die Container-Ports einem Host-Port zuzuordnen.

Sie können Ihre App der Öffentlichkeit zur Verfügung stellen, indem Sie einen öffentlichen Port zugänglich machen und entweder eine öffentliche IP-Adresse an Ihren einzelnen Container oder eine öffentliche Route an Ihre Containergruppe binden. Wenn Sie einen öffentlichen Port mit der Option -p im Befehl bx ic run zugänglich machen, erstellen Sie für Ihren Container auf diese Weise eine öffentliche Netzsicherheitsgruppe, die das Senden und Empfangen von öffentlichen Daten nur über den zugänglich gemachten Port zulässt. Alle anderen öffentlichen Ports sind geschlossen und können nicht für den Zugriff auf Ihre App aus dem Internet verwendet werden.

Kontinuierliche Überwachung des Unternehmensnetzes
Der gesamte eingehende und ausgehende Netzverkehr in IBM Bluemix Container Service wird von IBM überwacht, um Angriffe von außen sowie schädliche Aktivitäten zu erkennen und die entsprechenden Abhilfemaßnahmen treffen zu können.
Vollständige Unterstützung für die Docker-Befehle exec, attach und logs
Sie können sich bei Ihrem Container anmelden oder seine detaillierte Ausgabe während der Laufzeit anzeigen, indem Sie die Befehle bx ic exec, bx ic attach oder bx ic logs verwenden. Mit diesen Befehlen können Sie vermeiden, dass während der Laufzeit ein nicht sicherer SSH-Dämon für Containerzugriff und -überwachung ausgeführt wird.

Weitere Informationen zur sicheren Anmeldung bei Ihrem Container mithilfe von bx ic exec finden Sie unter Mit exec bei einem Container anmelden.

Mit exec bei einem Container anmelden

Sie können bx ic exec verwenden, um sich bei Ihrem aktiven Container anzumelden.

Der Befehl bx ic exec wird häufig zum Öffnen eines Bash-Fensters verwendet, um die Dateien eines Datenträgers anzuzeigen, der einem Container zugeordnet ist. Der Befehl bx ic exec kann nur für einen Container ausgeführt werden, der sich im aktiven Status befindet. Damit ein Container betriebsbereit bleibt, muss der Container mit einem Befehl mit langer Laufzeit gestartet werden. Wenn ein Container angehalten oder gestoppt wird, muss er erneut mit einem Befehl mit langer Laufzeit gestartet werden, bevor Sie sich bei dem Container anmelden können.

Befehl und Parameter

bx ic exec [-d] [-it ] [-u USER] CONTAINER CMD
Anmerkung: *In diesem Befehl können Sie bx ic durch docker ersetzen, wenn Sie bei IBM Bluemix Container Service angemeldet sind und Ihre Umgebungsvariablen zur Verwendung nativer Docker-Befehle definiert haben.

Beispiele

Dieser Beispielbefehl wird in einem interaktiven Bash-Terminal im Container mein_container ausgeführt.
bx ic exec -it mein_container bash
Dieser Beispielbefehl führt den Datumsbefehl im Container mein_container aus. 
bx ic exec mein_container date

Virtuelles privates Netz (VPN) zur Verbindungsherstellung zwischen Containern und dem Rechenzentrum eines Unternehmens einrichten

Mit dem durch IBM® Virtual Private Network bereitgestellten VPN-Service können Sie die einzelnen Container und Containergruppen in einem privaten Containernetz in Bluemix mit dem Rechenzentrum eines Unternehmens sicher verbinden. IBM VPN stellt einen sicheren End-to-End-Kommunikationskanal über das Internet bereit, der auf der standardisierten IPsec-Protokollsuite (IPsec - Internet Protocol Security) basiert. Das IPsec-Protokoll bietet Peerauthentifizierung auf Netzebene, Datenintegrität und Datenvertraulichkeit durch Verschlüsselung der Pakete, die zwischen den VPN-Endpunkten ausgetauscht werden. Um eine sichere Verbindung zwischen den Containern in Bluemix und dem Rechenzentrum des Unternehmens einrichten zu können, muss in Ihrem Rechenzentrum vor Ort ein IPsec-VPN-Gateway oder SoftLayer-Server installiert sein. Mit dem IBM VPN-Service können Sie ein VPN-Gateway pro Bereich konfigurieren und bis zu 16 Verbindungen zu verschiedenen Zielen definieren.

Gehen Sie wie folgt vor, um Container mit dem Rechenzentrum eines Unternehmens zu verbinden:

  1. Erstellen Sie mindestens einen einzelnen Container oder eine Containergruppe in Ihrem Bereich, der bzw. die mit dem Rechenzentrum Ihres Unternehmens verbunden werden soll.
  2. Erstellen Sie eine Instanz des IBM VPN-Service in Ihrem Bereich, indem Sie die Schritte 1-5 im Abschnitt Service binden ausführen.
    Anmerkung: Die IBM VPN-Serviceinstanz muss in demselben Bereich erstellt werden, in dem Sie die Container erstellt haben, die mit dem Rechenzentrum Ihres Unternehmens verbunden werden sollen.
  3. Führen Sie die Anweisungen im Abschnitt Einführung in IBM VPN aus, um das VPN-Gateway zu konfigurieren und Verbindungen zum Rechenzentrum Ihres Unternehmens einzurichten.

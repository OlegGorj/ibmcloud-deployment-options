IBM Bluemix Container Service-Plug-in (bx ic) installieren

Zum Ausführen von Befehlen muss Ihr Betriebssystem die folgenden Anforderungen erfüllen. Diese Task enthält Informationen zur Installation dieser Anforderungen. Ältere Versionen dieser Tools werden für die Verwendung mit dem IBM Bluemix Container Service-Plug-in (bx ic) nicht unterstützt.

  • Docker Version 1.8 bis 1.12
  • Cloud Foundry-CLI Version 6.14.0 - 6.22.0
    Anmerkung: macOS Sierra-Benutzer müssen Cloud Foundry CLI Version 6.22.0 oder eine höhere Version installieren.
  • Bluemix-CLI
  • IBM Bluemix Container Service-Plug-in

Gehen Sie wie folgt vor, um die Befehlszeilenschnittstellen (CLIs) zu installieren:

  1. Installieren Sie Docker 1.12. Bei Windows- und OS X-Betriebssystemen wird Docker zusammen mit Docker Toolbox installiert, die Docker Machine, Docker Engine und Kitematic umfasst. Bei Linux-Betriebssystemen variieren die Installationsmechanismen in Abhängigkeit von der verwendeten Linux-Distribution. Informationen zur Installation oder, falls Docker bereits installiert ist, zum Upgrade auf Docker 1.12 enthalten die folgenden Links.
    Anmerkung: Wenn Sie Docker bereits auf Ihrem Computer installiert haben und für Ihre aktuelle Version kein Upgrade durchführen möchten, können Sie diesen Schritt überspringen und mit der Installation der Cloud Foundry-CLI fortfahren.
    • OS X

      https://github.com/docker/toolbox/releases/download/v1.12/DockerToolbox-1.12.pkg

    • Windows

      https://github.com/docker/toolbox/releases/download/v1.12/DockerToolbox-1.12.exe

    • Linux

      Anweisungen zur Installation von Docker Version 1.12 auf der betreffenden Linux-Distribution finden Sie in der Docker-Installationsdokumentation.

  2. Starten Sie Docker.
    • Doppelklicken Sie bei Verwendung von OS X oder Windows auf das Symbol Docker Quickstart Terminal. Der Docker-Dämon wird gestartet und die Umgebungsvariablen werden für Sie festgelegt.
    • Überprüfen Sie bei Verwendung von Linux mit einem Befehlszeilendienstprogramm, ob der Docker-Dämon aktiv ist. Konfigurieren Sie außerdem den Docker-Dämon für den automatischen Neustart. Weitere Informationen finden Sie in der Docker-Dokumentation für unterstützte Betriebssysteme.
  3. Stellen Sie sicher, dass Ihre Docker-Installation ordnungsgemäß eingerichtet ist. Führen Sie hierzu in Ihrem Terminal den Befehl docker run hello-world aus. 
    docker run hello-world
    Das Image hello-world, von dem aus die Ausführung eines Containers erfolgt, wird von Docker lokal gesucht. Wenn Sie diesen Befehl zum ersten Mal ausführen, wird daher die folgende Nachricht angezeigt. 
    Unable to find image 'hello-world:latest' locally
    Das Image wird aus Docker Hub extrahiert, einem öffentlichen Repository für Images; nach Ausführung des Befehls wird eine Ausgabe angezeigt, die der folgenden ähnelt: 
    $ docker run hello-world

    executable that produces the output you are currently reading.
 4. The Docker daemon streamed that output to the Docker client, which sent it
    to your terminal.

To try something more ambitious, you can run an Ubuntu container with:
 $ docker run -it ubuntu bash

    Wenn der Befehl nicht ordnungsgemäß abgeschlossen wird, führen Sie docker version aus, um zu prüfen, ob Docker ordnungsgemäß installiert wurde.
  4. Installieren Sie die Cloud Foundry-Befehlszeilenschnittstelle (CLI) für Cloud Foundry-CLI Version 6.14.0 - 6.22.0 aus dem GitHub-Repository. Sie müssen die Cloud Foundry-CLI an der Standardposition für Ihr Betriebssystem installieren, da ansonsten die Umgebungsvariable PATH nicht mit Ihrem Installationsverzeichnis übereinstimmt.
    Anmerkung: macOS Sierra-Benutzer müssen Cloud Foundry CLI Version 6.22.0 oder eine höhere Version installieren.
    Führen Sie den folgenden Befehl aus, um zu überprüfen, ob die Cloud Foundry-Befehlszeilenschnittstelle ordnungsgemäß installiert wurde, oder welche Version installiert ist (wenn die Cloud Foundry-CLI bereits installiert war): 
    cf -v
  5. Installieren Sie die Bluemix-CLI.
  6. Melden Sie sich an der Bluemix-CLI an. 
    bx login -a https://api.ng.bluemix.net
  7. Installieren Sie das IBM Bluemix Container Service-Plug-in. 
    bx plugin install IBM-Containers -r Bluemix
    Überprüfen Sie durch Ausführen des folgenden Befehls, ob das Plug-in ordnungsgemäß installiert wurde: 
    bx plugin list

    Das Plug-in wird in den Ergebnissen unter dem Namen 'IBM-Containers' angezeigt.

Sie können sich jetzt anmelden und die Container verwalten.

Beim IBM Bluemix Container Service-CLI-Plug-in anmelden(bx ic)

Melden Sie sich nach der Installation der Befehlszeilenschnittstelle (CLI) an, um sie zu verwenden.

Führen Sie zunächst die folgenden Tasks aus, bevor Sie mit den eigentlichen Schritten beginnen.
  • Installieren Sie das IBM Bluemix Container Service-Plug-in (bx ic).
  • Entscheiden Sie, welche Bluemix-Organisation und welchen Bereich Sie zum Anmelden bei Cloud Foundry verwenden wollen. Weitere Informationen zu Organisationen und Bereichen finden Sie unter Organisationen und Bereiche.
  • Weitere Informationen zu den Optionen, die Sie für den Cloud Foundry-Befehl login angeben können, finden Sie unter Login command in der Cloud Foundry-Dokumentation.
  1. Öffnen Sie ein Befehlszeilendienstprogramm.
    • Doppelklicken Sie bei Verwendung von OS X oder Windows auf das Symbol Docker Quickstart Terminal. Der Docker-Dämon wird gestartet und die Umgebungsvariablen werden für Sie festgelegt.
    • Öffnen Sie bei Verwendung von Linux ein beliebiges Befehlszeilendienstprogramm.
  2. Melden Sie sich über die Cloud Foundry-Befehlszeilenschnittstelle bei Bluemix an. Um eine bestimmte Region anzugeben, geben Sie die Option -a mit dem API-Endpunkt für die Region an. Wenn Sie die Befehlszeilenschnittstelle (CLI) zum ersten Mal verwenden und die Option nicht angeben, werden Sie dazu aufgefordert, einen API-Endpunkt einzugeben. Nachdem Sie den API-Endpunkt angegeben haben, müssen Sie diesen bei nachfolgenden Anmeldungen nicht mehr eingeben. 
    bx login [-a api.DomainName] [-sso]
    Um sich bei einer bestimmten Region anzumelden, geben Sie den API-Endpunkt für die betreffende Region an, wenn Sie sich bei der Cloud Foundry-CLI anmelden.
    • Vereinigte Staaten (Süden)
      bx login -a api.ng.bluemix.net
    • Großbritannien
      bx login -a api.eu-gb.bluemix.net
    Der Parameter --sso für Single Sign-on ist erforderlich, wenn Sie sich mit einer eingebundenen ID anmelden. Rufen Sie bei Verwendung dieser Option https://login.ng.bluemix.net/UAALoginServerWAR/passcode auf, um einen nur für eine einzige Anmeldung gültigen Kenncode abzurufen. Führen Sie die folgenden untergeordneten Schritte aus, wenn Sie die Option --sso nicht angeben:
    1. Geben Sie für Email Ihre IBM ID für Bluemix an.
    2. Geben Sie für Kennwort das Kennwort für die IBM ID an. Daraufhin werden die Bluemix-Organisationen und -Bereiche abgerufen.
    3. Geben Sie die Nummer für eine Ihrer Bluemix-Organisationen ein.
    4. Geben Sie die Nummer für einen Ihrer bestehenden Bluemix-Bereiche an.
  3. Legen Sie einen Namensbereich für Ihre Organisation fest. Bei einem Namensbereich handelt es sich um einen eindeutigen Namen, der Ihre private Bluemix-Registry für Images kennzeichnet. Wenn Sie einen Container erstellen, müssen Sie die Imageposition durch Einschließen des Namensbereichs in den Imagenamen angeben.
    Anmerkung: Wenn Sie einen Namensbereich erstellt haben, können Sie ihn nicht mehr ändern, nur löschen. Wenn Benutzer in Ihrer Bluemix-Organisation bereits Container bereitgestellt haben und einen Namensbereich eingestellt haben, wird eine Nachricht angezeigt, die den Namensbereich für Ihre Organisation angibt. In diesem Fall können Sie den Vorgang mit dem nächsten Schritt fortsetzen. 
    bx ic namespace-set <mein_namensbereich>
    Tipp: Bei Namensbereichen sind die folgenden Regeln zu beachten:
    • Der Name darf nur Kleinbuchstaben, Ziffern und Unterstreichungszeichen (_) enthalten.
    • Der Name kann 4 bis 30 Zeichen umfassen. Wenn Sie Container über die Befehlszeile verwalten möchten, sollten Sie dem Namensbereich einen kurzen Namen zuordnen, der schnell eingegeben werden kann.
    • Der Name muss in Bluemix eindeutig (einmalig) sein.
  4. Initialisieren Sie IBM Bluemix Container Service. 
    bx ic init
  5. Entscheiden Sie, ob Sie bx ic-Befehle oder native Docker-Befehle ausführen möchten. Sie können eine dieser Optionen bei jeder Anmeldung beim Plug-in verwenden; alternativ können Sie den Befehl shellinit verwenden, um immer Option 2 auszuwählen.
    • Option 1 (Standardoption)

      Eine einfache Shell für IBM Bluemix Container Service-Wrapper Docker. Der lokale Docker-Dämon ist von IBM Bluemix Container Service nicht betroffen. bx ic-Befehle werden verwendet, um Container in Ihrem Bluemix-Bereich zu verwalten und docker-Befehle werden verwendet, um Container lokal zu verwalten. docker-Befehle sind jedoch immer für die lokale Imageentwicklung erforderlich.

      Mit Option 1 erstellt der Befehl bx ic run beispielsweise einen Container in Ihrem Bluemix-Bereich und docker run erstellt einen lokalen Container. Möglicherweise bevorzugen Sie diese Option, wenn Sie sowohl bei Docker als auch bei IBM Bluemix Container Service neu sind oder wenn Sie Container lieber lokal testen möchten.

    • Option 2

      Leiten Sie Ihren lokalen Docker-Dämon für die aktuelle CLI-Sitzung auf IBM Bluemix Container Service um, sodass die docker-Befehle anstelle der bx ic-Befehle verwendet werden können, wenn sie für die Verwaltung von Containern in Ihrem Bluemix-Bereich zur Verfügung stehen.

      Mit der Option 2 können Sie zum Beispiel entweder den Befehl bx ic run oder den Befehl docker run verwenden, um einen Container in Ihrem Bereich zu erstellen. Möglicherweise bevorzugen Sie diese Option, wenn Sie ein erfahrener Docker-Benutzer sind, der mit docker-Befehlen vertraut ist, oder wenn Sie automatische Scripts adaptieren, um mit IBM Bluemix Container Service zu arbeiten.

      In einigen Ausnahmefällen ist weiterhin die Verwendung der bx ic-Befehle erforderlich, um spezifische Tasks für IBM Bluemix Container Service auszuführen. Um zum Beispiel einen Namensbereich zu definieren, müssen Sie den Befehl bx ic namespace-set verwenden. Der Befehl docker namespace-set ist kein gültiger Befehl. Eine Liste der docker-Befehle, die von dieser Konfiguration unterstützt werden, finden Sie unter Unterstützte Docker-Befehle für das IBM Bluemix Container Service-Plug-in (bx ic).

      Um diese Option zu konfigurieren, kopieren Sie die Umgebungsvariablen, die während der Anmeldung im Terminal bereitgestellt werden, und fügen Sie ein.
      • DOCKER_HOST: Stellt für den Docker-Host IBM Bluemix Container Service ein.
      • DOCKER_CERT_PATH: Stellt die Position der Authentifizierungsschlüssel für das IBM Bluemix Container Service-Plug-in für Cloud Foundry ein.
      • DOCKER_TLS_VERIFY: Verwendet TLS und überprüft den fernen Server.
      Tipp: Wenn Sie die Tasks in der IBM Bluemix Container Service-Dokumentation ausführen, können Sie docker-Befehle in allen durch einen Stern (*) markierten Schritten ausführen.

Zweck
Einen aktiven Container steuern oder die zugehörige Ausgabe anzeigen. Drücken Sie die Tastenkombination STRG + C, um den Container zu verlassen und zu stoppen. Weitere Informationen finden Sie unter dem Befehl attach in der Docker-Hilfe.
Parameter
--no-stdin
(Optional) Die Standardeingabe nicht einschließen.
--sig-proxy
(Optional) Der Standardwert ist true.
CONTAINER
(Erforderlich) Ein Containername oder eine ID.
Beispiel

bx ic build -t TAG [--no-cache] [--pull] [-q] DOCKERFILE_LOCATION

Zweck
Erstellen Sie ein Docker-Image lokal oder in der privaten Bluemix-Registry. Weitere Informationen finden Sie unter dem Befehl build in der Docker-Hilfe.
Parameter
DOCKERFILE_LOCATION
(Erforderlich) Der Pfad zur Dockerfile und zu den Kontextdateien auf dem lokalen Host.
--no-cache
(Optional) Wenn ein Image zum ersten Mal erstellt wird, werden die Ebenen für das Image für die gesamte Organisation im Cache gespeichert. Wenn Sie nachfolgende Versionen des Image erstellen, werden diese im Cache gespeicherten Versionen des Image zum Beschleunigen des Erstellungsprozesses verwendet. Wenn Sie beim Erstellen einer neuen Version eines Image die Option --no-cache einschließen, können Sie sicherstellen, dass jede Ebene des Image neu erstellt wird.
--pull
(Optional) Das Basisimage wird standardmäßig nur aus der Registry extrahiert, wenn es noch nicht in den Image-Cache der Organisation gestellt wurde. Wenn Sie diese Option einschließen, können Sie sicherstellen, dass die neueste Version des Basisimage zum Erstellen des neuen Image verwendet wird.
-q, --quiet
(Optional) Die umfangreiche Ausgabe unterdrücken, die von den Containern generiert wird. Der Standardwert ist false.
-t TAG, --tag TAG
(Erforderlich) Der Pfad zu Ihrer privaten Image-Registry, der auf das zu erstellende Image angewendet wird.
Beispiel
Das folgende Beispiel ist eine Anforderung zum Erstellen eines Images mit dem Namen myimage. Die Dockerfile und andere Artefakte, die im Build verwendet werden, befinden sich in demselben Verzeichnis, in dem der Befehl ausgeführt wird. Da die Registry und der Namensbereich im Imagenamen enthalten sind, wird das Image in der privaten Bluemix-Image-Registry Ihrer Organisation erstellt.
Tipp: Führen Sie bx ic namespace-get aus, um Ihren Namensbereich abzurufen, und ersetzen Sie <mein_namensbereich> durch Ihren Namensbereich.
bx ic build -t registry.DomainName/<mein_namensbereich>/mein_image.

bx ic cp SRC_PATH CONTAINER:DEST_PATH | bx ic cp CONTAINER:SRC_PATH DEST_PATH

Zweck
Kopieren einer Datei oder eines Ordners von einem aktiven oder gestoppten Container auf Ihre lokale Maschine und umgekehrt. Weitere Informationen finden Sie unter dem Befehl cp in der Docker-Hilfe.
Anmerkung: Die Unterstützung für diesen Befehl auf die Verwendung mit einzelnen Containern beschränkt. Wenn Sie Dateien aus oder in eine Gruppe von Containern kopieren möchten, müssen Sie die Dateien jeweils aus jeder oder in jede Containerinstanz der Gruppe kopieren.
Parameter
DEST_PATH
(Erforderlich) Der Pfad zu der Datei oder dem Ordner auf Ihrer lokalen Maschine oder in Ihrem Container, wohin die Informationen aus dem Quellenpfad kopiert werden sollen. Die Datei oder der Ordner müssen auf der lokalen Maschine oder im Container vorhanden sein, bevor Sie mit dem Kopieren beginnen können. Wenn Sie auf Ihre lokale Maschine kopieren möchten, können Sie den absoluten oder den relativen Pfad verwenden. Wenn Sie in Ihren einzelnen Container kopieren möchten, müssen Sie den Namen des Containers, das Begrenzungszeichen ":" und den Pfad zu der Zieldatei bzw. zum Zielordner einschließen. Der Pfad muss relativ zum Stammverzeichnis des Containers angegeben werden.
SRC_PATH
(Erforderlich) Der Pfad zur Datei oder zu dem Ordner, den Sie kopieren möchten. Wenn die Datei oder der Ordner sich auf Ihrer lokalen Maschine befindet, können Sie den absoluten oder relativen Pfad verwenden. Wenn sich die Datei oder der Ordner in Ihrem einzelnen Container befindet, müssen Sie den Namen des Containers, das Begrenzungszeichen ":" und den Pfad zu der Datei oder dem Ordner, die bzw. den Sie kopieren möchten, einschließen. Der Pfad muss relativ zum Stammverzeichnis des Containers angegeben werden.
Anmerkung: Sie können nicht bestimmte Systemdateien kopieren, die unter dem Verzeichnis /dev, /etc, /proc und /sys Ihres Containers gespeichert sind.
Beispiel
Dieser Beispielbefehl kopiert eine Datei mit dem Namen config.txt aus dem aktuellen Verzeichnis Ihrer lokalen Maschine in das Verzeichnis config eines einzelnen Containers mit dem Namen mein_container. 
bx ic cp ./config.txt mein_container:/config

bx ic group-create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session-affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

Zweck
Erstellung einer Skalierungsgruppe. Standardmäßig werden Container in Bluemix im Modus mit aufgehobener Zuordnung ausgeführt.
Parameter
--anti
(Optional) Bewirkt die Verwendung von Anti-Affinität, um die Hochverfügbarkeit Ihrer Containergruppe zu steigern. Durch die Option --anti wird erzwungen, dass jede Containerinstanz in Ihrer Gruppe auf einem separaten physischen Rechenknoten angeordnet werden muss, was das Risiko eines Absturzes aller Container in einer Gruppe infolge eines Hardwarefehlers verringert. Diese Option kann in Verbindung mit größeren Gruppengrößen möglicherweise nicht verwendet werden, da für jede Bluemix-Region und -Organisation nur eine begrenzte Anzahl von Rechenknoten für die Bereitstellung verfügbar ist. Falls Ihre Bereitstellung nicht erfolgreich ist, reduzieren Sie entweder die Anzahl der Containerinstanzen in der Gruppe oder entfernen Sie die Option --anti.
--auto
(Optional) Wenn die automatische Wiederherstellung für eine Containergruppe aktiviert ist und die Gruppe 10 Minuten lang ausgeführt wurde, beginnt die Lastausgleichsfunktion der Gruppe mit der regelmäßigen Prüfung des Status jeder Containerinstanz in der Gruppe durch HTTP-Anforderungen. Wenn eine Containerinstanz nicht innerhalb von 100 Sekunden antwortet, wird sie als inaktiv markiert. Inaktive Containerinstanzen werden durch die automatische Wiederherstellung aus der Gruppe entfernt und erneut erstellt. Die automatische Wiederherstellung versucht dreimal, Containerinstanzen in einer Gruppe wiederherzustellen. Nach dem dritten Versuch wird die automatische Wiederherstellung für Containerinstanzen für 60 Minuten ausgesetzt. Nach 60 Minuten wird der automatische Wiederherstellungsprozess wieder gestartet.Weitere Informationen zur Statusprüfung können Sie den Erläuterungen zur Option --http_monitor_enabled entnehmen.

--desired DESIRED
(Optional) Die Anzahl von Instanzen in Ihrer Gruppe. Der Standardwert ist 2. Welche Anzahl von Containergruppeninstanzen Ihre App erfordert, richtet sich nach der erwarteten Arbeitslast und dem Zeitrahmen, in dem eine Anforderung von der App verarbeitet werden kann. Multitasking oder Netzkonnektivität wirken sich beispielsweise darauf aus, wie viele Instanzen für Ihre App erforderlich sind. Sie bestimmen die Anzahl der gewünschten Instanzen, indem Sie einen einzelnen Container in Bluemix bereitstellen und für diesen einen Belastungstest ausführen. Wenn die App zum Beispiel in der Lage ist, 100 Anforderungen pro Sekunde zu verarbeiten, jedoch eine durchschnittliche Arbeitslast von 300 Anforderungen pro Sekunde zu erwarten ist, so beläuft sich die gewünschte Anzahl von Instanzen für die Containergruppe auf 3. Um Ihre Containergruppe hoch verfügbar zu machen und mit erhöhter Ausfallsicherheit zu versehen, sollten Sie mehr Instanzen als das Minimum einbeziehen, damit die erwartete Arbeitslast verarbeitet werden kann. Zusätzliche Instanzen können die Arbeitslast übernehmen, falls andere Instanzen ausfallen. Um die automatische Wiederherstellung einer fehlgeschlagenen Instanz ohne Auswirkung auf die Arbeitslast sicherzustellen, beziehen Sie eine zusätzliche Instanz ein. Zum Schutz vor zwei gleichzeitigen Ausfällen beziehen Sie zwei zusätzliche Instanzen ein. Diese Konfiguration folgt dem Muster N+2, wobei N für die Anzahl der Instanzen steht, die zur Verarbeitung der Anforderungen dienen, und +2 für die beiden zusätzlichen Instanzen.
Wichtig: Eine Containergruppe sollte mindestens zwei Containerinstanzen enthalten. Wenn Sie lediglich eine Instanz in die Gruppe einbeziehen und diese Instanz irgendwann nicht verfügbar ist, treten bei den Benutzern möglicherweise Ausfallzeiten für die App auf.

IBM Bluemix Container Service-Plug-in (bx ic) deinstallieren

Wenn Sie das IBM Bluemix Container Service-Plug-in nicht mehr benötigen, können Sie es mit dem Befehl bx deinstallieren.

Anmerkung: Führen Sie den folgenden Befehl aus, um das IBM Bluemix Container Service Cloud Foundry-Plug-in zu deinstallieren. 
cf uninstall-plugin IBM-Containers
  1. Führen Sie den folgenden Befehl aus, um das IBM Bluemix Container Service-Plug-in für die Bluemix-CLI zu deinstallieren. 
    bx plugin uninstall IBM-Containers
  2. Stellen Sie sicher, dass das Plug-in auch tatsächlich deinstalliert wurde, indem Sie den folgenden Befehl ausführen und die Liste der installierten Plug-ins überprüfen. 
    bx plugin list

    Das IBM Bluemix Container Service-Plug-in wird in den Ergebnissen nicht angezeigt.

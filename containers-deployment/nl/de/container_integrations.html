Service über die Bluemix-GUI binden

IBM Bluemix verfügt über eine Liste mit Services und verwaltet diese für die Entwickler von Apps. Um einen Bluemix-Service hinzuzufügen, der von Ihrem Container verwendet werden soll, müssen Sie eine Instanz dieses Service anfordern und den Service an den Container binden.

Fügen Sie zunächst den Service zu Ihrem Bluemix-Bereich hinzu. Weitere Informationen finden Sie unter Neue Serviceinstanz anfordern.
Einschränkung: Einige Bluemix-Services unterstützen das direkte Binden an einen Container nicht und führen dazu, dass die Containererstellung fehlschlägt. Binden Sie den Bluemix-Service in diesem Fall zunächst an eine Cloud Foundry-App und binden Sie dann die App an den Container. Die App dient somit als Brücke zwischen Service und Container. Weitere Informationen finden Sie unter Bluemix-Service wird nicht direkt an einen Container gebunden.
  1. Klicken Sie im Bluemix-Katalog auf Container.
  2. Klicken Sie auf ein Container-Image.
  3. Erweitern Sie Erweiterte Optionen und wählen Sie unter Service auswählen einen verfügbaren Bluemix-Service aus.
  4. Klicken Sie auf Hinzufügen. Der Bluemix-Service wird im Feld Ist bereits an Ihren Container gebunden angezeigt.
  5. Füllen Sie die anderen Felder aus und klicken Sie anschließend auf Erstellen.
  6. Nachdem der Container erstellt worden ist, überprüfen Sie, ob die VCAP_SERVICES-Informationen angezeigt werden. Suchen Sie auf der Registerkarte mit der Übersicht für den Container den gebundenen Service und klicken Sie auf Berechtigungsnachweise anzeigen. Bei den Informationen, die angezeigt werden, handelt es sich um die VCAP_SERVICES-Informationen.

Service über die Befehlszeilenschnittstelle (CLI) an einen Container binden

IBM Bluemix verfügt über eine Liste mit Services und verwaltet diese für die Entwickler von Apps. Um einen Bluemix-Service hinzuzufügen, der von Ihrem Container verwendet werden soll, müssen Sie eine Instanz dieses Service anfordern und den Service an den Container binden.

Fügen Sie zunächst den Service zu Ihrem Bluemix-Bereich hinzu. Weitere Informationen finden Sie unter Neue Serviceinstanz anfordern.
Einschränkung: Einige Bluemix-Services unterstützen das direkte Binden an einen Container nicht und führen dazu, dass die Containererstellung fehlschlägt. Binden Sie den Bluemix-Service in diesem Fall zunächst an eine Cloud Foundry-App und binden Sie dann die App an den Container. Die App dient somit als Brücke zwischen Service und Container. Weitere Informationen finden Sie unter Bluemix-Service wird nicht direkt an einen Container gebunden.
  1. Verwenden Sie einen der folgenden Befehle, um die Bluemix-Serviceinstanz an einen neuen Container zu binden.
    Tipp: Führen Sie bx ic namespace-get aus, um Ihren Namensbereich abzurufen, und ersetzen Sie <mein_namensbereich> durch Ihren Namensbereich.
    • Wenn Sie einen einzelnen Container erstellen, verwenden Sie folgenden Befehl.
      bx ic run --name containername -e "CCS_BIND_SRV=serviceinstanz1,serviceinstanz2" registry.DomainName/<mein_namensbereich>/imagename
    • Wenn Sie eine Containergruppe erstellen, verwenden Sie folgenden Befehl.
      bx ic group-create --name containergruppenname -e "CCS_BIND_SRV=serviceinstanz1,serviceinstanz2" registry.DomainName/<mein_namensbereich>/imagename
    • Wenn Sie einen Service an eine Containergruppe binden, die bereits ausgeführt wird, verwenden Sie den folgenden Befehl. Dieser Befehl steht nur für Containergruppen zur Verfügung. Es kann nur jeweils ein Service zur Zeit gebunden werden. 
      bx ic service-bind containergruppenname serviceinstanz
  2. Verwenden Sie einen der folgenden Befehle, um zu überprüfen, ob die Serviceinstanz an den Container gebunden ist.
    • Verwenden Sie folgenden Befehl für einen einzelnen Container. 
      bx ic inspect CONTAINER
    • Verwenden Sie folgenden Befehl für eine Containergruppe. 
      bx ic group-inspect GROUP
    Stellen Sie sicher, dass die VCAP_SERVICES-Informationen für den Service angezeigt werden. 
    {
 "VCAP_SERVICES": {
  "<service_name>": [
   {
          "credentials": {
     "hostname": "<hostname_id>",
     "jdbcUrl": "<jdbc_URL>",
     "name": "<encoded_name>",
     "password": "<encoded_password>",
     "port": "<port_number>",
     "uri": "<service_URI>",
     "username": "<encoded_username>"
    },
    "label": "<service_name>",
    "name": "<service_instance>",
    "plan": "<plan_name>",
    "tags": [
     "<tag1>",
     "<tag2>",
     "<tag3>"
    ]
   }

Alerts für Apps konfigurieren, die mithilfe von Alert Notification in Containern bereitgestellt werden

Konfigurieren Sie Ihre App, um Benachrichtigungen zu senden, wenn eine bestimmte Bedingung oder ein bestimmtes Kriterium erfüllt ist, und verwenden Sie den Alert Notification-Service, um Ihre Alertbenachrichtigungen zu verwalten. Mit dem Alert Notification-Service können Sie angepasste Benachrichtigungsrichtlinien erstellen, die für Sie interessante Alerts herausfiltern. Sie können die Richtlinienfilter nach Bedarf optimieren, um Einzelpersonen oder Gruppen als Empfänger für Alerts sowie bevorzugte Benachrichtigungsmethoden und die Benachrichtigungshäufigkeit anzugeben.

Lesen Sie zunächst den Abschnitt Neue Serviceinstanz anfordern und erstellen Sie eine Instanz des Service 'Alert Notification' in Ihrem Bluemix-Bereich.

Die folgenden Schritte zeigen beispielhaft, wie ein Container erstellt wird, der eine Beispielknotenapp von Alert Notification ausführt. Sie können Testalerts an den Alert Notification-Service senden und diese im Tool Alert Viewer überprüfen.

  1. Laden Sie das Beispiel Alert Notification-App auf Ihre lokale Maschine herunter und klonen Sie es. 
    git clone https://github.com/IBM-Bluemix/alert-notification-helloworld.git
    Anmerkung: Wenn Sie das Git-Projekt klonen, wird ein neuer Ordnername alert-notification-helloworld auf Ihrer lokalen Maschine erstellt.
  2. Erstellen Sie ein Containerimage, das die Beispielapp Alert Notification enthält und übertragen Sie es mit der Push-Operation in Ihre private Bluemix-Image-Registry.
    1. Melden Sie sich bei der IBM Bluemix Container Service-CLI an.
    2. Navigieren Sie zum Verzeichnis alert-notification-helloworld.
    3. Erstellen Sie Ihr Containerimage und übertragen Sie es mit einer Push-Operation in Ihre private Bluemix-Image-Registry. Führen Sie bx ic namespace-get aus, um die Informationen zum Namensbereich abzurufen und <mein_namensbereich> durch Ihren Namensbereich zu ersetzen. 
      bx ic build -t registry.DomainName/<mein_namensbereich>/alert .
    4. Überprüfen Sie, ob Ihr Containerimage erfolgreich erstellt und mit der Push-Operation übertragen wurde. 
      bx ic images
  3. Erstellen Sie einen Container vom Image und binden Sie den Alert Notification-Service daran. Ersetzen Sie <mein_namensbereich> durch die Informationen zu Ihrem Namensbereich und <mein_alert> durch den Namen der Alert Notification-Serviceinstanz, die Sie zu Beginn erstellt haben. 
    bx ic run --name alertcontainer -p 8080 -e "CCS_BIND_SRV=<mein_alert>" registry.DomainName/<mein_namensbereich>/alert
  4. Überprüfen Sie, ob Ihr Container aktiv ist. 
    bx ic ps
  5. Binden Sie eine öffentliche IP-Adresse an Ihren Container.
    1. Listen Sie die öffentlichen IP-Adressen auf, die in Ihrem Bereich verfügbar sind, und notieren Sie die IP-Adresse, die Sie verwenden möchten. 
      bx ic ips
      Tipp: Wenn keine öffentlichen IP-Adressen in Ihrem Bereich verfügbar sind, führen Sie bx ic ip-request aus, um eine neue IP-Adresse anzufordern. Sie können auch die Bindung für eine nicht verwendete IP-Adresse im Container aufheben, indem Sie den Befehl bx ic unbind <IP-ADRESSE> <CONTAINER> ausführen.
    2. Binden Sie die öffentliche IP-Adresse an Ihren Container. Ersetzen Sie <IP_ADDRESS> durch die IP-Adresse, die Sie im vorherigen Schritt notiert haben. 
      bx ic ip-bind <IP-ADRESSE> alertcontainer
  6. Erstellen Sie einen Testalert.
    1. Öffnen Sie den Web-Browser.
    2. Geben Sie die IP-Adresse und den Port in Ihrem Webbrowser wie folgt ein: <IP_ADDRESS>:8080. Die Begrüßungsseite Ihrer Alert Notification-Web-App wird angezeigt.

    3. Klicken Sie auf die Schaltfläche Beispielalert senden, um einen Testalert an Alert Viewer zu senden.
  7. Öffnen Sie den Alert Viewer und überprüfen Sie Ihren Testalert.
    1. Wählen Sie in der Bluemix-GUI Ihre aktive Containerinstanz aus.
    2. Klicken Sie auf den Alert Notification-Service unter Gebundene Services.
    3. Klicken Sie auf Starten, um den Alert Viewer zu öffnen.
    4. Überprüfen Sie Ihren Testalert.
  8. Untersuchen Sie das Leistungsspektrum des Alert Notification-Service. Lesen Sie die Alert Notification-Dokumentation, um weitere Informationen zur Vorgehensweise bei folgenden Aufgaben zu finden: Benachrichtigungsrichtlinien erstellen und Ihre Alerts verwalten.

Containerbereitstellungen mithilfe von Delivery Pipeline im Continuous Delivery-Service automatisieren

Automatisieren Sie Ihre App-Builds und Containerbereitstellungen in Bluemix mithilfe von Delivery Pipeline, einem Bestandteil von IBM Bluemix Continuous Delivery.

Delivery Pipeline unterstützt:
  • Erstellen von Docker-Images
  • Bereitstellen von in Containern enthaltenen Images in Bluemix

Leitfaden für den Schnelleinstieg - Container mit Delivery Pipeline bereitstellen:

  1. Erstellen Sie ein Git-Repository, das Ihren Code enthält.
  2. Nehmen Sie eine Dockerfile in das Stammverzeichnis Ihres Repositorys auf.
  3. Klicken Sie im Menü auf Services > DevOps und dann auf Pipelines.
  4. Klicken Sie auf Pipeline erstellen > Delivery Pipeline.
  5. Klicken Sie auf Erstellen.
  6. Fügen Sie Ihr Git-Repository zur Toolchain hinzu.
    1. Klicken Sie auf Tool hinzufügen.
    2. Klicken Sie auf den Typ von Git-Service, der Ihr Repository hostet.
    3. Stellen Sie die Konfigurationsdetails für Ihr Repository bereit.
    4. Klicken Sie auf Integration erstellen.
  7. Klicken Sie auf der Übersichtsseite der Toolchain auf Delivery Pipeline.
  8. Erstellen Sie eine Build-Stage mit einem Build-Job.
    1. Klicken Sie auf Stage hinzufügen.
    2. Wählen Sie auf der Registerkarte EINGABE für Eingabetyp die Option Git-Repository aus.
    3. Fügen Sie auf der Registerkarte JOBS einen Build-Job hinzu. Wählen Sie für Buildertyp die Option IBM Containers in Bluemix aus.
  9. Erstellen Sie eine Bereitstellungsstage mit einem Bereitstellungsjob.
    1. Wählen Sie auf der Registerkarte EINGABE für Eingabetyp die Option Buildartefakte aus.
    2. Wählen Sie auf der Registerkarte JOBS für Bereitstellertyp den Eintrag IBM Containers Service for Bluemix aus.
  10. Führen Sie eine Bereitstellung aus, indem Sie entweder eine Codeänderung per Push-Operation durchführen oder indem Sie auf das Symbol 'Stage ausführen' in der Build-Stage von Delivery Pipeline klicken.

Einzelnen Container über Delivery Pipeline erstellen

Automatisieren Sie die Erstellung und die Bereitstellung Ihrer einzelnen Container mit Delivery Pipeline.

Build-Stages für Container mithilfe von Delivery Pipeline erstellen

IBM Bluemix Container Service beinhaltet einen Build-Service, der Quellcode als Eingabe verwendet und anschließend ein Docker-Image erstellt und in der Container Registry veröffentlicht. Delivery Pipeline, ein Bestandteil von IBM Bluemix Continuous Delivery, beinhaltet einen Build-Job, der den Container-Build-Service verwendet. Dieser Build-Job verwendet den Quellcode aus einem Repository, erstellt anhand dieses Codes ein Docker-Image, taggt das betreffende Image mit Versionsangaben, die der Buildnummer entsprechen, und überträgt das entstandene Image schließlich mit einer Push-Operation in die IBM Bluemix Container Service-Registry der Organisation in Bluemix. Anschließend können Mitarbeiter der Organisation das entstandene Image in Bereichen innerhalb der Organisation bereitstellen.

Lesen Sie zunächst folgende Informationen.
  • Weitere Informationen zu Builds, Stages und zur Basisfunktionalität von Pipelines finden Sie im Abschnitt Informationen zu Delivery Pipeline in der Dokumentation zu Continuous Delivery.
  • Ermitteln Sie ein bestehendes Image in der privaten Registry Ihrer Organisation, das in Ihrem Container verwendet werden soll, indem Sie den Bluemix-Katalog überprüfen.
  • Wenn Sie einen Service an den Container binden möchten, erstellen Sie eine App, die als Brücke fungiert. Weitere Informationen finden Sie im Abschnitt Service mit Delivery Pipeline an einen Container binden.

Erstellen Sie über Delivery Pipeline ein Docker-Image.

  1. Erstellen Sie eine Dockerfile im Stammverzeichnis des Git-Repositorys, das Ihrer Toolchain zugeordnet ist, und schreiben Sie sie in Ihrem Repository fest.
  2. Klicken Sie auf der Übersichtsseite der Toolchain auf Delivery Pipeline.
  3. Erstellen Sie eine Build-Stage.
    1. Klicken Sie auf Stage hinzufügen.
    2. Geben Sie der Stage einen Namen. Beispiel: Build.
    3. Wählen Sie als Eingabetyp die Option Git-Repository aus.
    4. Prüfen Sie die Werte in den anderen Feldern und aktualisieren Sie sie bei Bedarf.
    5. Klicken Sie auf die Registerkarte JOBS.
  4. Erstellen Sie in der Build-Stage einen Erstellungsjob.
    1. Klicken Sie auf der Registerkarte JOBS auf Job hinzufügen.
    2. Wählen Sie den Jobtyp Erstellen aus.
    3. Wählen Sie für Buildertyp die Option IBM Containers in Bluemix aus.
    4. Aktualisieren Sie das Bluemix-Ziel, die Organisation und den Bereich, falls erforderlich.
    5. Wählen Sie das Feld Cacheebenen verwenden ab, wenn Sie eine Aktualisierung von Ebenen erzwingen müssen, die Inhalte dynamisch mit Pull-Operationen aus anderen Quellen extrahieren.

      Enthält die Dockerfile beispielsweise den Befehl apt-get install oder wget, dann können Sie sicherstellen, dass diese dynamischen Abhängigkeiten bei jedem Build zusammengestellt werden, indem Sie das Kontrollkästchen Cacheebenen verwenden abwählen.

      Anmerkung: Wenn das Feld Cacheebenen verwenden abgewählt ist, sind die Buildzeiten länger, weil beim Erstellen des Images keine Cacheebenen verwendet werden.
    6. Geben Sie im Feld Imagename einen Namen für das zu erstellende Image ein. Verwenden Sie entsprechend der Docker-Namenskonvention nur Kleinbuchstaben, Zahlen und Gedankenstriche (-). Nach dem Build wird das Image mit dem Imagenamen und einer Buildnummer gekennzeichnet. Wenn Sie beispielsweise den Imagenamen myappimage eingeben, wird das entstandene Image für Build Nr. 4 als myappimage:4 bezeichnet.
    7. Fügen Sie an den folgenden Stellen den Pfad zur Dockerfile hinzu, wenn sich Ihre Dockerfile nicht im Stammverzeichnis des Projekts befindet.
      1. Fügen Sie den Ordnernamen vor dem Abschnitt für das Ausführen von Komponententests mit einem Verzeichniswechselbefehl hinzu. 
        cd <ordner>
      2. Hängen Sie den Ordnernamen bei allen Stellen, bei denen die Variable ${WORKSPACE} angezeigt wird, an die Variable an. Das Stammverzeichnis des Projekts wird automatisch von der Variablen ${WORKSPACE} erkannt.
        Beispiel
        BUILD_COMMAND="build --pull --tag ${FULL_REPOSITORY_NAME} ${WORKSPACE}/<folder>"
  5. Klicken Sie auf Speichern.
  6. Klicken Sie auf das Symbol Stage ausführen, um zu prüfen, ob die Stage fehlerfrei erstellt wird. Wenn der Build fehlschlägt, prüfen Sie die Protokolle und korrigieren Sie die Fehler. Wenn die Stage die Prüfung besteht, überprüfen Sie die Image-Registry Ihrer Organisation und vergewissern Sie sich, dass das Image erstellt wurde.

(Optional) Imagesicherheit als Teil der Pipeline überprüfen

Suchen Sie nach Sicherheits- und Konfigurationsproblemen in Ihrem Image, bevor der Container mithilfe von Vulnerability Advisor bereitgestellt wird.
Allgemeiner Aufbau einer Pipeline, die einen derartigen Scan beinhaltet:
  • Build-Stage zum Erstellen der Quellendateien
  • Verarbeitende Stage mit den folgenden Jobs:
    • Build-Job zum Ausführen eines Container-Builds
    • Testjob zum Ausführen von Vulnerability Advisor für ein Image
  • Bereitstellungsstage zum Bereitstellen des Containers
Tipp: Anhand dieser Task wird erläutert, wie Sie einen Testjob für Vulnerability Advisor in einer anderen Stage als der des Jobs, der das Container-Image erstellt, erstellen. Alternativ können Sie auch dieselbe Stage verwenden. Wenn Sie eine separate Stage erstellen, müssen Sie nicht eine Umgebungsvariable festlegen.

Erstellen Sie den Vulnerability Advisor-Scan.

  1. Erstellen Sie eine verarbeitende Stage.
    1. Klicken Sie auf Stage hinzufügen.
    2. Geben Sie der Stage einen Namen. Beispiel: Processing.
    3. Wählen Sie für den Eingabetyp die Option Buildartefakte aus.
    4. Überprüfen Sie die Werte für Stage und Job und aktualisieren Sie sie gegebenenfalls.
    5. Klicken Sie auf die Registerkarte JOBS.
  2. Fügen Sie in der verarbeitenden Stage einen Testjob zum Ausführen des Code-Scans hinzu.
    1. Klicken Sie auf der Registerkarte JOBS auf Job hinzufügen.
    2. Wählen Sie für den Job die Option Test aus.
    3. Wählen Sie als Testertyp die Advisorfunktion gegen Sicherheitslücken - IBM Vulnerability Advisor - aus.
    4. Aktualisieren Sie das Bluemix-Ziel, die Organisation und den Bereich, falls erforderlich.
    5. Geben Sie für Anzahl der Minuten für die Durchführung der Analyse einen Wert zwischen 0 - 59 Minuten ein. Der Standardwert ist mit 5 Minuten festgelegt. Eine URL zu dem Dashboard ist nach Beendigung des Jobs in den Konsolenprotokollen enthalten.

      Ist der Scan nach Ablauf dieser Zeit nicht abgeschlossen, schlägt der Job fehl. Die Scananalyse wird jedoch weiter ausgeführt und kann im Dashboard angezeigt werden. Wird der Job nach dem Scan erneut ausgeführt, wird die Scananforderung nicht erneut eingereicht. Sie können die Pipeline stattdessen auch so konfigurieren, dass sie bei einem erfolgreichen Scan nicht blockiert wird. Erläuterungen hierzu folgen beim nächsten Schritt.

    6. Wählen Sie das Kontrollkästchen Stage-Ausführung bei Fehlschlagen des Jobs stoppen aus oder ab, um das Verhalten bei einer Zeitlimitüberschreitung festzulegen.

      Sie können den Job so konfigurieren, dass nachfolgende Jobs in der Stage und nachfolgende Stages an der Ausführung gehindert werden, wenn der Job fehlschlägt. Sie können die Stage jedoch auch so konfigurieren, dass die Ausführung ohne eine Blockierung nachfolgender Jobs und Stages fortgesetzt wird. Der Status des Jobs richtet sich nach dem Vorhandensein größerer Sicherheitsprobleme.

      Bedeutendere Sicherheitsprobleme können ein Fehlschlagen des Jobs zur Folge haben. Je nach der Auswahl für Stage-Ausführung bei Fehlschlagen des Jobs stoppen führt ein fehlgeschlagener Job dazu, dass die Verarbeitung von Stage und Pipeline beendet wird oder dass die Verarbeitung fortgesetzt werden kann. Wenn Sie wissen, dass der Bericht viele Sicherheitsprobleme enthält, die bearbeitet werden müssen, empfiehlt es sich, die Stage für eine Fortsetzung zu konfigurieren, da der Scan sehr zeitaufwendig sein kann. Bei diesem Szenario ist es möglicherweise ungünstig, wenn die übrigen Jobs und Stages nur beendet werden, weil der Scan zu viel Zeit beansprucht.

  3. Wenn sich der Job in derselben Stage wie der Build-Job befinden, definieren Sie eine Umgebungsvariable für den Repository-Namen.
    1. Klicken Sie auf die Registerkarte UMGEBUNGSEIGENSCHAFTEN.
    2. Klicken Sie auf EIGENSCHAFT HINZUFÜGEN und wählen Sie Texteigenschaft aus.
    3. Geben Sie im Feld Name den Namen FULL_REPOSITORY_NAME ein.
    4. Lassen Sie das Feld Wert leer. Der Imagename, der im Build-Job für den Container angegeben ist, wird über diese Umgebungsvariable weitergegeben.
  4. Klicken Sie auf Speichern.
  5. Klicken Sie auf das Symbol Stage ausführen, um zu prüfen, ob die Stage fehlerfrei erstellt wird.
Wenn Sie die Ergebnisse des Scans durch Vulnerability Advisor anzeigen möchten, klicken Sie nach Abschluss des Jobs auf die Option zum Anzeigen der Ergebnisse und des Protokolls. Suchen Sie in der Registerkarte PROTOKOLLE nach dem folgenden Abschnitt: 
image registry.DomainName/namensbereich/image vulnerability results found

Eine Liste der gefährdeten Pakete, Sicherheitsprobleme und vorgeschlagenen Verbesserungen wird wie im Abschnitt Imagebericht prüfen von Vulnerability Advisor beschrieben aufgeführt. Dockerfile-Code, mit dem einige der häufigeren Probleme behoben werden können, finden Sie im Abschnitt Probleme in Images beheben.

(Optional) Images mit Delivery Pipeline bereinigen

Nicht verwendete Imageversionen werden vom Build-Job von IBM Bluemix Container Service automatisch bereinigt. Images, die im Einsatz sind, werden nicht entfernt. Sie können eine beliebige Anzahl von Imageversionen angegeben, die beibehalten werden sollen, oder Sie die Imagebereinigung inaktivieren.

  6. Klicken Sie auf Speichern.

Containergruppen mithilfe von Delivery Pipeline erstellen

Automatisieren Sie die Erstellung und die Bereitstellung Ihrer Containergruppen mithilfe von Delivery Pipeline.

Build-Stages für Containergruppen mithilfe von Delivery Pipeline erstellen

IBM Bluemix Container Service beinhaltet einen Build-Service, der Quellcode als Eingabe verwendet und anschließend ein Docker-Image erstellt und in der Container Registry veröffentlicht. Delivery Pipeline, ein Bestandteil von IBM Bluemix Continuous Delivery, beinhaltet einen Build-Job, der den Container-Build-Service verwendet. Dieser Build-Job verwendet den Quellcode aus einem Repository, erstellt anhand dieses Codes ein Docker-Image, taggt das betreffende Image mit Versionsangaben, die der Buildnummer entsprechen, und überträgt das entstandene Image schließlich mit einer Push-Operation in die IBM Bluemix Container Service-Registry der Organisation in Bluemix. Anschließend können Mitarbeiter der Organisation das entstandene Image in Bereichen innerhalb der Organisation bereitstellen.

Lesen Sie zunächst folgende Informationen.
  • Weitere Informationen zu Builds, Stages und zur Basisfunktionalität von Pipelines finden Sie im Abschnitt Informationen zu Delivery Pipeline in der Dokumentation zu Continuous Delivery.
  • Ermitteln Sie ein bestehendes Image in der privaten Registry Ihrer Organisation, das in Ihrem Container verwendet werden soll, indem Sie den Bluemix-Katalog überprüfen.
  • Wenn Sie einen Service an den Container binden möchten, erstellen Sie eine App, die als Brücke fungiert. Weitere Informationen finden Sie im Abschnitt Service mit Delivery Pipeline an einen Container binden.

Erstellen Sie über Delivery Pipeline ein Docker-Image.

  1. Erstellen Sie eine Dockerfile im Stammverzeichnis des Git-Repositorys, das Ihrer Toolchain zugeordnet ist, und schreiben Sie sie in Ihrem Repository fest.
  2. Klicken Sie auf der Übersichtsseite der Toolchain auf Delivery Pipeline.
  3. Erstellen Sie eine Build-Stage.
    1. Klicken Sie auf Stage hinzufügen.
    2. Geben Sie der Stage einen Namen. Beispiel: Build.
    3. Wählen Sie als Eingabetyp die Option Git-Repository aus.
    4. Prüfen Sie die Werte in den anderen Feldern und aktualisieren Sie sie bei Bedarf.
    5. Klicken Sie auf die Registerkarte JOBS.
  4. Erstellen Sie in der Build-Stage einen Erstellungsjob.
    1. Klicken Sie auf der Registerkarte JOBS auf Job hinzufügen.
    2. Wählen Sie den Jobtyp Erstellen aus.
    3. Wählen Sie für Buildertyp die Option IBM Containers in Bluemix aus.
    4. Aktualisieren Sie das Bluemix-Ziel, die Organisation und den Bereich, falls erforderlich.
    5. Wählen Sie das Feld Cacheebenen verwenden ab, wenn Sie eine Aktualisierung von Ebenen erzwingen müssen, die Inhalte dynamisch mit Pull-Operationen aus anderen Quellen extrahieren.

      Enthält die Dockerfile beispielsweise den Befehl apt-get install oder wget, dann können Sie sicherstellen, dass diese dynamischen Abhängigkeiten bei jedem Build zusammengestellt werden, indem Sie das Kontrollkästchen Cacheebenen verwenden abwählen.

      Anmerkung: Wenn das Feld Cacheebenen verwenden abgewählt ist, sind die Buildzeiten länger, weil beim Erstellen des Images keine Cacheebenen verwendet werden.
    6. Geben Sie im Feld Imagename einen Namen für das zu erstellende Image ein. Verwenden Sie entsprechend der Docker-Namenskonvention nur Kleinbuchstaben, Zahlen und Gedankenstriche (-). Nach dem Build wird das Image mit dem Imagenamen und einer Buildnummer gekennzeichnet. Wenn Sie beispielsweise den Imagenamen myappimage eingeben, wird das entstandene Image für Build Nr. 4 als myappimage:4 bezeichnet.
    7. Fügen Sie an den folgenden Stellen den Pfad zur Dockerfile hinzu, wenn sich Ihre Dockerfile nicht im Stammverzeichnis des Projekts befindet.
      1. Fügen Sie den Ordnernamen vor dem Abschnitt für das Ausführen von Komponententests mit einem Verzeichniswechselbefehl hinzu. 
        cd <ordner>
      2. Hängen Sie den Ordnernamen bei allen Stellen, bei denen die Variable ${WORKSPACE} angezeigt wird, an die Variable an. Das Stammverzeichnis des Projekts wird automatisch von der Variablen ${WORKSPACE} erkannt.
        Beispiel
        BUILD_COMMAND="build --pull --tag ${FULL_REPOSITORY_NAME} ${WORKSPACE}/<folder>"
  5. Klicken Sie auf Speichern.
  6. Klicken Sie auf das Symbol Stage ausführen, um zu prüfen, ob die Stage fehlerfrei erstellt wird. Wenn der Build fehlschlägt, prüfen Sie die Protokolle und korrigieren Sie die Fehler. Wenn die Stage die Prüfung besteht, überprüfen Sie die Image-Registry Ihrer Organisation und vergewissern Sie sich, dass das Image erstellt wurde.

(Optional) Imagesicherheit als Teil der Pipeline überprüfen

Suchen Sie nach Sicherheits- und Konfigurationsproblemen in Ihrem Image, bevor der Container mithilfe von Vulnerability Advisor bereitgestellt wird.
Allgemeiner Aufbau einer Pipeline, die einen derartigen Scan beinhaltet:
  • Build-Stage zum Erstellen der Quellendateien
  • Verarbeitende Stage mit den folgenden Jobs:
    • Build-Job zum Ausführen eines Container-Builds
    • Testjob zum Ausführen von Vulnerability Advisor für ein Image
  • Bereitstellungsstage zum Bereitstellen des Containers
Tipp: Anhand dieser Task wird erläutert, wie Sie einen Testjob für Vulnerability Advisor in einer anderen Stage als der des Jobs, der das Container-Image erstellt, erstellen. Alternativ können Sie auch dieselbe Stage verwenden. Wenn Sie eine separate Stage erstellen, müssen Sie nicht eine Umgebungsvariable festlegen.

Erstellen Sie den Vulnerability Advisor-Scan.

  1. Erstellen Sie eine verarbeitende Stage.
    1. Klicken Sie auf Stage hinzufügen.
    2. Geben Sie der Stage einen Namen. Beispiel: Processing.
    3. Wählen Sie für den Eingabetyp die Option Buildartefakte aus.
    4. Überprüfen Sie die Werte für Stage und Job und aktualisieren Sie sie gegebenenfalls.
    5. Klicken Sie auf die Registerkarte JOBS.
  2. Fügen Sie in der verarbeitenden Stage einen Testjob zum Ausführen des Code-Scans hinzu.
    1. Klicken Sie auf der Registerkarte JOBS auf Job hinzufügen.
    2. Wählen Sie für den Job die Option Test aus.
    3. Wählen Sie als Testertyp die Advisorfunktion gegen Sicherheitslücken - IBM Vulnerability Advisor - aus.
    4. Aktualisieren Sie das Bluemix-Ziel, die Organisation und den Bereich, falls erforderlich.
    5. Geben Sie für Anzahl der Minuten für die Durchführung der Analyse einen Wert zwischen 0 - 59 Minuten ein. Der Standardwert ist mit 5 Minuten festgelegt. Eine URL zu dem Dashboard ist nach Beendigung des Jobs in den Konsolenprotokollen enthalten.

      Ist der Scan nach Ablauf dieser Zeit nicht abgeschlossen, schlägt der Job fehl. Die Scananalyse wird jedoch weiter ausgeführt und kann im Dashboard angezeigt werden. Wird der Job nach dem Scan erneut ausgeführt, wird die Scananforderung nicht erneut eingereicht. Sie können die Pipeline stattdessen auch so konfigurieren, dass sie bei einem erfolgreichen Scan nicht blockiert wird. Erläuterungen hierzu folgen beim nächsten Schritt.

    6. Wählen Sie das Kontrollkästchen Stage-Ausführung bei Fehlschlagen des Jobs stoppen aus oder ab, um das Verhalten bei einer Zeitlimitüberschreitung festzulegen.

      Sie können den Job so konfigurieren, dass nachfolgende Jobs in der Stage und nachfolgende Stages an der Ausführung gehindert werden, wenn der Job fehlschlägt. Sie können die Stage jedoch auch so konfigurieren, dass die Ausführung ohne eine Blockierung nachfolgender Jobs und Stages fortgesetzt wird. Der Status des Jobs richtet sich nach dem Vorhandensein größerer Sicherheitsprobleme.

      Bedeutendere Sicherheitsprobleme können ein Fehlschlagen des Jobs zur Folge haben. Je nach der Auswahl für Stage-Ausführung bei Fehlschlagen des Jobs stoppen führt ein fehlgeschlagener Job dazu, dass die Verarbeitung von Stage und Pipeline beendet wird oder dass die Verarbeitung fortgesetzt werden kann. Wenn Sie wissen, dass der Bericht viele Sicherheitsprobleme enthält, die bearbeitet werden müssen, empfiehlt es sich, die Stage für eine Fortsetzung zu konfigurieren, da der Scan sehr zeitaufwendig sein kann. Bei diesem Szenario ist es möglicherweise ungünstig, wenn die übrigen Jobs und Stages nur beendet werden, weil der Scan zu viel Zeit beansprucht.

  3. Wenn sich der Job in derselben Stage wie der Build-Job befinden, definieren Sie eine Umgebungsvariable für den Repository-Namen.
    1. Klicken Sie auf die Registerkarte UMGEBUNGSEIGENSCHAFTEN.
    2. Klicken Sie auf EIGENSCHAFT HINZUFÜGEN und wählen Sie Texteigenschaft aus.
    3. Geben Sie im Feld Name den Namen FULL_REPOSITORY_NAME ein.
    4. Lassen Sie das Feld Wert leer. Der Imagename, der im Build-Job für den Container angegeben ist, wird über diese Umgebungsvariable weitergegeben.
  4. Klicken Sie auf Speichern.
  5. Klicken Sie auf das Symbol Stage ausführen, um zu prüfen, ob die Stage fehlerfrei erstellt wird.
Wenn Sie die Ergebnisse des Scans durch Vulnerability Advisor anzeigen möchten, klicken Sie nach Abschluss des Jobs auf die Option zum Anzeigen der Ergebnisse und des Protokolls. Suchen Sie in der Registerkarte PROTOKOLLE nach dem folgenden Abschnitt: 
image registry.DomainName/namensbereich/image vulnerability results found

Eine Liste der gefährdeten Pakete, Sicherheitsprobleme und vorgeschlagenen Verbesserungen wird wie im Abschnitt Imagebericht prüfen von Vulnerability Advisor beschrieben aufgeführt. Dockerfile-Code, mit dem einige der häufigeren Probleme behoben werden können, finden Sie im Abschnitt Probleme in Images beheben.

      Mit der Umgebungseigenschaft CONCURRENT_VERSIONS wird die Anzahl der App-Bereitstellungsversionen festgelegt, die als aktive Bereitstellungen beibehalten werden sollen. Der Standardwert für die Versionsanzahl liegt bei einer einzigen Version. Ist für CONCURRENT_VERSIONS z. B. der Wert 2 angegeben, werden beim Bereitstellen einer Containergruppe zwei Gruppen der Route zugeordnet, die dann Datenverkehr empfangen.

      Sollen immer alle Bereitstellungen der App entfernt werden, müssen Sie auf der Registerkarte UMGEBUNGSEIGENSCHAFTEN den Wert 0 für CONCURRENT_VERSIONS festlegen.

      DESIRED_INSTANCES Eine Zahl

      Beispiel: 4

      Welche Anzahl von Containergruppeninstanzen Ihre App erfordert, richtet sich nach der erwarteten Arbeitslast und dem Zeitrahmen, in dem eine Anforderung von der App verarbeitet werden kann. Multitasking oder Netzkonnektivität wirken sich beispielsweise darauf aus, wie viele Instanzen für Ihre App erforderlich sind.

      Um Ihre Containergruppe hoch verfügbar zu machen und mit erhöhter Ausfallsicherheit zu versehen, sollten Sie mehr Instanzen als das Minimum einbeziehen, damit die erwartete Arbeitslast verarbeitet werden kann. Zusätzliche Instanzen können die Arbeitslast übernehmen, falls andere Instanzen ausfallen. Um die automatische Wiederherstellung einer fehlgeschlagenen Instanz ohne Auswirkung auf die Arbeitslast sicherzustellen, beziehen Sie eine zusätzliche Instanz ein. Zum Schutz vor zwei gleichzeitigen Ausfällen beziehen Sie zwei zusätzliche Instanzen ein. Diese Konfiguration folgt dem Muster N+2, wobei N für die Anzahl der Instanzen steht, die zur Verarbeitung der Anforderungen dienen, und +2 für die beiden zusätzlichen Instanzen.
      ROUTE_DOMAIN Eine Zeichenfolge oder none

      Beispiel: mycustomdomain.com

      Die Standardsystemdomäne ist AppDomainName und stellt bereits ein SSL-Zertifikat bereit, sodass Sie ohne weitere Konfiguration mit HTTPS auf Ihre Containergruppen zugreifen können. Wenn Sie eine angepasste Domäne verwenden möchten, müssen Sie die angepasste Domäne bei einem öffentlichen DNS-Server registrieren, die angepasste Domäne in Bluemix konfigurieren und die angepasste Domäne anschließend der Systemdomäne von Bluemix auf dem öffentlichen DNS-Server zuweisen.

      Ist keine Domäne angegeben, wird die Domäne AppDomainName verwendet. Bei Angabe von none wird keine Domäne erstellt.

      Ist für Hostname und Domäne none angegeben, ist die Gruppe für andere Container in demselben Bereich zugänglich, jedoch nicht öffentlich verfügbar.

      ROUTE_HOSTNAME Eine Zeichenfolge oder none

      Beispiel: myapp

      Der Hostname, z. B. mycontainerhost. Der Hostname darf keine Unterstreichungszeichen (_) enthalten. Der Host und die Domäne werden zu einer vollständigen URL für eine öffentliche Route kombiniert, zum Beispiel http://mycontainerhost.AppDomainName.

      Ist kein Hostname angegeben, wird ein Hostname für Sie erstellt. Bei Angabe von none wird kein Hostname erstellt.

      Ist für Hostname und Domäne none angegeben, ist die Gruppe für andere Container in demselben Bereich zugänglich, jedoch nicht öffentlich verfügbar.

  6. Klicken Sie auf Speichern.

Lernprogramm: Cloud Foundry-App Node.js in einem Container bereitstellen

Sie können eine App, die Sie zuvor unter Verwendung von Cloud Foundry bereitgestellt haben, mithilfe von IBM Bluemix Container Service in einem Container bereitstellen.

Ziele

  • Erfahren Sie mehr über den allgemeinen Prozess der Containerbereitstellung.
  • Lernen Sie, was eine Dockerfile für das Erstellen eines Images enthalten muss.

Erforderlicher Zeitaufwand

30 Minuten

Zielgruppe

Cloud Foundry-App-Entwickler

Voraussetzungen

  • Installieren Sie die IBM Bluemix Container Service-Befehlszeilenschnittstelle.
  • Melden Sie sich bei der IBM Bluemix Container Service-Befehlszeilenschnittstelle an.

Lerneinheit 1: Cloud Foundry-Node.js-App-Code herunterladen

Bereiten Sie Ihren Node.js-Code für den Einsatz vor. Wenn Sie noch keinen Code haben, können Sie Startcode zur Verwendung in diesem Lernprogramm von Bluemix herunterladen.

Bevor Sie beginnen, installieren Sie die IBM Bluemix Container Service-Befehlszeilenschnittstelle und melden Sie sich an.

Gehen Sie wie folgt vor, um den App-Code abzurufen und zu organisieren:

  1. Erstellen Sie ein Verzeichnis mit dem Namen cf-pi-meine_initialen und navigieren Sie dorthin. In diesem Verzeichnis speichern Sie alle Dateien, die zum Erstellen des Docker-Images und zum Ausführen Ihrer Node.js-App erforderlich sind. Sie können einen beliebigen Namen für das Verzeichnis wählen. Wenn Sie den Namen aus dem Beispiel verwenden, ersetzen Sie meine_initialen durch Ihre Initialen. Beispiel: cf-pi-ms. 
    mkdir cf-pi-meine_initialen && cd cf-pi-meine_initialen
  2. Kopieren Sie Ihren Node.js-App-Code und alle zugehörigen Dateien in das Verzeichnis. Sie können Ihren eigenen Node.js-App-Code verwenden oder die Personal Insights-App von Bluemix herunterladen.

    Um den Personal Insights-App-Code von Bluemix herunterzuladen, führen Sie die folgenden Schritte aus.

    1. Klicken Sie im Bluemix-Katalog in Boilerplates auf Personality Insights Node.js Web Starter. Diese Boilerplate enthält eine Node.js-App, die den IBM Watson Personality Insights-Service verwendet, um kognitive und soziale Merkmale aus einem Text zu extrahieren, den eine Person durch ihre Blogs, Tweets oder Forenbeiträge generiert.
    2. Geben Sie den App-Namen cf-pi-meine_initialen ein und klicken Sie auf Erstellen. Um auf den App-Code für die Boilerplate zuzugreifen, müssen Sie die CF-App zuerst unter Bluemix bereitstellen. Sie können einen beliebigen Namen für die App wählen. Wenn Sie den Namen aus dem Beispiel verwenden, ersetzen Sie meine_initialen durch Ihre Initialen (z. B. cf-pi-ms), damit der Name eindeutig ist.

      Diese Bereitstellung erstellt eine App mit dem Namen cf-pi-meine_initialen sowie eine Instanz des IBM Watson Personality Insights-Service mit dem Namen cf-pi-meine_initialen-personality_insights.

      Beim Bereitstellen der App werden Anweisungen zum Bereitstellen Ihrer App über die Befehlszeilenschnittstelle angezeigt.

    3. Klicken Sie in Schritt 1 auf Startcode herunterladen.
    4. Extrahieren Sie die komprimierte Datei und speichern Sie die App-Dateien in Ihrem Verzeichnis.
Ihr Cloud Foundry-App-Code kann nun containerisiert werden.

Lerneinheit 2: Docker-Image mit Ihrem CF-App-Code erstellen

Sie können eine Dockerfile erstellen, die Ihren Node.js-App-Code sowie die erforderlichen Konfigurationen für Ihren Container enthält. Anschließend können Sie ein Docker-Image aus dieser Dockerfile erstellen und es mit einer Push-Operation an Ihre private Bluemix-Registry übertragen.

Führen Sie die folgenden Schritte aus, um ein Docker-Image mit Ihrem App-Code zu erstellen.

  1. Erstellen Sie eine Dockerfile, die die Basis zum Erstellen eines Containers darstellt. Sie können die Dockerfile mithilfe Ihres bevorzugten CLI-Editors oder einem Texteditor auf dem Computer erstellen. Im folgenden Beispiel wird gezeigt, wie Sie eine Dockerfile mit dem nano-Editor erstellen. 
    nano Dockerfile
  2. Kopieren Sie das folgende Script in die Dockerfile. 
    #Verwenden Sie das IBM Node-Image als Basisimage
FROM registry.DomainName/ibmnode:latest

#Machen Sie den Port für Ihre Personal Insights-App zugänglich und legen Sie ihn
#als von cf-Apps erwartete Umgebungsvariable fest.
ENV PORT=3000
EXPOSE 3000
ENV NODE_ENV production

#Kopieren Sie alle App-Dateien aus dem aktuellen Verzeichnis in das App-Verzeichnis
#in Ihrem Container. Legen Sie das App-Verzeichnis als Arbeitsverzeichnis
#fest.
ADD . /app
WORKDIR /app

#Installieren Sie die vorausgesetzte Software aus package.json.
RUN npm install

#'sleep' vor dem Start der App. Dieser Befehl stellt sicher, dass der
#IBM Containers-Netzbetrieb vor dem Starten der App beendet wird.
CMD (sleep 60; npm start)
#Starten Sie die Personal Insight-App.
CMD ["node", "/app/app.js"]
  3. Speichern Sie die Änderungen mit Strg + o. Bestätigen Sie die Änderungen durch Drücken der Eingabetaste.Verlassen Sie den nano-Editor mit Strg + x.
  4. Erstellen Sie das Docker-Image aus dieser App und übertragen Sie es mit einer Push-Operation an Ihre private Bluemix-Registry. 
    bx ic build -t registry.DomainName/namensbereich/cf-pi-meine_initialen .
    Erklärung der Bestandteile dieses Befehls
    build Der Befehl build.
    registry.DomainName/namensbereich/cf-pi-meine_initialen Der Pfad Ihrer privaten Bluemix-Registry, der aus dem eindeutigen Namensbereich und dem Namen des Image besteht. In diesem Beispiel wird für das Image der Name der CF-Knoten-App cf-pi-meine_initialen verwendet, aber Sie können für das Image in Ihrer privaten Registry jeden beliebigen Namen verwenden. Wenn Sie nicht sicher sind, welches Ihr Namensbereich ist, führen Sie den folgenden Befehl aus, um dies herauszufinden. 
    bx ic namespace-get
    . Die Position der Dockerfile. Wenn Sie den Befehl 'build' in dem Verzeichnis ausführen, in dem sich die Dockerfile befindet, geben Sie einen Punkt (.) ein. Verwenden Sie andernfalls einen relativen Pfad zu der Dockerfile.
Das Image wird in Ihrer Registry erstellt. Sie können mit dem Befehl bx ic images prüfen, ob das Image erstellt wurde. 
$ bx ic images
REPOSITORY                                                  TAG                 IMAGE ID            CREATED             SIZE
registry.DomainName/namensbereich/cf-pi-meine_initialen  latest  b02b7a24c35f  48 seconds ago  251.3 MB

Lerneinheit 3: Container aus Ihrem Image bereitstellen

Sie können Ihre App als Containergruppe in Bluemix bereitstellen.

Führen Sie die folgenden Schritte aus, um Ihre App in einer Containergruppe bereitzustellen.

  1. Erstellen Sie eine Containergruppe aus dem Image, das Sie in der vorherigen Lerneinheit erstellt haben, indem Sie den folgenden Befehl ausführen. 
    bx ic group-create --name cf-pi-meine_initialen -m 64 -n pi-app-meine_initialen -d AppDomainName -p 3000 -e CCS_BIND_SRV=cf-pi-meine_initialen-personal_insights --desired 3 --auto --anti registry.DomainName/namensbereich/cf-pi-meine_initialen:latest
    Erklärung der Bestandteile dieses Befehls
    group-create Der Befehl group-create.
    --name cf-pi-meine_initialen Der Name des Containers, wie er in der Bluemix-GUI angezeigt werden soll.
    -m 64 Jeder Instanz in der Gruppe werden 64 MB Speicher zugeordnet.
    -n pi-app-meine_initialen Der Hostname in der URL für die Route.
    Tipp: Wenn Sie den Startcode einsetzen, müssen Sie für die Bereitstellung der Cloud Foundry-App einen anderen Namen für den Hostnamen verwenden als in der vorhergehenden Lerneinheit. Wenn derselbe Host verwendet wird und wenn die CF-App immer noch ausgeführt wird, erfolgt ein Lastausgleich zwischen der Containergruppe und der App.
    -d AppDomainName Der Domänenname in der URL für die Route. Verwenden Sie AppDomainName nur, wenn Sie über eine angepasste Domäne verfügen.
    -p 3000 Die Portnummer, die zugänglich gemacht werden soll. Verbindungen zur URL für die Route werden an Port 3000 des Containers weitergeleitet.
    -e CCS_BIND_SRV=cf-pi-meine_initialen-personal_insights Binden Sie die IBM Watson Personality Insights-Serviceinstanz an den Container, sodass Sie kognitive und soziale Merkmale des Texts analysieren können, den Sie in der Personal Insights-App eingeben. Die Serviceinstanz wurde als Bestandteil der Bereitstellung der CP-App in Lerneinheit 1 erstellt.
    Wenn Sie nicht sicher sind, wie der Name Ihrer Serviceinstanz lautet, führen Sie den folgenden Befehl aus. 
    bx service list
    --desired 3 Die Anzahl einzelner Containerinstanzen, die in der Containergruppe erstellt werden sollen. Alle Instanzen sind gleich.
    --auto Aktivieren der automatischen Wiederherstellung für die Containergruppe. Wenn eine der Instanzen in der Gruppe nicht mehr verfügbar ist, wird automatisch eine andere Instanz erstellt, die die nicht antwortende Instanz ersetzt.
    --anti Aktivieren der Anti-Affinität für die Containergruppe. Jede Instanz der Gruppe wird auf einem separaten physischen Rechenknoten angeordnet. Dadurch wird das Risiko eines Absturzes aller Container in einer Gruppe infolge eines Hardwarefehlers verringert.
    registry.DomainName/namensbereich/cf-pi-meine_initialen:latest Der Repository-Pfad zum Ziel, der aus dem eindeutigen Namensbereich und dem Namen des Zielimage besteht. In diesem Beispiel wird für das Image der Name der CF-Knoten-App cf-pi-meine_initialen verwenden.
  2. Warten Sie einige Minuten, bis der Container bereitgestellt und das Netz eingerichtet ist. Wenn der Container bereitgestellt ist, können Sie Ihre Personal Insights-App in einem Browser anzeigen. Geben Sie dazu die folgende URL ein: http://pi-app-meine_initialen.AppDomainName/.
  3. Entfernen Sie die CF-App. Wenn Ihr CF-App-Code in einem Container bereitgestellt ist, können Sie die CF-App entfernen, um die Nutzung Ihres Kontingents zu maximieren. 
    cf delete cf-pi-meine_initialen
    Bestätigen Sie den Löschvorgang, indem Sie y eingeben.
Ihre App ist jetzt in einem Container bereitgestellt.

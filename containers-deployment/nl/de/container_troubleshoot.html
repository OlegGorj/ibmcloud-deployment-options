Debugging für Container durchführen und Protokolle überprüfen

Informieren Sie sich über die Optionen, die Ihnen für die Fehlerbehebung bei Ihren Containern und zum Eingrenzen der jeweiligen Fehlerquelle zur Verfügung stehen.

  1. Listen Sie Ihre Container auf und überprüfen Sie den aktuellen Status.
    • Einzelner Container
      bx ic ps -a
    • Containergruppe
      bx ic groups
  2. Optional: Informieren Sie sich über die Bedeutung der einzelnen Statusangaben und die Tipps, mit denen sich Containerausfälle vermeiden lassen.
  3. Untersuchen Sie Ihre Container und grenzen Sie die Fehlerquelle ein, wenn ein Container ausfällt. Die Fehlernachricht wird in der Antwort im Abschnitt zum Fehler (Failure) aufgeführt.
    • Einzelner Container
      bx ic inspect CONTAINER
    • Containergruppe
      1. Überprüfen Sie die Containergruppe. 
        bx ic group-inspect GROUP
      2. Weitere Informationen erhalten Sie, wenn Sie die Instanzen der Containergruppe auflisten und die einzelnen Instanzen untersuchen. 
        bx ic group-instances GROUP
        bx ic inspect GROUP_INSTANCE
  4. Rufen Sie die Protokolle für die App ab, wenn die App mit einem Fehler beendet wurde.
    Anmerkung: Dieser Befehl kann nur verwendet werden, wenn die App Protokolle in die Standardausgabedatenströme STDOUT und STDERR schreibt. Das Schreiben in diese Standarddatenströme für die Ausgabe ermöglicht es Ihnen, Containerprotokolle auch dann anzuzeigen, wenn ein Container beendet wird oder ausfällt.
    • Einzelner Container
      bx ic logs CONTAINER
    • Containergruppe
      1. Listen Sie die Containerinstanzen einer Containergruppe auf. 
        bx ic group-instances GROUP
      2. Rufen Sie die Protokolle für die einzelnen Instanzen ab. 
        bx ic logs GROUP_INSTANCE
  5. Wenn Sie auf Protokolle eines Containers zugreifen müssen, der nicht mehr verfügbar ist, wie zum Beispiel auf die Instanz einer Containergruppe, die mittels automatischer Wiederherstellung neu erstellt wurde, verwenden Sie Kibana zum Anzeigen Ihrer Protokolle.

    https://logmet.DomainName/

    Protokolle sind in Kibana bis zu sieben Tage verfügbar.

  6. Wenn die Protokolle in einer Datei im Container gespeichert wurden, müssen Sie sich am Container anmelden, um die Protokolle überprüfen zu können.
    Wichtig: Sie können sich nur an einem aktiven Container anmelden. Ist Ihr Container ausgefallen und kann nicht erneut gestartet werden, können Sie möglicherweise nicht mehr auf die Protokolle in dem Container zugreifen. Ziehen Sie deshalb in Betracht, die App so zu ändern, dass in die Standardausgabedatenströme STDOUT und STDERR geschrieben wird, sodass Sie auch dann auf die Protokolldateien zugreifen können, wenn der Container nicht mehr verfügbar ist.
    1. Wenn Ihr Container nicht aktiv ist, starten Sie ihn. 
      bx ic start CONTAINER
    2. Melden Sie sich an dem aktiven Container an. 
      bx ic exec -it CONTAINER bash
    3. Navigieren Sie zu Ihrer Protokolldatei und überprüfen Sie die Protokolle.
  7. Wenn die Protokolle auf einem Datenträger gespeichert wurden und Ihr Container nicht mehr gestartet werden kann, müssen Sie einen neuen Container erstellen, um auf den Datenträger zugreifen zu können.
    1. Erstellen Sie einen neuen Container, der an den Datenträger mit den gespeicherten Protokolldateien angehängt wird. 
      bx ic run --name logcontainer -m 64 --volume MOUNT_PATH_TO_VOLUME registry.DomainName/ibmliberty
    2. Melden Sie sich an dem aktiven Container an. 
      bx ic exec -it logcontainer bash
    3. Navigieren Sie zu dem Mountpfad für den Datenträger und überprüfen Sie Ihre Protokolldateien.
  8. Öffnen Sie ein Support-Ticket für Bluemix, wenn Sie bei der Fehlerbehebung für Ihre Container weitere Unterstützung benötigen.

Fehlerbehebung für die Konfiguration der Befehlszeilenschnittstelle (CLI)

Methodenfehler (rpc) während der Ausführung von 'init'

Wenn Sie versuchen, bx ic init auszuführen, wird eine der folgenden Nachrichten angezeigt.
  • bx ic rpc: Methode CliRpcCmd.IsLoggedIn kann nicht gefunden werden
  • bx ic rpc: Methode CliRpcCmd.IsMinCliVersion kann nicht gefunden werden
Die installierte Version von Cloud Foundry ist eine frühere Version, die von IBM Bluemix Container Service nicht unterstützt wird.

sudo usermod -a -G docker <AKTUELLER_BENUTZER>

Fehlerbehebung für Kontingente

Container kann wegen unzureichenden Organisationskontingents nicht erstellt werden

Wenn Sie einen Container erstellen, wird die Nachricht angezeigt, dass im Speicherbereich kein ausreichendes Kontingent vorhanden ist. 
BXNUI0110E: IBM Containers-Ressourcen konnten nicht zugeordnet werden. Unzureichendes Organisationskontingent für Zuordnung zu neuem Bereich. Anzahl der variabel verknüpften IPs des Typs 'organization' darf nicht überschritten werden. Verwenden Sie 'Organisationen verwalten' in IBM Bluemix, um die Kontingentzuordnung anzupassen.
Diese Nachricht kann angezeigt werden, wenn Sie versuchen, Container in einem Bereich zu erstellen, dem ein zu geringes Kontingent zugeordnet wurde. Für jeden Bereich sind standardmäßig 2 IP-Adressen und 2 GB Speicher erforderlich, die diesem Bereich zugeordnet werden müssen, damit er mit dem Container-Service arbeiten kann.
Gehen Sie wie folgt vor, um den Speicher und das IP-Kontingent auf mehrere Bereiche zu verteilen:
  1. Wählen Sie in der Anzeige 'Organisationen verwalten' bei Ihren Kontodetails eine Organisation aus.
  2. Sehen Sie sich im Abschnitt 'Kontingent' die Details für Container an.
  3. Klicken Sie auf Bearbeiten und ordnen Sie den Speicher und die IP-Adressen den Bereichen in der Organisation zu.
  4. Erstellen Sie einen Container in einem Bereich der Organisation.

Kontingent kann trotz zahlungspflichtigem Konto nicht bearbeitet werden

Sie können Ihre Containerkontingente nicht ändern.
Es kann sein, dass Abweichungen zwischen dem Manager der Organisation und den Rechnungsinformationen der Grund sind.
Der angemeldete Benutzer muss der Manager der Organisation sein. Führen Sie zur Prüfung die folgenden Schritte aus.
  1. Wählen Sie in der Anzeige 'Organisationen verwalten' bei Ihren Kontodetails eine Organisation aus.
  2. Überprüfen Sie in der Benutzerliste, ob der Manager der Organisation derselbe ist wie der mit der ID, mit der Sie sich anmelden.
  3. Melden Sie sich am Konto ab und erneut an.
Wenden Sie sich an den IBM Support, wenn Sie das Containerkontingent auch nach Ausführen der angegebenen Schritte nicht bearbeiten können.

Möglicherweise ist das Image nicht ordnungsgemäß eingerichtet.

Image kann nicht aus der privaten Bluemix-Image-Registry extrahiert werden

Sie haben einen Container aus einem Container-Image in Ihrer privaten Bluemix-Registry erstellt, aber der Container verbleibt im Status Wird erstellt. Beim Ausführen des Befehls bx ic inspect wird die folgende Fehlernachricht im Bereich Env Ihrer CLI-Antwort angezeigt: 
"Env": [
            "metadata_exit_code=-1",
            "logging_password=",
            "space_id=0e65c436-9413-46e2-8b42-ff06aa74ac21",
            "logstash_target=logmet.opvis.bluemix.net:9091",
            "metadata_err_msg=Cannot pull image from registry: Pull failed with error message failed 
             to register layer: ApplyLayer exit status 1 stdout:  stderr: Container ID 1234567890 
             cannot be mapped to a host ID",
            "metrics_target=logmet.opvis.bluemix.net:9095",
            "metadata_running=1"
Im IBM Bluemix Container Service ist die Docker-Funktion für Benutzernamensbereiche in Docker Engine aktiviert. Die Funktion für Benutzernamensbereiche wurde erstmals in Version 1.10 von Docker eingeführt und bietet die Möglichkeit, einen Rootbenutzer im Container einem Benutzer ohne Rootberechtigung außerhalb des Containers zuzuordnen. Aus diese Weise können die Berechtigungen eines Benutzers, der in dem Container Rootaktionen ausführen darf, außerhalb des Containers eingeschränkt werden, sodass kein Zugriff auf Dateien auf dem Host und in anderen Containern möglich ist und diese Dateien auch nicht geändert oder entfernt werden können.
Anmerkung: Der Docker-Benutzernamensbereich unterscheidet sich von dem Namensbereich, den Sie in Bluemix zur Angabe Ihrer privaten Image-Registry eingerichtet haben.

Wenn Sie mit dem Befehl bx ic run einen Container aus einem Container-Image erstellen, wird das Image von Ihrer privaten Bluemix-Image-Registry in IBM Cloud Docker Engine geladen. Docker Engine beginnt dann mit der Erstellung der jeweiligen Containerebenen. Abhängig von den Angaben im Container-Image werden gegebenenfalls App-Dateien von Ihrer lokalen Maschine zum Container hinzugefügt. Alle diese Dateien werden von einer Benutzer-ID (UID) und Gruppen-ID (GID) auf Ihrer lokalen Maschine erstellt. Die Benutzer-ID und die Gruppen-ID geben die Berechtigungen des Benutzers auf der lokalen Maschine an. Wenn Sie diese Dateien zum Container hinzufügen möchten, werden die betreffende Benutzer- bzw. Gruppen-ID einer Benutzer- bzw. Gruppen-ID im Container zugeordnet. Die zugeordnete Benutzer-ID und die zugeordnete Gruppen-ID verfügen im Container unter Umständen über andere Berechtigungen als auf der lokalen Maschine. Die Benutzer-ID und die Gruppen-ID des Benutzers der lokalen Maschine ohne Rootberechtigung werden einem Rootbenutzer im Container zugeordnet. Auf diese Weise kann der Benutzer im Container wie ein Rootbenutzer agieren und auf Containerdateien zugreifen, Änderungen an diesen vornehmen und Containerdateien löschen. Da der Container in einer isolierten Umgebung ausgeführt wird, ist über die UID und GID kein Zugriff auf Dateien auf dem Host möglich, da diesen IDs außerhalb des Containers keine Rootberechtigungen zugewiesen sind.

Um die UID und die GID jeweils einer entsprechenden ID in einem Container zuzuordnen, erfordert die Funktion für Benutzernamensbereiche, dass alle Dateien mit einer UID und GID zwischen 0 und 65536 erstellt worden sind.

Der angezeigte Fehler tritt auf, wenn eine Datei mit einer Benutzer- oder einer Gruppen-ID jenseits dieses Zahlenbereichs erstellt und dann in eine Image-Ebene des Containers aufgenommen wurde. Wenn Sie einen Container aus dem Image erstellen, wird jede einzelne Image-Ebene von Docker Engine erstellt. Wird eine Datei mit einer Benutzer- oder Gruppen-ID erstellt, die nicht aus dem angegebenen Bereich stammt, tritt bei Docker Engine ein Fehler auf, weil diese Benutzer-ID und Gruppen-ID nicht zu einer gültigen ID innerhalb des Containers zugeordnet werden konnte. Es tritt ebenfalls ein Fehler bei Docker Engine auf, wenn eine Datei in einer Image-Ebene verwendet und in der nächsten Image-Ebene entfernt wird.

Benutzernamensbereich in lokaler Docker Engine-Installation aktivieren
Wenn Sie Ihre Container-Images lokal erstellen, sollten Sie sicherstellen, dass in der lokalen Docker Engine-Installation die Funktion für Benutzernamensbereiche aktiviert ist. Auf diese Weise werden alle Benutzer-IDs (UIDs) und Gruppen-IDs (GIDs) auf Ihrer lokalen Maschine, die sich außerhalb des gültigen Benutzernamensbereichs befinden, automatisch einer gültigen ID innerhalb des Bereichs zugeordnet. Testen Sie Ihr Container-Image lokal, bevor Sie es mit einer Push-Operation in Ihre private Bluemix-Image-Registry übertragen und mit IBM Bluemix Container Service verwenden. Weitere Informationen dazu, wie Sie die Funktion für Benutzernamensbereiche aktivieren, erfahren Sie im Abschnitt Starting the daemon with user namespaces enabled.
Betroffene Image-Ebene auf Kommentar setzen und Eigentumsrecht betroffener Dateien ändern
Versuchen Sie, die Image-Ebene in der Dockerfile auf Kommentar zu setzen und bei einer Ebene nach der anderen die Kommentarzeichen zu entfernen, wenn Sie die Datei, durch die das Problem verursacht wird, nicht ermitteln können. Erstellen Sie Ihr Image und einen Container, um die betroffene Image-Ebene zu ermitteln.

Führen Sie den Befehl chown [-R] root:root <datei_oder_ordner> aus, um das Eigentumsrecht der betroffenen Datei zu ändern, die auf Ihrer lokalen Maschine gespeichert ist, oder fügen Sie diesen Befehl zu der betroffenen Image-Ebene in der Dockerfile hinzu.

Beispiel-Dockerfile

RUN curl -fsSL "https://www.kernel.org/pub/software/scm/git/git-${GIT_VERSION}.tar.gz" \
        | tar -xzC /tmp \
  && cd /tmp/git-${GIT_VERSION} \
  && ./configure \
  && make all \
  && make install \
  && chown -R root:root /tmp

Image kann nicht aus der privaten Bluemix-Image-Registry exportiert werden

Sie sind bei IBM Bluemix Container Service angemeldet und wollen ein Image aus Ihrer privaten Bluemix-Registry auf Ihre lokale Maschine mit dem Befehl bx ic pull extrahieren. Der Befehl schlägt mit der Nachricht fehl, dass der Befehl bx ic pull nicht unterstützt wird.
Der Befehl bx ic pull wird nicht unterstützt, um ein Image aus Ihrer privaten Bluemix-Registry auf Ihre lokale Maschine zu extrahieren.
Führen Sie zum Exportieren eines Images aus Ihrer privaten Bluemix-Registry die folgenden Schritte aus.
  1. Melden Sie sich bei IBM Bluemix Container Service an. Wenn Sie bereits angemeldet sind, stellen Sie sicher, dass Ihre Umgebungsvariablen nicht so eingestellt sind, dass native Docker-Befehle für IBM Bluemix Container Service verwendet werden.
  2. Listen Sie Ihre Images auf. 
    bx ic images
  3. Extrahieren Sie ein Image aus Ihrer privaten Bluemix-Registry auf Ihre lokale Maschine. Verwenden Sie den Namen oder die ID des Images, der bzw. die im vorherigen Schritt zurückgegeben wurde. 
    docker pull <name_oder_id_des_images>
  4. Überprüfen Sie, ob das Image auf Ihre lokale Maschine extrahiert wurde. 
    docker images
  5. Speichern Sie Ihr Image in einer .tar-Datei. 
    docker export <name_oder_id_des_images> > <name_des_exportierten_images>.tar

Anmeldeaufforderung während docker push

Wenn Sie versuchen, docker push auszuführen, um ein Image zu Ihrer privaten Bluemix-Image-Registry hinzuzufügen, werden Sie zur Eingabe von Anmeldeberechtigungsnachweisen aufgefordert. Wenn Sie Docker- oder Bluemix-Berechtigungsnachweise eingeben, schlägt die Authentifizierung fehl.
Die Authentifizierung kann möglicherweise aus folgenden Gründen fehlschlagen.
  • Das Trägertoken, das die Cloud Foundry-Authentifizierungsinformationen enthält, ist abgelaufen.
  • Der Bluemix-Berechtigungsservice ist nicht verfügbar.
  • Gehen Sie wie folgt vor, wenn Sie das IBM Bluemix Container Service-Plug-in verwenden.
    1. Melden Sie sich bei der Cloud Foundry-CLI an, indem Sie den Befehl zum Aktualisieren des Trägertokens auf Ihrem Computer ausführen. 
      bx login [-a api.DomainName] [-sso]
      Der Parameter --sso für Single Sign-on ist erforderlich, wenn Sie sich mit einer eingebundenen ID anmelden. Rufen Sie bei Verwendung dieser Option https://login.ng.bluemix.net/UAALoginServerWAR/passcode auf, um einen nur für eine einzige Anmeldung gültigen Kenncode abzurufen. Führen Sie die folgenden untergeordneten Schritte aus, wenn Sie die Option --sso nicht angeben:
      1. Geben Sie unter E-Mail die IBM ID an, die Sie zum Anmelden bei Bluemix verwenden.
      2. Geben Sie unter Kennwort das Kennwort für die IBM ID an, mit der Sie sich bei Bluemix anmelden.Durch das Eingeben Ihres Benutzernamens und Kennworts werden die Organisation und der Bereich von Bluemix abgerufen.
      3. Geben Sie die Nummer für eine Ihrer Bluemix-Organisationen ein.
      4. Geben Sie die Nummer für einen Ihrer bestehenden Bluemix-Bereiche an.
    2. Initialisieren Sie die IBM Bluemix Container Service-CLI, um das Trägertoken erneut zu erstellen. 
      bx ic init

Wenn die Eingabeaufforderung immer noch angezeigt wird, nachdem Sie sich erneut angemeldet haben, ist der Bluemix-Berechtigungsservice nicht verfügbar. Versuchen Sie es zu einem späteren Zeitpunkt erneut.

Fehlerbehebung für Container

Fehler 502 Bad Gateway beim Zugriff auf eine App

Sie haben eine Containergruppe erfolgreich bereitgestellt; beim Öffnen der URL für die App wird jedoch der folgende Fehler angezeigt. 
502 Bad Gateway: Registered endpoint failed to handle the request.
  1. Das Festlegen der Routing-Einstellungen ist noch nicht abgeschlossen.
  2. Möglicherweise antwortet der Container nicht.
  3. Möglicherweise liegt ein Routing-Fehler vor, obwohl die Containergruppe erfolgreich ausgeführt wird.
  1. Warten Sie noch ein paar Minuten, bis das Routing abgeschlossen ist. Möglicherweise kann die Verbindung über die angegebene Route erst 5 Minuten nach dem Ausführen des Containerprozesses hergestellt werden.
  2. Wenn Sie nach 5 Minuten Wartezeit vermuten, dass ein Fehler vorliegt, können Sie prüfen, ob der Container antwortet. Binden Sie zu diesem Zweck eine IP-Adresse an den aktiven Container und führen Sie anschließend einen Befehl 'curl' für diese IP-Adresse und den zugehörigen Port aus.
    1. Führen Sie den folgenden Befehl aus, um die ID für eine der Containerinstanzen in der Gruppe abzurufen. Notieren Sie die Container-ID und den Port. 
      bx ic ps
    2. Führen Sie den folgenden Befehl aus, um eine IP-Adresse anzufordern. 
      bx ic ip-request
    3. Führen Sie den folgenden Befehl aus, um die IP-Adresse an die Containerinstanz zu binden. 
      bx ic ip-bind IP-adresse container-ID
    4. Geben Sie in einem Browser die URL der Anwendung, mit der IP-Adresse und dem Port ein. Beispiel: http://IP-adresse:port

      Wenn Sie nicht auf die Anwendung zugreifen können, liegt ein Problem mit der Containergruppe vor und die Gruppe muss erneut erstellt werden. Wenn Sie auf die Anwendung zugreifen können, wird der Container erfolgreich ausgeführt und es liegt ein Routing-Problem vor. Wenden Sie sich an den IBM Bluemix-Support.

Containergruppeninstanzen werden während des Starts vom automatischen Wiederherstellungsmodus von IBM Bluemix Container Service wiederholt neu erstellt.

Sie haben eine Containergruppe erstellt und die automatische Wiederherstellung aktiviert. Während des Starts der Containergruppe, werden nicht antwortende Gruppeninstanzen entfernt und wiederholt neu erstellt.

Die App, die in Ihrer Containergruppe ausgeführt wird, braucht zu lange, bis sie an dem zugänglichen Port empfangsbereit ist. Wenn Sie die automatische Wiederherstellung für Ihre Containergruppe aktiviert haben, prüft IBM Bluemix Container Service den Zustand jeder Instanz mit einer HTTP-Anforderung an den zugänglichen TCP-Port, nachdem die Gruppe erstellt wurde. Wenn eine Instanz innerhalb von drei Minuten nicht am zugänglichen Port empfangsbereit ist, wird die Instanz als 'nicht antwortend' gekennzeichnet und anschließend entfernt, neu erstellt und erneut gestartet. Dieser Prozess wiederholt sich, bis die maximale Anzahl neu erstellter Instanzen erreicht ist. Anschließend wird die automatische Wiederherstellung inaktiviert.

Um eine App in einer Containergruppe mit inaktivierter automatischer Wiederherstellung auszuführen, stellen Sie sicher, dass die App innerhalb von zwei Minuten gestartet wird und am zugänglichen Port empfangsbereit ist. Abhängig vom Typ der App gibt es verschiedene Verfahren, mit denen ein schneller Start Ihrer Anwendung sichergestellt wird.

Bluemix-Service wird nicht direkt an einen Container gebunden

Ihr Bluemix-Service wird unter Verwendung der Servicebindung nicht direkt an einen Container gebunden.
Einige Bluemix-Services unterstützen das direkte Binden an einen Container nicht und führen dazu, dass die Containererstellung fehlschlägt. Bei diesen Services übergibt die Servicebindung die Umgebungsvariable VCAP_SERVICES nicht an den Container. Zur Umgehung dieses Problems können Sie mit der CLI eine Cloud Foundry-App erstellen, die beim Senden von VCAP_SERVICES vom Service an den Container als Brücke fungiert.
  1. Wählen Sie einen Bereich aus, in dem ein Container bereitgestellt werden soll (z. B. dev).
  2. Suchen Sie eine bestehende App oder erstellen Sie eine Cloud Foundry-App, an die der Bluemix-Service gebunden werden soll. Diese App wird verwendet, um die VCAP_SERVICES-Informationen zu übergeben, sodass sie als Brücken-App bezeichnet wird.
    Gehen Sie wie folgt vor, um eine Cloud Foundry-App zu erstellen:
    Tipp: Die App, die durch Ausführen dieser Schritte erstellt wird, dient als Platzhalter zum Speichern der VCAP_SERVICES-Services. Damit weniger Kosten als durch eine aktive App anfallen, wird diese App nicht gestartet oder ausgeführt. Die App muss nicht ausgeführt werden, um dem Container die VCAP_SERVICES-Services bereitzustellen.
    1. Erstellen Sie ein Verzeichnis und eine Textdatei in Ihrem Arbeitsverzeichnis für Apps. Beispiel: <verzeichnis>/<leere_datei.txt>. Diese Datei dient als Quellendatei für die App, sie muss jedoch keinen Inhalt enthalten.
      • Beispiel unter Windows: C:\Benutzer\benutzer\containerbridge\empty_file.txt
      • Beispiel unter Linux: /ausgangsverzeichnis_des_benutzers/containerbridge/leere_Datei.txt
      • Beispiel unter OS X: /Benutzer/ausgangsverzeichnis_des_benutzers/containerbridge/leere_Datei.txt
    2. Führen Sie den folgenden Befehl aus, um eine gebührenfreie App ohne Laufzeit (zero-runtime-App) zu erstellen. Die App kann über einen beliebigen Namen verfügen. Häufig wird jedoch der Name containerbridge verwendet, da die App als Brücke zwischen dem Container und einem Service dient. 

      cf push <appname> -p <verzeichnis> -i 1 -d AppDomainName -k 1M -m 64M --no-hostname --no-manifest --no-route --no-start
      Beispiel: 
      cf push containerbridge -p containerbridge -i 1 -d AppDomainName -k 1M -m 64M --no-hostname --no-manifest --no-route --no-start
  3. Wenn der Bluemix-Service, den Sie binden möchten, noch nicht hinzugefügt wurde, fügen Sie den Service im Bluemix-Bereich hinzu.
    1. Um eine Liste der verfügbaren Bluemix-Services abzurufen, führen Sie den Befehl cf marketplace aus. Die daraufhin angezeigte Liste enthält den Namen des Bluemix-Service, den zugehörigen Plantyp und eine Beschreibung des Service.
    2. Führen Sie den Befehl create-service aus und geben Sie darin den Servicenamen und den Serviceplan aus der mit cf marketplace abgerufenen Liste an. Geben Sie außerdem Ihren eigenen Instanznamen für den Service an. 
      cf create-service <service> <plan> <serviceinstanz>
      Beispiel: 
      cf create-service clear-db spark meine_clear-db
  4. Binden Sie den Bluemix-Service an die App, indem Sie den Befehl 'bind-service' ausführen. Wiederholen Sie den Befehl, um weitere Bluemix-Services an die Brücken-App zu binden. 
    cf bind-service <app-name> <serviceinstanz>
    Beispiel: 
    cf bind-service containerbridge meine_clear-db
  5. Wenn Sie aufgefordert werden, ein erneutes Staging für die App durchzuführen, führen Sie den folgenden Befehl aus. 
    cf restage containerbridge
  6. Überprüfen Sie mit dem folgenden Befehl die Informationen für den Bluemix-Service, der über die containerbridge-App übergeben wird. 
    cf env <app-name>
    Beispiel: 
    cf env containerbridge

    Die Ergebnisse umfassen die eingefügten Bluemix-Serviceparameter mit der Umgebungsvariablen VCAP_SERVICES.

    Beispiel: 
    {
 "VCAP_SERVICES": {
  "<service_name>": [
   {
          "credentials": {
     "hostname": "<hostname_id>",
     "jdbcUrl": "<jdbc_URL>",
     "name": "<encoded_name>",
     "password": "<encoded_password>",
     "port": "<port_number>",
     "uri": "<service_URI>",
     "username": "<encoded_username>"
    },
    "label": "<service_name>",
    "name": "<service_instance>",
    "plan": "<plan_name>",
    "tags": [
     "<tag1>",
     "<tag2>",
     "<tag3>"
    ]
   }
Erstellen Sie nach dem Erstellen der Bindung zwischen den Bluemix-Services und der containerbridge-App einen Container und binden Sie die containerbridge-App an den Container. Weitere Informationen finden Sie unter dem Flag CCS_BIND_APP in den Umgebungsvariablen unter Einzelnen Container erstellen und Containergruppe erstellen.
Tipp: Löschen Sie die Brücken-App nicht, nachdem sie an einen Container gebunden wurde. Das Löschen einer gebundenen Brücken-App führt zu einer Blockierung des Containers. Wenn Sie die Brücken-App nicht mehr benötigen, löschen Sie zuerst den Container sowie alle anderen Container, die die Bridge-App verwenden, und löschen Sie erst dann die Brücken-App.

Meldung Cannot create child process mit dem Befehl with su

Wenn der Linux-Befehl zur Verwendung eines Ersatzbenutzers (substitute user, su) zum Bearbeiten der Benutzer für einen Container verwendet wird, empfangen Sie die Nachricht su: cannot create child process: Resource temporarily unavailable, obwohl der Befehl su in der lokalen Docker-Umgebung funktioniert hat. Dieser Fehler kann beim Anmelden an einem Container auftreten oder beim Ausführen des Befehls su als Bestandteil eines Scripts innerhalb des Containers.

Beispiel: 

bx ic exec -it mein_container bash
adduser benutzer
su - benutzer
Ergebnis: 
su: cannot create child process: Resource temporarily unavailable

Dieser Fehler tritt häufig in Containern auf, die mit Centos 7-Images erstellt wurden. Der Fehler kann jedoch auch bei anderen Imagetypen auftreten.

Um diesen Fehler zu beheben, bearbeiten Sie die Dockerfile und erstellen Sie das Image erneut oder melden Sie sich bi dem aktiven Container an.
So beheben Sie den Fehler durch Bearbeiten der Dockerfile:
  1. Erstellen sie in dem lokalen Verzeichnis, in dem Sie das Image erstellen, einen Ordner etc, sofern dieser Ordner noch nicht vorhanden ist. 
    mkdir etc
  2. Navigieren Sie zum Verzeichnis etc. 
    cd etc
  3. Erstellen Sie im Ordner etc ein Verzeichnis pam.d. 
    mkdir pam.d
  4. Navigieren Sie zum Verzeichnis pam.d. 
    cd pam.d
  5. Erstellen Sie im Verzeichnis pam.d eine Datei mit dem Namen su und öffnen Sie die Datei in einem Texteditor.
  6. Kopieren Sie den folgenden Text mit der Einstellung optional für alle Variablen session in die Datei su und speichern Sie die geänderte Datei. 
    #%PAM-1.0
auth            sufficient      pam_rootok.so
# Entfernen Sie die Kommentarzeichen aus der folgenden Zeile, damit Benutzern der Gruppe "wheel" implizit vertraut wird.
#auth           sufficient      pam_wheel.so trust use_uid
# Entfernen Sie die Kommentarzeichen aus der folgenden Zeile, damit ein Benutzer Mitglied in der Gruppe "wheel" sein muss.
#auth           required        pam_wheel.so use_uid
auth            substack        system-auth
auth            include         postlogin
account         sufficient      pam_succeed_if.so uid = 0 use_uid quiet
account         include         system-auth
password        include         system-auth
session         optional        system-auth
session         optional        postlogin
session         optional        pam_xauth.so
  7. Geben Sie in der Dockerfile für das Image die folgenden Informationen an. 
    RUN sed -i 's/session.*include/session        optional/' /etc/pam.d/su
  8. Erstellen Sie das Image erneut. Weitere Informationen finden Sie unter Image erstellen.
  9. Erstellen Sie den einzelnen Container oder die Containergruppe erneut. Weitere Informationen finden Sie unter Einzelnen Container erstellen oder Containergruppe erstellen.
  10. Führen Sie den folgenden Befehl aus, um das Ergebnis zu prüfen. 
    bx ic logs containername
    Ergebnis: 
    %Changing password for user benutzername.
passwd: all authentication tokens updated successfully.
Gehen Sie wie folgt vor, um das Problem durch die Anmeldung am aktiven Container zu beheben:
  1. Erstellen Sie den Container.
  2. Melden Sie sich am Container an. 
    bx ic exec -it mein_container bash
  3. Navigieren Sie zum Verzeichnis etc. 
    cd etc
  4. Navigieren Sie zum Verzeichnis pam.d. 
    cd pam.d
  5. Öffnen Sie den visuellen Editor durch Absetzen des Linux-Befehls vi. 
    vi su
  6. Ändern Sie in der Datei su alle Sitzungsvariablen in optional und speichern Sie die Änderungen in der Datei. Weitere Informationen zum Verwenden des Editors vi finden Sie unter How to Use the vi Editor. 
    #%PAM-1.0
auth            sufficient      pam_rootok.so
# Entfernen Sie die Kommentarzeichen aus der folgenden Zeile, damit Benutzern der Gruppe "wheel" implizit vertraut wird.
#auth           sufficient      pam_wheel.so trust use_uid
# Entfernen Sie die Kommentarzeichen aus der folgenden Zeile, damit ein Benutzer Mitglied in der Gruppe "wheel" sein muss.
#auth           required        pam_wheel.so use_uid
auth            substack        system-auth
auth            include         postlogin
account         sufficient      pam_succeed_if.so uid = 0 use_uid quiet
account         include         system-auth
password        include         system-auth
session         optional        system-auth
session         optional        postlogin
session         optional        pam_xauth.so
  7. Melden Sie sich als Benutzer an. 
    su - benutzer
  8. Führen Sie den Linux-Befehl id aus, um die Konfigurationsinformationen für den Benutzer anzuzeigen. 
    id
    Ergebnis: 
    uid=1000(benutzer) gid=1000(benutzer) groups=1000(benutzer)

Container stoppt nach Erstellung und wird beendet

Sie haben einen Container erstellt, jedoch wird dieser sofort beendet.

Sie haben einen Container lokal ausgeführt und die Ausführung war erfolgreich. Wenn Sie den Container jedoch in Bluemix ausführen, tritt eine der folgenden Situationen auf.
  • Im Bluemix-Dashboard ändert sich der Status des Containers von Aktiv in Gestoppt.
  • Wenn Sie den Befehl bx ic inspect container über die CLI ausführen, wird der Status mit Beendet (Shutdown) angegeben.
Sie haben außerdem den Befehl bx ic logs ausgeführt, jedoch ließen sich dadurch keine Probleme mit dem Container finden.

Wenn Sie einen Container in Bluemix ausführen, wird der Container immer im Modus mit aufgehobener Zuordnung ausgeführt, ähnlich wie bei der Ausführung des Befehls docker run -d, wenn ein Container lokal ausgeführt wird. Infolgedessen werden auch Container, die lokal gestartet werden, nicht kontinuierlich in Bluemix ausgeführt.

Sie müssen den Container zu einer Anwendung mit langer Laufzeit machen, indem Sie die Dockerfile bearbeiten. Wenn Sie zum Beispiel einen Befehl hinzugefügt haben, um die Phrase "hello world" alle zehn Sekunden auszugeben, macht diese fortlaufende Aktion Ihren Container zu einer Anwendung mit langer Laufzeit. 
ENTRYPOINT ["/bin/sh"]
CMD ["-c","while true; do echo hello world; sleep 10;done"]

Gelöschter Container wird weiterhin in der Bluemix-GUI angezeigt

Sie haben in der Bluemix-GUI einen Container gelöscht, der Container wird jedoch weiterhin in der Bluemix-GUI angezeigt und belegt Kontingente.

Der Löschvorgang konnte über die Bluemix-GUI nicht abgeschlossen werden.

Löschen Sie den Container über die Befehlszeile.

Gehen Sie bei einzelnen Containern wie folgt vor.
  1. Löschen Sie den betreffenden Container, indem Sie eine der folgenden Befehlszeilenoptionen ausführen:
    • IBM Bluemix Container Service-Plug-in mit bx ic-Befehlen 
      bx ic rm [-f] CONTAINER [CONTAINER]
    • IBM Bluemix Container Service-Plug-in mit docker-Befehlen 
      docker rm [-f] CONTAINER
  2. Überprüfen Sie, ob der Container entfernt wurde, indem Sie einen der folgenden Befehle ausführen, der alle Container in Ihrem Bereich auflistet:
    • IBM Bluemix Container Service-Plug-in mit bx ic-Befehlen 
      bx ic ps -a
    • IBM Bluemix Container Service-Plug-in mit docker-Befehlen 
      docker ps -a
Gehen Sie bei Containergruppen wie folgt vor.
  1. Löschen Sie die Containergruppe, indem Sie die den folgenden Befehl ausführen.
    • IBM Bluemix Container Service-Plug-in
      bx ic group-remove [-f] GROUP [GROUP]
  2. Stellen Sie sicher, dass der Container entfernt wurde, indem Sie den Befehl ausführen, der alle Container in Ihrem Bereich auflistet.
    • IBM Bluemix Container Service-Plug-in
      bx ic groups [-q]

Wenn der Container nach dem Ausführen der Befehle weiterhin in der Bluemix-GUI angezeigt wird, warten Sie ab, ob er später entfernt wird.

Containergruppe zeigt Status HAS_INACTIVE_INSTANCES an

Beim Ausführen von bx ic group-inspect wird für die Containergruppe der Status HAS_INACTIVE_INSTANCES angezeigt.

Wenn der Netzbetrieb beim Erstellen der Gruppe nicht konfiguriert werden kann oder wenn die Statusprüfung für mindestens eine Instanz fehlschlägt, dann wird der Instanzstatus in INACTIVE geändert. Wenn eine Gruppe eine inaktive Instanz enthält, so wird der Gruppenstatus in HAS_INACTIVE_INSTANCES geändert.

Wenn das Problem auftritt, wenn die Statusprüfung für eine Instanz fehlschlägt, dann kann es behoben werden, ohne dass hierzu eine Aktion erforderlich ist. Tritt der Fehler bei der Ersterstellung der Gruppe auf, dann können Sie versuchen, die Gruppe erneut zu erstellen. Tritt das Problem nach der Neuerstellung erneut auf, kann der Netzbetrieb für die App möglicherweise nicht konfiguriert werden, wenn die App in einer Gruppe bereitgestellt wird. Versuchen Sie, die App stattdessen in einem einzelnen Container bereitzustellen.

Fehlerbehebung für Datenspeicher

Berechtigungsfehler bei Änderung des Eigners für das Verzeichnis eines angehängten Datenträgers in einem Container

Wenn Sie versuchen, das Eigentumsrecht für eine Datei oder ein Verzeichnis eines Datenträgers zu ändern, der an einen Container angehängt ist, wird ein Berechtigungsfehler ausgegeben.
Bei IBM Bluemix Container Service ist die Funktion für Benutzernamensbereiche für Docker Engine aktiviert. Das effektive Stammelement innerhalb des Containers wird einem Benutzer ohne Rootberechtigung auf dem Rechenhost zugeordnet. Die Quelle des angehängten Datenträgers ist ein Netzdateisystem (NFS). Da das NFS keine Benutzernamensbereiche kennt, werden für die Operationen in angehängten Dateien und Verzeichnissen innerhalb des Containers alle nicht benutzerbezogenen Prüfungen ausgeführt.

  1. Fügen Sie in der Dockerfile für ein Image die folgenden Zeilen hinzu, um einen Benutzer mit einer verfügbaren Benutzer-ID (UID) zu erstellen. Sie können dem Benutzer beispielsweise den Namen bearbeiter_des_datenträgers geben.
    RUN groupadd --gid 1010 bearbeiter_des_datenträgers
RUN useradd --uid 1010 --gid 1010 -m --shell /bin/bash bearbeiter_des_datenträgers
  2. Erstellen Sie das Image und übertragen Sie es mit einer Push-Operation in Ihre private Bluemix-Registry.
  3. Erstellen Sie einen Container mit dem Image, an das der Datenträgerpfad angehängt ist.
    Tipp: Führen Sie bx ic namespace-get aus, um Ihren Namensbereich abzurufen, und ersetzen Sie <mein_namensbereich> durch Ihren Namensbereich.
    bx ic run --volume datenträgername:/path/zu/dateien --name korrektur_datenträgerberechtigungen registry.DomainName/<meine_namensbereich>/image
  4. Melden Sie sich am Container an.
    bx ic exec -it containername bash
  5. Definieren Sie eine Umgebungsvariable namens MOUNTPATH für den Mountpfad.
    MOUNTPATH="/pfad/zu/dateien"
  6. Fügen Sie den Benutzer zur Gruppe root hinzu.
    adduser bearbeiter_des_datenträgers root
  7. Legen Sie Berechtigungen so fest, dass alle Gruppenmitglieder Lese- und Schreibzugriff auf den Datenträgermount haben. 
    chmod 775 $MOUNTPATH
  8. Erstellen Sie einen Ordner im Mountpfad und legen Sie die Lese- und Schreibberechtigungen fest. In diesem Beispiel heißt der Ordner daten.
    su -c "mkdir -p $MOUNTPATH/daten" bearbeiter_des_datenträgers
    su -c "chmod 700 $MOUNTPATH/daten" bearbeiter_des_datenträgers
  9. Stellen Sie sicher, dass der Benutzer bearbeiter_des_datenträgers Zugriff auf die im Mountpfad enthaltenen Dateien und Verzeichnisse hat. 
    sudo -u bearbeiter_des_datenträgers ls -al $MOUNTPATH
  10. Optional: Verlinken Sie einen Ordner im Container mit dem Ordner daten auf dem Datenträger. Dieser Link bewirkt die Zuordnung der Schreibberechtigungen von $MOUNTPATH/daten zu allen Ordnern im Container. 
    ln -sf $MOUNTPATH/daten /ordner/in/container/
  11. Wenn die Änderungen ohne Berechtigungsfehler ausgeführt werden können, entfernen Sie den Rootbenutzer bearbeiter_des_datenträgers der Gruppe.
    deluser bearbeiter_des_datenträgers root

Protokolle von Containergruppeninstanzen werden in einem Datenträger überschrieben

Als Sie eine Containergruppe erstellt haben, haben Sie einen Datenträger angehängt und für jede Instanz der Containergruppe Protokolle an den Datenträger ausgeben lassen. Jede Containergruppeninstanz schreibt in dieselben Dateien und die Dateien werden durch jede Instanz überschrieben.
Beispiel: 
bx ic group-create –p 9080 –m 64 -v log_vol:/opt/ibm/logs -e LOG_LOCATIONS=/opt/ibm/logs/messages.log,/opt/ibm/logs/debug.log,/opt/ibm/logs/trace.log -–name meine_containergruppe registry.domänenname/ibmliberty:latest
Jede Containerinstanz schreibt eine Datei auf den Datenträger, als handele es sich bei ihr um das einzige Ausgabeprogramm. Jede Instanz verfügt im Hauptspeicher über lokalen Cache. Die Dateien im Datenträger werden wiederholt durch die im Cache zwischengespeicherten Daten aus jeder Containerinstanz ersetzt.
Passen Sie den Namen der Protokolldatei so an, dass er eine Kennung (ID) enthält, damit sich die Dateien für die einzelnen Instanzen jeweils unterschieden. Schließen Sie zum Beispiel die Variable $HOSTNAME in den Protokolldateinamen ein, der von der App generiert und bei der Erstellung der Containergruppe angegeben wird.
Beispiel: 
bx ic group-create –p 9080 –m 64 -v log_vol:/opt/ibm/logs -e LOG_LOCATIONS=/opt/ibm/logs/messages.$HOSTNAME.log,/opt/ibm/logs/debug.$HOSTNAME.log,/opt/ibm/logs/trace.$HOSTNAME.log -–name meine_containergruppe registry.domänenname/ibmliberty:latest

Die Übertragung von Daten zu oder von einem Datenträger dauert länger als erwartet

Das Schreiben auf oder Lesen aus einem Datenträger dauert länger als erwartet.
In IBM Bluemix Container Service können Datenträger nicht auf den lokalen Host zugreifen. Daher werden alle Datenträger an eine NFS-Freigabe (NFS - Network File System, Netzdateisystem) angehängt. Die Leistung des Datenträgers wird durch die Größe und den IOPS-Wert (Input/Output per Second - E/A-Operationen pro Sekunde) Ihrer Dateifreigabe vorgegeben und ist darüber hinaus von der Arbeitslast Ihrer Anwendung abhängig. Möglicherweise ist die Dateifreigabe, auf der Ihr Datenträger gehostet ist, nicht für die Arbeitslast Ihrer Datenträger und Container geeignet. Darüber hinaus können mehrere Datenträger auf einer Dateifreigabe um die Ressourcen der Dateifreigabe konkurrieren.

Zur Erstellung einer neuen Dateifreigabe müssen Sie über die Rolle des Organisationsmanagers verfügen. Detaillierte Informationen finden Sie unter Benutzer und Rollen.

  1. Bestimmen Sie die ID der Dateifreigabe, auf der Ihr Datenträger gehostet wird. 
    bx ic volume-inspect mein_datenträger
    {
    "fsId": "fa277ff4-8a64-435b-9b75-0f11d59a3ae4", 
    "hostPath": "/vol/fa277ff4-8a64-435b-9b75-0f11d59a3ae4/mein_datenträger",
    "otherSpaceVisibility": [], 
    "spaceGuid": "fa277ff4-8a64-435b-9b75-0f11d59a3ae4", 
    "volName": "mein_datenträger"
}
  2. Zeigen Sie die Dateifreigabe an, um ihre Größe und IOPS-Werte zu bestimmen. 
    bx ic volume-fs
    Name                                   Size   IOPS/GB   Created               State   
fa277ff4-8a64-435b-9b75-0f11d59a3ae4   20     4.00      2016-05-20 13:48:10   READY
  3. Zeigen Sie die verfügbaren Größen in GB für Ihre Dateifreigaben an. 
    bx ic volume fs-flavor
    [
    20, 
    40, 
    80, 
    100, 
    250, 
    500, 
    1000, 
    2000, 
    4000, 
    8000, 
    12000
]
  4. Ermitteln Sie, welche Größe und IOPS-Werte für die Arbeitslast Ihrer App erforderlich sind. Gültige IOPS-Werte sind 0,25, 2 oder 4 E/A-Operationen pro Sekunde pro GB. Die Dateifreigabegröße und die E/A-Operationen pro Sekunde pro GB werden miteinander multipliziert und bilden die Summe der E/A-Operationen pro Sekunde. Diese Tabelle enthält Beispiele für Dateifreigaben verschiedener Größen bei 4 E/A-Operationen pro Sekunde pro GB.
    Tabelle 1. Beispiele für Dateifreigabegrößen und E/A-Operationen pro Sekunde (IOPS) pro GB

  5. Erstellen Sie eine neue Dateifreigabe. Die erste Zahl gibt die Größe in GB an. Die zweite Zahl gibt den IOPS-Wert für die Dateifreigabe an. 
    bx ic volume-fs-create meine_neue_dateifreigabe 1000 4
  6. Überprüfen Sie die neue Dateifreigabe und warten Sie, bis ihr Status 'Bereit' lautet. 
    bx ic volume-fs

    bx ic volume-create mein_neuer_datenträger meine_neue_dateifreigabe
  8. Führen Sie einen Container oder eine Containergruppe von Ihrem Image aus mit dem neuen angehängten Datenträger aus.
  9. Sichern Sie die Daten von mein_datenträger und kopieren Sie die Daten nach neuer_datenträger. Weitere Informationen finden Sie unter Daten sichern und wiederherstellen.

Parsing-Fehler beim Erstellen eines Containers mit einem Datenträger über die Befehlszeilenschnittstelle (CLI)

Wenn Sie einen einzelnen Container oder eine Containergruppe über die Befehlszeilenschnittstelle (CLI) erstellen, wird eine Fehlernachricht gemeldet.
Beispiel
bx ic group-create --name mein_container -n mein_host -d AppDomainName -m 2048 --min 1 --max 2 --desired 1 -p 9080 --auto -v mein_volume:/mein/pfad registry.DomainName/ibmliberty
Ergebnis
docker: Error parsing reference: "mein_datenträger:/mein/pfad" is not a valid repository/tag.
Die Befehlszeilenschnittstelle (CLI) für Cloud Foundry Version 6.17 und höher verwendet den Parameter -v für eine andere Funktion.
Ändern Sie in Ihrem Befehl die Option -v in die Option --volume.
Beispiel
bx ic group-create --name mein_container -n mein_host -d AppDomainName -m 2048 --min 1 --max 2 --desired 1 -p 9080 --auto --volume mein_datenträger:/mein/pfad registry.DomainName/ibmliberty

Bekannte Probleme

Erfahren Sie mehr über bekannte Probleme im Zusammenhang mit IBM Containers.

Docker und IBM Bluemix Container Service

Einige Docker-Befehle werden vom IBM Bluemix Container Service-Plug-in (bx ic) nicht unterstützt.

Nur bestimmte Docker-Befehle werden in (bx ic) unterstützt. Eine vollständige Liste der unterstützten Befehle finden Sie in der Docker-Befehlsreferenz. Beispielsweise ist tag in bx ic nicht verfügbar, aber docker tag kann lokal verwendet werden; anschließend kann das aktualisierte Image mit einer Push-Operation in die Bluemix-Registry übertragen werden.

Allgemeine Probleme zwischen Docker und Windows

Es gibt eine Reihe allgemeiner Fehler, die beim Installieren und Konfigurieren von Docker auf Windows-Computern auftreten. Weitere Informationen zu diesen allgemeinen Fehlern finden Sie im Blog: Installing Docker for Windows: Fixes for common problems und unter Stack Overflow posts tagged with boot2docker.

Images

Push- und Pull-Operationen für umfangreiche Images schlagen fehl

Wenn Sie einen Docker-Befehl ausführen, um für ein umfangreiches Image aus der privaten Bluemix-Registry Ihrer Organisation Push- oder Pull-Operationen durchzuführen, schlägt die Anforderung fehl. Zum Ermitteln, ob ein Problem mit dem Image vorliegt, überprüfen Sie zunächst, dass für das Image eine Push-Operation an DockerHub durchgeführt werden kann. Falls mit dem Image kein Problem vorliegt, verringern Sie, falls möglich, die Größe des Images.

Die Größe eines Images in Bluemix stimmt nicht mit der lokalen Größe überein

Wenn Sie ein Image mit einer Push-Operation in die private Bluemix-Registry übertragen, ist die angegebene Größe für das Image kleiner als die Größe desselben Images in der lokalen Docker-Engine. Der Größenunterschied ist kein Indikator dafür, dass während der Push-Operation ein Problem aufgetreten ist. Die komprimierte Größe des Image wird in IBM Bluemix Container Service angegeben.

Überwachung und Protokollierung

Zusätzliche Zeichen in der Ausgabe für den Befehl bx ic logs
Wenn Sie den Befehl bx ic logs ausführen, werden zusätzliche Zeichen wie & am Anfang jeder Zeile angezeigt. Dieses Problem tritt auf, wenn eingehende Docker-Protokolle von NGINX geparst werden.
Protokolldateizusätze können nicht mit Ausdrücken festgelegt werden

Die Protokolldateien, die erfasst werden sollen, können nicht mithilfe von Filtern oder Ausdrücken mit Platzhalterzeichen angegeben werden. Sie können zum Beispiel nicht /var/log/*.log angeben. Listen Sie stattdessen jede Protokolldatei einzeln auf.

Einzelne Container und Containergruppen

Verbindungsausfall nach dem Erstellen eines Containers

Aufgrund einer durch die Docker-Netzarchitektur von IBM Bluemix Container Service bedingten Einschränkung funktioniert der abgehende Netzzugriff beim ersten Start des Containers möglicherweise nicht. Versucht Ihr Container eine abgehende Netzverbindung herzustellen, um einen Fehlercode zurückzugeben oder eine Ausnahmebedingung auszulösen, wenn das Netz nicht verfügbar ist, wird der Container beendet und der Status 'Ausgefallen' wird angezeigt.

Damit der Netzbetrieb initialisiert und die Containerbereitstellung fortgesetzt werden kann, fügen Sie eine Inaktivitätsanweisung ein, bevor Prozesse gestartet werden, die Konnektivität abgehender Daten erfordern. Erstellen Sie eine angepasste Dockerfile zum Einschließen aller vorhandenen Serverstartbefehle in der Basis-Dockerfile. Ein Beispiel hierfür finden Sie unter Bluemix-Beispielanwendung 'Lets Chat'.

Konnektivität ankommender Daten ist nicht sofort gegeben

Die Konnektivität ankommender Daten, einschließlich Routen und externer Apps, ist weder für Containergruppen noch für einzelne Container sofort gegeben. Warten Sie, bis die Konnektivität verfügbar ist. Dies kann bis zu 5 Minuten dauern.

Verwendung verschiedener Maßeinheiten in docker run -m und in bx ic run -m
Wenn keine Maßeinheit in Verbindung mit docker run -m angegeben ist, wird 'Byte' als Standardmaßeinheit verwendet. Ist für bx ic run -m keine Maßeinheit angegeben, so wird 'MB' als Standardmaßeinheit verwendet.
Grenzwert für Benutzerdaten verhindert das Erstellen eines Containers
Wenn Sie zu viele Services an einen Container gebunden haben, kann der Container nicht erstellt werden. Von jedem Service werden Daten in den Container eingefügt. Die eingefügten Daten werden mit dem Grenzwert von 64 KB abgeglichen. Wenn Sie einen Container erstellen, der diesen Grenzwert überschreitet, wird in einer Nachricht darauf hingewiesen, dass die Ressourcenerstellung aufgrund zu umfangreicher Benutzerdaten fehlgeschlagen ist (Resource CREATE failed... User data too large). Entfernen Sie einen an den Container gebundenen Service, wenn diese Nachricht angezeigt wird.
Port 25 ist nicht zum Bereitstellen in einem Container verfügbar
Port 25 wird von SoftLayer für die Verwendung gesperrt. Dies ist Bestandteil der IBM Bluemix Container Service-Architektur. Dieses Problem hat Auswirkungen auf die Verwendung mancher SMTP-Services in Containern. Weiter Informationen zu den Gründen für das Sperren dieses Ports enthält der Artikel Outbound Email on Port 25 in der SoftLayer-Dokumentation.
Containererstellung oder Vernetzungsprozess wird nicht abgeschlossen
Wenn ein Container zu lange im Erstellungs- oder Vernetzungsstatus bleibt, müssen Sie den Container möglicherweise entfernen und einen neuen Container von demselben Image erstellen. Sie können die folgenden bx ic-CLI-Befehle verwenden:
  1. Eine Liste mit Containern abrufen, die in dem Bereich aktiv sind. Standardmäßig werden nur aktive Container angezeigt.Weitere Informationen finden Sie in der CLI-Referenz zu 'bx ic ps'.
    bx ic ps -a
  2. Entfernung von mindestens einem Container.Weitere Informationen finden Sie in der CLI-Referenz zu 'bx ic rm'.
    bx ic rm -f container-id
  3. Neuen Container im Container-Cloud-Service über einen Imagenamen starten.Weitere Informationen finden Sie in der CLI-Referenz zu 'bx ic run'.
    bx icrun
Automatische Wiederherstellung ist für Containergruppen nicht sofort verfügbar

Bei der Wiederanlauffunktion für Containergruppen kann es bis zu 15 Minuten dauern, bis neue Systeme online sind. Warten Sie, bis die automatische Wiederherstellung verfügbar ist. Dies kann bis zu 15 Minuten dauern.

Mit dem Befehl cf delete-orphaned-routes werden Routen aus Containergruppen gelöscht

Cloud Foundry erkennt Routen, die Containergruppen zugeordnet sind, nicht auf dieselbe Weise, wie es erkennt, dass Routen an Anwendungen gebunden sind. Führen Sie den Befehl cf delete-orphaned-routes nicht aus, wenn Sie Containergruppen verwenden.

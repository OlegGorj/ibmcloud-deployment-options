Hoch verfügbare Prozesse als Containergruppen ausführen

Hoch verfügbare Container bedeuten weniger Ausfallfallzeit für Ihre App. Je weiter Sie Ihre Containerkonfiguration verteilen, umso weniger wahrscheinlich ist es, dass Ihre Benutzer Ausfallzeiten mit Ihrer App verzeichnen. Sie können eine höhere Verfügbarkeit erzielen, indem Sie integrierte IBM® Bluemix Container Service-Funktionen verwenden, um die Ausfallsicherheit gegenüber potenziellen Fehlerbedingungen wie Hostfehlern, Netzfehlern oder Anwendungssoftwarefehlern zu steigern.
Achtung: Kostenlose Testversionen für einzelne und skalierbare Container sind nicht verfügbar. Lesen Sie die Informationen zum Cluster-Management und zur Migration auf Kubernetes in IBM Bluemix Container Service unter Bluemix Public.
Diese Seite enthält folgende Abschnitte:
     • Optionen für das Einrichten von Containergruppen nach Verfügbarkeit
     • Langfristige Services als Containergruppen über die GUI oder CLI ausführen
     • Gruppen automatisch skalieren (veraltet)
     • Gruppen in einen anderen Bereich verschieben
     • In der Containergruppe ausgeführte App über die GUI oder CLI aktualisieren
     • Hoch verfügbare Gruppen in demselben Bereich, in unterschiedlichen Bereichen oder unterschiedlichen Regionen ausführen
     • Verbindung zu einer privaten Gruppe herstellen
     • Gruppen entfernen

Optionen für das Einrichten von Containergruppen nach Verfügbarkeit

Machen Sie sich mit diesen potenziellen Containergruppenkonfigurationen vertraut, die nach dem Grad der Verfügbarkeit aufsteigend sortiert sind:

  1. Eine Containergruppe mit automatischer Wiederherstellung und Muster für n+1 Instanzen
  2. Eine Containergruppe mit automatischer Wiederherstellung, Muster für n+2 Instanzen und Anti-Affinität
  3. Zwei Containergruppen mit automatischer Wiederherstellung, Muster für n+2 Instanzen und Anti-Affinität, die im selben Bereich ausgeführt werden und dieselbe Route gemeinsam nutzen
  4. Zwei Containergruppen mit automatischer Wiederherstellung, Muster für n+2 Instanzen und Anti-Affinität, die in verschiedenen Bereichen ausgeführt werden und dieselbe Route gemeinsam nutzen
  5. Zwei Containergruppen mit automatischer Wiederherstellung, Muster für n+2 Instanzen und Anti-Affinität, die in verschiedenen IBM Bluemix Container Service-Verfügbarkeitszonen ausgeführt werden und dieselbe Route gemeinsam nutzen
  6. Zwei Containergruppen mit automatischer Wiederherstellung, Muster für n+2 Instanzen und Anti-Affinität, die in verschiedenen Regionen ausgeführt werden und dieselbe Route gemeinsam nutzen
Erfahren Sie mehr über die Verwendung dieser Verfahren, um die Verfügbarkeit Ihrer App zu verbessern:
Apps in Containergruppen anstatt in einem einzelnen Container bereitstellen
Eine Containergruppe enthält zwei oder mehr Container, von denen dasselbe Image ausgeführt wird. Die Verwendung von Containergruppen ist ratsam zur Ausführung von langfristigen Services mit Arbeitslasten (Workloads), die skalierbar und zuverlässig sein müssen, oder zum Testen im erforderlichen Umfang. Sie können Containergruppen über die Bluemix-GUI oder die Befehlszeilenschnittstelle erstellen. 
bx ic group-create -p 9080 -n mein_liberty-host -d AppDomainName --name mein_liberty-image --volume myvolume:/mydata/path registry.DomainName/mein_namensbereich/mein_liberty-image
Automatische Wiederherstellung in der Containergruppe aktivieren
Wenn die automatische Wiederherstellung für eine Containergruppe aktiviert ist und die Gruppe 10 Minuten lang ausgeführt wurde, beginnt die Lastausgleichsfunktion der Gruppe mit der regelmäßigen Prüfung des Status jeder Containerinstanz in der Gruppe durch HTTP-Anforderungen. Wenn eine Containerinstanz nicht innerhalb von 100 Sekunden antwortet, wird sie als inaktiv markiert. Inaktive Containerinstanzen werden durch die automatische Wiederherstellung aus der Gruppe entfernt und erneut erstellt. Die automatische Wiederherstellung versucht dreimal, Containerinstanzen in einer Gruppe wiederherzustellen. Nach dem dritten Versuch wird die automatische Wiederherstellung für Containerinstanzen für 60 Minuten ausgesetzt. Nach 60 Minuten wird der automatische Wiederherstellungsprozess wieder gestartet.Sie können Containergruppen mit automatischer Wiederherstellung über die Bluemix-GUI oder die Befehlszeilenschnittstelle erstellen. 
bx ic group-create -p 9080 --auto -n mein_liberty-host -d AppDomainName --name mein_liberty-image --volume myvolume:/mydata/path registry.DomainName/mein_namensbereich/mein_liberty-image
Ausreichende Instanzen für die Workload Ihrer App (plus zwei) einbeziehen
Um Ihre Containergruppe hoch verfügbar zu machen und mit erhöhter Ausfallsicherheit zu versehen, sollten Sie mehr Instanzen als das Minimum einbeziehen, damit die erwartete Arbeitslast verarbeitet werden kann. Zusätzliche Instanzen können die Arbeitslast übernehmen, falls andere Instanzen ausfallen. Um die automatische Wiederherstellung einer fehlgeschlagenen Instanz ohne Auswirkung auf die Arbeitslast sicherzustellen, beziehen Sie eine zusätzliche Instanz ein. Zum Schutz vor zwei gleichzeitigen Ausfällen beziehen Sie zwei zusätzliche Instanzen ein. Diese Konfiguration folgt dem Muster N+2, wobei N für die Anzahl der Instanzen steht, die zur Verarbeitung der Anforderungen dienen, und +2 für die beiden zusätzlichen Instanzen.Sie können angeben, wie viele Instanzen einbezogen werden sollen, wenn Sie eine Containergruppe über die Bluemix-GUI oder die Befehlszeilenschnittstelle erstellen. 
bx ic group-create -p 9080 --desired 5 --auto -n mein_liberty-host -d AppDomainName --name mein_liberty-image --volume myvolume:/mydata/path registry.DomainName/mein_namensbereich/mein_liberty-image
Containerinstanzen über Rechenknoten verteilen
Wenn Sie eine Containergruppe erstellen, wird jede Instanz in der Gruppe möglicherweise auf demselben physischen Rechenknoten bereitgestellt. Diese Konfiguration, bei der sich die Containerinstanzen auf demselben Rechenknoten befinden, wird als Affinität oder Kollokation (Zusammenstellung) bezeichnet. Wenn Sie die Containergruppe über die CLI erstellen, können Sie die Instanzen über verschiedene physische Rechenknoten verteilen, um die Verfügbarkeit Ihrer App zu erhöhen, indem Sie die Option für die Anti-Affinität verwenden (--anti).
Anmerkung: Um die Option --anti von der CLI verwenden zu können, müssen Sie das IBM Bluemix Container Service-Plug-in (bx ic) in Version 0.8.934 oder höher installiert haben.
bx ic group-create -p 9080 --anti --desired 5 --auto -n mein_liberty-host -d AppDomainName --name mein_liberty-image --volume myvolume:/mydata/path registry.DomainName/mein_namensbereich/mein_liberty-image
Mehrere Containergruppen erstellen, die dieselbe Route verwenden
Sie können zwei Containergruppen aus demselben Image in demselben Bluemix-Bereich erstellen, die dieselbe Route verwenden. Wenn ein Fehler auftritt, bei dem eine der Instanzen in einer Containergruppe nicht erreichbar ist, wird der Benutzerdatenverkehr automatisch an die zweite Containergruppe umgeleitet. Weitere Informationen zur gemeinsamen Nutzung einer Route zwischen Containergruppen finden Sie unter Hoch verfügbare Containergruppen in demselben Bereich ausführen. 
bx ic group-create -p 9080 --anti --desired 5 --auto -n mein_liberty-host -d AppDomainName --name mein_liberty-image1 --volume myvolume:/mydata/path registry.DomainName/mein_namensbereich/mein_liberty-image
bx ic group-create -p 9080 --anti --desired 5 --auto -n mein_liberty-host -d AppDomainName --name mein_liberty-image2 --volume myvolume:/mydata/path registry.DomainName/mein_namensbereich/mein_liberty-image

Wenn die Containergruppe, wenn Hardware oder wenn gar ein komplettes Rechenzentrum in einer Region ausfällt, wird der Datenverkehr an die Containergruppe umgeleitet, die in einer anderen Region bereitgestellt ist.

Langfristige Services als Containergruppen über die Bluemix-GUI ausführen

Erstellen Sie eine skalierbare Containergruppe über die Bluemix-GUI und stellen Sie die Gruppe bereit. Eine Containergruppe enthält zwei oder mehr Container, von denen dasselbe Image ausgeführt wird. Die Verwendung von Containergruppen ist ratsam zur Ausführung von langfristigen Services mit Arbeitslasten (Workloads), die skalierbar und zuverlässig sein müssen, oder zum Testen im erforderlichen Umfang.

Prüfen Sie zunächst die folgenden Voraussetzungen.
  • Sie müssen über die Rolle eines Entwicklers für den Bluemix-Bereich verfügen, damit Sie einen Container in ihm erstellen können. Weitere Informationen zu Berechtigungen für jede Rolle und dazu, wie Rollen bearbeitet werden, finden Sie unter Benutzer und Rollen.
  • Über die Bluemix-GUI erstellen Sie einen Container aus den von IBM bereitgestellten öffentlichen Images oder aus einem bestehenden Image in der privaten Bluemix-Registry Ihrer Organisationen. Weitere Informationen zum Hinzufügen von Images zu Ihrem Katalog finden Sie unter Docker-Images zur privaten Bluemix-Image-Registry Ihrer Organisation hinzufügen.
  • Wenn Sie einen Bluemix-Service an den Container binden möchten, fügen Sie den Bluemix-Service zu Ihrem Bereich hinzu. Weitere Informationen finden Sie unter Service über die Bluemix-GUI binden.
  • (Optional) Wenn Sie eine gemeinsam genutzte Platte benötigen, die Dateien von einer Containergruppe aus zugänglich macht, erstellen Sie einen Datenträger, bevor Sie die Containergruppe erstellen. Weitere Informationen finden Sie unter Datenträger über die Befehlszeile (CLI) erstellen.

Gehen Sie wie folgt vor, um eine Containergruppe zu erstellen und bereitzustellen:

  1. Wählen Sie im Katalog Container und anschließend ein Image aus. Es werden sowohl die von IBM bereitgestellten Images als auch die in Ihrer privaten Bluemix-Registry gespeicherten Images angezeigt.
  2. Wenn bereits ein Namensbereich für Ihre Organisation festgelegt ist, werden Sie nicht zur Erstellung eines Namensbereichs aufgefordert und können mit dem nächsten Schritt fortfahren. Wenn für Ihre Organisation noch kein Namensbereich angegeben ist, werden Sie aufgefordert, einen Namensbereich für die Image-Registry festzulegen. Bei einem Namensbereich handelt es sich um einen eindeutigen Namen, der Ihre private Registry in Bluemix kennzeichnet. Der Namensbereich für eine Organisation wird einmalig zugeordnet und kann nach seiner Erstellung nicht mehr geändert werden. Wenn Sie IBM Bluemix Container Service in mehreren Bluemix-Regionen verwenden wollen, müssen Sie einen Namensbereich für jede Region einrichten.
    Tipp: Bei Namensbereichen sind die folgenden Regeln zu beachten:
    • Der Name darf nur Kleinbuchstaben, Ziffern und Unterstreichungszeichen (_) enthalten.
    • Der Name kann 4 bis 30 Zeichen umfassen. Wenn Sie Container über die Befehlszeile verwalten möchten, sollten Sie dem Namensbereich einen kurzen Namen zuordnen, der schnell eingegeben werden kann.
    • Der Name muss in Bluemix eindeutig (einmalig) sein.
  3. Wählen Sie einen Bluemix-Bereich in Ihrer Organisation aus.
  4. Definieren Sie als Nächstes Ihre Containergruppe. Im Fenster wird standardmäßig die Registerkarte Einzeln angezeigt. Wählen Sie daher die Registerkarte Skalierbar aus, um eine Containergruppe zu erstellen.
  5. Optional: Wählen Sie unter dem Imagenamen den Tag oder die Version des Image aus, das verwendet werden soll.
  6. Geben Sie im Feld Name der Containergruppe einen Namen für den Container ein.
    Tipp: Der Containername muss mit einem Buchstaben beginnen und kann danach Großbuchstaben, Kleinbuchstaben, Ziffern, Punkte (.), Unterstriche (_) und Bindestriche (-) enthalten.
  7. Geben Sie im Feld Instanzen die Anzahl der Instanzen in der Gruppe an. Der Standardwert lautet 2. Jede Instanz verfügt über dieselbe Größe und muss dasselbe Image verwenden. Welche Anzahl von Containergruppeninstanzen Ihre App erfordert, richtet sich nach der erwarteten Arbeitslast und dem Zeitrahmen, in dem eine Anforderung von der App verarbeitet werden kann. Multitasking oder Netzkonnektivität wirken sich beispielsweise darauf aus, wie viele Instanzen für Ihre App erforderlich sind.

    Sie bestimmen die Anzahl der gewünschten Instanzen, indem Sie einen einzelnen Container in Bluemix bereitstellen und für diesen einen Belastungstest ausführen. Wenn die App zum Beispiel in der Lage ist, 100 Anforderungen pro Sekunde zu verarbeiten, jedoch eine durchschnittliche Arbeitslast von 300 Anforderungen pro Sekunde zu erwarten ist, so beläuft sich die gewünschte Anzahl von Instanzen für die Containergruppe auf 3.

    Um Ihre Containergruppe hoch verfügbar zu machen und mit erhöhter Ausfallsicherheit zu versehen, sollten Sie mehr Instanzen als das Minimum einbeziehen, damit die erwartete Arbeitslast verarbeitet werden kann. Zusätzliche Instanzen können die Arbeitslast übernehmen, falls andere Instanzen ausfallen. Um die automatische Wiederherstellung einer fehlgeschlagenen Instanz ohne Auswirkung auf die Arbeitslast sicherzustellen, beziehen Sie eine zusätzliche Instanz ein. Zum Schutz vor zwei gleichzeitigen Ausfällen beziehen Sie zwei zusätzliche Instanzen ein. Diese Konfiguration folgt dem Muster N+2, wobei N für die Anzahl der Instanzen steht, die zur Verarbeitung der Anforderungen dienen, und +2 für die beiden zusätzlichen Instanzen. Falls Sie feststellen, dass Sie mehr Instanzen benötigen, nachdem die Gruppe bereitgestellt wurde, können Sie weitere Instanzen hinzufügen, ohne die Gruppe erneut erstellen zu müssen.

    Wichtig: Eine Containergruppe sollte mindestens zwei Containerinstanzen enthalten. Wenn Sie lediglich eine Instanz in die Gruppe einbeziehen und diese Instanz irgendwann nicht verfügbar ist, treten bei den Benutzern möglicherweise Ausfallzeiten für die App auf.
  8. Wählen Sie im Feld Größe eine Containergröße aus. Die Standardgröße ist Micro (256 MB Hauptspeicher, 16 GB Plattenspeicher). Die von Ihnen ausgewählte Größe wirkt sich auf die Menge an Hauptspeicher und Plattenspeicher aus, die der Rechenhost zur Laufzeit erhält, und kann nach der Erstellung des Containers nicht mehr geändert werden. Die folgenden Größen sind ebenfalls verfügbar:
    • Pico (64 MB Hauptspeicher, 4 GB Plattenspeicher)
    • Nano (128 MB Hauptspeicher, 8 GB Plattenspeicher)
    • Tiny (512 MB Hauptspeicher, 32 GB Plattenspeicher)
    • S (Klein) (1 GB Hauptspeicher, 64 GB Plattenspeicher)
    • M (Mittel) (2 GB Hauptspeicher, 128 GB Plattenspeicher)
    • L (Groß) (4 GB Hauptspeicher, 256 GB Plattenspeicher)
    • XL (8 GB Hauptspeicher, 512 GB Plattenspeicher)
    • XXL (16 GB Hauptspeicher, 1 TB Plattenspeicher)
  9. Wählen Sie eine vollständige, öffentliche URL aus. Diese URL wird auch als Route bezeichnet. Wenn Sie anstatt einer IP-Adresse für eine einzelne Containerinstanz einen Host und eine Domäne für die Containergruppe verwenden, stellen Sie einen Lastausgleich für die App bereit. Ihre Containergruppe ist möglicherweise nicht unmittelbar nach der Erstellung unter der öffentlichen URL verfügbar, da die Netzkonnektivität bis zu fünf Minuten für die Auflösung benötigt.
    1. Geben Sie im Feld Host den Hostnamen an (z. B. meincontainerhost). Der Hostname darf keine Unterstreichungszeichen (_) enthalten.
    2. Wählen Sie für die Domäne des Systems die Option AppDomainName aus. Da von der Systemdomäne bereits ein SSL-Zertifikat bereitgestellt wird, können Sie ohne weitere Konfiguration über HTTPS auf die Containergruppe zugreifen. Sie können auch eine angepasste Domäne auswählen, die Sie für Ihre Organisation erstellt haben. Wenn Sie eine angepasste Domäne erstellen möchten, müssen Sie die angepasste Domäne bei einem öffentlichen DNS-Server registrieren, die angepasste Domäne in Bluemix konfigurieren und die angepasste Domäne anschließend der Systemdomäne von Bluemix auf dem öffentlichen DNS-Server zuweisen. Nachdem die angepasste Domäne der Bluemix-Systemdomäne zugeordnet wurde, werden Anforderungen für die angepasste Domäne an die Anwendung in Bluemix weitergeleitet. Bei Erstellung einer angepassten Domäne darf der Domänenname keine Unterstreichungszeichen (_) enthalten. Weitere Informationen zu angepassten Domänen finden Sie unter Angepasste Domäne erstellen und verwenden. Wenn Sie die angepasste Domäne schützen möchten, laden Sie ein SSL-Zertifikat hoch, sodass auf die Containergruppen über HTTPS zugegriffen werden kann.

      Die Routendomäne und der Host werden zu einer vollständigen URL für eine öffentliche Route kombiniert, zum Beispiel http://meincontainerhost.AppDomainName, die in Bluemix eindeutig sein muss.

  10. Geben Sie im Feld HTTP-Port einen öffentlichen Port an, der veröffentlicht werden soll. Sie können einen Port eingeben oder einen Port in der Dropdown-Liste auswählen. Wenn Sie einen Port eingeben, können Sie zwischen UDP und TCP zum Angeben des IP-Protokolls auswählen, das Sie verwenden möchten. Wenn Sie kein Protokoll angeben, wird der Port automatisch für den TCP-Datenverkehr zugänglich gemacht. Die Dropdown-Liste wurde mit den Ports vorab gefüllt, die in der Dockerdatei (Dockerfile) zu dem Zeitpunkt für das Image angegeben wurden, als das Image erstellt wurde. In Containergruppen können nicht mehrere Ports eingefügt werden. Wenn Sie eine Route binden, müssen Container in Ihrer Gruppe an dem zugänglich gemachten Port der Gruppe für HTTP-Datenverkehr empfangsbereit sein. Nur HTTP-Ports können öffentlich zugänglich gemacht werden. Wenn HTTPS-Datenverkehr am zugänglich gemachten Port eintrifft, führt (Go)Router die HTTPS-Beendigung durch. Anschließend wird im privaten virtuellen Netz zwischen (Go)Router und den Containern das HTTP-Protokoll verwendet.
    Wenn in der Dockerfile für das von Ihnen verwendete Image ein Port angegeben ist, schließen Sie diesen Port ein.
    Tipp:
    • Geben Sie für das von IBM zertifizierte Liberty Server-Image oder eine geänderte Version dieses Images den Port 9080 ein.
    • Geben Sie für das von IBM zertifizierte Node.js-Image oder eine geänderte Version dieses Images einen beliebigen Port, z. B. Port 8000, ein.
  11. Optional: Aktivieren Sie die automatische Wiederherstellung. Wählen Sie Automatische Wiederherstellung aktivieren aus. Bei der automatischen Wiederherstellung werden nicht verfügbare Containerinstanzen entfernt und durch die neuen Containerinstanzen ersetzt.

    Wenn die automatische Wiederherstellung für eine Containergruppe aktiviert ist und die Gruppe 10 Minuten lang ausgeführt wurde, beginnt die Lastausgleichsfunktion der Gruppe mit der regelmäßigen Prüfung des Status jeder Containerinstanz in der Gruppe durch HTTP-Anforderungen. Wenn eine Containerinstanz nicht innerhalb von 100 Sekunden antwortet, wird sie als inaktiv markiert. Inaktive Containerinstanzen werden durch die automatische Wiederherstellung aus der Gruppe entfernt und erneut erstellt. Die automatische Wiederherstellung versucht dreimal, Containerinstanzen in einer Gruppe wiederherzustellen. Nach dem dritten Versuch wird die automatische Wiederherstellung für Containerinstanzen für 60 Minuten ausgesetzt. Nach 60 Minuten wird der automatische Wiederherstellungsprozess wieder gestartet.

  12. Optional: Definieren Sie Daten- oder Dateispeicher. Ordnen Sie einen vorhandenen Datenträger zu einem Container bzw. einer Containergruppe zu. Ein Datenträger ist eine gemeinsam genutzte Platte, die ermöglicht, dass auf Dateien von einem Container aus zugegriffen werden kann. Der Pfad im Container gibt an, wo im Dateisystem des Containers der Datenträger angehängt ist.
    Einschränkung: Sie können Datenträger aus Ihrem Bluemix-Bereich nur über die Befehlszeile löschen.
    1. Klicken Sie auf Vorhandenen Datenträger zuweisen.
    2. Wählen Sie den Datenträger aus der Dropdown-Liste aus.
    3. Geben Sie im nächsten Feld einen Pfad in dem Container an, um den Datenträger zuzuordnen. Beispiel: /var/lib/mein_datenträger
    4. Um anzugeben, dass ein Datenträger schreibgeschützt ist, wählen Sie die Option Schreibgeschützt aus. Standardmäßig verfügen Datenträger über Lese- und Schreibberechtigung.
    5. Um mehrere Datenträger hinzuzufügen, klicken Sie auf Vorhandenen Datenträger zuweisen und wiederholen Sie die folgenden Schritte.
  13. Optional: Definieren Sie Umgebungsvariablen.
    1. Klicken Sie auf Neue Umgebungsvariable hinzufügen.
    2. Geben Sie einen Namen für die Umgebungsvariable ein.
    3. Geben Sie im nächsten Feld einen Wert für die Umgebungsvariable an.
    4. Um mehrere Umgebungsvariablen hinzuzufügen, klicken Sie auf das Symbol Neue Umgebungsvariable hinzufügen und wiederholen Sie die angegebenen Schritte.

Zur Autoskalierung von Kubernetes migrieren

Mit Kubernetes können Sie die Funktion zum automatischen horizontalen Skalieren von Pods aktivieren, um Ihre Apps CPU-basiert zu skalieren.

  1. Installieren Sie die erforderlichen Benutzerschnittstellen (CLIs).
  2. Erstellen Sie einen Cluster, indem Sie die Schritte in Einführung in Kubernetes-Cluster in Bluemix ausführen. Stellen Sie unbedingt sicher, dass Sie beim Einrichten Ihres Clusters die aktuelle Containergruppe einschließlich Protokollierung, Überwachung, Advisorfunktion für Schwachstellen (Vulnerability Advisor) und allen übrigen vorhandenen Anpassungen replizieren.
  3. Legen Sie den Kontext für Ihren Cluster fest.
  4. Stellen Sie Ihre vorhandene App über die Benutzerschnittstelle (CLI) auf dem Cluster bereit. Bei der Bereitstellung Ihrer App müssen Sie CPU anfordern. 
    kubectl run <name> --image=<image> --requests=cpu=<cpu> --expose --port=<portnummer>
    Erklärung der Bestandteile dieses Befehls
    --image Die Anwendung, die bereitgestellt werden soll.
    --requests=cpu Die erforderliche CPU für Ihren Container. Die Angabe erfolgt in Millicore. Beispiel: --requests=200m.
    --expose Wenn für diese Befehlskomponente 'true' definiert ist, wird ein externer Service erstellt.
    --port Der Port, an dem Ihre App extern verfügbar ist.
    Anmerkung: Für Bereitstellungen mit einem höheren Grad an Komplexität ist möglicherweise die Erstellung eines Bereitstellungsscripts erforderlich.
  5. Erstellen Sie eine Funktion zur automatischen horizontalen Skalierung von Pods (Horizontal Pod Autoscaler) und definieren Sie Ihre Richtlinie. Weitere Informationen zum Arbeiten mit dem Befehl kubetcl autoscale enthält die Kubernetes-Dokumentation. 
    kubectl autoscale deployment <bereitstellungsname> --cpu-percent=<prozentsatz> --min=<mindestwert> --max=<höchstwert>
    Erklärung der Bestandteile dieses Befehls
    --cpu-percent Die durchschnittliche CPU-Auslastung, die durch die Funktion zur automatischen horizontalen Skalierung von Pods (Horizontal Pod Autoscaler) aufrechterhalten wird. Die Angabe erfolgt als Prozentsatz.
    --min Die Mindestanzahl von bereitgestellten Pods, die zur Aufrechterhaltung der angegebenen CPU-Auslastung verwendet werden.
    --max Die maximale Anzahl von bereitgestellten Pods, die zur Aufrechterhaltung der angegebenen CPU-Auslastung verwendet werden.
  6. Überprüfen Sie, ob Ihre Anwendung ordnungsgemäß ausgeführt wird.
    • Stellen Sie über das Dashboard für Services sicher, dass Ihre App aktiv ist.
    • Prüfen Sie Ihre App, um sicherzustellen, dass Schlüsseländerungen implementiert wurden.
  7. Nehmen Sie die Einrichtung Ihrer ursprünglichen Instanz von Containern zurück und entfernen Sie sie.

Containergruppen in andere Bluemix-Bereiche verschieben

Wenn Sie eine Containergruppe in derselben Organisation in einen anderen Bereich verschieben müssen, können Sie eine zweite Containergruppe mit derselben Route erstellen. Dann können Sie die erste Gruppe entfernen, ohne dass hierdurch Ausfallzeiten für Ihre App verursacht werden.

Bei der Zuordnung einer vorhandenen Route zu einer neuen Containergruppe werden die Referenzen der Containerinstanzen zur Lastausgleichsfunktion des Bereichs hinzugefügt. Demzufolge wird die neue Containergruppe bei der Weiterleitung des eingehenden Datenverkehrs von der Lastausgleichsfunktion berücksichtigt. Beim Löschen einer Containergruppe werden die Referenzen auf zugehörige Containerinstanzen von der Lastausgleichsfunktion automatisch entfernt.
  1. Melden Sie sich bei der Cloud Foundry-CLI an. Geben Sie nach entsprechender Aufforderung Ihren Bluemix-Benutzernamen und das zugehörige Kennwort ein. 
    bx login [-a api.DomainName] [-sso]

    Der Parameter --sso für Single Sign-on ist erforderlich, wenn Sie sich mit einer eingebundenen ID anmelden. Rufen Sie bei Verwendung dieser Option https://login.ng.bluemix.net/UAALoginServerWAR/passcode auf, um einen nur für eine einzige Anmeldung gültigen Kenncode abzurufen. Führen Sie die folgenden untergeordneten Schritte aus, wenn Sie die Option --sso nicht angeben:

    1. Geben Sie für Email Ihre IBM ID für Bluemix an.
    2. Geben Sie für Kennwort das Kennwort für die IBM ID an. Daraufhin werden die Bluemix-Organisationen und -Bereiche abgerufen.
    3. Geben Sie die Nummer für eine Ihrer Bluemix-Organisationen ein.
    4. Geben Sie die Nummer für einen Ihrer bestehenden Bluemix-Bereiche an.
  2. Initialisieren Sie IBM Bluemix Container Service. 
    bx ic init
  3. Erstellen Sie eine Containergruppe und ordnen Sie dieser eine Route zu. Mit dem folgenden Befehl wird eine Containergruppe namens meine_gruppe_1 aus dem Image ibmliberty erstellt. Der Zugriff auf die Containergruppe wird über die öffentliche Route myhost.AppDomainName bereitgestellt.
    Anmerkung: Der Hostname (in diesem Beispiel mein_host) muss für alle Organisationen in Bluemix eindeutig sein. Deshalb müssen Sie für mein_host einen neuen Namen auswählen, bevor Sie fortfahren können.
    bx ic group-create -p 9080 -n mein_host -d AppDomainName --name meine_gruppe_1 registry.DomainName/ibmliberty
  4. Stellen Sie sicher, dass die Containergruppe erstellt wurde. 
    bx ic groups
  5. Ändern Sie den Organisationsbereich. Ersetzen Sie BEREICH durch den Namen des Bereichs, in den die Containergruppe verschoben werden soll. 
    cf target -s BEREICH
  6. Reinitialisieren Sie Ihren IBM Bluemix Container Service, damit dieser auf den neuen Bereich verweist. 
    bx ic init
  7. Erstellen Sie in dem neuen Bereich eine weitere Containergruppe und ordnen Sie dieser die vorhandene Route zu. Mit dem folgenden Befehl wird eine Containergruppe namens meine_gruppe_2 aus dem Image ibmliberty erstellt. Diese Gruppe wird derselben öffentlichen Route zugeordnet, der auch die zuerst erstellte Containergruppe zugeordnet wurde.
    Anmerkung: Der Name der Containergruppe muss innerhalb eines Organisationsbereichs eindeutig sein. Der Name kann bei der Erstellung einer neuen Containergruppe in einem anderen Bereich Ihrer Organisation wiederverwendet werden.
    bx ic group-create -p 9080 -n mein_host -d AppDomainName --name meine_gruppe_2 registry.DomainName/ibmliberty
  8. Stellen Sie sicher, dass die Containergruppe erstellt wurde. 
    bx ic groups
    Nachdem Sie nun eine neue Containergruppe in Ihrem neuen Bereich erstellt und die vorhandene Route zugeordnet haben, können Sie die erste Containergruppe entfernen.
  9. Wechseln Sie in den Organisationsbereich, in dem Sie die erste Containergruppe erstellt haben. Ersetzen Sie BEREICH durch den Namen des betreffenden Bereichs. 
    cf target -s BEREICH
  10. Reinitialisieren Sie Ihren IBM Bluemix Container Service, damit dieser auf den ausgewählten Bereich verweist. 
    bx ic init
  11. Heben Sie die Zuordnung der Route zur Containergruppe auf. Ersetzen Sie mein_host durch den Namen des ausgewählten Hosts. Wenn Sie die Zuordnung der Route zur Containergruppe aufheben, werden alle Containerreferenzen aus der Lastausgleichsfunktion gelöscht und bei der Weiterleitung des eingehenden Datenverkehrs nicht mehr berücksichtigt. 
    bx ic route-unmap -n mein_host -d AppDomainName meine_gruppe_1
  12. Optional: Entfernen Sie die Containergruppe. 
    bx ic group-remove meine_gruppe_1
  13. Optional: Stellen Sie sicher, dass die Containergruppe entfernt wurde. 
    bx ic groups

Hoch verfügbare Containergruppen in demselben Bereich ausführen

Sie können zwei Containergruppen aus demselben Image in demselben Bluemix-Bereich erstellen und beiden Gruppen dieselbe Route zuordnen. Diese Konfiguration stellt sicher, dass der Datenverkehr an die jeweils andere Containergruppe weitergeleitet wird, falls eine Containergruppe oder die Lastausgleichsfunktion für eine Gruppe ausfällt.

Lesen Sie vor Beginn die Informationen zum Erstellen von Containergruppen mit der CLI oder der Bluemix-GUI.

So erstellen Sie hoch verfügbare Containergruppen in demselben Bluemix-Bereich:

  1. Erstellen Sie eine Containergruppe, die die folgenden Anforderungen erfüllt.
    • Die Anzahl von Instanzen in der Container muss die erwartete Arbeitslast auf Ihrer App unterstützen. Sie bestimmen die Anzahl der gewünschten Instanzen, indem Sie einen einzelnen Container in Bluemix bereitstellen und für diesen einen Belastungstest ausführen. Wenn die App zum Beispiel in der Lage ist, 100 Anforderungen pro Sekunde zu verarbeiten, jedoch eine durchschnittliche Arbeitslast von 300 Anforderungen pro Sekunde zu erwarten ist, so beläuft sich die gewünschte Anzahl von Instanzen für die Containergruppe auf 3. Um Ihre Containergruppe hoch verfügbar zu machen und mit erhöhter Ausfallsicherheit zu versehen, sollten Sie mehr Instanzen als das Minimum einbeziehen, damit die erwartete Arbeitslast verarbeitet werden kann. Zusätzliche Instanzen können die Arbeitslast übernehmen, falls andere Instanzen ausfallen. Um die automatische Wiederherstellung einer fehlgeschlagenen Instanz ohne Auswirkung auf die Arbeitslast sicherzustellen, beziehen Sie eine zusätzliche Instanz ein. Zum Schutz vor zwei gleichzeitigen Ausfällen beziehen Sie zwei zusätzliche Instanzen ein. Diese Konfiguration folgt dem Muster N+2, wobei N für die Anzahl der Instanzen steht, die zur Verarbeitung der Anforderungen dienen, und +2 für die beiden zusätzlichen Instanzen.
    • Aktivieren Sie die automatische Wiederherstellung für die Containergruppe, um sicherzustellen, dass bei ausgefallenen (abgestürzten) Containern die Wiederherstellung automatisch durchgeführt wird.
    • Aktivieren Sie die Anti-Affinität. Wenn die Anti-Affinität aktiviert ist, werden die Containerinstanzen auf mehrere getrennte physische Rechenknoten verteilt, wodurch die Wahrscheinlichkeit von Containerausfällen durch Hardwarefehler gesenkt wird. Die Anti-Affinität erfordert die Verwendung der CLI und das Angeben der Option --anti im Befehl bx ic group-create. Um die Option --anti von der CLI verwenden zu können, müssen Sie das IBM Bluemix Container Service-Plug-in (bx ic) in Version 0.8.934 oder höher installiert haben. Diese Option kann in Verbindung mit größeren Gruppengrößen möglicherweise nicht verwendet werden, da für jede Bluemix-Region und -Organisation nur eine begrenzte Anzahl von Rechenknoten für die Bereitstellung verfügbar ist. Falls Ihre Bereitstellung nicht erfolgreich ist, reduzieren Sie entweder die Anzahl der Containerinstanzen in der Gruppe oder entfernen Sie die Option --anti.
    • Sie können Dateien auf einem Datenträger speichern und den Datenträger an den Container anhängen oder vorzugsweise die Daten in einem externen Datenspeicher wie etwa einer Cloudant-Datenbank speichern. Schließen Sie keine persistenten Dateien in das lokale Dateisystem für den Container ein. Es müssen erst Datenträger erstellt werden, bevor eine Containergruppe erstellt werden kann.
    • Erstellen Sie keinen einzelnen Container anstelle einer Containergruppe. Einzelne Container sind generell zur Verwendung für kurzfristige Prozesse vorgesehen und sind im Gegensatz zu Containergruppen nicht mit integrierten Hochverfügbarkeitsfunktionen ausgestattet.
    CLI-Befehlsbeispiel mit Berücksichtigung der genannten Hinweise. 
    bx ic group-create --min 3 --max 5 --desired 3 --auto --anti --volume mein_datenträger:/verzeichnis/pfad registry.DomainName/mein_namensbereich/mein_image
  2. Erstellen Sie in demselben Bereich wie die erste Containergruppe eine zweite Containergruppe, die die folgenden Anforderungen erfüllt.
    • Die Containergruppe muss einen anderen Namen aufweisen, kann aber denselben Hostnamen verwenden wie die Containergruppe, die Sie zuvor erstellt haben.
    • Um die zu erwartende Arbeitslast der App zu unterstützen, verwenden Sie das beschriebene Muster N+2, um die Anzahl der Instanzen für die Containergruppe zu ermitteln.
    • Aktivieren Sie die automatische Wiederherstellung für die Containergruppe, um sicherzustellen, dass bei ausgefallenen (abgestürzten) Containern die Wiederherstellung automatisch durchgeführt wird.
    • Aktivieren Sie die Anti-Affinität, um die Containerinstanzen auf separate physische Rechenknoten zu verteilen.
    • Die Containergruppe muss derselben Route zugeordnet werden wie die erste Containergruppe, die Sie im vorherigen Schritt erstellt haben.
    • Schließen Sie keine persistenten Dateien in den Container ein. An diesen Container können Sie denselben Datenträger anhängen, den Sie an den ersten Container angehängt haben, oder Sie können dieselbe Cloudant-Datenbank verwenden.
Informationen dazu, wie Sie Ihre App weiter verteilen, finden Sie in container_ha_org.html.

Verbindung zu einer privaten Containergruppe herstellen

Durch die Angabe der IP-Adresse der Lastausgleichsfunktion für eine Containergruppe können Sie auf die Gruppe zugreifen, ohne sie öffentlich zugänglich zu machen.

Jede Containergruppe wird mit einer integrierten Lastausgleichsfunktion bereitgestellt, die die Anforderungen gleichmäßig auf die einzelnen Container in einer Gruppe verteilt. Häufig wird die Containergruppe mit einer URL, die der Gruppe zugeordnet ist, ungeschützt öffentlich zugänglich gemacht. In Situationen, bei denen es sich nicht um eine öffentliche Gruppe handelt, kann für den Zugriff auf die Containergruppe eine private IP-Adresse verwendet werden, die der Lastausgleichsfunktion automatisch zugewiesen wird. Diese private IP-Adresse lässt auch für Apps, die über TCP oder HTTPS kommunizieren, den Zugriff auf die Containergruppe zu. Mit den folgenden Schritten können Sie die private IP-Adresse der Lastausgleichsfunktion ermitteln.
  1. Rufen Sie eine Liste der Containergruppen ab, die im Bereich ausgeführt werden. Notieren Sie sich den zugänglichen Port für die Containergruppe (hier 9080). Wenn die Containergruppe keinen zugänglichen Port aufweist, müssen Sie eine neue Containergruppe erstellen und einen Port zugänglich machen. Die Lastausgleichsfunktion der Gruppe steht nur an dem zugänglichen Port zur Verfügung. 
    bx ic groups
    Group ID                               Name                      Status            Created                         Updated   Port   
cb628ddd-a1e7-4944-9a42-02abc80ea6ee   container_group_example   CREATE_COMPLETE   2016-09-26 15:43:00 -0400 EDT             9080
  2. Überprüfen Sie die Containergruppe. Ersetzen Sie container_group_example durch den Namen Ihrer Containergruppe. Prüfen Sie das Feld private_ip_address im Abschnitt Loadbalancer. 
    bx ic group-inspect containergruppenbeispiel
        "Loadbalancer": {
        "intermediate_ip_address": "192.0.2.1", 
        "private_ip_address": "198.51.100.27"
    },
    Wichtig: Das Feld intermediate_ip_address wird von GoRouter dazu verwendet, den Datenverkehr der Route vom öffentlichen Netz sicher an die Containergruppe weiterzuleiten, wenn die Containergruppe öffentlich verfügbar ist. Verwenden Sie die zwischengeschaltete IP-Adresse (intermediate_ip_address) der Lastausgleichsfunktion nicht für den Zugriff auf Ihre Containergruppe im privaten Netz, da in diesem Fall zusätzliche Suchvorgänge in Routertabellen erforderlich werden und Netzlatenz verursacht wird.
  3. Ermitteln Sie die private IP-Adresse der Lastausgleichsfunktion. Eine App in demselben Bereich kann diese private IP-Adresse nutzen, um mit der Containergruppe nicht öffentlich zu kommunizieren. Sie müssen den zugänglichen Port wie im folgenden Format gezeigt hinzufügen, um auf die Lastausgleichsfunktion zuzugreifen: 198.51.100.27:9080.

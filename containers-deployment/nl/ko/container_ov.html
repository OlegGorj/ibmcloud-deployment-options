단일 및 확장 가능 컨테이너 정보

IBM® Bluemix®는 일부 지역에서 IBM Bluemix Container Service 인프라를 앱 개발을 위한 기능으로 제공합니다. 컨테이너를 사용하면 임의의 프로그래밍 도구를 사용하여 임의의 언어로 앱을 빌드할 수 있습니다. 각 컨테이너는 격리된 안전한 앱 플랫폼입니다. 컨테이너는 OS X 또는 Windows를 실행 중인 개발 워크스테이션, Ubuntu를 실행 중인 서버, Red Hat을 실행 중인 프로덕션 데이터 센터 가상 머신 또는 Bluemix와 같은 클라우드 플랫폼 등 어디서나 실행될 수 있습니다.
주의: 단일 및 확장 가능 컨테이너의 무료 평가판은 사용할 수 없습니다. 클러스터 관리에 대해 알아보고 Bluemix 퍼블릭에서 IBM Bluemix Container Service의 Kubernetes로 마이그레이션하십시오.

컨테이너의 개념

컨테이너는 앱 실행에 필요한 모든 요소를 포함하는 가상 소프트웨어 오브젝트입니다. 컨테이너는 이미지로부터 빌드되며, 이미지는 컨테이너 작성을 위한 읽기 전용 템플리트입니다. 각 이미지는 앱 및 앱의 종속 항목만 포함하며, 호스트 운영 체제에서 격리된 프로세스로 실행됩니다. 따라서 컨테이너는 리소스 격리 및 할당이라는 이점 외에, 예를 들면, 가상 머신보다 이식 가능성 및 효율성 면에서도 우수합니다. 컨테이너를 사용하면 고품질의 앱을 신속하게 빌드하는 데 도움이 됩니다.

컨테이너는 민첩합니다.

컨테이너는 개발 및 프로덕션 팀을 위한 표준화된 환경을 제공함으로써 시스템 관리를 단순화합니다. 엔진의 경량 런타임을 사용하면 수요 변경에 따른 응답으로 신속한 확장 및 축소가 가능합니다. 또한 서로 다른 운영 체제 플랫폼 및 기본 인프라를 관리해야 하는 복잡도를 제거하는 데 도움이 됩니다. 컨테이너는 임의의 인프라에서 임의의 앱을 신속하고 확실하게 개발하고 실행할 수 있도록 도움을 줍니다.

컨테이너는 크기가 작습니다.

하나의 가상 머신에 필요한 크기의 공간에 더 많은 컨테이너를 배치할 수 있습니다. 컨테이너의 크기를 선택할 수 있으므로 앱에 필요한 크기의 공간만 사용할 수 있습니다.

컨테이너는 이식 가능성을 제공합니다.

다른 이미지를 기반으로 다른 컨테이너를 위한 이미지를 빌드할 수 있습니다. 다른 사람이 이미지에 대한 작업의 대부분을 수행하고 여러분의 용도에 맞게 이미지를 변경하도록 할 수 있습니다. 또한 스테이징 환경에서 프로덕션 환경으로 앱 코드를 신속하게 마이그레이션할 수 있습니다. Delivery Pipeline이나 UrbanCode Deploy와 같은 도구를 사용하여 마이그레이션 프로세스를 자동화할 수 있습니다.

다음 그림은 Bluemix에서 컨테이너를 실행하는 IBM Bluemix Container Service의 아키텍처를 보여줍니다.

Docker 및 IBM Bluemix Container Service

IBM Bluemix Container Service는 Docker 기술을 활용하는 Bluemix 런타임입니다. 기존 Docker 사용자와 Docker를 처음 사용하는 사용자들은 IBM Bluemix Container Service에서 다음 기능을 기대할 수 있습니다.

  • 기존 Docker Hub 이미지를 IBM에서 호스트하는 개인용 레지스트리에 추가합니다.
  • 이미지의 유효성을 검증하여 잠재적 취약점을 찾아냅니다.
  • 이미 편하게 사용 중인 Docker CLI 명령을 사용하도록 IBM Bluemix Container Service에 대한 명령행 인터페이스(CLI)를 구성합니다.
  • 추가 구성 없이 컨테이너에 대한 로깅 및 모니터링 데이터를 봅니다.

이전에 Docker를 사용한 경우에는 IBM Bluemix Container Service를 시작할 때 유용한 이 일반적인 질문에 대한 해답을 찾을 수 있습니다.

IBM Bluemix Container Service에서 Docker Hub의 이미지를 어떻게 사용합니까?
IBM Bluemix Container Service CLI를 사용하여 Docker Hub의 이미지를 개인용 Bluemix 레지스트리로 복사할 수 있습니다. 이미지가 복사되면 해당 이미지를 사용하여 컨테이너를 작성할 수 있습니다.
고가용성 컨테이너를 어떻게 작성합니까?
IBM Bluemix Container Service는 Docker 컨테이너와 비슷한 단일 컨테이너를 제공합니다. 또한 IBM Bluemix Container Service는 애플리케이션을 고가용성 애플리케이션으로 만드는 데 사용할 수 있는 확장 가능 컨테이너 그룹을 제공합니다. 컨테이너 그룹은 워크로드가 컨테이너 인스턴스 사이에서 라우팅되고 균형을 유지하도록 하기 위해 기본 제공 로드 밸런서를 제공합니다. 그룹의 가용성을 늘리기 위해 연관관계 방지 기능으로 여러 실제 컴퓨팅 호스트에 걸쳐 컨테이너 그룹 인스턴스를 분산시키고 비정상적인 인스턴스의 자동 복구를 사용으로 설정하십시오. 또한, CPU 및 메모리 사용량을 기반으로 Auto-Scale 컨테이너 그룹 인스턴스에 정책을 설정할 수 있습니다. 자세한 정보는 고가용성 프로세스를 컨테이너 그룹으로 실행의 내용을 참조하십시오.
어떻게 컨테이너 포트를 호스트 포트에 맵핑합니까?
IBM Bluemix Container Service는 사설 컨테이너 네트워크와 공용 네트워크 사이에 차이가 있습니다. 사설 컨테이너 네트워크에서는 모든 컨테이너 포트가 추가 구성 없이 기본적으로 노출됩니다. 동일한 사설 컨테이너 네트워크에 연결되는 단일 컨테이너와 컨테이너 그룹은 지정된 사설 IP 주소를 사용하여 통신할 수 있습니다. 포트 맵핑은 필요하지 않습니다.

앱을 공용 네트워크에서 사용할 수 있도록 하려면 공용 포트를 노출하고, 공인 IP주소를 단일 컨테이너에 바인드하거나 공용 라우트를 컨테이너 그룹에 바인드해야 합니다. 공용 포트를 노출하면 노출된 포트에서만 공개 데이터를 전송 및 수신할 수 있습니다. 다른 모든 공용 포트는 닫힙니다.

컨테이너를 인터넷에서 공개적으로 사용할 수 있게 하려면 어떻게 해야 합니까?
  • 단일 컨테이너의 경우, 하나 이상의 포트를 노출하고 컨테이너에 유동 IP 주소를 지정하십시오.
  • 컨테이너 그룹의 경우, 단일 HTTP 포트를 노출하고 라우트를 지정하십시오.
로컬 Docker와 Bluemix가 컨테이너를 빌드하는 데 필요한 시간이 서로 다른 이유는 무엇입니까?
로컬 Docker 설치에서 컨테이너 빌드가 Bluemix에서 컨테이너 빌드보다 더 빠른 것을 알게 되었을 수도 있습니다. 첫 번째 배치에서 컨테이너 이미지는 호스트의 레지스트리에 다운로드되어야 합니다. 이미지의 후속 배치는 더 빨라집니다. 배치 속도가 서로 다른 컨테이너 구성에 영향을 받는 방법에 대한 자세한 정보는 컨테이너 유형의 내용을 참조하십시오.
컨테이너를 공개적으로 노출하지 않고 사설 네트워크에서 작성하려면 어떻게 해야 합니까?
컨테이너를 작성할 때 IBM Bluemix Container Service 기본 컨테이너 사설 네트워크에 연결되고 사설 IP 주소가 사용자의 컨테이너에 지정됩니다. 이 사설 네트워크에서는 모든 컨테이너 포트가 기본적으로 노출됩니다. 호스트 포트를 컨테이너 포트에 맵핑할 필요가 없습니다. 인터넷을 통해 컨테이너에 액세스하지 못하게 하려면 컨테이너를 작성할 때 IP 주소를 바인드하거나 포트를 노출하지 마십시오.
컨테이너에 볼륨을 어떻게 마운트합니까?
단일 컨테이너는 단기적으로만 지속되도록 디자인되었습니다. 그러나 볼륨을 사용하여 컨테이너를 재시작하는 사이에 데이터를 지속할 수 있습니다. Docker에서 볼륨을 마운트할 때 볼륨이 로컬 파일 시스템에 마운트됩니다. IBM Bluemix Container Service에서 컴퓨팅 호스트에 대한 액세스가 제한되므로 호스트 디렉토리를 컨테이너에 마운트할 수 없습니다. 대신 조직 범위의 볼륨을 사용하여 컨테이너를 다시 시작하는 사이에 데이터를 유지합니다. 앱 데이터를 안전하게 저장하고 파일에 대한 액세스와 권한을 관리하는 격리된 파일 공유에서 볼륨을 호스팅합니다. 볼륨을 컨테이너에 마운트하려면 먼저 볼륨을 작성해야 합니다.CLI에서 컨테이너를 작성할 때 --volume 옵션을 사용하여 기존 볼륨을 마운트하십시오. Bluemix GUI에서 고급 옵션을 선택하고 기존 볼륨을 지정하십시오. 자세한 정보는 명령행 인터페이스(CLI)를 사용하여 볼륨에 파일 추가의 내용을 참조하십시오.
IBM Bluemix Container Service에서 Docker Compose를 어떻게 사용합니까?
Docker Compose 명령을 실행하여 IBM Bluemix Container Service에서 다중 컨테이너 배치를 작성하고 관리할 수 있습니다. 자세한 정보는 튜토리얼: Docker Compose 및 IBM Bluemix Container Service를 사용하여 다중 컨테이너 배치 작성의 내용을 참조하십시오.
내 Docker Compose 클라이언트가 IBM Containers 서비스를 가리키도록 하려면 어떻게 해야 합니까?
IBM Bluemix Container Service CLI에 로그인하여 기본 Docker 명령을 실행하도록 CLI를 구성하십시오. 자세한 정보는 bx ic 플러그인에 로그인을 참조하십시오.
내 실행 중인 컨테이너에 로그인하려면 어떻게 해야 합니까? (SSH로 연결하려면 어떻게 해야 합니까?)
bx ic exec를 사용하여 컨테이너에 로그인할 수 있습니다.
내 컨테이너가 분리 모드에서 실행되는 이유는 무엇입니까?
기본적으로 Bluemix의 컨테이너는 분리 모드로 실행되므로 실행 중인 프로세스가 있는 컨테이너를 종료하지 않고 백그라운드로 실행할 수 있습니다. 분리 모드에서 실행 중인 컨테이너는 클라우드 환경의 표준 모델입니다. bx ic attach [--no-stdin] [--sig-proxy] CONTAINER를 실행하여 컨테이너에 연결할 수 있습니다.

Bluemix 데디케이티드 및 Bluemix 로컬의 IBM Bluemix Container Service

Bluemix 데디케이티드 및 Bluemix 로컬에서 IBM Bluemix Container Service를 사용할 수 있습니다.

Bluemix 로컬의 IBM Bluemix Container Service 하드웨어 요구사항

기본 하드웨어 요구사항
기본 설정은 기본적으로 64GB 메모리의 컨테이너 컴퓨팅 용량으로 구성됩니다.
  1. 사용자의 환경은 Bluemix 로컬 하드웨어 요구사항을 충족해야 합니다.
  2. IBM Bluemix Container Service를 사용하려면, 각 설정에 다음 요구사항을 충족하는 제어 영역이 필요합니다. 모니터링과 로깅을 위해서는 동일한 하드웨어 요구사항을 충족하는 두 개의 인스턴스도 필요합니다.
    • VMware: 각각 256GB 메모리 및 32개 실제 코어(하이퍼스레딩 사용 64 vCore)가 있는 ESX 서버 3개. 3개 서버에 걸쳐, 4TB의 클러스터 공유 스토리지가 필요함.
추가 하드웨어 증가를 위한 요구사항
추가 하드웨어 없이 최대 12개의 확장 VM을 추가하여 16GB의 증분 단위로 최대 256GB까지 컨테이너 컴퓨팅 용량을 늘릴 수 있습니다. 각 VM에는 20GB 메모리, 4 vCPU 및 200GB의 디스크 공간이 있어야 합니다. 단, 클러스터에 배치된 각 10개 확장 VM의 경우 네트워킹을 처리하는 데 하나의 VM이 추가로 필요합니다. 이 네트워킹 VM에는 4vCPU, 16GB 메모리 및 250GB의 디스크 공간이 필요합니다.

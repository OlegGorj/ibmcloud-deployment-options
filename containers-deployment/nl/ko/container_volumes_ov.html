단일 및 확장 가능 컨테이너의 볼륨에 지속적 데이터 저장

스토리지 볼륨은 조직의 Bluemix® 영역에 있는 컨테이너 사이에서 데이터를 공유하거나 지속시키는 데 사용됩니다.
주의: 단일 및 확장 가능 컨테이너의 무료 평가판은 사용할 수 없습니다. 클러스터 관리에 대해 알아보고 Bluemix 퍼블릭에서 IBM® Bluemix Container Service의 Kubernetes로 마이그레이션하십시오.
이 페이지의 내용:
     • 볼륨 정보
     • GUI 또는 CLI를 사용하여 볼륨 작성
     • 볼륨에 파일 추가
     • 볼륨 백업 및 복원
     • 볼륨에 비루트 사용자 액세스 권한 추가
     • 다른 데이터 센터로 데이터 마이그레이션
     • 볼륨 제거
     • 파일 공유 추가 및 제거

볼륨 정보

컨테이너는 임시적이므로 컨테이너에서 애플리케이션 데이터를 지속할 수 없습니다. 하지만 볼륨을 사용하면 컨테이너를 다시 시작하는 사이에 데이터를 지속할 수 있고 영역의 컨테이너 사이에서 데이터를 공유할 수 있고 컨테이너를 사용할 수 없을 때 애플리케이션 데이터에 액세스할 수 있습니다. 컨테이너를 삭제해도 스토리지 볼륨은 삭제되지 않습니다. 스토리지 볼륨의 라이프사이클은 컨테이너의 라이프사이클로부터 독립적입니다. Bluemix의 Docker 컨테이너는 로컬 호스트에 대한 액세스 권한이 없기 때문에 볼륨은 프로비저닝된 파일 공유에 마운트됩니다.

지속적 스토리지 라이프사이클에는 다음과 같은 태스크가 있습니다.

  1. 옵션: GUI 또는 CLI를 사용하여 볼륨을 호스트하는 사용자 정의 파일 공유를 작성하십시오. 파일 공유를 작성하려면 조직 관리자 역할이 있어야 합니다. 조직 관리자가 아니면 볼륨 작성을 진행하십시오. Bluemix 조직마다 기본적으로 하나의 20GB의 파일 공유가 제공됩니다.
  2. GUI 또는 CLI를 사용하여 볼륨을 작성하십시오.
  3. ibm-backup-restore image를 사용하여 볼륨의 자동 백업을 작성하십시오.
  4. 볼륨을 삭제하십시오.
  5. 파일 공유를 삭제하십시오.

Bluemix GUI로 볼륨 작성

볼륨은 앱이 작성하는 데이터 또는 앱이 실행해야 하는 파일의 지속적 스토리지 위치입니다. Bluemix GUI에서 컨테이너의 볼륨을 작성할 수 있습니다.

단일 컨테이너는 단기적으로만 지속되도록 디자인되었습니다. 그러나 영역의 컨테이너 간에 데이터를 공유하고 그룹의 컨테이너 인스턴스 간에 데이터를 공유하도록 볼륨을 사용하여 컨테이너 다시 시작 사이에 데이터를 지속할 수 있습니다. Docker에서 볼륨을 마운트할 때 볼륨이 로컬 파일 시스템에 마운트됩니다. IBM Bluemix Container Service에서 컴퓨팅 호스트에 대한 액세스가 제한되므로 호스트 디렉토리를 컨테이너에 마운트할 수 없습니다. 대신 조직 범위의 볼륨을 사용하여 컨테이너를 다시 시작하는 사이에 데이터를 유지합니다. 앱 데이터를 안전하게 저장하고 파일에 대한 액세스와 권한을 관리하는 격리된 파일 공유에서 볼륨을 호스팅합니다. 컨테이너 또는 이미지에 대해 수행되는 업데이트는 볼륨에 영향을 미치지 않습니다. 컨테이너를 삭제해도 스토리지 볼륨은 삭제되지 않습니다.

  1. 카탈로그에서 컨테이너를 선택하고 이미지를 선택하십시오.
  2. 고급 옵션을 선택하여 볼륨 작성 필드를 펼치십시오.
  3. 볼륨 작성을 클릭하십시오.
  4. 볼륨 이름 필드에 이름을 입력하십시오. 볼륨은 영역별로 고유하므로 영역에서 볼륨 이름을 재사용할 수 없습니다. 볼륨 이름에는 대문자, 소문자, 숫자, 밑줄(_), 하이픈(-)을 사용할 수 있습니다.
  5. 파일 공유가 존재하는 경우 파일 공유 드롭 다운 메뉴에서 기존 파일 공유를 선택하십시오. 파일 공유가 존재하지 않는 경우에는 20GB 크기의 기본 파일 공유가 자동으로 작성됩니다. 조직 관리자는 특정 스토리지 크기 및 IOPS를 사용하여 파일 공유를 할당할 수 있습니다. 자세한 정보는 볼륨에 대한 파일 공유 추가를 참조하십시오. 볼륨의 크기는 제한이 없습니다. 볼륨은 호스트되는 파일 공유의 크기만큼 스케일링할 수 있습니다.
  6. 작성을 클릭하십시오. 볼륨을 작성하는 데 몇 분이 걸릴 수 있습니다.
볼륨은 사용자의 영역에서 사용 가능하며 고급 옵션의 기존 볼륨 지정에 있는 드롭 다운 메뉴에서 선택하여 단일 컨테이너 또는 컨테이너 그룹에 마운트할 수 있습니다. 볼륨을 선택했으면 컨테이너 내부로 볼륨을 마운트할 파일 경로(/var/lib/my_volume)를 지정하십시오. 컨테이너를 백업해야 하는 경우 데이터 백업 및 복원을 참조하십시오.

명령행(CLI)에서 볼륨 작성

볼륨은 앱이 작성하는 데이터 또는 앱이 실행해야 하는 파일의 지속적 스토리지 위치입니다. 명령행에서 컨테이너의 볼륨을 작성할 수 있습니다.

단일 컨테이너는 단기적으로만 지속되도록 디자인되었습니다. 그러나 영역의 컨테이너 간에 데이터를 공유하고 그룹의 컨테이너 인스턴스 간에 데이터를 공유하도록 볼륨을 사용하여 컨테이너 다시 시작 사이에 데이터를 지속할 수 있습니다. Docker에서 볼륨을 마운트할 때 볼륨이 로컬 파일 시스템에 마운트됩니다. IBM Bluemix Container Service에서 컴퓨팅 호스트에 대한 액세스가 제한되므로 호스트 디렉토리를 컨테이너에 마운트할 수 없습니다. 대신 조직 범위의 볼륨을 사용하여 컨테이너를 다시 시작하는 사이에 데이터를 유지합니다. 앱 데이터를 안전하게 저장하고 파일에 대한 액세스와 권한을 관리하는 격리된 파일 공유에서 볼륨을 호스팅합니다. 컨테이너 또는 이미지에 대해 수행되는 업데이트는 볼륨에 영향을 미치지 않습니다. 컨테이너를 삭제해도 스토리지 볼륨은 삭제되지 않습니다.

  1. 다음 명령을 실행하여 볼륨을 작성하십시오. 볼륨은 호스트되는 파일 공유의 크기만큼 스케일링할 수 있습니다. 파일 공유를 지정하는 것은 선택사항입니다. bx ic volume-fs를 실행하여 기존 파일 공유를 볼 수 있습니다. 파일 공유가 존재하지 않는 경우에는 20GB 크기의 기본 파일 공유가 자동으로 작성됩니다. 조직 관리자는 특정 스토리지 크기 및 IOPS를 사용하여 파일 공유를 할당할 수 있습니다. 자세한 정보는 볼륨에 대한 파일 공유 추가를 참조하십시오. 
    bx ic volume-create VOLNAME [FSNAME]
    bx ic volume-create 매개변수
    FSNAME
    (선택사항) 파일 공유 이름입니다.사용 가능하거나 이름 지정된 파일 공유가 없는 경우에는 영역의 기본 파일 공유를 기반으로 볼륨이 빌드됩니다.
    VOLNAME
    (필수) 볼륨 이름입니다.이름에는 대문자, 소문자, 숫자, 밑줄(_) 및 파이픈(-)이 포함될 수 있습니다.
  2. 옵션: 다음 명령을 실행하여 볼륨이 올바르게 작성되었는지 확인하십시오. 
    bx ic volumes [--table]

그다음에, 볼륨을 단일 컨테이너 또는 컨테이너 그룹에 지정하십시오. 볼륨은 run 명령 및 group-create 명령에서 각각 --volume 옵션을 사용하여 지정됩니다. 컨테이너를 백업해야 하는 경우 데이터 백업 및 복원을 참조하십시오.

명령행 인터페이스(CLI)를 사용하여 볼륨에 파일 추가

볼륨을 작성한 후에는 볼륨을 컨테이너에 마운트하고 볼륨이 마운트된 컨테이너에서 액세스할 수 있는 파일을 추가할 수 있습니다.

시작하기 전에 Bluemix GUI를 사용하거나 CLI를 사용하여 볼륨을 작성하십시오.
  1. 마운트된 볼륨으로 컨테이너를 작성하십시오.* my_volume을 사용자 볼륨의 이름으로, container_path를 볼륨의 마운트 경로로, my_container를 컨테이너의 이름으로, registry.DomainName/my_namespace/my_image를 이미지로 바꾸십시오. 
    bx ic run --volume my_volume:/container_path --name my_container registry.DomainName/<my_namespace>/my_image
    팁: bx ic namespace-get을 실행하여 네임스페이스를 검색하고 <my_namespace>를 자신의 네임스페이스 정보로 대체하십시오.
    참고: *이 명령에서 IBM Bluemix Container Service에 로그인하고 환경 변수를 기본 Docker 명령을 사용하도록 설정한 경우 bx ic를 docker로 대체할 수 있습니다. 이 주제에서 별표(*)로 표시된 모든 단계에서 기본 Docker 명령을 사용할 수 있습니다.
  2. bx ic cp 명령을 사용하여 로컬 디렉토리에서 볼륨으로 파일을 복사하십시오.*

    이 예에서, 컨테이너 안에서 /src/usr/tmp 경로에 마운트되는 볼륨으로 config.txt 파일을 복사하고 있습니다. 복사 중인 자체 파일 또는 디렉토리로 이 파일을 대체하고 컨테이너 경로가 사용자의 볼륨을 가리키도록 변경하십시오.명령에서 소스 파일이나 경로가 먼저 오고 대상 경로가 두 번째입니다. 컨테이너에/에서 복사할 때 컨테이너의 이름, 구분 기호 ":" 및 대상 파일 또는 폴더의 경로를 포함해야 합니다. 경로는 컨테이너의 루트 디렉토리에 상대적이어야 합니다. 자세한 정보는 bx ic cp 문서를 참조하십시오. 

    bx ic cp ./config.txt my_container:/src/usr/tmp
  3. bx ic exec 명령으로 실행 중인 컨테이너에 로그인하여 파일이 볼륨에 복사되었는지 확인하십시오.* 
    bx ic exec -it my_container bash
  4. 볼륨이 마운트된 디렉토리로 이동하여 파일을 나열하십시오. 파일이 출력에 나열됩니다. 이 볼륨이 마운트되는 컨테이너에서 이 파일에 액세스해야 합니다. 
    # cd container_path
# ls
config.txt

명령행 인터페이스(CLI)를 사용하여 데이터 백업 및 복원

컨테이너 볼륨은 지속적 스토리지에서 호스팅되지만 백업은 없습니다. 볼륨 데이터를 백업행 하는 경우 bx ic cp를 사용하여 수동 백업을 작성할 수 있습니다.

  1. 백업하고 있는, 실행 중이거나 중지된 컨테이너의 마운트 경로를 식별하십시오.* 
    bx ic inspect my_container

    출력에는 볼륨 이름과 컨테이너 마운트 지점이 있는 이 예와 비슷한 섹션이 포함되어 있습니다. 이 예제에서 마운트 지점은 /mnt/data 폴더입니다. 

            "Mounts": [
            {
                "Destination": "/mnt/data",
                "Driver": "",
                "Mode": "rw",
                "Name": "",
                "RW": true,
                "Source": "/vol/fa277ff4-8a64-435b-9b75-0f11d59a3ae4/example_container_id"
            }
    참고: *이 명령에서 IBM Bluemix Container Service에 로그인하고 환경 변수를 기본 Docker 명령을 사용하도록 설정한 경우 bx ic를 docker로 대체할 수 있습니다. 이 주제에서 별표(*)로 표시된 모든 단계에서 기본 Docker 명령을 사용할 수 있습니다.
  2. 옵션: 볼륨에 쓰고 있는 실행 중인 모든 컨테이너를 중지하십시오.* 컨테이너를 중지하여 쓰고 있는 데이터가 복사되지 않도록 합니다. 
    bx ic stop my_container
  3. 단일 컨테이너에서 로컬 컴퓨터로 파일을 복사하십시오.* 컨테이너 그룹의 볼륨을 백업하려는 경우 모든 인스턴스가 볼륨에 대한 동일한 마운트 지점을 공유하므로 그룹의 한 인스턴스에서만 이 명령을 실행해도 충분합니다.

    이제 볼륨 데이터의 로컬 백업이 생깁니다.

    bx ic cp my_container:/mnt/data my_local_folder
    표 1. bx ic cp 명령의 요소
    이 명령 이해
    cp 복사 명령입니다.
    my_container:/mnt/data 첫 번째는 소스 경로입니다. 이 경우 컨테이너의 데이터를 로컬 대상에 복사하므로 컨테이너 파일 경로가 소스입니다. 컨테이너에 컨테이너 이름, 콜론 및 파일 경로를 포함해야 합니다. 이 예에서는 포함된 모든 파일을 복사할 수 있도록 이전 단계에서 볼륨 마운트 지점을 선택했습니다.
    my_local_folder 두 번째는 대상 경로입니다. 로컬 경로는 절대적이거나 상대적일 수 있습니다. 이 예에서는 컨테이너 데이터를 상대 경로 my_local_folder에 복사합니다.
  4. 옵션: 로컬 폴더에서 새 볼륨으로 데이터를 복사하십시오.* 이 명령은 데이터를 새 컨테이너나 새 볼륨에 복원합니다. 데이터를 새 볼륨에 전송하는 이유 중 하나는 이전 파일 공유보다 스토리지가 많거나 IOPS가 높은 새로 작성된 파일 공유에 볼륨이 마운트되기 때문입니다. 새 파일 공유 작성은 명령행 인터페이스(CLI)를 사용하여 파일 공유 추가의 내용을 참조하고 새 볼륨 작성은 명령행(CLI)에서 볼륨 작성의 내용을 참조하십시오. 
    bx ic cp my_local_folder my_new_container:/mnt/data
이제 데이터가 로컬 폴더에 복사되며 수동 백업이 생깁니다. 또한 데이터가 새 컨테이너의 볼륨에 복사되었습니다.

명령행 인터페이스(CLI)를 사용하여 볼륨에 대한 비루트 사용자 액세스 권한 추가

비루트 사용자에게는 볼륨 마운트 경로에 대한 쓰기 권한이 없습니다. 쓰기 권한을 부여하려면, 이미지의 Dockerfile을 편집하여 올바른 권한으로 마운트 경로에서 디렉토리를 작성해야 합니다.

IBM Bluemix Container Service의 경우, 사용자 네임스페이스 기능이 Docker Engine에 대해 사용됩니다. 컨테이너 루트 사용자가 기타 컨테이너 또는 컴퓨팅 호스트에 대한 액세스 권한을 받을 수 없도록, 사용자 네임스페이스는 격리를 제공합니다. 그러나 볼륨은 컨테이너 컴퓨팅 호스트에 대해 외부인 NFS 파일 공유에 있습니다. 볼륨은 컨테이너의 루트 사용자를 인식하도록 설정되어 있습니다. 그러나 볼륨이 컨테이너에 대해 외부이므로 이는 사용자 네임스페이스를 인식하지 않습니다.

일부 애플리케이션의 경우, 컨테이너 내의 유일한 사용자는 루트 사용자입니다. 그러나 다수의 애플리케이션은 컨테이너 마운트 경로에 쓰는 비루트 사용자를 지정합니다. 볼륨에 대한 쓰기 권한이 필요한 비루트 사용자로 애플리케이션을 디자인하는 경우, 다음 프로세스를 Dockerfile 및 시작점 스크립트에 추가해야 합니다.

  • 비루트 사용자를 작성하십시오.
  • 임시로 사용자를 루트 그룹에 추가하십시오.
  • 올바른 사용자 권한으로 볼륨 마운트 경로에서 디렉토리를 작성하십시오.

  1. 로컬 디렉토리에서 Dockerfile을 작성하십시오. 이 예제 Dockerfile은 이름이 myguest인 비루트 사용자를 작성합니다. 
    FROM registry.domain_name/ibmnode:latest

#Create group and user with GID & UID 1010.
#In this case your are creating a group and user named myguest. 
#The GUID and UID 1010 is unlikely to create a conflict with any existing user GUIDs or UIDs in the image.
#The GUID and UID must be between 0 and 65536. Otherwise, container creation fails.
RUN groupadd --gid 1010 myguest
RUN useradd --uid 1010 --gid 1010 -m --shell /bin/bash myguest

ENV MY_USER=myguest

COPY entrypoint.sh /sbin/entrypoint.sh
RUN chmod 755 /sbin/entrypoint.sh

EXPOSE 22
ENTRYPOINT ["/sbin/entrypoint.sh"]
  2. Dockerfile과 동일한 로컬 폴더에 시작점 스크립트를 작성하십시오. 이 예제 시작점 스크립트는 볼륨 마운트 경로로 /mnt/myvol을 지정합니다. 
    #!/bin/bash
set -e

#This is the mount point for the shared volume.
# Use this mount point with the bx ic run command.
#By default the mount point is owned by the root user.
MOUNTPATH="/mnt/myvol"
MY_USER=${MY_USER:-"myguest"}

#This function creates a subdirectory that is owned by 
#the non-root user under the shared volume mount path.
create_data_dir() {
  #Add the non-root user to primary group of root user.
  usermod -aG root $MY_USER

  #Provide read-write-execute permission to the group for the shared volume mount path.
  chmod 775 $MOUNTPATH
  #Create a directory under the shared path owned by non-root user myguest.
  su -c "mkdir -p ${MOUNTPATH}/mydata" -l $MY_USER
  su -c "chmod 700 ${MOUNTPATH}/mydata" -l $MY_USER
  ls -al ${MOUNTPATH}

  #For security, remove the non-root user from root user group.
  deluser $MY_USER root

  #Change the shared volume mount path back to its original read-write-execute permission.
  chmod 755 $MOUNTPATH
  echo "Created Data directory..."
}

create_data_dir

#This command creates a long-running process for the purpose of this example.
tail -F /dev/null
  3. IBM Bluemix에 로그인하십시오. 
    bx login [-a api.DomainName] [-sso]
    싱글 사인온 매개변수 -sso는 연합 ID로 로그인할 때 필요합니다. 이 옵션이 사용되면 https://login.ng.bluemix.net/UAALoginServerWAR/passcode를 열고 일회성 패스코드를 얻으십시오. --sso 옵션을 포함하지 않는 경우에는 다음의 하위 단계를 완료하십시오.
    1. 이메일에 대해 Bluemix의 IBM ID를 입력하십시오.
    2. 비밀번호에 대해 IBM ID의 비밀번호를 입력하십시오. 사용자의 Bluemix 조직과 영역이 검색됩니다.
    3. Bluemix 조직 중 하나를 표시하는 번호를 입력하십시오.
    4. 기존 Bluemix 영역 중 하나를 표시하는 번호를 입력하십시오.
  4. IBM Bluemix Container Service CLI를 초기화하십시오. 
    bx ic init
  5. Bluemix 레지스트리에 이미지를 추가하십시오.* <my_namespace>를 개인용 이미지 레지스트리의 네임스페이스로 대체하십시오. 네임스페이스를 찾아야 하는 경우에는 bx ic namespace-get을 실행하십시오. 
    bx ic build -t registry.domain_name/<my_namespace>/nonroot .
    참고: *이 명령에서 IBM Bluemix Container Service에 로그인하고 환경 변수를 기본 Docker 명령을 사용하도록 설정한 경우 bx ic를 docker로 대체할 수 있습니다. 이 주제에서 별표(*)로 표시된 모든 단계에서 기본 Docker 명령을 사용할 수 있습니다.
  6. 볼륨을 작성하십시오. 
    bx ic volume-create my_volume
  7. 볼륨을 마운트하고 nonroot 이미지에서 컨테이너를 실행하십시오.* 볼륨 마운트 경로 /mnt/myvol은 Dockerfile에 지정된 마운트 경로와 일치합니다. 
    bx ic run --name my_container --volume my_volume:/mnt/myvol registry.ng.bluemix.net/<my_namespace>/nonroot
  8. my_container가 실행 중인지 확인하십시오.* 컨테이너의 상태가 BUILDING인 경우에는 아직 이에 로그인할 수 없습니다. 
    bx ic ps
    $ bx ic ps
CONTAINER ID        IMAGE                                           COMMAND             CREATED             STATUS                  PORTS                         NAMES
d92ee43d-2af        registry.domain_name/<my_namespace>/nonroot:latest   ""                  41 seconds ago      Running 29 seconds ago               my_container
  9. 컨테이너가 실행된 후에 컨테이너에 로그인하십시오.* 
    bx ic exec -it my_container /bin/bash
  10. 볼륨 마운트 경로의 권한을 보십시오. 
    ls -al /mnt/myvol/
    root@instance-006ff76b:/# ls -al /mnt/myvol/                                                                                                                                                             
total 12
drwxr-xr-x 3 root    root    4096 Jul 13 19:03 .
drwxr-xr-x 3 root    root    4096 Jul 13 19:03 ..
drwx------ 2 myguest myguest 4096 Jul 13 19:03 mydata

    이 출력은 해당 루트에 볼륨 마운트 경로 mnt/myvol/에서 읽기, 쓰기 및 실행이 있는 것으로 표시하지만, 비루트 myguest 사용자에게는 mnt/myvol/mydata 폴더에 대한 읽기 및 쓰기 권한이 있습니다. 이 업데이트된 권한 때문에 비루트 사용자는 이제 데이터를 지속적 볼륨에 쓸 수 있습니다.

볼륨 데이터를 다른 데이터 센터로 마이그레이션

현재 Bluemix 영역을 새로운 데이터 센터로 이동할 계획인 경우, 새로운 데이터 센터에서 데이터를 복원하여 사용할 수 있도록 볼륨이 백업되었는지 확인하십시오.

주의: bx ic reprovision 명령을 사용하여 영역을 새로운 데이터 센터에 이동하는 경우, 이 영역에 있는 모든 컨테이너, 공인 IP 주소 및 볼륨이 영구적으로 제거되며 새 데이터 센터에 마이그레이션되지 않습니다. 새 데이터 센터로 이동한 후 데이터가 유실되지 않게 하려면, 모든 볼륨을 백업하고 영역이 새 데이터 센터에서 다시 프로비저닝된 후에 데이터를 복원해야 합니다.
  1. 다시 프로비저닝하려는 Bluemix 영역에 있는 모든 볼륨을 나열하십시오. 
    bx ic volumes
  2. 새 데이터 센터에서 사용할 모든 볼륨을 백업하십시오. 다음 옵션 중에 선택하십시오.
    • bx ic cp 명령을 사용하여 로컬 머신에 볼륨을 백업합니다.
    • IBM Backup and Restore 이미지에서 백업 컨테이너를 작성하여 IBM Object Storage for Bluemix 인스턴스에 데이터를 저장합니다.
    주의: 볼륨을 백업하지 않고 영역을 새 데이터 센터로 다시 프로비저닝한 경우 데이터가 유실되며 복구할 수 없습니다.
  3. 영역에서 모든 볼륨을 제거하십시오.
  4. 영역에서 모든 파일 공유를 제거하십시오.
  5. 새 데이터 센터로 영역을 다시 프로비저닝하십시오. <data_center>를 새 영역을 작성하려는 데이터 센터로 바꾸십시오. 
    bx ic reprovision <data_center>
  6. 새 영역에 파일 공유를 작성하십시오.
  7. 백업된 데이터를 복원하는 데 사용할 수 있는 볼륨을 작성하십시오.
  8. 데이터를 새 볼륨에 복원하십시오. 볼륨을 백업하는 데 사용한 방법에 따라 다음 옵션 중에서 선택하십시오.
    • bx ic cp 명령을 사용하여 로컬 머신에서 데이터를 복원합니다.
    • IBM Backup and Restore 이미지에서 복원 컨테이너를 작성하여 IBM Object Storage for Bluemix 인스턴스에서 데이터를 복원합니다.

명령행(CLI)에서 볼륨 제거

컨테이너에 대한 데이터를 저장하기 위한 볼륨이 더 이상 필요하지 않으면 볼륨을 제거할 수 있습니다.

  • 중요사항: 볼륨을 제거하면 볼륨에 저장된 지속적 데이터 또는 파일이 영구적으로 제거됩니다. 볼륨은 자동으로 백업되지 않으므로 볼륨을 제거하기 전에 볼륨 백업을 수행하십시오.
  1. 볼륨 이름을 식별하십시오. 
    bx ic volumes [--table]
  2. 볼륨을 조사하여 볼륨이 마운트된 컨테이너를 검토하십시오. 
    bx ic volume-inspect example_volume

    출력에는 mounted_on_containers라는 섹션이 포함되어 있으며 볼륨이 마운트된 컨테이너가 나열됩니다. 

        "mounted_on_containers": [
        "example_container"
    ],
  3. 볼륨이 컨테이너로 마운트된 경우, 볼륨을 삭제하기 전에 기밀 데이터를 제거하십시오.
    1. 옵션: 컨테이너를 검사하여 볼륨이 마운트된 마운트 경로를 찾으십시오. 
      bx ic inspect example_container
      CLI 응답의 Volumes 섹션에서 마운트 경로를 찾을 수 있습니다. 
      "Volumes": {
        "/mount_path": "/vol/0e65c452-62413-46f2-8b22-ff06ab74ac21/example_volume"
    2. 컨테이너에 로그인하십시오. 
      bx ic exec -it example_container bash
    3. 마운트 경로로 이동하십시오. 
      cd /mount_path
    4. 볼륨에서 기밀 데이터를 모두 제거하십시오. 
      rm -rf *
    5. 모든 데이터가 제거되었는지 확인하십시오. 
      ls
      표시되는 디렉토리 또는 파일이 없다면, 데이터가 볼륨 마운트 경로에서 제거된 것입니다.
    6. 볼륨에 마운트되고 해당 볼륨 마운트 경로에 기밀 데이터가 있는 모든 컨테이너에 대해 a - e 단계를 반복하십시오.
  4. 볼륨을 제거하기 전에 볼륨에 마운트된 모든 컨테이너를 중지하십시오.* 
    bx ic stop example_container
    참고: *이 명령에서 IBM Bluemix Container Service에 로그인하고 환경 변수를 기본 Docker 명령을 사용하도록 설정한 경우 bx ic를 docker로 대체할 수 있습니다. 이 주제에서 별표(*)로 표시된 모든 단계에서 기본 Docker 명령을 사용할 수 있습니다.
  5. 컨테이너가 더 이상 필요하지 않으면 제거하십시오.* 
    bx ic rm example_container
  6. 볼륨이 마운트된 컨테이너가 모두 중지되면 볼륨을 제거하십시오. 
    bx ic volume-remove VOLNAME
    bx ic volume-remove 매개변수
    VOLNAME
    (필수) 볼륨 이름입니다.
  7. 옵션: 볼륨이 제거되었는지 확인하십시오. 
    bx ic volumes [--table]

Bluemix GUI로 볼륨의 파일 공유 추가

파일 공유는 Bluemix 영역에서 Docker 볼륨을 호스트하는 지속적 NFS 기반(네트워크 파일 시스템) 스토리지 시스템이며, 이 시스템을 통해 사용자가 컨테이너와 앱 관련 파일을 저장하고 액세스할 수 있습니다. 파일 공유에 파일을 저장하려면 컨테이너 볼륨을 작성하고 이 볼륨에 데이터를 저장해야 합니다.
참고: 이미 최대 할당량 한계에 도달한 경우에는 Bluemix 지원 티켓을 열어서 할당량 추가를 요청할 수 있습니다.

Bluemix의 컨테이너에 로컬 호스트에 대한 액세스 권한이 없기 때문에 볼륨은 프로비저닝된 NFS 파일 공유에 마운트됩니다. 파일 공유의 볼륨은 호스트 컴퓨터의 디렉토리와 유사합니다. 파일 공유 및 볼륨 둘 다 영역마다 고유하여, 컨테이너가 동일한 영역에서 작성되어야 볼륨의 지속적 데이터에 액세스할 수 있습니다. 조직 및 영역 설정에 대한 자세한 정보는 Bluemix 퍼블릭 관리를 참조하십시오. 조직의 볼륨을 처음 작성할 때, 4 IOPS 및 20GB의 파일 공유가 자동으로 프로비저닝되어 볼륨을 호스팅합니다. 데이터의 성능 요구사항을 충족하기 위해, 조직 관리자는 0.25, 2 또는 4의 GB당 IOPS 및 20GB - 12TB의 새 파일 공유 크기를 작성할 수 있습니다.

  1. 계정 세부사항의 조직 관리에서 조직을 선택하십시오.
  2. 할당량 섹션에서 컨테이너의 세부사항을 보십시오.
  3. 파일 공유를 작성하십시오. 테이블 끝에 행이 작성됩니다.
  4. 파일 공유가 작성될 기존 영역을 선택하십시오.
  5. 파일 공유의 이름을 입력하십시오. 이름에는 대문자, 소문자, 숫자, 밑줄(_) 및 파이픈(-)이 포함될 수 있습니다.
  6. 파일 공유에 필요한 용량을 판별하십시오. 파일 공유 크기와 GB당 IOPS(초당 입출력(I/O) 오퍼레이션)를 곱하여 총 IOPS가 됩니다. 테이블에서 파일 공유 크기 및 IOPS 간의 관계에 대해 설명합니다.
    표 2. 파일 공유 크기와 GB당 IOPS의 예
    파일 공유 크기 GB당 IOPS 총 IOPS
    20GB 4 80
    100GB 4 400
    1,000GB 4 4,000
    4,000GB 4 16,000
    보다 높은 워크로드의 앱에서는 보다 큰 총 IOPS가 필요하므로, 파일 공유 구성을 선택할 때 볼륨에 쓰는 앱을 고려하십시오.
    1. 크기에 대해 기가바이트 단위의 값을 선택하십시오.
    2. IOPS에 대해 기가바이트당 IOPS 단위의 값을 선택하십시오. 크기 및 IOPS에 대한 입력을 기반으로 매월 가격이 계산됩니다. 계속하기 전에 크기 및 IOPS의 값을 조정하여 매월 가격 계산을 변경할 수 있습니다.
  7. 저장을 클릭하십시오.

파일 공유 제거

조직 관리자는 기존 파일 공유를 제거하고 파일 공유에서 마운트된 볼륨을 볼 수 있습니다.

조직 관리자는 Bluemix GUI와 CLI를 모두 사용해서 파일 공유를 검토하여 파일 공유에서 마운트된 볼륨을 확인하고 파일 공유를 제거할 수 있습니다. 조직 관리자는 개발 또는 프로덕션에서 현재 사용되고 있지 않은 파일 공유를 제거할 수 있습니다.
참고: 청구 처리 주기 때문에 월말에는 파일 공유를 삭제할 수 없습니다. 월말에 파일 공유를 삭제할 때 오류 메시지가 표시되면 다음 달 첫 번째 날이 올 때까지 기다렸다가 파일 공유를 삭제하십시오.

Bluemix GUI 사용

  1. 계정 세부사항의 조직 관리에서 조직을 선택하십시오.
  2. 할당량 섹션에서 컨테이너의 세부사항을 보십시오.
파일 스토리지 테이블에서 ACTION 열 및 NAME 열을 찾을 수 있습니다.
  • 파일 공유의 이름을 클릭하여 연결된 볼륨을 보십시오.
  • 삭제를 클릭하여 파일 공유를 삭제하십시오. 파일 공유를 삭제하려면 먼저 연결된 볼륨을 제거해야 합니다. 파일 공유는 삭제되고 나면 계정에서 영구적으로 제거됩니다. 볼륨을 제거하려면 명령행(CLI)에서 볼륨 제거의 내용을 참조하십시오.

명령행 사용

CLI를 사용하여 파일 공유를 제거하려면 다음의 명령을 실행하십시오.
  1. 파일 공유 이름을 식별하십시오. 
    bx ic volume-fs
  2. 파일 공유를 검사하여 파일 공유에 볼륨이 존재하는지 판별하십시오. 
    bx ic volume-fs-inspect FSNAME
  3. 필요한 경우 기존 볼륨을 제거하십시오. 
    bx ic volume-remove VOLNAME
  4. 기존 파일 공유를 제거하십시오. 
    bx ic volume-fs-remove FSNAME
  5. 기존 파일 공유를 나열하여 파일 공유가 제거되는지 확인하십시오. 
    bx ic volume-fs

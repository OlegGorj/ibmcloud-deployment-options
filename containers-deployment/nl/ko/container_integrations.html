Bluemix GUI에서 서비스 바인딩

IBM Bluemix에는 서비스 목록이 있으며 앱 개발자 대신 해당 서비스를 관리합니다. 사용할 컨테이너의 Bluemix 서비스를 추가하려면 이 서비스의 인스턴스를 요청하고 서비스를 컨테이너에 바인드해야 합니다.

시작하기 전에 Bluemix 영역에 서비스를 추가하십시오. 자세한 정보는 새 서비스 인스턴스 요청을 참조하십시오.
제한사항: 일부 Bluemix 서비스는 컨테이너에 대한 직접적 바인딩을 지원하지 않으며 컨테이너 작성 동안 실패를 일으킵니다. 이 경우, 먼저 Bluemix 서비스를 Cloud Foundry 앱에 바인드한 다음 앱을 컨테이너에 바인드하십시오. 이 앱은 서비스와 컨테이너 간 브릿지 역할을 수행합니다. 자세한 정보는 Bluemix 서비스가 컨테이너에 직접 바인드되지 않음의 내용을 참조하십시오.
  1. Bluemix 카탈로그에서 컨테이너를 클릭하십시오.
  2. 컨테이너 이미지를 클릭하십시오.
  3. 고급 옵션을 펼치고, 서비스 선택에 대해 사용 가능한 Bluemix 서비스를 선택하십시오.
  4. 추가를 클릭하십시오. Bluemix 서비스가 컨테이너에 이미 바인드됨 필드에 표시됩니다.
  5. 다른 필드를 완성한 후 작성을 클릭하십시오.
  6. 컨테이너를 작성할 때 VCAP_SERVICES 정보가 표시되는지 확인하십시오. 컨테이너의 개요 탭에서 바인딩된 서비스를 찾고 신임 정보 보기를 클릭하십시오. 표시된 정보는 VCAP_SERVICES입니다.

CLI에서 컨테이너에 서비스 바인드

IBM Bluemix에는 서비스 목록이 있으며 앱 개발자 대신 해당 서비스를 관리합니다. 사용할 컨테이너의 Bluemix 서비스를 추가하려면 이 서비스의 인스턴스를 요청하고 서비스를 컨테이너에 바인드해야 합니다.

시작하기 전에 Bluemix 영역에 서비스를 추가하십시오. 자세한 정보는 새 서비스 인스턴스 요청을 참조하십시오.
제한사항: 일부 Bluemix 서비스는 컨테이너에 대한 직접적 바인딩을 지원하지 않으며 컨테이너 작성 동안 실패를 일으킵니다. 이 경우, 먼저 Bluemix 서비스를 Cloud Foundry 앱에 바인드한 다음 앱을 컨테이너에 바인드하십시오. 이 앱은 서비스와 컨테이너 간 브릿지 역할을 수행합니다. 자세한 정보는 Bluemix 서비스가 컨테이너에 직접 바인드되지 않음의 내용을 참조하십시오.
  1. 다음 명령 중 하나를 사용하여 Bluemix 서비스 인스턴스를 새 컨테이너에 바인드하십시오.
    팁: bx ic namespace-get을 실행하여 네임스페이스를 검색하고 <my_namespace>를 자신의 네임스페이스 정보로 대체하십시오.
    • 단일 컨테이너를 작성하는 경우에는 다음 명령을 사용하십시오.
      bx ic run --name container_name -e "CCS_BIND_SRV=service_instance1,service_instance2" registry.DomainName/<my_namespace>/image_name
    • 컨테이너 그룹을 작성하는 경우에는 다음 명령을 사용하십시오.
      bx ic group-create --name container_group_name -e "CCS_BIND_SRV=service_instance1,service_instance2" registry.DomainName/<my_namespace>/image_name
    • 이미 실행 중인 컨테이너 그룹에 서비스를 바인딩하는 경우 다음 명령을 사용하십시오. 이 명령은 컨테이너 그룹에 대해서만 사용할 수 있습니다. 한 번에 하나의 서비스만 바인드할 수 있습니다.
      bx ic service-bind container_group_name service_instance
  2. 다음 명령 중 하나를 사용하여 서비스 인스턴스가 컨테이너에 바인드되었는지 확인하십시오.
    • 단일 컨테이너의 경우 다음 명령을 사용하십시오.
      bx ic inspect CONTAINER
    • 컨테이너 그룹의 경우 다음 명령을 사용하십시오.
      bx ic group-inspect GROUP

Alert Notification을 사용하여 컨테이너에 배치된 앱의 경보 구성

앱을 구성하여 특정한 조건이나 기준에 맞으면 알림을 보내고 Alert Notification 서비스를 사용하여 경보 알림을 관리하십시오. Alert Notification 서비스를 사용하면 관련된 경보를 필터링하는 사용자 정의 알림 정책을 작성할 수 있습니다. 필요에 따라 정책 필터를 세분화하여 선호하는 알림 방법 및 빈도뿐 아니라 개인 또는 그룹을 경보 수신인으로 지정하십시오.

시작하기 전에 새 서비스 인스턴스 요청 주제를 검토하고 Bluemix 영역에서 Alert Notification 서비스의 인스턴스를 작성하십시오.

다음 단계에서는 샘플 Alert Notification 노드 앱을 실행하는 컨테이너를 작성하는 방법을 보여줍니다. 테스트 경보를 Alert Notification 서비스에 보내고 경보 뷰어 도구에서 이를 검토할 수 있습니다.

  1. 샘플 경보 알림 앱을 로컬 시스템에 다운로드하여 복제하십시오. 
    git clone https://github.com/IBM-Bluemix/alert-notification-helloworld.git
    참고: Git 프로젝트를 복제할 때 이름이 alert-notification-helloworld인 새 폴더가 로컬 시스템에 작성됩니다.
  2. 샘플 경보 알림 앱이 포함된 컨테이너 이미지를 빌드하고 이를 개인용 Bluemix 이미지 레지스트리에 푸시하십시오.
    1. IBM Bluemix Container Service CLI로 로그인하십시오.
    2. alert-notification-helloworld 디렉토리로 이동하십시오.
    3. 컨테이너 이미지를 작성하여 개인용 Bluemix 이미지 레지스트리에 푸시하십시오. bx ic namespace-get을 실행하여 네임스페이스 정보를 검색하고 <my_namespace>를 사용자 네임스페이스로 바꾸십시오. 
      bx ic build -t registry.DomainName/<my_namespace>/alert .
    4. 컨테이너 이미지가 성공적으로 작성되어 푸시되는지 확인하십시오. 
      bx ic
images
  3. 이미지에서 컨테이너를 작성하고 Alert Notification 서비스를 이에 바인드하십시오. <my_namespace>는 네임스페이스 정보로 바꾸고 <my_alert>는 시작 시 작성한 경보 알림 서비스 인스턴스의 이름으로 바꾸십시오. 
    bx ic run --name alertcontainer -p 8080 -e "CCS_BIND_SRV=<my_alert>" registry.DomainName/<my_namespace>/alert
  4. 컨테이너가 실행 중인지 확인하십시오. 
    bx ic ps
  5. 공인 IP 주소를 컨테이너에 바인드하십시오.
    1. 해당 영역에서 사용 가능한 공인 IP 주소를 나열하고 사용하려는 IP 주소를 기록해 놓으십시오. 
      bx ic ips
      팁: 공인 IP 주소가 해당 영역에서 사용 가능하지 않은 경우 bx ic ip-request를 실행하여 새 IP 주소를 요청하십시오. 또한 bx ic unbind <IP_ADDRESS> <CONTAINER> 명령을 실행하여 컨테이너에서 사용된 IP 주소를 바인드 해제할 수 있습니다.
    2. 공인 IP 주소를 컨테이너에 바인드하십시오. <IP_ADDRESS>를 이전 단계에서 기록한 IP 주소로 바꾸십시오. 
      bx ic ip-bind <IP_ADDRESS> alertcontainer
  6. 테스트 경보를 작성하십시오.
    1. 웹 브라우저를 여십시오.
    2. 다음과 같이 웹 브라우저에서 IP 주소 및 포트를 입력하십시오. <IP_ADDRESS>:8080 Alert Notification 웹 앱의 시작 페이지가 열립니다.

    3. 샘플 경보 발송 단추를 클릭하여 경보 뷰어에 테스트 경보를 보내십시오.
  7. 경보 뷰어를 열고 테스트 경보를 검토하십시오.
    1. Bluemix GUI에서 실행 중인 컨테이너 인스턴스를 선택하십시오.
    2. 바인딩된 서비스에서 Alert Notification 서비스를 클릭하십시오.
    3. 실행을 클릭하여 경보 뷰어를 여십시오.
    4. 테스트 경보를 검토하십시오.
  8. Alert Notification 서비스의 기능을 탐색하십시오. Alert Notification 문서를 검토하여 알림 정책 작성 및 경보 관리 방법에 대한 자세한 정보를 찾으십시오.

Continuous Delivery 서비스에서 Delivery Pipeline을 사용하여 컨테이너 배치 자동화

IBM Bluemix Continuous Delivery의 일부인 Delivery Pipeline을 사용하여 Bluemix에 대한 앱 빌드 및 컨테이너 배치를 자동화합니다.

Delivery Pipeline은 다음을 지원합니다.
  • Docker 이미지 빌드
  • 컨테이너의 이미지를 Bluemix에 배치

Delivery Pipeline을 사용하여 컨테이너를 배치하기 위한 빠른 시작 안내:

  1. 사용자의 코드를 포함하는 Git 저장소를 작성하십시오.
  2. 저장소의 루트 디렉토리에 Dockerfile을 포함하십시오.
  3. 메뉴에서 서비스 > DevOps를 클릭한 다음에 파이프라인을 클릭하십시오.
  4. 파이프라인 작성 > Delivery Pipeline을 클릭하십시오.
  5. 작성을 클릭하십시오.
  6. 사용자의 Git 저장소를 도구 체인에 추가하십시오.
    1. 도구 추가를 클릭하십시오.
    2. 사용자의 저장소를 호스팅하는 Git 서비스의 유형을 클릭하십시오.
    3. 사용자의 저장소에 대한 구성 세부사항을 제공하십시오.
    4. 통합 작성을 클릭하십시오.
  7. 도구 체인의 개요 페이지에서 Delivery Pipeline을 클릭하십시오.
  8. 빌드 작업을 포함하는 빌드 단계를 작성하십시오.
    1. 단계 추가를 클릭하십시오.
    2. 입력 탭에서 입력 유형으로 Git 저장소를 선택하십시오.

  9. 배치 작업을 포함하는 배치 단계를 작성하십시오.
    1. 입력 탭에서 입력 유형으로 빌드 아티팩트를 선택하십시오.
    2. 작업 탭에서 배치자 유형으로 IBM Container Service for Bluemix를 선택하십시오.
  10. 코드 변경을 푸시하거나 Delivery Pipeline의 빌드 단계에서 단계 실행 아이콘을 클릭하여 배치를 실행하십시오.

Delivery Pipeline을 사용하여 단일 컨테이너 작성

Delivery Pipeline에서 단일 컨테이너의 빌드 및 배치를 자동화합니다.

Delivery Pipeline을 사용하여 컨테이너의 빌드 단계 작성

IBM Bluemix Container Service는 소스 코드에서 사용된 후 Docker 이미지를 빌드해서 컨테이너 레지스트리에 공개하는 빌드 서비스를 포함하고 있습니다. Delivery Pipeline은 IBM Bluemix Continuous Delivery의 일부이며, 컨테이너 빌드 서비스를 사용하는 빌드 작업을 제공합니다. 이 빌드 작업은 저장소에서 소스 코드를 가져와서 해당 코드로부터 Docker 이미지를 빌드하고 빌드 번호와 일치하도록 해당 이미지를 버전화한(태그 지정) 다음, 결과 이미지를 Bluemix에 있는 조직의 IBM Bluemix Container Service 레지스트리에 푸시합니다. 그런 다음, 조직의 구성원은 결과 이미지를 조직 내의 영역에 배치할 수 있습니다.

시작하기 전에 다음 정보를 고려하십시오.
  • 빌드, 단계 및 기본 파이프라인 기능에 대해 자세히 알아보려면 Continuous Delivery 문서에서 Delivery Pipeline 정보 주제를 참조하십시오.
  • Bluemix 카탈로그를 확인하여 조직의 개인용 이미지 레지스트리에서 컨테이너에서 사용할 기존 이미지를 식별하십시오.
  • 컨테이너에 서비스를 바인드하려는 경우에는 브릿지로 사용할 앱을 작성하십시오. 자세한 정보는 Delivery Pipeline을 사용하여 컨테이너에 서비스 바인딩을 참조하십시오.

Delivery Pipeline에서 Docker 이미지를 빌드하십시오.

  1. 도구 체인과 연관된 Git 저장소의 루트 디렉토리에서 Dockerfile을 작성하고 사용자의 저장소에 커미트하십시오.
  2. 도구 체인의 개요 페이지에서 Delivery Pipeline을 클릭하십시오.
  3. 빌드 단계를 작성하십시오.
    1. 단계 추가를 클릭하십시오.
    2. 단계에 이름을 지정하십시오. 예: Build.
    3. 입력 유형으로 Git 저장소를 선택하십시오.
    4. 필요하면 다른 필드의 값과 업데이트를 확인하십시오.
    5. 작업 탭을 클릭하십시오.
  4. 빌드 단계에서 빌드 작업을 작성하십시오.
    1. 작업 탭에서 작업 추가를 클릭하십시오.
    2. 빌드 작업 유형을 선택하십시오.
    3. 빌더 유형으로 IBM Containers on Bluemix를 선택하십시오.
    4. 필요한 경우 Bluemix 대상, 조직 및 영역을 업데이트하십시오.
    5. 캐시된 계층 사용의 경우, 다른 소스에서 컨텐츠를 동적으로 가져오는 계층의 새로 고치기를 강제 실행해야 할 때는 필드를 선택 취소하십시오.

      예를 들어, Dockerfile에 apt-get install 또는 wget 명령이 포함된 경우 캐시된 계층 사용 선택란을 선택 취소하여 각 빌드에 대해 이러한 동적 종속 항목이 수집되도록 하십시오.

      참고: 캐시된 계층 사용 필드가 선택 취소되면, 이미지를 빌드할 때 캐시된 계층을 사용하지 않으므로 빌드 시간이 더 오래 걸립니다.
    6. 이미지 이름 필드에서 빌드할 이미지의 이름을 입력하십시오. Docker 이름 지정 규칙을 충족하려면 소문자, 숫자 및 하이픈(-)만 사용하십시오. 이미지가 빌드되면 이미지 이름과 빌드 번호로 이미지에 태그가 지정됩니다. 예를 들어, 이미지 이름 myappimage를 입력하면 빌드 번호 4의 경우 결과로 얻는 이미지는 myappimage:4로 불립니다.
    7. Dockerfile이 프로젝트의 루트 디렉토리에 없으면 빌드 스크립트 섹션에서 다음 위치의 Dockerfile에 경로를 추가하십시오.
      1. 단위 테스트 실행 섹션 앞에 디렉토리 변경 명령을 사용하여 폴더 이름을 추가하십시오. 
        cd <folder>
      2. ${WORKSPACE} 변수가 표시되는 곳마다 폴더 이름을 추가하십시오. ${WORKSPACE} 변수는 프로젝트의 루트 디렉토리를 자동으로 발견합니다.
        예
        BUILD_COMMAND="build --pull --tag ${FULL_REPOSITORY_NAME} ${WORKSPACE}/<folder>"
  5. 저장을 클릭하십시오.
  6. 단계 실행 아이콘을 클릭하여 오류 없이 단계가 빌드되는지 확인하십시오. 빌드에 실패하면 로그를 검토하고 오류를 정정하십시오. 단계를 패스하는 경우 조직의 이미지 레지스트리를 확인하고 이미지가 빌드되었는지 확인하십시오.

(선택사항) 파이프라인의 일부로 이미지 보안 확인

Vulnerability Advisor를 사용하여 컨테이너를 배치하기 전에 이미지에서 보안 또는 구성 문제를 찾으십시오.
이 스캔을 포함하는 공통 파이프라인 설정은 다음과 같습니다.
  • 소스 파일을 빌드하는 빌드 단계
  • 처리 단계. 처리 단계는 다음과 같은 작업을 포함합니다.
    • 컨테이너 빌드를 실행하는 빌드 작업
    • 이미지에 대해 Vulnerability Advisor를 실행하는 테스트 작업
  • 컨테이너를 배치하는 배치 단계
팁: 이 태스크는 컨테이너 이미지를 빌드하는 작업과 다른 단계에서 Vulnerability Advisor의 테스트 작업을 작성하는 방법에 대해 개략적으로 설명합니다. 또는 동일한 단계를 사용할 수 있습니다. 별도 단계를 작성하기로 결정하는 경우에는 환경 변수를 설정하지 않아도 됩니다.

Vulnerability Advisor 스캔을 작성하십시오.

  1. 처리 단계를 작성하십시오.
    1. 단계 추가를 클릭하십시오.
    2. 단계에 이름을 지정하십시오. 예: Processing.
    3. 입력 유형으로 빌드 아티팩트를 선택하십시오.
    4. 필요하면 단계 및 작업에 대해 값과 업데이트를 확인하십시오.
    5. 작업 탭을 클릭하십시오.
  2. 처리 단계에서 코드 스캔을 실행할 테스트 작업을 추가하십시오.
    1. 작업 탭에서 작업 추가를 클릭하십시오.
    2. 작업 유형으로 테스트를 선택하십시오.
    3. 테스터 유형으로 IBM Vulnerability Advisor를 선택하십시오.
    4. 필요한 경우 Bluemix 대상, 조직 및 영역을 업데이트하십시오.
    5. 분석이 완료될 때까지 대기하는 시간(분)으로 값 0 - 59분을 입력하십시오. 기본값은 5분입니다. 대시보드에 대한 URL은 작업이 종료될 때 콘솔 로그에 제공됩니다.

      이 시간이 종료되기 전에 스캔이 완료되지 않으면 작업은 실패하지만, 스캔 분석은 계속 실행되며 대시보드에서 스캔 분석을 볼 수 있습니다. 스캔이 완료된 후 작업을 다시 실행하면 스캔 요청은 다시 제출되지 않으며 파이프라인 작업이 성공적으로 완료될 수 있습니다. 또는 성공적인 스캔 결과에 대해 파이프라인이 차단되지 않도록 구성할 수 있으며, 이 구성 작업은 다음 단계에서 설명합니다.

    6. 제한시간 초과 시 작동을 결정하려면 이 작업에 실패하는 경우 이 단계 실행 중지를 선택하거나 선택 취소하십시오.

      이 작업이 실패하는 경우 단계에서 나중 작업을 차단할 뿐만 아니라 나중 단계 실행을 차단하도록 작업을 구성할 수도 있고 나중 작업과 단계를 차단하지 않고 계속하도록 단계를 구성할 수도 있습니다. 작업의 상태는 주요 보안 문제가 있는지에 따라 결정됩니다.

      주된 보안 문제가 있으면 작업이 실패합니다. 이 작업에 실패하는 경우 이 단계 실행 중지에 대한 선택사항에 따라 실패한 작업은 단계와 파이프라인이 진행되지 않도록 하거나 계속 진행되도록 허용합니다. 보고서에 처리할 문제가 많이 포함되어 있음을 아는 경우 스캔에 오랜 시간이 걸릴 수 있으므로 단계가 계속되도록 구성할 수 있습니다. 이 시나리오에서 스캔에 시간이 너무 오래 걸린다는 이유만으로 나머지 작업과 단계가 중지되는 것을 원하지 않을 수도 있습니다.

  3. 빌드 작업과 동일한 단계에 작업이 있으면 저장소 이름의 환경 변수를 설정하십시오.
    1. 환경 특성 탭을 클릭하십시오.
    2. 특성 추가를 클릭하고 텍스트 특성을 선택하십시오.
    3. 이름 필드에 FULL_REPOSITORY_NAME을 입력하십시오.
    4. 값 필드에는 아무것도 입력하지 마십시오. 컨테이너 빌드 작업에 지정하는 이미지 이름은 이 환경 변수를 통해 전달됩니다.
  4. 저장을 클릭하십시오.
  5. 단계 실행 아이콘을 클릭하여 오류 없이 단계가 빌드되는지 확인하십시오.
Vulnerability Advisor 스캔 결과를 보려면 작업이 완료된 후 결과 및 히스토리 보기를 클릭하십시오. 로그 탭에서 다음 섹션을 찾으십시오. 
image registry.DomainName/namespace/image vulnerability results found

Vulnerability Advisor 주제 이미지 보고서 검토에 취약성 패키지, 보안 문제 및 제안된 개선사항이 설명되어 있습니다. 이미지의 문제점 해결에서 일부 공통적인 문제를 해결하기 위한 Dockerfile 코드가 제공됩니다.

(선택사항) Delivery Pipeline으로 이미지 정리

IBM Bluemix Container Service 빌드 작업은 사용하지 않는 이미지 버전을 자동으로 정리합니다. 사용 중인 이미지는 제거되지 않습니다. 유지할 이미지 버전 수를 지정하거나 이미지 정리를 사용 안함으로 설정할 수 있습니다.

프로젝트에 대해 유지되는 이미지 버전의 수를 변경하려면 다음을 수행하십시오.

  1. 파이프라인에서 빌드 단계에 대한 구성을 여십시오.
  2. 환경 특성 탭에서 특성 추가를 클릭하십시오.
  3. 텍스트 특성을 선택하십시오.
  4. 이름 필드에 IMAGE_LIMIT 명령을 입력하십시오.
  5. 레지스트리에 유지하기 위한 이 프로젝트의 이미지 버전 수를 값 필드에 입력하십시오. 기본적으로 프로젝트에 대해 다섯 개의 최신 이미지 버전이 유지됩니다. 이미지 정리를 사용하지 않고 모든 이미지 버전을 유지하려면 값을 -1로 설정하십시오.
  6. 저장을 클릭하십시오.

(선택사항) Delivery Pipeline을 사용하여 컨테이너에 서비스 바인드

서비스는 컨테이너에 직접 바인드할 수 없습니다. 대안으로서 앱에 서비스를 바인드한 후 파이프라인 단계에서 환경 특성을 구성하여 컨테이너에 해당 앱을 바인드할 수 있습니다. 앱은 서비스와 컨테이너 간에 브릿지 역할을 합니다.

Bluemix는 ClearDB MySQL 데이터베이스, Question and Answer 같이 앱에 바인딩되어 있는 서비스의 이름과 세부사항을 정의하는 환경 변수를 제공합니다. 이러한 환경 변수를 VCAP_SERVICES라고 합니다. 컨테이너가 서비스에 바인드된 경우, VCAP_SERVICES는 컨테이너 및 서비스 간에 브릿지 역할을 하는 앱을 통해 환경 변수로 컨테이너에 전달됩니다. 그러면 컨테이너가 서비스에서 엔드포인트, 사용자, 비밀번호 등의 정보를 조회할 수 있습니다.

환경 특성을 구성하기 전에, 이미 바인딩된 서비스가 있는 기존 앱을 식별하거나 Bluemix GUI 또는 명령행을 사용하여 앱을 작성하십시오. 자세한 정보는 Bluemix 서비스가 컨테이너에 직접 바인드되지 않음의 내용을 참조하십시오.

브릿지 앱을 컨테이너에 바인드하십시오.

  1. 도구 체인의 개요 페이지에서 Delivery Pipeline을 클릭하십시오.
  2. 파이프라인 단계를 열고 환경 특성 탭을 클릭하십시오.
  3. 특성 추가를 클릭하고 텍스트 특성을 선택하십시오.
  4. 이름 필드에 명령 BIND_TO를 입력하십시오.

팁: 브릿지 앱이 컨테이너에 바인딩된 후 브릿지 앱을 삭제하지 마십시오. 바인딩된 브릿지 앱을 삭제하면 컨테이너가 정지됩니다. 브릿지 앱을 삭제하기 전에 컨테이너 및 해당 브릿지 앱을 사용 중인 기타 모든 컨테이너를 삭제하십시오.

Delivery Pipeline으로 컨테이너의 배치 단계 작성

Delivery Pipeline을 사용하여 컨테이너의 배치 단계를 작성할 수 있습니다.

시작하기 전에:
  • 빌드 단계를 작성하십시오.
  • 단일 컨테이너에 대한 기본 배치 스크립트를 검토하십시오. 기본 스크립트를 포킹하고 이를 편집하거나 대신 사용할 자체 스크립트를 작성할 수 있습니다.
  • Delivery Pipeline 서비스에서 제공하는 사용 가능한 환경 변수의 목록을 검토하십시오.
  1. 도구 체인의 개요 페이지에서 Delivery Pipeline을 클릭하십시오.
  2. 배치 단계를 작성하십시오.
    1. 단계 추가를 클릭하십시오.
    2. 단계에 이름을 지정하십시오. 예를 들어, Deploy입니다.
    3. 입력 유형으로 빌드 아티팩트를 선택하십시오.
  3. 배치 단계에서 배치 작업을 작성하십시오.
    1. 작업 탭에서 작업 추가를 클릭하십시오.
    2. 작업 유형으로 배치를 선택하십시오.
    3. 배치자 유형으로 IBM Container Service on Bluemix를 선택하십시오.
    4. 필요하면 조직 및 영역에 대해 값과 업데이트를 확인하십시오.
    5. 배치 전략 필드에 다음 값 중 하나를 입력하십시오.
      • red_black - 앱의 새 버전을 배치합니다. 배치에 성공하면 앱의 이전 배치는 제거되고 IP 주소 또는 라우트는 새 버전에 맵핑됩니다.
      • clean - 애플리케이션의 이전 배치를 모두 제거합니다.
      팁: CONCURRENT_VERSIONS 환경 특성은 앱 배치 버전 수가 활성 상태를 유지하도록 설정합니다. 기본값은 하나의 버전입니다. 예를 들어, CONCURRENT_VERSIONS가 2로 설정되고 IP 주소를 사용 중인 경우 최신 배치 버전이 IP 주소에 바인딩되지만, 이전 버전도 빠른 대기를 위해 배치됩니다.

      환경 특성 탭에 있는 앱 배치를 항상 모두 제거하려면 CONCURRENT_VERSIONS를 0으로 설정합니다.

    6. 이름 필드에 배치할 컨테이너의 이름을 입력하십시오.
    7. 포트에 컨테이너에 대해 공개된 포트를 입력하십시오. 여러 개의 포트는 쉼표(,)로 구분해야 합니다.
      팁: 포트가 지정되지 않고 기본 배치 스크립트를 사용하는 경우 포트 80이 자동으로 공개됩니다. 기본 배치 스크립트로 컨테이너를 사설로 만들려면 다음 단계를 완료하십시오.
      1. 배치 전략에 simple을 입력하십시오.
      2. 포트 필드를 공백으로 두십시오.
      3. 단순 배치가 이전 배치를 정리하지 않으므로 이를 정리하기 위해 다른 배치 단계를 작성할 수 있습니다. 두 번째 배치 단계의 배치 전략을 정리로 설정하십시오.
    8. 선택적 배치 인수가 있는 경우 이를 입력하십시오. 예를 들어, 앱에 백엔드 데이터베이스의 위치에 대한 환경 변수가 필요한 경우 다음 행을 포함할 수 있습니다. 
      --env MY_DB_HOSTNAME_PROPERTY=mydbhostname
  4. 자체 배치 스크립트를 작성한 경우 배치자 스크립트 섹션을 편집하여 이를 사용하십시오.
    1. 다음 행을 복사하고 사용자의 저장소를 사용하도록 이를 업데이트하십시오. 
      git_retry clone https://github.com/Osthanes/deployscripts.git deployscripts
    2. 다음 행을 편집하여 사용자의 스크립트를 사용하십시오. 
      /bin/bash deployscripts/deploycontainer.sh
  5. 저장을 클릭하십시오.

Delivery Pipeline을 사용하여 컨테이너 그룹 작성

Delivery Pipeline을 사용하여 컨테이너 그룹의 빌드 및 배치를 자동화합니다.

Delivery Pipeline을 사용하여 컨테이너 그룹의 빌드 단계 작성

IBM Bluemix Container Service는 소스 코드에서 사용된 후 Docker 이미지를 빌드해서 컨테이너 레지스트리에 공개하는 빌드 서비스를 포함하고 있습니다. Delivery Pipeline은 IBM Bluemix Continuous Delivery의 일부이며, 컨테이너 빌드 서비스를 사용하는 빌드 작업을 제공합니다. 이 빌드 작업은 저장소에서 소스 코드를 가져와서 해당 코드로부터 Docker 이미지를 빌드하고 빌드 번호와 일치하도록 해당 이미지를 버전화한(태그 지정) 다음, 결과 이미지를 Bluemix에 있는 조직의 IBM Bluemix Container Service 레지스트리에 푸시합니다. 그런 다음, 조직의 구성원은 결과 이미지를 조직 내의 영역에 배치할 수 있습니다.

시작하기 전에 다음 정보를 고려하십시오.
  • 빌드, 단계 및 기본 파이프라인 기능에 대해 자세히 알아보려면 Continuous Delivery 문서에서 Delivery Pipeline 정보 주제를 참조하십시오.
  • Bluemix 카탈로그를 확인하여 조직의 개인용 이미지 레지스트리에서 컨테이너에서 사용할 기존 이미지를 식별하십시오.
  • 컨테이너에 서비스를 바인드하려는 경우에는 브릿지로 사용할 앱을 작성하십시오. 자세한 정보는 Delivery Pipeline을 사용하여 컨테이너에 서비스 바인딩을 참조하십시오.

Delivery Pipeline에서 Docker 이미지를 빌드하십시오.

  1. 도구 체인과 연관된 Git 저장소의 루트 디렉토리에서 Dockerfile을 작성하고 사용자의 저장소에 커미트하십시오.
  2. 도구 체인의 개요 페이지에서 Delivery Pipeline을 클릭하십시오.
  3. 빌드 단계를 작성하십시오.
    1. 단계 추가를 클릭하십시오.
    2. 단계에 이름을 지정하십시오. 예: Build.
    3. 입력 유형으로 Git 저장소를 선택하십시오.
    4. 필요하면 다른 필드의 값과 업데이트를 확인하십시오.
    5. 작업 탭을 클릭하십시오.
  4. 빌드 단계에서 빌드 작업을 작성하십시오.
    1. 작업 탭에서 작업 추가를 클릭하십시오.
    2. 빌드 작업 유형을 선택하십시오.
    3. 빌더 유형으로 IBM Containers on Bluemix를 선택하십시오.
    4. 필요한 경우 Bluemix 대상, 조직 및 영역을 업데이트하십시오.
    5. 캐시된 계층 사용의 경우, 다른 소스에서 컨텐츠를 동적으로 가져오는 계층의 새로 고치기를 강제 실행해야 할 때는 필드를 선택 취소하십시오.

      예를 들어, Dockerfile에 apt-get install 또는 wget 명령이 포함된 경우 캐시된 계층 사용 선택란을 선택 취소하여 각 빌드에 대해 이러한 동적 종속 항목이 수집되도록 하십시오.

      참고: 캐시된 계층 사용 필드가 선택 취소되면, 이미지를 빌드할 때 캐시된 계층을 사용하지 않으므로 빌드 시간이 더 오래 걸립니다.
    6. 이미지 이름 필드에서 빌드할 이미지의 이름을 입력하십시오. Docker 이름 지정 규칙을 충족하려면 소문자, 숫자 및 하이픈(-)만 사용하십시오. 이미지가 빌드되면 이미지 이름과 빌드 번호로 이미지에 태그가 지정됩니다. 예를 들어, 이미지 이름 myappimage를 입력하면 빌드 번호 4의 경우 결과로 얻는 이미지는 myappimage:4로 불립니다.
    7. Dockerfile이 프로젝트의 루트 디렉토리에 없으면 빌드 스크립트 섹션에서 다음 위치의 Dockerfile에 경로를 추가하십시오.
      1. 단위 테스트 실행 섹션 앞에 디렉토리 변경 명령을 사용하여 폴더 이름을 추가하십시오. 
        cd <folder>
      2. ${WORKSPACE} 변수가 표시되는 곳마다 폴더 이름을 추가하십시오. ${WORKSPACE} 변수는 프로젝트의 루트 디렉토리를 자동으로 발견합니다.
        예
        BUILD_COMMAND="build --pull --tag ${FULL_REPOSITORY_NAME} ${WORKSPACE}/<folder>"
  5. 저장을 클릭하십시오.
  6. 단계 실행 아이콘을 클릭하여 오류 없이 단계가 빌드되는지 확인하십시오. 빌드에 실패하면 로그를 검토하고 오류를 정정하십시오. 단계를 패스하는 경우 조직의 이미지 레지스트리를 확인하고 이미지가 빌드되었는지 확인하십시오.

(선택사항) 파이프라인의 일부로 이미지 보안 확인

Vulnerability Advisor를 사용하여 컨테이너를 배치하기 전에 이미지에서 보안 또는 구성 문제를 찾으십시오.
이 스캔을 포함하는 공통 파이프라인 설정은 다음과 같습니다.
  • 소스 파일을 빌드하는 빌드 단계
  • 처리 단계. 처리 단계는 다음과 같은 작업을 포함합니다.
    • 컨테이너 빌드를 실행하는 빌드 작업
    • 이미지에 대해 Vulnerability Advisor를 실행하는 테스트 작업
  • 컨테이너를 배치하는 배치 단계
팁: 이 태스크는 컨테이너 이미지를 빌드하는 작업과 다른 단계에서 Vulnerability Advisor의 테스트 작업을 작성하는 방법에 대해 개략적으로 설명합니다. 또는 동일한 단계를 사용할 수 있습니다. 별도 단계를 작성하기로 결정하는 경우에는 환경 변수를 설정하지 않아도 됩니다.

Vulnerability Advisor 스캔을 작성하십시오.

  1. 처리 단계를 작성하십시오.
    1. 단계 추가를 클릭하십시오.
    2. 단계에 이름을 지정하십시오. 예: Processing.
    3. 입력 유형으로 빌드 아티팩트를 선택하십시오.
    4. 필요하면 단계 및 작업에 대해 값과 업데이트를 확인하십시오.
    5. 작업 탭을 클릭하십시오.
  2. 처리 단계에서 코드 스캔을 실행할 테스트 작업을 추가하십시오.
    1. 작업 탭에서 작업 추가를 클릭하십시오.
    2. 작업 유형으로 테스트를 선택하십시오.
    3. 테스터 유형으로 IBM Vulnerability Advisor를 선택하십시오.
    4. 필요한 경우 Bluemix 대상, 조직 및 영역을 업데이트하십시오.
    5. 분석이 완료될 때까지 대기하는 시간(분)으로 값 0 - 59분을 입력하십시오. 기본값은 5분입니다. 대시보드에 대한 URL은 작업이 종료될 때 콘솔 로그에 제공됩니다.

      이 시간이 종료되기 전에 스캔이 완료되지 않으면 작업은 실패하지만, 스캔 분석은 계속 실행되며 대시보드에서 스캔 분석을 볼 수 있습니다. 스캔이 완료된 후 작업을 다시 실행하면 스캔 요청은 다시 제출되지 않으며 파이프라인 작업이 성공적으로 완료될 수 있습니다. 또는 성공적인 스캔 결과에 대해 파이프라인이 차단되지 않도록 구성할 수 있으며, 이 구성 작업은 다음 단계에서 설명합니다.

    6. 제한시간 초과 시 작동을 결정하려면 이 작업에 실패하는 경우 이 단계 실행 중지를 선택하거나 선택 취소하십시오.

      이 작업이 실패하는 경우 단계에서 나중 작업을 차단할 뿐만 아니라 나중 단계 실행을 차단하도록 작업을 구성할 수도 있고 나중 작업과 단계를 차단하지 않고 계속하도록 단계를 구성할 수도 있습니다. 작업의 상태는 주요 보안 문제가 있는지에 따라 결정됩니다.

      주된 보안 문제가 있으면 작업이 실패합니다. 이 작업에 실패하는 경우 이 단계 실행 중지에 대한 선택사항에 따라 실패한 작업은 단계와 파이프라인이 진행되지 않도록 하거나 계속 진행되도록 허용합니다. 보고서에 처리할 문제가 많이 포함되어 있음을 아는 경우 스캔에 오랜 시간이 걸릴 수 있으므로 단계가 계속되도록 구성할 수 있습니다. 이 시나리오에서 스캔에 시간이 너무 오래 걸린다는 이유만으로 나머지 작업과 단계가 중지되는 것을 원하지 않을 수도 있습니다.

  3. 빌드 작업과 동일한 단계에 작업이 있으면 저장소 이름의 환경 변수를 설정하십시오.
    1. 환경 특성 탭을 클릭하십시오.
    2. 특성 추가를 클릭하고 텍스트 특성을 선택하십시오.
    3. 이름 필드에 FULL_REPOSITORY_NAME을 입력하십시오.
    4. 값 필드에는 아무것도 입력하지 마십시오. 컨테이너 빌드 작업에 지정하는 이미지 이름은 이 환경 변수를 통해 전달됩니다.
  4. 저장을 클릭하십시오.
  5. 단계 실행 아이콘을 클릭하여 오류 없이 단계가 빌드되는지 확인하십시오.
Vulnerability Advisor 스캔 결과를 보려면 작업이 완료된 후 결과 및 히스토리 보기를 클릭하십시오. 로그 탭에서 다음 섹션을 찾으십시오. 
image registry.DomainName/namespace/image vulnerability results found

Vulnerability Advisor 주제 이미지 보고서 검토에 취약성 패키지, 보안 문제 및 제안된 개선사항이 설명되어 있습니다. 이미지의 문제점 해결에서 일부 공통적인 문제를 해결하기 위한 Dockerfile 코드가 제공됩니다.

(선택사항) Delivery Pipeline으로 이미지 정리

IBM Bluemix Container Service 빌드 작업은 사용하지 않는 이미지 버전을 자동으로 정리합니다. 사용 중인 이미지는 제거되지 않습니다. 유지할 이미지 버전 수를 지정하거나 이미지 정리를 사용 안함으로 설정할 수 있습니다.

프로젝트에 대해 유지되는 이미지 버전의 수를 변경하려면 다음을 수행하십시오.

  1. 파이프라인에서 빌드 단계에 대한 구성을 여십시오.
  2. 환경 특성 탭에서 특성 추가를 클릭하십시오.
  3. 텍스트 특성을 선택하십시오.
  4. 이름 필드에 IMAGE_LIMIT 명령을 입력하십시오.
  5. 레지스트리에 유지하기 위한 이 프로젝트의 이미지 버전 수를 값 필드에 입력하십시오. 기본적으로 프로젝트에 대해 다섯 개의 최신 이미지 버전이 유지됩니다. 이미지 정리를 사용하지 않고 모든 이미지 버전을 유지하려면 값을 -1로 설정하십시오.
  6. 저장을 클릭하십시오.

(선택사항) Delivery Pipeline을 사용하여 컨테이너에 서비스 바인드

서비스는 컨테이너에 직접 바인드할 수 없습니다. 대안으로서 앱에 서비스를 바인드한 후 파이프라인 단계에서 환경 특성을 구성하여 컨테이너에 해당 앱을 바인드할 수 있습니다. 앱은 서비스와 컨테이너 간에 브릿지 역할을 합니다.

Bluemix는 ClearDB MySQL 데이터베이스, Question and Answer 같이 앱에 바인딩되어 있는 서비스의 이름과 세부사항을 정의하는 환경 변수를 제공합니다. 이러한 환경 변수를 VCAP_SERVICES라고 합니다. 컨테이너가 서비스에 바인드된 경우, VCAP_SERVICES는 컨테이너 및 서비스 간에 브릿지 역할을 하는 앱을 통해 환경 변수로 컨테이너에 전달됩니다. 그러면 컨테이너가 서비스에서 엔드포인트, 사용자, 비밀번호 등의 정보를 조회할 수 있습니다.

환경 특성을 구성하기 전에, 이미 바인딩된 서비스가 있는 기존 앱을 식별하거나 Bluemix GUI 또는 명령행을 사용하여 앱을 작성하십시오. 자세한 정보는 Bluemix 서비스가 컨테이너에 직접 바인드되지 않음의 내용을 참조하십시오.

브릿지 앱을 컨테이너에 바인드하십시오.

  1. 도구 체인의 개요 페이지에서 Delivery Pipeline을 클릭하십시오.
  2. 파이프라인 단계를 열고 환경 특성 탭을 클릭하십시오.
  3. 특성 추가를 클릭하고 텍스트 특성을 선택하십시오.
  4. 이름 필드에 명령 BIND_TO를 입력하십시오.
  5. 값 필드에 브릿지 앱의 이름(예: pipeline_bridge_app)을 입력하십시오.
  6. 저장을 클릭하십시오.
  7. 파이프라인을 실행하여 구성을 테스트하십시오.
팁: 브릿지 앱이 컨테이너에 바인딩된 후 브릿지 앱을 삭제하지 마십시오. 바인딩된 브릿지 앱을 삭제하면 컨테이너가 정지됩니다. 브릿지 앱을 삭제하기 전에 컨테이너 및 해당 브릿지 앱을 사용 중인 기타 모든 컨테이너를 삭제하십시오.

Delivery Pipeline을 사용하여 컨테이너 그룹의 배치 단계 작성

Delivery Pipeline을 사용하여 컨테이너 그룹의 배치 단계를 작성할 수 있습니다.

시작하기 전에:
  • 빌드 단계를 작성하십시오.
  • 컨테이너 그룹에 대한 기본 배치 스크립트를 검토하십시오. 기본 스크립트를 포킹하고 이를 편집하거나 대신 사용할 자체 스크립트를 작성할 수 있습니다.
  • Delivery Pipeline 서비스에서 제공하는 사용 가능한 환경 변수의 목록을 검토하십시오.
  1. 배치 단계를 작성하십시오.
    1. 단계 추가를 클릭하십시오.
    2. 단계에 이름을 지정하십시오. 예를 들어, Deploy입니다.
    3. 입력 유형으로 빌드 아티팩트를 선택하십시오.
  2. 단계에서 배치 작업을 작성하십시오.
    1. 작업 탭에서 작업 추가를 클릭하십시오.
    2. 작업 유형으로 배치를 선택하십시오.
    3. 배치자 유형으로 IBM Container Service on Bluemix를 선택하십시오.
    4. 조직 및 영역을 선택하십시오.
    5. 배치 전략에 다음 값 중 하나를 입력하십시오.
      • red_black - 앱의 새 버전을 배치합니다. 배치에 성공하면 앱의 이전 배치는 제거되고 IP 주소 또는 라우트는 새 버전에 맵핑됩니다.
      • clean - 애플리케이션의 이전 배치를 모두 제거합니다.
    6. 이름에 배치할 컨테이너의 이름을 입력하십시오.
    7. 포트에 컨테이너에 대해 공개된 포트를 입력하십시오. 컨테이너 그룹의 경우 하나의 포트만 지정할 수 있습니다.
      팁: 포트가 지정되지 않고 기본 배치 스크립트를 사용하는 경우 포트 80이 자동으로 공개됩니다. 기본 배치 스크립트로 컨테이너를 사설로 만들려면 다음 단계를 완료하십시오.
      1. 배치 전략에 simple을 입력하십시오.
      2. 포트 필드를 공백으로 두십시오.
      3. 단순 배치가 이전 배치를 정리하지 않으므로 이를 정리하기 위해 다른 배치 단계를 작성할 수 있습니다. 두 번째 배치 단계의 배치 전략을 정리로 설정하십시오.
    8. 선택적 배치 인수가 있는 경우 이를 입력하십시오. 예를 들어, 앱에 백엔드 데이터베이스의 위치에 대한 환경 변수가 필요한 경우 다음 행을 포함할 수 있습니다. 
      --env MY_DB_HOSTNAME_PROPERTY=mydbhostname
  3. 단일 컨테이너 배치가 아닌 컨테이너 그룹 배치를 설정하십시오. "배치자 스크립트" 섹션에서 #을 제거하여 컨테이너 그룹 스크립트의 주석 처리를 해제하십시오. 
    /bin/bash deployscripts/deploygroup.sh
    #을 추가하여 단일 컨테이너 스크립트를 주석 처리하십시오. 
    #/bin/bash deployscripts/deploycontainer.sh
  4. 자체 배치 스크립트를 작성한 경우 배치자 스크립트 섹션을 편집하여 이를 사용하십시오.
    1. 다음 행을 복사하고 사용자의 저장소를 사용하도록 이를 업데이트하십시오. 
      git_retry clone https://github.com/Osthanes/deployscripts.git deployscripts
    2. 다음 행을 편집하여 사용자의 스크립트를 사용하십시오. 
      /bin/bash deployscripts/deploygroup.sh
  5. 옵션: 기본 컨테이너 그룹을 구성하려면 표에 있는 환경 변수 각각을 설정하십시오. 이러한 특성은 명령행 또는 Bluemix 사용자 인터페이스에서 컨테이너를 배치할 때 설정할 수 있는 일부 옵션을 설정할 수 있게 합니다. 기타 특성에 대해서는 사용자 인터페이스의 배치자 스크립트 섹션에 있는 주석을 참조하십시오.
    1. 환경 특성 탭을 클릭하고 특성 추가를 클릭하십시오.
    2. 텍스트 특성을 선택하십시오.
    3. 환경 특성의 이름과 값을 입력하십시오.
      특성 설명
      AUTO_RECOVERY true 또는 false

      컨테이너 그룹에 자동 복구를 사용하고 그룹이 10분 동안 실행된 경우, 그룹의 로드 밸런서가 HTTP 요청을 통해 그룹에 있는 각 컨테이너 인스턴스의 상태를 정기적으로 확인하기 시작합니다. 컨테이너 인스턴스가 100초 내에 응답하지 않으면 비활성으로 표시됩니다. 비활성 컨테이너 인스턴스는 그룹에서 제거되고 자동 복구를 통해 다시 작성됩니다. 자동 복구는 그룹에 있는 컨테이너 인스턴스의 복구를 3번 시도합니다. 자동 복구는 3번의 시도 후에 그룹에 있는 컨테이너 인스턴스를 60분 동안 복구하지 않습니다. 60분 후에 자동 복구 프로세스가 다시 시작됩니다.
      CONCURRENT_VERSIONS 숫자

      CONCURRENT_VERSIONS 환경 특성은 앱 배치 버전 수가 활성 상태를 유지하도록 설정합니다. 기본값은 하나의 버전입니다. 예를 들어, CONCURRENT_VERSIONS가 2로 설정되고 컨테이너 그룹을 배치하는 경우 두 개의 그룹이 라우트와 수신 트래픽에 맵핑됩니다.

      앱의 모든 배치를 항상 제거하려면 ENVIRONMENT PROPERTIES 탭에서 CONCURRENT_VERSIONS를 0로 설정할 수 있습니다.
      DESIRED_INSTANCES 숫자

      예제: 4

      앱에 필요한 컨테이너 그룹 인스턴스 수는 앱에서 요청을 처리할 수 있는 시간 및 예상 워크로드에 따라 달라집니다. 예를 들어, 멀티태스킹 또는 네트워크 연결은 앱에 필요한 인스턴스 수에 영향을 미칩니다.

      컨테이너 그룹에 고가용성을 부여하고 실패에 더 탄력적이게 하려면 예상된 워크로드를 다루기 위한 최소 수보다 많은 추가 인스턴스를 포함하는 것을 고려하십시오. 추가 인스턴스는 다른 인스턴스가 충돌하는 경우 워크로드를 처리할 수 있습니다. 워크로드에 영향을 주지 않고 실패한 인스턴스의 자동 복구를 보장하려면 하나의 추가 인스턴스를 포함하십시오. 두 개의 동시 실패로부터 보호하려면 두 개의 추가 인스턴스를 포함하십시오. 이 설정은 N+2 패턴이며 여기서 N은 요청을 다루기 위한 인스턴스의 수이고 +2는 두 개의 추가 인스턴스입니다.

      ROUTE_DOMAIN 문자열 또는 none

      예제: mycustomdomain.com

      기본 시스템 도메인은 AppDomainName이며 이미 SSL 인증서를 제공하므로, 추가 구성 없이 HTTPS로 컨테이너 그룹에 액세스할 수 있습니다. 사용자 정의 도메인을 사용하려면 공용 DNS 서버에서 사용자 정의 도메인을 등록하고, Bluemix에서 사용자 정의 도메인을 구성한 후에 사용자 정의 도메인을 공용 DNS 서버의 Bluemix 시스템 도메인에 맵핑해야 합니다.

      도메인이 지정되지 않으면 AppDomainName이 사용됩니다. none을 지정하면 도메인이 작성되지 않습니다.

      호스트 이름과 도메인에 none을 지정하면 동일한 영역에서 다른 컨테이너가 그룹에 액세스할 수 있지만 공용 액세스는 불가능합니다.

      ROUTE_HOSTNAME 문자열 또는 none

      예제: myapp

      호스트 이름은 mycontainerhost와 같습니다. 호스트 이름에는 밑줄(_)을 포함하지 마십시오. 호스트와 도메인이 결합되어 전체 공용 라우트 URL(예: http://mycontainerhost.AppDomainName)을 구성합니다.

      호스트 이름을 지정하지 않으면 사용자를 위해 호스트 이름이 작성됩니다. none을 지정하면 호스트 이름이 작성되지 않습니다.

      호스트 이름과 도메인에 none을 지정하면 동일한 영역에서 다른 컨테이너가 그룹에 액세스할 수 있지만 공용 액세스는 불가능합니다.

  6. 저장을 클릭하십시오.

튜토리얼: 컨테이너에 Cloud Foundry Node.js 앱 배치

이전에 Cloud Foundry를 사용하여 배치한 앱을 선택해서 IBM Bluemix Container Service를 사용하여 컨테이너에 배치할 수 있습니다.

목표

  • 컨테이너 배치 일반 프로세스 학습
  • 이미지를 빌드하기 위해 Dockerfile에 포함되어야 하는 항목 학습

소요 시간

30분

대상

Cloud Foundry 앱 개발자

전제조건

  • IBM Bluemix Container Service CLI 설치
  • IBM Bluemix Container Service CLI에 로그인

학습 1: Cloud Foundry Node.js 앱 코드 다운로드

Node.js 코드를 사용할 수 있게 준비하십시오. 아직 코드가 없습니까? Bluemix에서 이 튜토리얼에서 사용할 스타터 코드를 다운로드할 수 있습니다.

시작하기 전에 IBM Bluemix Container Service CLI를 설치하고 로그인하십시오.

다음과 같이 앱 코드를 가져와 구성하십시오.

  1. cf-pi-myinitials 디렉토리를 작성하고 이 디렉토리로 이동하십시오. 이 디렉토리에 Docker 이미지를 빌드하고 Node.js 앱을 실행하는 데 필요한 모든 파일을 저장합니다. 디렉토리에는 모든 이름을 사용할 수 있습니다. 예제의 이름을 사용할 경우 myinitials를 사용자의 이니셜(예: cf-pi-ms)로 바꾸십시오. 
    mkdir cf-pi-myinitials && cd cf-pi-myinitials
  2. Node.js 앱 코드와 모든 관련 파일을 디렉토리에 복사하십시오. 사용자 자체 Node.js 앱 코드를 사용하거나 Bluemix에서 Personal Insights 앱을 다운로드할 수 있습니다.

    Bluemix에서 Personal Insights 앱을 다운로드하려면 다음 단계를 수행하십시오.

    1. Bluemix 카탈로그의 표준 유형에서 Personality Insights Node.js 웹 스타터를 클릭하십시오. 이 표준 유형에는 Node.js 앱이 포함되어 있습니다. 이 앱에서는 IBM Watson™ Personality Insights 서비스를 사용하여 사용자가 블로그, 트윗 또는 포럼 게시물을 통해 생성하는 텍스트에서 소셜 특성과 코그너티브의 스펙트럼을 추출합니다.
    2. 앱 이름 cf-pi-myinitials를 입력하고 작성을 클릭하십시오. 표준 유형의 앱 코드에 액세스하려면 먼저 CF 앱을 Bluemix에 배치해야 합니다. 앱에는 모든 이름을 사용할 수 있습니다. 예제의 이름을 사용할 경우 이름이 고유하도록 myinitials를 사용자의 이니셜(예: cf-pi-ms)로 바꾸십시오.

      이 배치에서는 cf-pi-myinitials라는 앱과 cf-pi-myinitials-personality_insights라는 IBM Watson Personality Insights 서비스 인스턴스를 작성합니다.

      앱 배치 시 명령행 인터페이스를 사용하여 앱 배치와 관련된 지시사항이 표시됩니다.

    3. 1단계에서 스타터 코드 다운로드를 클릭하십시오.
    4. .zip 파일의 압축을 풀고 앱 파일을 디렉토리에 저장하십시오.
Cloud Foundry 앱 코드를 컨테이너화할 준비가 되었습니다.

학습 2: CF 앱 코드로 Docker 이미지 작성

컨테이너에 필요한 구성과 Node.js 앱 코드가 포함된 Dockerfile을 작성합니다. 그런 다음 해당 Dockerfile에서 Docker 이미지를 빌드하고 이를 개인용 Bluemix 레지스트리에 푸시합니다.

앱 코드를 사용하여 Docker 이미지를 작성하려면 다음 단계를 수행하십시오.

  1. 컨테이너 작성의 기초가 되는 Dockerfile을 작성하십시오. 컴퓨터에 있는 텍스트 편집기 또는 선호하는 CLI 편집기를 사용하여 Dockerfile을 작성할 수 있습니다. 다음 예에서는 nano 편집기를 사용하여 Dockerfile을 작성하는 방법을 보여줍니다. 
    nano Dockerfile
  2. 다음 스크립트를 Dockerfile에 복사하십시오. 
    #Use the IBM Node image as a base image
FROM registry.DomainName/ibmnode:latest

#Expose the port for your Personal Insights app, and set 
#it as an environment variable as expected by cf apps
ENV PORT=3000
EXPOSE 3000
ENV NODE_ENV production

#Copy all app files from the current directory into the app 
#directory in your container. Set the app directory
#as the working directory
ADD . /app
WORKDIR /app

#Install any necessary requirements from package.json
RUN npm install

#Sleep before the app starts. This command ensures that the
#IBM Containers networking is finished before the app starts. 
CMD (sleep 60; npm start)
#Start the Personal Insight app. 
CMD ["node", "/app/app.js"]
  3. Ctrl+O를 눌러 변경사항을 저장하십시오. ENTER를 눌러 변경사항을 확인하십시오. Ctrl+X를 눌러 nano 편집기를 종료하십시오.
  4. 앱 코드를 사용하여 Docker 이미지를 빌드한 후 이를 개인용 Bluemix 레지스트리에 푸시하십시오. 
    bx ic build -t registry.DomainName/namespace/cf-pi-myinitials .
    이 명령의 컴포넌트 이해
    build build 명령입니다.
    registry.DomainName/namespace/cf-pi-myinitials 개인용 Bluemix 레지스트리로 사용자 고유 네임스페이스와 이미지의 이름을 포함합니다. 이 예의 경우 CF 노드 앱 cf-pi-myinitials와 동일한 이름이 이미지에 사용되지만 개인용 레지스트리에서 이미지의 이름을 선택할 수 있습니다. 네임스페이스를 확실히 알 수 없는 경우 다음 명령을 실행하여 찾으십시오. 
    bx ic namespace-get
    . Dockerfile의 위치입니다. Dockerfile이 포함된 디렉토리에서 빌드 명령을 실행할 경우 마침표(.)를 입력하십시오. . 그렇지 않으면 Dockerfile의 상대 경로를 사용하십시오.
레지스트리에 이미지가 작성됩니다. bx ic images 명령을 실행하여 이미지가 작성되었는지 확인할 수 있습니다. 
$ bx ic images
REPOSITORY                                   TAG     IMAGE ID      CREATED         SIZE
registry.DomainName/namespace/cf-pi-myinitials  latest  b02b7a24c35f  48 seconds ago  251.3 MB

학습 3: 이미지를 사용하여 컨테이너 배치

앱을 Bluemix에 컨테이너 그룹으로 배치합니다.

컨테이너 그룹에 앱을 배치하려면 다음 단계를 수행하십시오.

  1. 다음 명령을 실행하여 이전 학습에서 작성한 이미지를 사용하여 컨테이너 그룹을 작성하십시오. 
    bx ic group-create --name cf-pi-myinitials -m 64 -n pi-app-myinitials -d AppDomainName -p 3000 -e CCS_BIND_SRV=cf-pi-myinitials-personal_insights --desired 3 --auto --anti registry.DomainName/namespace/cf-pi-myinitials:latest
    이 명령의 컴포넌트 이해
    group-create group-create 명령입니다.
    --name cf-pi-myinitials Bluemix GUI에서 표시할 컨테이너의 이름입니다.
    -m 64 그룹의 각 인스턴스에 64MB의 메모리가 지정됩니다.
    -n pi-app-myinitials 라우트 URL의 호스트 이름입니다.
    팁: 스타터 코드를 사용 중인 경우 이전 학습에서 사용한 이름과 다른 이름을 호스트 이름으로 사용하여 Cloud Foundry 앱을 배치하십시오. 동일한 호스트가 사용되고 CF 앱이 계속 실행 중이면 컨테이너 그룹과 앱 간에 로드 밸런싱이 발생합니다.
    -d AppDomainName 라우트 URL의 도메인 이름입니다. 사용자 정의 도메인이 있는 경우가 아니면 AppDomainName을 사용하십시오.
    -p 3000 노출할 포트 번호입니다. 라우트 URL에 대한 연결은 컨테이너의 포트 3000으로 전달됩니다.
    -e CCS_BIND_SRV=cf-pi-myinitials-personal_insights Personal Insights 앱에서 입력한 텍스트에서 코그너티브 특성과 소셜 특성을 분석할 수 있도록 IBM Watson Personality Insights 서비스 인스턴스를 컨테이너에 바인드하십시오. 서비스 인스턴스는 학습 1의 CF 앱 배치 과정에서 작성되었습니다.
    서비스 인스턴스의 이름이 불확실한 경우 다음 명령을 실행하십시오. 
    bx service list
    --desired 3 컨테이너 그룹에 작성할 단일 컨테이너 인스턴스의 수입니다. 모든 인스턴스는 동일합니다.
    --auto 컨테이너 그룹의 자동 복구를 사용합니다. 그룹의 인스턴스 중 하나를 사용할 수 없게 되는 경우 응답하지 않는 인스턴스를 대체하는 다른 인스턴스가 자동으로 작성됩니다.
    --anti 컨테이너 그룹의 연관관계 방지를 사용합니다. 그룹의 각 인스턴스는 개별 실제 컴퓨팅 노드에 배치되므로 하드웨어 장애 발생으로 인해 그룹에 있는 모든 컨테이너가 충돌할 가능성이 줄어듭니다.
    registry.DomainName/namespace/cf-pi-myinitials:latest 대상 저장소 경로이며, 사용자의 고유 네임스페이스 및 대상 이미지의 이름이 포함됩니다. 이 예의 경우 CF 노드 앱 cf-pi-myinitials와 동일한 이름이 이미지에 사용됩니다.
  2. 컨테이너가 배치되고 네트워크가 설정되도록 잠시 대기하십시오. 컨테이너가 배치되면 다음 URL을 입력하여 브라우저에서 Personal Insights 앱을 볼 수 있습니다. http://pi-app-myinitials.AppDomainName/.
  3. CF 앱을 제거하십시오. 이제 C 앱 코드가 컨테이너에 배치되었으므로 CF 앱을 제거하여 할당량을 최대로 사용할 수 있습니다. 
    cf delete cf-pi-myinitials
    y를 입력하여 삭제를 확인하십시오.
앱이 컨테이너에 배치되었습니다.

bx ic attach [--no-stdin] [--sig-proxy] CONTAINER

용도
실행 중인 컨테이너를 제어하거나 해당 출력을 표시합니다. 컨테이너를 종료하고 중지하려면 CTRL+C를 사용하십시오. 자세한 정보는 Docker 도움말에서 attach 명령을 참조하십시오.
매개변수
--no-stdin
(선택사항) 표준 입력이 포함되지 않습니다.
--sig-proxy
(선택사항) 기본값은 true입니다.
CONTAINER
(필수) 컨테이너 이름 또는 ID입니다.
예
다음은 my_container 컨테이너에 연결하도록 요청하는 예입니다. 
bx ic attach my_container

bx ic build -t TAG [--no-cache] [--pull] [-q] DOCKERFILE_LOCATION

용도
Docker 이미지를 로컬로 또는 개인용 Bluemix 이미지 레지스트리에서 빌드합니다. 자세한 정보는 Docker 도움말에서 build 명령을 참조하십시오.
매개변수
DOCKERFILE_LOCATION
(필수) 로컬 호스트의 컨텍스트 및 Dockerfile에 대한 경로입니다.
--no-cache
(선택사항) 이미지가 처음으로 빌드되면 이미지의 계층이 전체 조직에 대해 캐시됩니다. 이미지의 후속 버전을 빌드할 때 빌드 프로세스의 속도를 높이는 데 이러한 캐시된 버전의 이미지가 사용됩니다. 이미지의 새 버전을 빌드할 때 --no-cache 옵션을 포함시킴으로써 이미지의 각 계층도 다시 빌드되었음을 확인할 수 있습니다.
--pull
(선택사항) 기본적으로 기본 이미지가 조직의 이미지 캐시에 아직 없는 경우에만 레지스트리에서 가져옵니다. 이 옵션을 포함시킴으로써 새 이미지를 빌드하기 위해 기본 이미지의 최신 버전이 사용되었음을 확인할 수 있습니다.
-q, --quiet
(선택사항) 컨테이너에서 생성하는 상세 출력을 억제합니다. 기본값은 false입니다.
-t TAG, --tag TAG
(필수) 작성된 이미지에 적용할 개인용 이미지 레지스트리의 경로입니다.
예
다음은 이름이 myimage인 이미지의 빌드를 요청하는 예입니다. 빌드에 사용할 Dockerfile 및 기타 아티팩트는 명령이 실행되는 디렉토리와 동일한 디렉토리에 있습니다. 레지스트리와 네임스페이스가 이미지 이름과 함께 포함되므로 이미지는 조직의 개인용 Bluemix 이미지 레지스트리에서 빌드됩니다.
팁: bx ic namespace-get을 실행하여 네임스페이스를 검색하고 <my_namespace>를 자신의 네임스페이스 정보로 대체하십시오. 
bx ic build -t registry.DomainName/<my_namespace>/myimage.

bx ic cp SRC_PATH CONTAINER:DEST_PATH | bx ic cp CONTAINER:SRC_PATH DEST_PATH

용도
실행 중이거나 중지된 단일 컨테이너에서 로컬 시스템으로 또는 반대로 파일 또는 폴더를 복사하십시오. 자세한 정보는 Docker 도움말에서 cp 명령을 참조하십시오.
참고: 이 명령은 단일 컨테이너와만 사용하도록 지원됩니다. 컨테이너 그룹에/에서 파일을 복사하려는 경우 그룹의 각 컨테이너 인스턴스에/에서 파일을 복사해야 합니다.
매개변수
DEST_PATH
(필수) 소스 경로에서 정보를 복사하려는 로컬 시스템 또는 컨테이너의 파일 또는 폴더 경로입니다. 복사를 시작하려면 파일 또는 폴더가 로컬 시스템 또는 컨테이너에 있어야 합니다. 로컬 시스템에 복사하려는 경우 절대 또는 상대 경로를 사용할 수 있습니다. 단일 컨테이너에 복사하려는 경우 컨테이너의 이름, 구분 기호 ":" 및 대상 파일 또는 폴더의 경로를 포함해야 합니다. 경로는 컨테이너의 루트 디렉토리에 상대적이어야 합니다.
SRC_PATH
(필수) 복사하려는 파일 또는 폴더의 경로입니다. 파일 또는 폴더가 로컬 시스템에 있는 경우 절대 또는 상대 경로를 사용할 수 있습니다. 파일 또는 폴더가 단일 컨테이너에 있는 경우 컨테이너의 이름, 구분 기호 ":" 및 복사하려는 파일 또는 폴더의 경로를 포함해야 합니다. 경로는 컨테이너의 루트 디렉토리에 상대적이어야 합니다.
참고: 컨테이너의 /dev, /etc, /proc 및 /sys 폴더에 저장된 특정한 시스템 파일을 복사할 수 없습니다.
예
이 예제 명령은 이름이 config.txt인 파일을 로컬 시스템의 현재 디렉토리에서 이름이 my_container인 단일 컨테이너의 config 디렉토리로 복사합니다.
bx ic cp ./config.txt my_container:/config

bx ic cpi SRC DST

용도
로컬 레지스트리의 이미지 또는 Docker Hub 이미지에 액세스하여 이미지를 개인용 Bluemix 이미지 레지스트리에 복사합니다.
참고: 이 명령은 Bluemix 이미지 레지스트리에 이미지를 추가하기 위해 docker pull과 docker push를 함께 사용하는 방법의 대체입니다. docker pull 및 docker push 사용을 선호하는 경우 명령행에서 개인용 Bluemix 레지스트리에 로컬 이미지 푸시의 내용을 참조하십시오.
매개변수
DST
(필수) 개인용 Bluemix 이미지 레지스트리 URL이며 여기에는 네임스페이스와 대상 이미지 이름이 포함됩니다. 이미지에 대한 태그는 선택사항입니다.
SRC
(필수) 소스 저장소 및 이미지 이름입니다.
예
bx ic cpi source_repository/source_image_name private_registry_URL/destination_image_name:tag
팁: bx ic namespace-get을 실행하여 네임스페이스를 검색하고 <my_namespace>를 자신의 네임스페이스 정보로 대체하십시오. 
bx ic cpi training/sinatra registry.DomainName/<my_namespace>/mysinatra:v1

bx ic exec [-d] [-it ] [-u USER] CONTAINER CMD

용도
컨테이너 내에서 명령을 실행합니다. 자세한 정보는 Docker 도움말에서 exec 명령을 참조하십시오.
매개변수
CMD
(필수) 지정된 컨테이너에서 실행할 명령입니다.
CONTAINER
(필수) 컨테이너 이름 또는 ID입니다.
-d , --detach
(선택사항) 지정된 명령을 백그라운드에서 실행합니다.
-it
(선택사항) 대화식 모드입니다. 표준 입력 표시를 유지합니다. 종료하려면 exit를 입력하십시오.
-u USER, --user USER
(선택사항) 사용자 이름입니다.
예
이 예제 명령은 컨테이너 my_container에서 대화식 bash 터미널을 실행합니다. 
bx ic exec -it my_container bash
이 예제 명령은 컨테이너 my_container에서 date 명령을 실행합니다. 
bx ic exec my_container date

bx ic group-create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session-affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

용도
스케일링 그룹을 작성합니다. 기본적으로 Bluemix의 컨테이너는 분리 모드로 실행됩니다.
매개변수
--anti
(선택사항) 연관관계 방지를 사용하면 컨테이너 그룹의 가용성을 더욱 높입니다. --anti 옵션을 사용하면 그룹의 모든 컨테이너 인스턴스를 각각의 실제 컴퓨팅 노드에 두므로, 하드웨어 실패로 인해 그룹에 있는 모든 컨테이너가 충돌할 가능성을 줄입니다. 각 Bluemix 지역 및 조직에는 배치에 사용할 수 있는 컴퓨팅 노드 세트가 제한되어 있으므로, 보다 큰 그룹 크기에서는 이 옵션을 사용하지 못할 수 있습니다. 배치에 성공하지 못하면 그룹의 컨테이너 인스턴스 수를 줄이거나 --anti 옵션을 제거하십시오.
--auto
(선택사항) 컨테이너 그룹에 자동 복구를 사용하고 그룹이 10분 동안 실행된 경우, 그룹의 로드 밸런서가 HTTP 요청을 통해 그룹에 있는 각 컨테이너 인스턴스의 상태를 정기적으로 확인하기 시작합니다. 컨테이너 인스턴스가 100초 내에 응답하지 않으면 비활성으로 표시됩니다. 비활성 컨테이너 인스턴스가 그룹에서 제거되고 자동 복구를 통해 다시 작성됩니다. 자동 복구는 그룹에 있는 컨테이너 인스턴스의 복구를 3번 시도합니다. 자동 복구는 3번의 시도 후에 그룹에 있는 컨테이너 인스턴스를 60분 동안 복구하지 않습니다. 60분 후에 자동 복구 프로세스가 다시 시작됩니다.상태 검사에 대한 자세한 정보는 --http_monitor_enabled 옵션을 참조하십시오.
CMD
(선택사항) 실행할 컨테이너 그룹으로 명령 및 인수가 전달됩니다. 이 명령은 오래 실행되는 명령이어야 합니다. 오래 실행되지 않는 명령(예: /bin/date)은 컨테이너에 충돌을 일으킬 수 있으므로 사용하지 마십시오.
다음은 IBM Bluemix Container Service에서 제공하는 ibmnode 이미지를 사용하는 오래 실행되는 명령의 예입니다.
bx ic group-create --name my_container_group registry.DomainName/ibmnode ping localhost
bx ic group-create --name my_container_group registry.DomainName/ibmnode -- tail -f /dev/null
참고: CMD 및 해당 인수는 bx ic group-create 명령행의 끝에 있어야 합니다.
-d DOMAIN, --domain DOMAIN
(선택사항) 기본 시스템 도메인은 AppDomainName이며 이미 SSL 인증서를 제공해서 추가적인 구성 없이 HTTPS로 컨테이너 그룹에 액세스할 수 있습니다. 사용자 정의 도메인을 사용하려면 공용 DNS 서버에서 사용자 정의 도메인을 등록하고, Bluemix에서 사용자 정의 도메인을 구성한 후에 사용자 정의 도메인을 공용 DNS 서버의 Bluemix 시스템 도메인에 맵핑해야 합니다. 사용자 정의 도메인이 Bluemix 시스템 도메인에 맵핑된 후에 사용자 정의 도메인에 대한 요청이 Bluemix에 있는 사용자의 애플리케이션으로 라우팅됩니다. 사용자 정의 도메인을 작성할 때는 도메인 이름에 밑줄(_)을 포함하지 마십시오. 사용자 정의 도메인에 대한 자세한 정보는 사용자 정의 도메인 작성 및 사용을 참조하십시오. 사용자 정의 도메인의 보안을 유지하려면 컨테이너 그룹에 HTTPS로 액세스할 수 있도록 SSL 인증서를 업로드하십시오.

호스트와 도메인이 결합되어 전체 공용 라우트 URL(예: http://mycontainerhost.AppDomainName)을 구성하며 Bluemix 내에서 고유해야 합니다. bx ic group-inspect 명령을 사용하여 컨테이너 그룹의 세부사항을 검토할 때 호스트 및 도메인이 함께 라우트로 나열됩니다.

중요사항: 컨테이너 그룹에는 최소 두 개의 컨테이너 인스턴스가 있어야 합니다. 그룹에 인스턴스가 하나뿐이며 해당 인스턴스를 사용할 수 없게 되면 앱의 가동이 중단될 수 있습니다.

(선택사항) 인터넷에서 앱에 액세스할 수 있게 하려면 공용 포트를 노출시켜야 합니다. 공용 포트를 노출할 때 노출된 포트에서만 공개 데이터를 전송 및 수신할 수 있게 하는 컨테이너에 대한 공용 네트워크 보안 그룹을 작성합니다. 기타 공용 포트는 닫히고 인터넷에서 앱에 액세스하는 데 사용할 수 없습니다. 여러 번 -p 옵션을 사용하여 여러 포트를 포함할 수 있습니다. 포트는 맵핑되거나 전달될 수 없습니다.포트를 노출시키지 않으면, 컨테이너는 사설 컨테이너 네트워크에서만 액세스 가능합니다. 지정된 사설 IP 주소를 사용하여 사설 네트워크에 있는 컨테이너와 통신할 수 있습니다.

사용 중인 이미지의 Dockerfile에 포트가 지정된 경우 해당 포트를 포함하십시오.
팁:
  • IBM 인증 Liberty Server 이미지 또는 이 이미지의 수정된 버전에 대해서는 포트 9080을 입력하십시오.
  • IBM 인증 Node.js 이미지 또는 이 이미지의 수정된 버전에 대해서는 임의의 포트(예: 8000)를 입력하십시오.

사용하려는 IP 프로토콜을 표시하기 위해 UDP와 TCP 사이에서 선택할 수 있습니다. 프로토콜을 지정하지 않으면 포트가 자동으로 TCP 트래픽에 대해 노출됩니다.

예

  • -p 9080/udp
  • -p 9080/udp -o 8000/tcp

컨테이너에 바인드할 기존 IP 주소가 영역에 있는 경우 포트 정보를 사용하여 IP 주소를 지정할 수 있습니다. IP 주소, 포트 또는 IP 주소 및 컨테이너 포트 번호(예: <ip-address>:<port_on_IP>:<container-port>)를 포함하십시오. IP 주소의 포트는 Dockerfile에서 노출된 컨테이너 포트와 일치해야 합니다.

(선택사항) VOLUME:/DIRECTORY_PATH[:ro] 형식으로 세부사항을 지정하여 컨테이너에 볼륨을 연결합니다.

  • DIRECTORY_PATH: 컨테이너에 있는 디렉토리의 절대 경로입니다.
  • ro: 선택사항입니다. ro를 지정하면 볼륨을 기본 읽기/쓰기 대신 읽기 전용으로 만듭니다.

HAS_INACTIVE_INSTANCES 컨테이너 그룹의 하나 이상의 인스턴스가 유실되었거나 아직 네트워크 연결이 없고 INACTIVE 상태를 표시합니다. 수신 데이터가 ACTIVE로 표시된 남은 인스턴스로 라우팅됩니다. 컨테이너 그룹을 작성하는 경우 응답이 없는 인스턴스를 새 인스턴스로 자동으로 대체하고 HAS_INACTIVE_INSTANCES 상태를 해결하려면, --auto 옵션을 사용하여 자동 복구 모드를 사용으로 설정하십시오. 자동 복구를 사용하지 않는 경우 이 상태를 해결하려면 컨테이너 그룹을 다시 작성해야 합니다. 또한 컨테이너 그룹을 작성할 때 앱에서 포트를 노출해야 하면 -p 옵션을 사용하여 해당 포트 번호를 포함하도록 하십시오.
NO_ACTIVE_INSTANCES 활성 인스턴스가 남아 있지 않은 경우에는 컨테이너 그룹을 사용할 수 없게 되고 그룹 로드 밸런서의 사설 IP 주소 또는 라우트를 사용하여 액세스할 수 없습니다. 이 상태가 지속되는 경우, 앱이 그룹으로 배치될 때 네트워킹 구성이 가능하지 않은 것일 수 있습니다. 앱을 단일 컨테이너로 대신 배치해 보십시오.
UPDATE_
COMPLETE
 업데이트가 성공적으로 완료되었습니다. 그룹 인스턴스가 추가 또는 제거되면 할당 사용량이 자동으로 업데이트됩니다.
UPDATE_
FAILED
 OpenStack의 문제로 인해 컨테이너 그룹을 업데이트할 수 없습니다. 몇 분 대기한 후 다시 시도하십시오. 이 상태가 지속되면 Bluemix 지원 티켓을 여십시오.
UPDATE_IN_
PROGRESS
 이 상태는 bx ic group-update 명령을 사용하여 컨테이너 그룹의 업데이트가 요청된 경우 또는 응답이 없는 그룹 인스턴스를 복구하기 위해 자동 복구가 사용 설정된 경우에 표시됩니다. 컨테이너 그룹의 업데이트는 완료하는 데 몇 분이 걸릴 수 있습니다. 성공적으로 완료되면 UPDATE_COMPLETE 상태가 표시됩니다. 업데이트에 실패하면 UPDATE_FAILED가 표시됩니다.

(선택사항) 컨테이너 그룹에 자동 복구를 사용하고 그룹이 10분 동안 실행된 경우, 그룹의 로드 밸런서가 HTTP 요청을 통해 그룹에 있는 각 컨테이너 인스턴스의 상태를 정기적으로 확인하기 시작합니다. 컨테이너 인스턴스가 100초 내에 응답하지 않으면 비활성으로 표시됩니다. 비활성 컨테이너 인스턴스가 그룹에서 제거되고 자동 복구를 통해 다시 작성됩니다. 자동 복구는 그룹에 있는 컨테이너 인스턴스의 복구를 3번 시도합니다. 자동 복구는 3번의 시도 후에 그룹에 있는 컨테이너 인스턴스를 60분 동안 복구하지 않습니다. 60분 후에 자동 복구 프로세스가 다시 시작됩니다.상태 검사에 대한 자세한 정보는 --http_monitor_enabled 옵션을 참조하십시오.

호스트와 도메인이 결합되어 전체 공용 라우트 URL(예: http://mycontainerhost.AppDomainName)을 구성하며 Bluemix 내에서 고유해야 합니다. cf ic group inspect 명령을 사용하여 컨테이너 그룹의 세부사항을 검토하는 경우 호스트 및 도메인이 라우트로 함께 나열됩니다.

--desired DESIRED
(선택사항) 그룹의 인스턴스 수입니다. 기본값은 2입니다. 앱에 필요한 컨테이너 그룹 인스턴스 수는 앱에서 요청을 처리할 수 있는 시간 및 예상 워크로드에 따라 달라집니다. 예를 들어, 멀티태스킹 또는 네트워크 연결은 앱에 필요한 인스턴스 수에 영향을 미칩니다. 원하는 인스턴스 수를 판별하려면 앱을 실행하는 Bluemix에 단일 컨테이너를 배치하고 이 컨테이너에서 로드 테스트를 수행하십시오. 예를 들어, 앱이 초당 100개의 요청을 처리할 수 있지만 사용자가 초당 300개 요청의 평균 워크로드를 예상하는 경우에 컨테이너 그룹에 대한 적정 인스턴스 수는 3개입니다. 컨테이너 그룹에 고가용성을 부여하고 실패에 더 탄력적이게 하려면 예상된 워크로드를 다루기 위한 최소 수보다 많은 추가 인스턴스를 포함하는 것을 고려하십시오. 추가 인스턴스는 다른 인스턴스가 충돌하는 경우 워크로드를 처리할 수 있습니다. 워크로드에 영향을 주지 않고 실패한 인스턴스의 자동 복구를 보장하려면 하나의 추가 인스턴스를 포함하십시오. 두 개의 동시 실패로부터 보호하려면 두 개의 추가 인스턴스를 포함하십시오. 이 설정은 N+2 패턴이며 여기서 N은 요청을 다루기 위한 인스턴스의 수이고 +2는 두 개의 추가 인스턴스입니다.
중요사항: 컨테이너 그룹에는 최소 두 개의 컨테이너 인스턴스가 있어야 합니다. 그룹에 인스턴스가 하나뿐이며 해당 인스턴스를 사용할 수 없게 되면 앱의 가동이 중단될 수 있습니다.

(선택사항) 기본값은 true입니다. 컨테이너 그룹에 자동 복구를 사용하고 그룹이 10분 동안 실행된 경우, 그룹의 로드 밸런서가 HTTP 요청을 통해 그룹에 있는 각 컨테이너 인스턴스의 상태를 정기적으로 확인하기 시작합니다. 컨테이너 인스턴스가 100초 내에 응답하지 않으면 비활성으로 표시됩니다. 비활성 컨테이너 인스턴스가 그룹에서 제거되고 자동 복구를 통해 다시 작성됩니다. 자동 복구는 그룹에 있는 컨테이너 인스턴스의 복구를 3번 시도합니다. 자동 복구는 3번의 시도 후에 그룹에 있는 컨테이너 인스턴스를 60분 동안 복구하지 않습니다. 60분 후에 자동 복구 프로세스가 다시 시작됩니다. 일부 앱은 HTTP 대신 통신을 위해 TCP를 사용하는 앱과 같이 HTTP 상태 검사를 요구하지 않을 수도 있습니다.   • Small(1024MB 메모리, 64GB 디스크 공간)
  • Medium(2048MB 메모리, 128GB 디스크 공간)

컨테이너의 크기를 설정하지 않으면 각 컨테이너 인스턴스가 256MB로 작성됩니다.
중요사항: 단위 레이블 없이 컨테이너 메모리(MB)를 입력하십시오. 예를 들어, Pico 컨테이너를 작성하려는 경우에는 -m 64를 입력하십시오.

팁: 컨테이너 이름은 문자로 시작해야 하며 대문자, 소문자, 숫자, 마침표(.), 밑줄(_) 또는 하이픈(-)을 포함할 수 있습니다.

(선택사항) 인터넷에서 앱에 액세스할 수 있게 하려면 공용 포트를 노출시켜야 합니다. 공용 포트를 노출할 때 노출된 포트에서만 공개 데이터를 전송 및 수신할 수 있게 하는 컨테이너에 대한 공용 네트워크 보안 그룹을 작성합니다. 기타 공용 포트는 닫히고 인터넷에서 앱에 액세스하는 데 사용할 수 없습니다. 여러 번 -p 옵션을 사용하여 여러 포트를 포함할 수 있습니다. 포트는 맵핑되거나 전달될 수 없습니다. 포트를 노출시키지 않으면, 컨테이너는 사설 컨테이너 네트워크에서만 액세스 가능합니다. 지정된 사설 IP 주소를 사용하여 사설 네트워크에 있는 컨테이너와 통신할 수 있습니다.

팁:
  • IBM 인증 Liberty Server 이미지 또는 이 이미지의 수정된 버전에 대해서는 포트 9080을 입력하십시오.
  • IBM 인증 Node.js 이미지 또는 이 이미지의 수정된 버전에 대해서는 임의의 포트(예: 8000)를 입력하십시오.

사용하려는 IP 프로토콜을 표시하기 위해 UDP와 TCP 사이에서 선택할 수 있습니다. 프로토콜을 지정하지 않으면 포트가 자동으로 TCP 트래픽에 대해 노출됩니다.

  • -p 9080/udp

(선택사항) VOLUME:/DIRECTORY_PATH[:ro] 형식으로 세부사항을 지정하여 컨테이너에 볼륨을 연결합니다.

  • ro: 선택사항입니다. ro를 지정하면 볼륨을 기본 읽기/쓰기 대신 읽기 전용으로 만듭니다.

단일 컨테이너 또는 컨테이너 그룹이 비임시적 상태에 도달할 때까지 대기합니다. 이 대기 시간 중에는 명령행이 리턴되지 않으며 사용자가 명령을 입력할 수 없습니다. 컨테이너가 비임시적 상태에 도달하는 순간, 확인 메시지가 표시됩니다. 단일 컨테이너의 경우, 비임시적 상태에는 실행 중, 종료, 충돌, 일시정지 또는 일시중단 등이 포함됩니다. 컨테이너 그룹의 경우, 비임시적 상태에는 CREATE_COMPLETE, UPDATE_COMPLETE 또는 FAILED가 포함됩니다.
매개변수
CONTAINER
(필수) 컨테이너 또는 그룹 이름 또는 ID입니다.
예
다음은 이름이 my_container인 컨테이너가 비임시적 상태에 도달할 때까지 대기하도록 요청하는 예입니다. 
cf ic wait-status my_container

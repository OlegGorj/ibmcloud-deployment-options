컨테이너 디버깅 및 로그 보기

컨테이너를 디버그해야 하는 옵션을 보고 실패의 근본 원인을 찾습니다.

  1. 컨테이너를 나열하고 현재 상태를 검토하십시오.
    • 단일 컨테이너
      bx ic ps -a
    • 컨테이너 그룹
      bx ic groups
  2. 옵션: 각 상태가 의미하는 내용을 확인하고 컨테이너가 실패하지 않게 하는 팁을 검토하십시오.
  3. 컨테이너가 실패하거나 충돌되면 컨테이너를 검사하여 실패의 근본 원인을 찾으십시오. 오류 메시지는 응답의 실패 섹션에서 볼 수 있습니다.
    • 단일 컨테이너
      bx ic inspect CONTAINER
    • 컨테이너 그룹
      1. 컨테이너 그룹을 검사하십시오. 
        bx ic group-inspect GROUP
      2. 추가 정보를 보려면 컨테이너 그룹의 인스턴스를 나열하고 각 인스턴스를 검사하십시오. 
        bx ic group-instances GROUP
        bx ic inspect GROUP_INSTANCE
  4. 앱이 실패로 종료된 경우 앱의 로그를 검색하십시오.
    참고: 이 명령은 앱이 표준 STDOUT 및 STDERR 출력 스트림에 로그를 작성한 경우에만 사용할 수 있습니다. 이러한 표준 출력 스트림에 작성하면 컨테이너가 종료되거나 충돌된 경우에도 컨테이너 로그를 볼 수 있습니다.
    • 단일 컨테이너
      bx ic logs CONTAINER
    • 컨테이너 그룹
      1. 컨테이너 그룹의 컨테이너 인스턴스를 나열하십시오. 
        bx ic group-instances GROUP
      2. 각 인스턴스에 대해 로그를 검색하십시오. 
        bx ic logs GROUP_INSTANCE
  5. 자동 복구로 재작성된 컨테이너 그룹의 인스턴스와 같이 더 이상 사용 가능하지 않은 컨테이너의 로그에 액세스해야 하는 경우에는 Kibana를 사용하여 로그를 보십시오.

    https://logmet.DomainName/

    로그는 Kibana에서 최대 7일 동안 사용할 수 있습니다.

  6. 로그가 컨테이너 내부의 파일로 저장된 경우 컨테이너에 로그인하여 검토해야 합니다.
    중요사항: 실행 중인 컨테이너에만 로그인할 수 있습니다. 컨테이너가 충돌되고 다시 시작할 수 없는 경우 컨테이너의 로그에 액세스할 수 없을 수도 있습니다. 표준 STDOUT 및 STDERR 출력 스트림에 쓰도록 앱을 변경하여 컨테이너가 더 이상 액세스 가능하지 않더라도 로그 파일에 액세스할 수 있게 하십시오.
    1. 컨테이너가 실행 중이 아니면 컨테이너를 시작하십시오. 
      bx ic start CONTAINER
    2. 실행 중인 컨테이너에 로그인하십시오. 
      bx ic exec -it CONTAINER bash
    3. 로그 파일로 이동하여 로그를 검토하십시오.
  7. 로그가 볼륨에 저장되었고 컨테이너를 더 이상 시작할 수 없으면 새 컨테이너를 작성하여 볼륨에 액세스하십시오.
    1. 로그 파일이 저장된 볼륨에 마운트된 새 컨테이너를 작성하십시오. 
      bx ic run --name logcontainer -m 64 --volume MOUNT_PATH_TO_VOLUME registry.DomainName/ibmliberty
    2. 실행 중인 컨테이너에 로그인하십시오. 
      bx ic exec -it logcontainer bash
    3. 볼륨 마운트 경로로 이동하고 로그 파일을 검토하십시오.
  8. 컨테이너 디버깅에 대해 추가 지원이 필요하면 Bluemix지원 티켓을 여십시오.

CLI 설정 문제점 해결

init 중에 메소드 오류(rpc)

bx ic init을 실행하려고 시도하는 경우 다음 메시지 중 하나가 표시됩니다.
  • bx ic rpc: can't find method CliRpcCmd.IsLoggedIn
  • bx ic rpc: can't find method CliRpcCmd.IsMinCliVersion
설치된 Cloud Foundry 버전이 IBM Bluemix Container Service에서 지원하는 것보다 이전 버전입니다.
  1. cf -v를 실행하여 설치된 Cloud Foundry CLI 버전을 확인하십시오.
  2. 설치된 버전이 6.14.0 미만이면 Cloud Foundry CLI를 업데이트하십시오.

bx ic init 중에 레지스트리 인증 오류 발생

bx ic init을 실행하는 경우 다음 인증 오류 메시지 중 하나가 표시됩니다.

****exit status 1
****time="2015-07-31T08:58:32-04:00" level=fatal msg="Post http:///var/run/docker.sock/v1.18/auth: dial

bx ic 전제조건 중 하나가 충족되지 않았거나 IBM Bluemix Container Service를 사용할 수 없습니다.
  1. Docker 디먼이 실행 중인지 확인하십시오. 실행되고 있지 않은 경우에는 해당 디먼을 시작한 후 bx ic init을 다시 실행하십시오.
  2. bx login을 실행하여 Cloud Foundry CLI에 로그인되어 있는지 확인하십시오. 그런 다음 bx ic init을 실행하여 플러그인에 다시 로그인하십시오.
  3. IBM Bluemix Container Service의 가용성을 확인하십시오. Bluemix 상태 페이지에서 정보를 확인하십시오.

bx ic 명령에 Sudo가 필요함

bx ic 명령을 실행할 때는 모든 명령에서 sudo 접두부를 사용해야 합니다.
기본적으로, bx ic 명령이 루트 인증으로 실행되어야 하기 때문에 운영 체제에 따라 sudo 접두부가 필요할 수 있습니다.
sudo 없이 docker 및 bx ic 명령을 실행하려면, 다음 명령을 실행한 후에 (Linux인 경우에는) 로그아웃 이후 다시 로그인해야 합니다. 
sudo usermod -a -G docker <CURRENT_USER>

할당량 문제 해결

조직의 할당량이 충분하지 않으므로 컨테이너를 작성할 수 없음

컨테이너를 작성할 때 사용자 영역에 할당량이 충분하지 않다는 메시지가 표시됩니다.
BXNUI0110E: IBM Containers 리소스를 할당할 수 없습니다. 조직 할당량이 충분하지 않아서 새 영역에 할당할 수 없습니다. 조직 'organization' 유동 IP를 초과할 수 없습니다.
IBM Bluemix 조직 관리를 사용하여 할당량 할당을 조정하십시오.
이 메시지는 할당량이 충분히 지정되지 않은 영역에서 컨테이너 작성을 시도할 때 표시될 수 있습니다. 기본적으로, 컨테이너 서비스 관련 작업을 위해 해당 영역에 할당되려면 모든 영역에서 2개의 IP 주소와 2GB의 메모리가 필요합니다.
여러 영역에 걸쳐 메모리와 IP 할당량을 분배하려면 다음을 수행하십시오.
  1. 계정 세부사항의 조직 관리에서 조직을 선택하십시오.
  2. 할당량 섹션에서 컨테이너의 세부사항을 보십시오.
  3. 편집을 클릭하고 조직의 영역들 간에 메모리와 IP 주소를 할당하십시오.
  4. 조직의 임의 영역에서 컨테이너를 작성하십시오.

지불 계정이 있는데도 할당량을 편집할 수 없음

컨테이너 할당량을 수정할 수 없습니다.
조직의 관리자와 대금 청구 정보가 일치하지 않습니다.
로그인한 사용자가 조직의 관리자여야 합니다. 소유자를 검증하려면 다음 단계를 완료하십시오.
  1. 계정 세부사항의 조직 관리에서 조직을 선택하십시오.
  2. 사용자 목록에서 조직의 관리자와 로그인한 ID가 동일한지 확인하십시오.
  3. 계정에서 로그아웃한 후 다시 로그인하십시오.
이 단계를 완료하고도 컨테이너 할당량을 편집할 수 없으면 IBM 지원 센터에 문의하십시오.

IBM WebSphere Application Server Liberty 이미지의 문제점

IBM Bluemix Container Service에 있는 IBM WebSphere® Application Server Liberty 이미지 중 하나를 사용할 때 문제점이 발생합니다.

이미지가 올바르게 설정되지 않았을 수 있습니다.

특히 이 Dockerfile 이미지와 관련된 문제가 발생할 경우 GitHub 문제 트래커를 사용하십시오. IBM WebSphere Application Server Liberty에 관련한 더 일반적인 문제의 경우 WASdev 커뮤니티로 이동하여 도움을 받을 수 있습니다.

IBM Bluemix Container Service의 IBM WebSphere Application Server Liberty 이미지에 대한 자세한 정보는 IBM WebSphere Application Server Liberty 이미지를 참조하십시오.

개인용 Bluemix 이미지 레지스트리에서 이미지를 가져올 수 없음

개인용 Bluemix 레지스트리의 컨테이너 이미지에서 컨테이너를 작성했지만 컨테이너가 계속 빌드 중 상태입니다. bx ic inspect 명령을 실행하면 CLI 응답의 Env 섹션에 다음 오류 메시지가 표시됩니다. 
"Env": [
            "metadata_exit_code=-1",
            "logging_password=",
            "space_id=0e65c436-9413-46e2-8b42-ff06aa74ac21",
            "logstash_target=logmet.opvis.bluemix.net:9091",
            "metadata_err_msg=Cannot pull image from registry: Pull failed with error message failed 
             to register layer: ApplyLayer exit status 1 stdout:  stderr: Container ID 1234567890 
             cannot be mapped to a host ID",
            "metrics_target=logmet.opvis.bluemix.net:9095",
            "metadata_running=1"
IBM Bluemix Container Service에서는 Docker 사용자 네임스페이스 기능이 Docker Engine에서 사용으로 설정됩니다. 사용자 네임스페이스 기능은 처음에 Docker 버전 1.10에서 도입되었으며 컨테이너의 루트 사용자를 컨테이너 외부의 루트가 아닌 사용자에 맵핑할 수 있는 가능성을 제공합니다. 이 방법으로 컨테이너 내부에서 루트 조치를 수행할 수 있는 사용자의 권한은 컨테이너 외부에서 제한될 수 있으므로 호스트와 다른 컨테이너에 있는 파일은 액세스, 변경 또는 제거할 수 없습니다.
참고: Docker 사용자 네임스페이스는 개인용 이미지 레지스트리를 식별하도록 Bluemix에서 설정한 네임스페이스와는 다릅니다.

bx ic run 명령을 사용하여 컨테이너 이미지에서 컨테이너를 작성하는 경우 개인용 Bluemix 이미지 레지스트리에서 IBM Cloud Docker Engine으로 이미지를 로드합니다. 그러면 Docker Engine에서 각 컨테이너 계층을 작성하기 시작합니다. 컨테이너 이미지에 지정된 내용에 따라 로컬 시스템의 앱 파일을 컨테이너에 추가할 수도 있습니다. 이 모든 파일은 로컬 시스템에서 사용자 ID(UID)와 그룹 ID(GID)를 사용하여 작성합니다. UID와 GID는 로컬 시스템에서의 사용자 권한을 표시합니다. 이 파일을 컨테이너에 추가하려는 경우 각 UID와 GID가 컨테이너 내부의 UID와 GID에 맵핑합니다. 컨테이너의 맵핑된 UID와 GID의 권한은 로컬 시스템과 다를 수 있습니다. 예를 들어, 로컬 시스템의 루트가 아닌 UID와 GID가 컨테이너의 루트 사용자에 맵핑됩니다. 그러면 사용자가 컨테이너에서 루트 사용자의 역할을 수행하고 컨테이너 파일에 액세스, 변경 및 제거할 수 있습니다. 컨테이너가 격리된 환경에서 실행 중이므로, 컨테이너 외부에서는 루트가 아닌 권한이 지정되므로 UID와 GID가 호스트의 파일에 액세스할 수 없습니다.

로컬 시스템의 UID와 GID를 컨테이너 내부의 UID와 GID에 맵핑하려면 사용자 네임스페이스 기능은 0 - 65536의 UID와 GID로 모든 파일을 작성해야 합니다.

표시되는 오류는 이 범위를 벗어나는 UID 또는 GID에서 파일을 작성하고 이 파일이 컨테이너 이미지 계층 중 하나에 포함될 때 발생합니다. 이미지에서 컨테이너를 작성할 때 Docker Engine은 이미지의 각 계층을 빌드합니다.범위에서 벗어난 UID 또는 GID의 파일이 발견되면 UID 또는 GID가 컨테이너 내부에서 유효한 ID에 맵핑될 수 없으므로 Docker Engine이 실패합니다. Docker Engine은 또한 하나의 이미지 계층에 사용되고 다음 이미지 계층에서 제거되면 실패합니다.

로컬 Docker Engine에서 사용자 네임스페이스를 사용
로컬에서 컨테이너 이미지를 빌드할 때 로컬 Docker Engine에서 사용자 네임스페이스 기능을 사용하게 하십시오. 그러면 유효한 사용자 네임스페이스 범위 밖에 있는 로컬 시스템의 모든 UID 및 GID가 자동으로 범위 내에 있는 유효한 ID에 맵핑됩니다. 개인용 Bluemix 이미지 레지스트리에 컨테이너 이미지를 푸시하기 전에 이미지를 로컬로 테스트하고 IBM Bluemix Container Service에서 사용하십시오. 사용자 네임스페이스 기능을 사용하는 방법에 대한 자세한 정보는 사용자 네임스페이스를 사용으로 설정하여 디먼 시작을 참조하십시오.
영향을 받은 이미지 계층을 주석 처리하고 영향 받은 파일의 소유권을 변경
문제를 일으키는 파일을 찾을 수 없으면 Dockerfile의 이미지 계층을 주석 처리하고 계층을 차례로 주석 해제해 보십시오. 이미지를 빌드하고 컨테이너를 작성하여 영향 받은 이미지 계층을 찾으십시오.

chown [-R] root:root <file_or_folder> 를 실행하여 로컬 머신에 저장된 영향 받은 파일의 소유권을 변경하거나 이 명령을 Dockerfile의 영향 받은 이미지 계층에 추가하십시오.

예제 Dockerfile

RUN curl -fsSL "https://www.kernel.org/pub/software/scm/git/git-${GIT_VERSION}.tar.gz" \
        | tar -xzC /tmp \
  && cd /tmp/git-${GIT_VERSION} \
  && ./configure \
  && make all \
  && make install \
  && chown -R root:root /tmp

개인용 Bluemix 이미지 레지스트리에서 이미지를 내보낼 수 없음

IBM Bluemix Container Service에 로그인하고 bx ic pull 명령을 사용하여 개인용 Bluemix 레지스트리에서 로컬 시스템으로 이미지를 가져오려고 합니다. bx ic pull이 지원되지 않는다는 메시지와 함께 명령이 실패합니다.
개인용 Bluemix 레지스트리에서 로컬 시스템으로 이미지를 가져오는 데는 bx ic pull 명령이 지원되지 않습니다.
개인용 Bluemix 레지스트리에서 이미지를 내보내려면 다음 단계를 따르십시오.
  1. IBM Bluemix Container Service에 로그인하십시오. 이미 로그인한 경우 IBM Bluemix Container Service에 대해 기본 Docker 명령을 사용하도록 환경 변수가 설정되지 않게 하십시오.
  2. 이미지를 나열하십시오. 
    bx ic
images
  3. 개인용 Bluemix 레지스트리에서 로컬 시스템으로 이미지를 가져오십시오. 이전 단계에서 리턴된 이미지의 이름 또는 ID를 사용하십시오. 
    docker pull <image_name_or_id>
  4. 로컬 시스템에 이미지를 가져왔는지 확인하십시오. 
    docker images
  5. 이미지를 .tar 파일에 저장하십시오. 
    docker export <image_name_or_id> > <exported_image_name>.tar

작성 후에는 조직의 네임스페이스를 변경할 수 없습니다.
다른 조직을 작성하고 이 조직에 대한 네임스페이스를 원하는 이름으로 설정하십시오.

docker push 중에 로그인 프롬프트가 표시됨

개인용 Bluemix 이미지 레지스트리에 이미지를 추가하기 위해 docker push를 실행할 때 로그인 신임 정보를 입력하도록 프롬프트가 표시됩니다. Docker 또는 Bluemix 신임 정보를 입력하면 인증에 실패합니다.
다음과 같은 이유로 인해 인증에 실패할 수 있습니다.
  • Cloud Foundry 인증 정보가 포함된 운반자 토큰이 만료됨.
  • Bluemix 권한 부여 서비스를 사용할 수 없음.
  • IBM Bluemix Container Service 플러그인을 사용하는 경우 다음을 수행하십시오.
    1. 다음 명령을 실행해 Cloud Foundry CLI에 로그인하여 컴퓨터의 운반자 토큰을 새로 고치십시오.
      bx login [-a api.DomainName] [-sso]
      싱글 사인온 매개변수 -sso는 연합 ID로 로그인할 때 필요합니다. 이 옵션이 사용되면 https://login.ng.bluemix.net/UAALoginServerWAR/passcode를 열고 일회성 패스코드를 얻으십시오. --sso 옵션을 포함하지 않는 경우에는 다음의 하위 단계를 완료하십시오.
      1. 이메일에 대해 Bluemix에 로그인하는 데 사용하는 IBMId를 입력하십시오.
      2. 비밀번호에 대해 Bluemix에 로그인하는 데 사용하는 IBMId의 비밀번호를 입력하십시오. 사용자 이름 및 비밀번호를 입력하면 사용자의 Bluemix 조직 및 영역이 검색됩니다.
      3. Bluemix 조직 중 하나를 표시하는 번호를 입력하십시오.
      4. 기존 Bluemix 영역 중 하나를 표시하는 번호를 입력하십시오.
    2. 운반자 토큰을 다시 작성하려면 IBM Bluemix Container Service CLI를 초기화하십시오. 
      bx ic init

다시 로그인한 후에도 프롬프트가 표시되는 경우에는 현재 Bluemix 권한 부여 서비스를 사용할 수 없는 상태입니다. 나중에 다시 시도하십시오.

컨테이너 문제점 해결

앱에 액세스할 때 502 Bad Gateway 오류 발생

컨테이너 그룹을 정상적으로 배치했지만, 앱에 대해 URL을 열 때 다음 오류가 표시됩니다. 
502 Bad Gateway: Registered endpoint failed to handle the request.
  1. 라우팅이 아직 설정 중일 수 있습니다.
  2. 컨테이너가 응답하지 않을 수 있습니다.
  3. 컨테이너 그룹이 정상적으로 실행 중이지만 라우팅에 오류가 있을 수 있습니다.
  1. 라우팅이 완료될 충분한 시간이 허용되도록 추가로 수 분을 대기합니다. 컨테이너 그룹이 지정된 라우트에 연결할 수 있기 전에 컨테이너 프로세스 실행 이후 5분 동안 대기할 수 있습니다.
  2. 5분 동안 대기했으며 오류 발생이 의심되는 경우, IP 주소를 실행 중인 컨테이너에 바인딩하여 컨테이너가 응답 중인지 확인한 후에 해당 IP 주소 및 포트에 대해 curl 명령을 실행하십시오.
    1. 다음 명령을 실행하여 그룹에서 컨테이너 인스턴스 중 하나의 ID를 가져오십시오. 컨테이너 ID 및 포트를 기록해 놓으십시오. 
      bx ic ps
    2. 다음 명령을 실행하여 IP 주소를 요청하십시오. 
      bx ic ip-request
    3. 다음 명령을 실행하여 IP 주소를 컨테이너 인스턴스에 바인드하십시오. 
      bx ic ip-bind IP_address container_ID
    4. 브라우저에서 IP 주소 및 포트로 애플리케이션에 대한 URL을 입력하십시오. 예: http://IP_address:port

      애플리케이션에 액세스할 수 없으면 컨테이너 그룹에 문제점이 있으며 그룹을 재작성해야 합니다. 애플리케이션에 액세스할 수 있으면 컨테이너가 정상적으로 실행 중이지만 라우팅에 오류가 있습니다. IBM Bluemix 지원 센터에 문의하십시오.

IBM Bluemix Container Service 자동 복구 모드에서 시작 중에 컨테이너 그룹 인스턴스를 반복해서 다시 작성

컨테이너 그룹을 작성하고 자동 복구를 사용으로 설정했습니다. 컨테이너 그룹의 시작 동안 응답이 없는 그룹 인스턴스가 제거되고 반복해서 다시 작성됩니다.

컨테이너 그룹에서 실행 중인 앱이 노출된 포트 청취를 시작하는 데 시간이 너무 오래 걸립니다. 컨테이너 그룹에 자동 복구를 사용으로 설정한 경우 IBM Bluemix Container Service가 그룹 작성 후 노출된 TCP 포트에서 HTTP 요청으로 각 인스턴스의 상태를 검사합니다. 인스턴스가 3분 이내에 노출된 포트에서 청취하지 않을 경우 인스턴스를 응답 없음으로 표시한 후 제거하고 다시 작성하여 다시 시작합니다. 최대 다시 작성된 인스턴스 수에 도달할 때까지 이 프로세스를 반복합니다. 그런 다음 자동 복구를 사용 안함으로 설정합니다.

자동 복구를 사용하는 컨테이너 그룹에서 앱을 실행하려면 2분 이내에 앱이 시작되고 노출된 포트에서 청취하는지 확인하십시오. 앱의 유형에 따라 다른 방법으로 앱을 빨리 시작하게 할 수 있습니다.

Bluemix 서비스가 컨테이너에 직접 바인드되지 않음

Bluemix 서비스가 서비스 바인딩을 사용하여 컨테이너에 직접 바인드하지 않습니다.
일부 Bluemix 서비스는 컨테이너에 대한 직접적 바인딩을 지원하지 않으며 컨테이너 작성 동안 실패를 일으킵니다. 이 서비스의 경우 서비스 바인딩은 환경 변수 VCAP_SERVICES를 컨테이너에 전달하지 않습니다. 이 문제점을 임시로 해결하려면 CLI를 사용하여 서비스에서 컨테이너로 VCAP_SERVICES를 전송하는 데 필요한 브릿지 역할을 수행하는 Cloud Foundry 앱을 작성하십시오.
  1. 컨테이너를 배치할 영역(예: dev)을 선택하십시오.
  2. 기존 앱을 식별하거나 Bluemix 서비스가 바인드될 Cloud Foundry 앱을 작성하십시오. 이 앱이 VCAP_SERVICES 정보의 전달에 사용되므로, 이를 브릿지 앱이라고 합니다.
    Cloud Foundry 앱을 작성하려면 다음을 수행하십시오.
    팁: 이 단계를 완료하여 작성하는 앱이 VCAP_SERVICES 서비스를 저장하는 플레이스홀더입니다. 실행 중인 앱보다 비용을 줄이기 위해 이 앱을 시작하거나 실행하지 않습니다. 컨테이너에 VCAP_SERVICES를 제공하기 위해 이 앱을 실행할 필요가 없습니다.
    1. 작업 중인 앱 디렉토리에서 디렉토리와 텍스트 파일을 작성하십시오. 예: <directory>/<empty_file.txt>. 이 파일은 앱의 소스 파일로 사용되지만, 컨텐츠를 포함할 필요는 없습니다.
      • Windows 예: C:\Users\User\containerbridge\empty_file.txt
      • Linux 예: /user_home_directory/containerbridge/empty_file.txt
      • OS X 예: /Users/user_home_directory/containerbridge/empty_file.txt
    2. 아무런 비용 없이 제로 런타임 앱을 작성하려면 다음 명령을 실행하십시오. 이 앱에는 어떤 이름이든 지정할 수 있지만 이 앱은 컨테이너로부터 서비스로의 브릿지 역할을 수행하므로 일반적으로 사용되는 이름은 containerbridge입니다. 

      cf push <appname> -p <directory> -i 1 -d AppDomainName -k 1M -m 64M --no-hostname --no-manifest --no-route --no-start
      예: 
      cf push containerbridge -p containerbridge -i 1 -d AppDomainName -k 1M -m 64M --no-hostname --no-manifest --no-route --no-start
  3. 바인드할 Bluemix 서비스가 아직 추가되지 않은 경우에는 해당 서비스를 Bluemix 영역에 추가하십시오.
    1. 사용 가능한 Bluemix 서비스의 목록을 가져오려면 cf marketplace 명령을 실행하십시오. 결과 목록에는 Bluemix 서비스의 이름, 해당 플랜 유형 및 서비스에 대한 설명이 포함되어 있습니다.
    2. create-service 명령을 실행하고 cf marketplace 목록의 서비스 이름과 서비스 플랜을 포함하십시오. 또한 서비스에 대한 사용자 자체 인스턴스 이름을 포함하십시오.
      cf create-service <service> <plan> <service_instance>
      예
      cf create-service cleardb spark my-clear-db
  4. bind-service 명령을 실행하여 Bluemix 서비스를 앱에 바인드하십시오. 반복하여 기타 Bluemix 서비스를 브릿지 앱에 바인드하십시오. 
    cf bind-service <appname> <service_instance>
    예
    cf bind-service containerbridge my-clear-db
  5. 앱을 다시 스테이징하도록 프롬프트되면 다음 명령을 실행하십시오.
    cf restage containerbridge
  6. 다음 명령을 실행하여 containerbridge 앱을 통해 전달되는 Bluemix 서비스에 대한 정보를 검토하십시오. 
    cf env <appname>
    예
    cf env containerbridge

    VCAP_SERVICES 환경 변수를 사용하여 삽입된 Bluemix 서비스 매개변수가 결과에 포함됩니다.

    예:
    {
 "VCAP_SERVICES": {
  "<service_name>": [
   {
    "credentials": {
     "hostname": "<hostname_id>",
     "jdbcUrl": "<jdbc_URL>",
     "name": "<encoded_name>",
     "password": "<encoded_password>",
     "port": "<port_number>",
     "uri": "<service_URI>",
     "username": "<encoded_username>"
    },
    "label": "<service_name>",
    "name": "<service_instance>",
    "plan": "<plan_name>",
    "tags": [
     "<tag1>",
     "<tag2>",
     "<tag3>"
    ]
   }
Bluemix 서비스 및 containerbridge 앱 간에 바인딩을 작성한 후에 컨테이너를 작성하고 이에 containerbridge 앱을 바인드하십시오. 자세한 정보는 단일 컨테이너 작성 또는 컨테이너 그룹 작성의 환경 변수에 있는 CCS_BIND_APP 플래그를 참조하십시오.
팁: 브릿지 앱이 컨테이너에 바인딩된 후 브릿지 앱을 삭제하지 마십시오. 바인딩된 브릿지 앱을 삭제하면 컨테이너가 정지됩니다. 더 이상 브릿지 앱이 필요하지 않으면 브릿지 앱을 삭제하기 전에 컨테이너와 브릿지 앱을 사용하는 다른 컨테이너를 삭제하십시오.

su 명령으로 하위 프로세스를 작성할 수 없음

컨테이너에 대한 사용자 편집을 위해 대체 사용자(su) Linux 명령을 사용하면 su 명령이 로컬 Docker 환경에서 작동했더라도 su: 하위 프로세스를 작성할 수 없음. 리소스를 일시적으로 사용할 수 없음 메시지를 수신합니다. 컨테이너에 로그인하거나 su 명령이 컨테이너 내에서 스크립트의 일부로서 실행될 때 이 문제가 발생할 수 있습니다.

예:

bx ic exec -it my_container bash
adduser user
su - user
결과:
su: cannot create child process: Resource temporarily unavailable

기타 이미지 유형에서도 발생할 수 있지만, 이 문제는 종종 Centos 7 이미지로 빌드된 컨테이너에서 발생합니다.

Dockerfile을 편집하고 이미지를 재작성함으로써 또는 실행 중인 컨테이너에 로그인함으로써 문제를 해결할 수 있습니다.
Dockerfile을 편집하여 문제를 해결하려면 다음을 수행하십시오.
  1. 이미지가 작성되는 로컬 디렉토리에서, 아직 없으면 etc 폴더를 작성하십시오. 
    mkdir etc
  2. etc 디렉토리에 이동하십시오. 
    cd etc
  3. etc 폴더에서 pam.d 디렉토리를 작성하십시오. 
    mkdir pam.d
  4. pam.d 디렉토리에 이동하십시오. 
    cd pam.d
  5. pam.d 디렉토리에서 이름이 su인 파일을 작성하고 이를 텍스트 편집기에서 여십시오.
  6. su 파일에서, 모든 session 변수가 optional로 설정된 다음의 텍스트에서 복사하고 변경사항을 파일에 저장하십시오. 
    #%PAM-1.0
auth            sufficient      pam_rootok.so
# Uncomment the following line to implicitly trust users in the "wheel" group.
#auth           sufficient      pam_wheel.so trust use_uid
# Uncomment the following line to require a user to be in the "wheel" group.
#auth           required        pam_wheel.so use_uid
auth            substack        system-auth
auth            include         postlogin
account         sufficient      pam_succeed_if.so uid = 0 use_uid quiet
account         include         system-auth
password        include         system-auth
session         optional        system-auth
session         optional        postlogin
session         optional        pam_xauth.so
  7. 이미지의 Dockerfile에서 다음 정보를 포함하십시오. 
    RUN sed -i 's/session.*include/session        optional/' /etc/pam.d/su
  8. 이미지를 재작성하십시오. 자세한 정보는 이미지 빌드의 내용을 참조하십시오.
  9. 단일 컨테이너 또는 컨테이너 그룹을 재작성하십시오. 자세한 정보는 단일 컨테이너 작성 또는 컨테이너 그룹 작성을 참조하십시오.
  10. 확인하려면 다음 명령을 실행하십시오. 
    bx ic logs container_name
    결과:
    %Changing password for user user_name.
passwd: all authentication tokens updated successfully.
실행 중인 컨테이너에 로그인하여 문제를 해결하려면 다음을 수행하십시오.
  1. 컨테이너를 작성하십시오.
  2. 컨테이너에 로그인하십시오. 
    bx ic exec -it my_container bash
  3. etc 디렉토리에 이동하십시오. 
    cd etc
  4. pam.d 디렉토리에 이동하십시오. 
    cd pam.d
  5. Linux vi 명령을 사용하여 Visual Editor를 여십시오. 
    vi su
  6. su 파일에서 세션 변수를 모두 optional로 변경하고 파일에 변경사항을 저장하십시오. vi 편집기 사용에 대한 자세한 정보는 vi 편집기 사용 방법을 참조하십시오. 
    #%PAM-1.0
auth            sufficient      pam_rootok.so
# Uncomment the following line to implicitly trust users in the "wheel" group.
#auth           sufficient      pam_wheel.so trust use_uid
# Uncomment the following line to require a user to be in the "wheel" group.
#auth           required        pam_wheel.so use_uid
auth            substack        system-auth
auth            include         postlogin
account         sufficient      pam_succeed_if.so uid = 0 use_uid quiet
account         include         system-auth
password        include         system-auth
session         optional        system-auth
session         optional        postlogin
session         optional        pam_xauth.so
  7. 사용자로 로그인하십시오. 
    su - user
  8. 사용자를 위한 구성 정보를 보려면 id Linux 명령을 실행하십시오. 
    id
    결과:
    uid=1000(user) gid=1000(user) groups=1000(user)

컨테이너 작성 후 컨테이너가 중지 및 종료됨

컨테이너를 작성했지만 즉시 종료됩니다.

로컬에서 컨테이너를 성공적으로 실행했지만 Bluemix에서 컨테이너를 실행하면 다음 상황 중 하나가 발생합니다.
  • Bluemix 대시보드에서 컨테이너 상태가 실행 중에서 중지됨으로 변경됩니다.
  • CLI에서 bx ic inspect container를 실행할 때, 해당 상태가 종료로 표시됩니다.
bx ic logs도 실행했지만 컨테이너의 문제를 찾지 못했습니다.

Bluemix에서 컨테이너를 실행할 때 컨테이너는 항상 분리 모드로 실행됩니다. 즉, 컨테이너를 로컬로 실행할 때 docker run -d를 실행하는 경우와 비슷합니다. 결과적으로, 로컬에서 시작되는 컨테이너라도 Bluemix에서는 계속 실행되지 않습니다.

Dockerfile을 편집하여 컨테이너를 장기 실행 애플리케이션으로 만들어야 합니다. 예를 들어, 10초마다 "hello world"라는 문구를 출력하는 명령을 추가하면, 해당 조치를 통해 컨테이너가 장기 실행 애플리케이션이 됩니다. 
ENTRYPOINT ["/bin/sh"]
CMD ["-c","while true; do echo hello world; sleep 10;done"]

삭제된 컨테이너가 계속 Bluemix GUI에 표시됨

Bluemix GUI에서 컨테이너를 삭제했지만, 해당 컨테이너가 Bluemix GUI에 계속 표시되어 할당량을 사용합니다.

Bluemix GUI에서 삭제를 완료할 수 없습니다.

명령행에서 컨테이너를 삭제하십시오.

단일 컨테이너의 경우, 다음 단계를 따르십시오.
  1. 다음 명령행 옵션 중 하나를 실행하여 컨테이너를 삭제하십시오.
    • bx ic 명령의 IBM Bluemix Container Service 플러그인
      bx ic rm [-f] CONTAINER [CONTAINER]
    • docker 명령의 IBM Bluemix Container Service 플러그인 
      docker rm [-f] CONTAINER
  2. 다음 명령 중 하나를 실행하고 영역에 있는 모든 컨테이너를 나열하여 해당 컨테이너가 제거되었는지 확인하십시오.
    • bx ic 명령의 IBM Bluemix Container Service 플러그인
      bx ic ps -a
    • docker 명령의 IBM Bluemix Container Service 플러그인 
      docker ps -a
컨테이너 그룹의 경우, 다음 단계를 따르십시오.
  1. 다음 명령을 실행하여 컨테이너 그룹을 삭제하십시오.
    • IBM Bluemix Container Service 플러그인
      bx ic group-remove [-f] GROUP [GROUP]
  2. 명령을 실행하여 영역에 있는 모든 컨테이너를 나열하여 컨테이너가 제거되었는지 확인하십시오.
    • IBM Bluemix Container Service 플러그인
      bx ic groups [-q]

명령을 실행한 후에도 컨테이너가 Bluemix GUI에 계속 표시될 경우 잠시 기다렸다 컨테이너가 나중에 제거되었는지 확인하십시오.

컨테이너 그룹이 HAS_INACTIVE_INSTANCES 상태를 표시함

bx ic group-inspect를 실행하는 경우 컨테이너 그룹이 HAS_INACTIVE_INSTANCES 상태를 표시합니다.

그룹이 작성될 때 네트워킹이 구성에 실패하거나 하나 이상의 인스턴스에 대한 상태 검사에 실패하는 경우 인스턴스 상태가 INACTIVE로 변경됩니다. 그룹의 임의의 인스턴스가 비활성이면, 그룹 상태가 HAS_INACTIVE_INSTANCES로 변경됩니다.

인스턴스에 대한 상태 검사에 실패하는 경우 문제가 발생하면 필요한 조치 없이 문제가 해결될 수 있습니다. 그룹이 처음으로 작성될 때 문제가 발생하는 경우 그룹 재작성을 시도해 볼 수 있습니다. 재작성 후에도 문제가 다시 발생하는 경우, 앱이 그룹으로 배치될 때 네트워킹 구성이 가능하지 않은 것일 수 있습니다. 앱을 단일 컨테이너로 대신 배치해 보십시오.

데이터 스토리지 문제점 해결

컨테이너에서 마운트된 볼륨 디렉토리의 소유자를 변경하는 중에 권한 오류 발생

컨테이너에서 마운트된 볼륨에 대한 파일 또는 디렉토리의 소유권을 변경할 때 권한 오류가 발생합니다.
IBM Bluemix Container Service의 경우, 사용자 네임스페이스 기능이 Docker Engine에 대해 사용됩니다. 컨테이너에 있는 유효 루트는 컴퓨팅 호스트의 루트가 아닌 사용자에 맵핑됩니다. 마운트된 볼륨의 소스는 NFS(Network File System)입니다. NFS는 사용자 네임스페이스를 인식하지 않으므로 컨테이너에서 마운트된 파일 및 디렉토리에 대한 조작에 대해 모든 비사용자 검사가 수행됩니다.

  1. 이미지에 대한 Dockerfile에서 다음 행을 추가하여 사용 가능한 사용자 ID(UID)를 가진 사용자를 작성하십시오. 예를 들어, 사용자의 이름을 volumeeditor로 지정할 수 있습니다. 
    RUN groupadd --gid 1010 volumeeditor
RUN useradd --uid 1010 --gid 1010 -m --shell /bin/bash volumeeditor
  2. 이미지를 빌드하고 개인용 Bluemix 이미지 레지스트리에 푸시하십시오.
  3. 볼륨 경로가 마운트되는 이미지를 사용하여 컨테이너를 작성하십시오.
    팁: bx ic namespace-get을 실행하여 네임스페이스를 검색하고 <my_namespace>를 자신의 네임스페이스 정보로 대체하십시오. 
    bx ic run --volume volume_name:/path/to/files --name fixVolumePermissions registry.DomainName/<my_namespace>/image
  4. 컨테이너에 로그인하십시오. 
    bx ic exec -it container_name bash
  5. 마운트 경로에 대해 이름이 MOUNTPATH인 환경 변수를 정의하십시오. 
    MOUNTPATH="/path/to/files"
  6. root 그룹에 사용자를 추가하십시오. 
    adduser volumeeditor root
  7. 모든 그룹 구성원에 볼륨 마운트에 대한 읽기 및 쓰기 액세스 권한이 있도록 권한을 설정하십시오. 
    chmod 775 $MOUNTPATH
  8. 마운트 경로에서 폴더를 작성하고 읽기 및 쓰기 권한을 설정하십시오. 이 예제에서 폴더의 이름은 data입니다.
    su -c "mkdir -p $MOUNTPATH/data" volumeeditor
    su -c "chmod 700 $MOUNTPATH/data" volumeeditor
  9. 사용자 volumeeditor가 마운트 경로에 있는 파일 및 디렉토리에 액세스할 수 있는지 확인하십시오. 
    sudo -u volumeeditor ls -al $MOUNTPATH
  10. 선택사항: 컨테이너의 폴더를 볼륨의 data 폴더에 링크하십시오. 이 링크는 $MOUNTPATH/data의 쓰기 권한을 컨테이너의 폴더와 연관시킵니다.
    ln -sf $MOUNTPATH/data /folder/in/container/
  11. 권한 오류 없이 편집을 수행할 수 있는 경우 volumeeditor 그룹 루트 사용자를 제거하십시오. 
    deluser volumeeditor root

볼륨에서 컨테이너 그룹 인스턴스의 로그 겹쳐쓰기

컨테인 그룹을 작성할 때 볼륨을 마운트하고 각 컨테이너 그룹 인스턴스에서 볼륨으로 로그를 출력했습니다. 각 컨테이너 그룹 인스턴스는 동일한 파일에 작성되고 각 인스턴스가 파일을 겹쳐씁니다.
예:
bx ic group-create –p 9080 –m 64 -v log_vol:/opt/ibm/logs -e LOG_LOCATIONS=/opt/ibm/logs/messages.log,/opt/ibm/logs/debug.log,/opt/ibm/logs/trace.log -–name my_container_group registry.domain_name/ibmliberty:latest
각 컨테이너 인스턴스가 유일한 기록기인 것처럼 볼륨에 파일을 작성합니다. 각 인스턴스는 메모리에 로컬 캐시가 있습니다. 각 컨테이너 인스턴스에서 캐시된 데이터가 볼륨의 파일을 반복적으로 대체합니다.
인스턴스마다 파일이 달라지게 ID를 포함하도록 로그 파일의 이름을 조정하십시오. 예를 들어, 컨테이너 그룹을 작성할 때 지정하고 앱에서 생성한 로그 파일 이름에 $HOSTNAME 변수를 포함하십시오.
예:
bx ic group-create –p 9080 –m 64 -v log_vol:/opt/ibm/logs -e LOG_LOCATIONS=/opt/ibm/logs/messages.$HOSTNAME.log,/opt/ibm/logs/debug.$HOSTNAME.log,/opt/ibm/logs/trace.$HOSTNAME.log -–name my_container_group registry.domain_name/ibmliberty:latest

볼륨에/에서 데이터를 전송하는 데 예상보다 시간이 오래 걸림

볼륨에/에서 읽거나 쓰는 데 예상보다 오래 걸립니다.
IBM Bluemix Container Service에서 볼륨은 로컬 호스트에 대한 액세스 권한이 없습니다. 그러므로 모든 볼륨이 NFS 파일 공유에 마운트됩니다. 볼륨의 성능은 애플리케이션의 작업 로드와 함께 파일 공유의 크기 및 IOPS(초당 입출력(I/O))로 나타냅니다. 볼륨이 호스트되는 파일 공유가 볼륨과 컨테이너의 워크로드에 적합하지 않을 수도 있습니다. 또한 한 파일 공유에 있는 여러 볼륨이 파일 공유 리소스를 사용하기 위해 경쟁할 수 있습니다.

새 파일 공유를 작성하려면 조직 관리자 역할이 있어야 합니다. 자세한 정보는 사용자 및 역할을 참조하십시오.

  1. 볼륨이 호스트되는 파일 공유의 ID를 판별합니다.
    bx ic volume-inspect my_volume
    {
    "fsId": "fa277ff4-8a64-435b-9b75-0f11d59a3ae4", 
    "hostPath": "/vol/fa277ff4-8a64-435b-9b75-0f11d59a3ae4/my_volume", 
    "otherSpaceVisibility": [], 
    "spaceGuid": "fa277ff4-8a64-435b-9b75-0f11d59a3ae4", 
    "volName": "my_volume"
}
  2. 파일 공유를 확인하여 해당 크기와 IOPS를 판별합니다.
    bx ic volume-fs
    Name                                   Size   IOPS/GB   Created               State
fa277ff4-8a64-435b-9b75-0f11d59a3ae4   20     4.00      2016-05-20 13:48:10   READY
  3. 파일 공유에 사용 가능한 크기(GB)를 확인합니다.
    bx ic volume fs-flavor
    [
    20, 
    40, 
    80, 
    100, 
    250, 
    500, 
    1000, 
    2000, 
    4000, 
    8000, 
    12000
]
  4. 앱의 워크로드에 필요한 크기와 IOPS을 판별하십시오. 올바른 IOPS 값은 GB당 0.25IOPS, 2IOPS 또는 4IOPS입니다. 파일 공유 크기와 GB당 IOPS를 함께 곱하여 파일 공유의 총 IOPS를 판별합니다. 이 테이블에는 GB당 4IOPS의 여러 다른 공유 크기가 스케일링하는 방식의 예가 들어 있습니다.
    표 1. 파일 공유 크기와 GB당 IOPS의 예
    파일 공유 크기 GB당 IOPS 총 IOPS
    20GB 4 80
    100GB 4 400
    1,000GB 4 4,000
    4,000GB 4 16,000
  5. 새 파일 공유를 작성합니다. 첫 번째 숫자는 GB 단위의 크기입니다. 두 번째 숫자는 파일 공유 IOPS입니다.
    bx ic volume-fs-create my_new_fs 1000 4
  6. 새 파일 공유를 확인하고 상태가 준비로 표시될 때까지 기다립니다.
    bx ic volume-fs
  7. 새 파일 공유에 볼륨을 작성합니다.
    bx ic volume-create my_new_volume my_new_fs
  8. 새 볼륨이 마운트된 이미지에서 컨테이너 또는 컨테이너 그룹을 실행합니다.
  9. my_volume에서 데이터를 백업하고 데이터를 my_new_volume에 복사합니다. 자세한 정보는 데이터 백업 및 복원을 참조하십시오.

CLI에서 볼륨을 사용하여 컨테이너를 작성할 때 구문 분석 오류 발생

CLI에서 단일 컨테이너 또는 컨테이너 그룹을 작성할 때 실패 메시지가 표시됩니다.
예
bx ic group-create --name my_container -n my_host -d AppDomainName -m 2048 --min 1 --max 2 --desired 1 -p 9080 --auto -v my_volume:/my/path registry.DomainName/ibmliberty
결과 
docker: Error parsing reference: “my_volume:/my/path” is not a valid repository/tag.
Cloud Foundry 버전 6.17 이상용 CLI에서는 다른 용도로 -v 매개변수를 사용합니다.
명령에서 -v 옵션을 --volume 옵션으로 변경하십시오.
예
bx ic group-create --name my_container -n my_host -d AppDomainName -m 2048 --min 1 --max 2 --desired 1 -p 9080 --auto --volume my_volume:/my/path registry.DomainName/ibmliberty

알려진 문제점

IBM Containers에서 발생하는 알려진 문제점을 알아봅니다.

Docker 및 IBM Bluemix Container Service

일부 Docker 명령은 IBM Bluemix Container Service 플러그인(bx ic)에서 지원되지 않습니다.

특정 Docker 명령만 (bx ic)에서 지원됩니다. 지원되는 명령의 전체 목록은 docker 명령 참조에서 볼 수 있습니다. 예를 들어, tag는 bx ic에서 사용할 수 없지만 docker tag를 로컬로 사용한 후 업데이트된 이미지를 Bluemix 레지스트리에 푸시할 수 있습니다.

Docker와 Windows의 공통 문제

Windows 컴퓨터에서 Docker의 설치 및 구성과 관련된 공통 문제가 있습니다. 이 공통 문제에 대한 자세한 정보를 확인하려면 블로그: Windows용 Docker 설치: 공통 문제점에 대한 수정사항 및 boot2docker 태그가 지정된 Stack Overflow 게시물을 검토하십시오.

이미지

큰 이미지의 푸시 및 가져오기에 실패함

Docker 명령을 실행하여 조직의 개인용 Bluemix 레지스트리에서 큰 이미지를 푸시하거나 가져오는 경우 요청에 실패합니다. 이미지에 문제가 있는지 판별하려면 먼저 이미지를 DockerHub에 푸시할 수 있는지 확인하십시오. 이미지에 문제가 없는 경우 (가능하면) 이미지의 크기를 줄이십시오.

Bluemix에서 이미지 크기가 로컬에서 이미지 크기와 일치하지 않음

개인용 Bluemix 레지스트리에 이미지를 푸시하면 이미지와 관련하여 보고된 크기가 로컬 Docker 엔진에 있는 동일한 이미지의 크기보다 작습니다. 크기가 다르다고 해서 이미지를 푸시할 때 문제가 발생한 것은 아닙니다. 이미지의 압축 크기는 IBM Bluemix Container Service에서 보고됩니다.

모니터링 및 로깅

bx ic logs의 출력에 있는 추가 문자
bx ic logs를 실행하면 추가 문자(예: &)가 각 행의 시작에 표시됩니다. NGINX에서 수신 Docker 로그를 구문 분석할 때 이 문제가 발생합니다.
표현식을 사용하여 로그 파일 추가를 지정할 수 없음

수집할 로그 파일을 지정하기 위해 필터 또는 와일드카드 표현식을 사용할 수 없습니다. 예를 들면, /var/log/*.log는 지정할 수 없습니다. 대신, 각 로그 파일을 개별적으로 나열하십시오.

단일 컨테이너 및 컨테이너 그룹

컨테이너 작성 후 연결 실패

IBM Bluemix Container Service의 Docker 네트워킹 아키텍처 제한사항으로 인해 컨테이너가 처음 시작될 때 아웃바운드 네트워크 액세스가 작동하지 않을 수 있습니다. 컨테이너가 아웃바운드 네트워크 연결을 시도할 때 네트워크를 사용할 수 없는 경우 오류 코드를 리턴하거나 예외 처리(throw)를 하면 컨테이너가 종료하고 "충돌됨" 상태가 표시됩니다.

네트워킹 초기화를 허용하고 컨테이너 배치를 계속 진행하도록 하려면 아웃바운드 연결이 필요한 프로세스를 시작하기 전에 휴면 명령문이 있어야 합니다. 기본 Dockerfile에서 기존 서버 스타트업 명령을 랩핑하는 사용자 정의 Dockerfile을 작성하십시오. 예는 Let's Chat Bluemix 샘플 애플리케이션을 참조하십시오.

인바운드 연결이 즉각적이지 않음

인바운드 연결은 컨테이너 그룹 및 단일 컨테이너 둘 다(라우트 및 외부 앱 포함)에 대해 즉각적이지 않습니다. 연결이 사용 가능해질 때까지 기다리십시오. 최대 5분이 소요될 수 있습니다.

docker run -m 및 bx ic run -m에서 서로 다른 기본 측정 단위가 사용됨
docker run -m에서 측정 단위를 지정하지 않으면, 바이트가 기본 측정 단위로서 사용됩니다. bx ic run -m에서 측정 단위를 지정하지 않으면, MB가 기본 측정 단위로서 사용됩니다.
사용자 데이터 한계는 컨테이너가 작성되는 것을 방지
컨테이너에 너무 많은 서비스가 바인드된 경우 컨테이너를 작성할 수 없습니다. 각 서비스는 컨테이너에 데이터를 삽입하고 64KB 한계에 포함되어 계산됩니다. 이 한계를 초과하는 컨테이너를 작성하면 Resource CREATE failed... User data too large 메시지를 수신합니다. 이 메시지가 표시되면 컨테이너에서 서비스를 제거하십시오.
포트 25를 컨테이너에서 노출에 사용할 수 없음
포트 25는 IBM Bluemix Container Service 아키텍처의 일부인 SoftLayer에서 사용하지 못하도록 차단됩니다. 이 문제는 컨테이너에서 일부 SMTP 서비스의 사용에 영향을 주지 않습니다. 이 포트가 차단되는 이유에 대한 자세한 정보는 SoftLayer 문서에서 포트 25의 아웃바운드 이메일 항목을 참조하십시오.
컨테이너 빌딩 또는 네트워킹 프로세스가 완료되지 않음
컨테이너가 빌딩 또는 네트워킹 상태에 너무 오래 머물러 있는 경우에는 해당 컨테이너를 제거한 후 동일한 이미지로부터 새 컨테이너를 작성해야 할 수도 있습니다. 다음과 같은 CLI bx ic 명령을 사용할 수 있습니다.
  1. 해당 영역에서 실행 중인 컨테이너의 목록을 검색합니다. 기본적으로는 실행 중인 컨테이너만 표시됩니다. 자세한 정보를 보려면 CLI bx ic ps 참조를 검토하십시오. 
    bx ic ps -a
  2. 하나 이상의 컨테이너를 제거하십시오. 자세한 정보를 보려면 CLI bx ic rm 참조를 검토하십시오. 
    bx ic rm -f container_ID
  3. 이미지 이름으로부터 컨테이너 클라우드 서비스에서 새 컨테이너를 시작합니다. 자세한 정보를 보려면 CLI bx ic run 참조를 검토하십시오. 
    bx ic run
컨테이너 그룹에 대해 자동 복구가 즉각적이지 않음

컨테이너 그룹의 자동 복구는 새 시스템이 온라인 상태가 될 때까지 15분 이상 소요될 수 있습니다.자동 복구가 사용 가능해질 때까지 기다리십시오. 15분 이상이 소요될 수 있습니다.

cf delete-orphaned-routes 명령으로 컨테이너 그룹에서 라우트를 삭제함

Cloud Foundry가 애플리케이션에 바인드된 라우트를 인식하는 것과 같은 방식으로, 컨테이너 그룹에 맵핑된 라우트를 인식하지 않습니다. 컨테이너 그룹을 사용할 때에는 cf delete-orphaned-routes 명령을 실행하지 마십시오.

安装 IBM Bluemix Container Service 插件 (bx ic)

要运行命令，您的操作系统必须满足以下需求。此任务包含用于安装这些所需工具的信息。这些工具的较低版本无法与 IBM Bluemix Container Service 插件 (bx ic) 配合使用。

  • Docker V1.8 到 V1.12
  • Cloud Foundry CLI V6.14.0 - 6.22.0
    注： MacOS Sierra 用户必须安装 Cloud Foundry CLI V6.22.0 或更新版本。
  • Bluemix CLI
  • IBM Bluemix Container Service 插件

要安装 CLI，请执行以下操作：

  1. 安装 Docker V1.12。 对于 Windows 和 OS X 操作系统，Docker 会随 Docker Toolbox 一起安装，其中包含 Docker Machine、Docker Engine 和 Kitematic。对于 Linux 操作系统，安装机制根据使用的 Linux 分发版的类型不同而变化。请参阅以下链接以安装 Docker V1.12；或者如果已安装 Docker，请升级到该版本。
    注： 如果已在计算机上安装 Docker，但并不想升级当前版本，那么可以跳过此步骤，继续安装 Cloud Foundry CLI。
    • OS X

      https://github.com/docker/toolbox/releases/download/vV1.12/DockerToolbox-V1.12.pkg

    • Windows

      https://github.com/docker/toolbox/releases/download/vV1.12/DockerToolbox-V1.12.exe

    • Linux

      请参阅 Docker 安装文档以查找有关如何在使用的 Linux 分发版上安装 Docker VV1.12。

  2. 启动 Docker。
    • 对于 OS X 和 Windows，双击 Docker 快速启动终端图标。此时将启动 Docker 守护程序，并为您设置环境变量。
    • 对于 Linux，在命令行实用程序中，验证 Docker 守护程序是否在运行。此外，将 Docker 守护程序设置为自动重新启动。有关更多信息，请参阅适用于您的操作系统的 Docker 文档。
  3. 通过在终端运行 docker run hello-word，验证 Docker 安装已正确设置。 
    docker run hello-world
    Docker 会在本地查找 hello-world 映像，以从中运行容器，因此如果这是您首次运行该命令，那么将会收到以下消息。 
    Unable to find image 'hello-world:latest' locally
    映像是从公共映像存储库 Docker Hub 中拉取的，该命令完成后，输出将类似于以下内容： 
    $ docker run hello-world

    executable that produces the output you are currently reading.
 4. The Docker daemon streamed that output to the Docker client, which sent it
    to your terminal.

To try something more ambitious, you can run an Ubuntu container with:
 $ docker run -it ubuntu bash

For more examples and ideas, visit:
 https://docs.docker.com/userguide/
    如果命令未正确完成，请运行 docker version，以查看是否正确安装 Docker。
  4. 从 GitHub 存储库安装 Cloud Foundry CLI VCloud Foundry CLI V6.14.0 - 6.22.0。 必须将 Cloud Foundry CLI 安装在操作系统的缺省位置，否则 PATH 环境变量会与安装目录不匹配。
    注： MacOS Sierra 用户必须安装 Cloud Foundry CLI V6.22.0 或更新版本。
    要验证 Cloud Foundry CLI 是否已正确安装，或者要检查所安装的 Cloud Foundry CLI 是哪个版本，请运行以下命令：
    cf -v
  5. 安装 Bluemix CLI。
  6. 登录到 Bluemix CLI。 
    bx login -a https://api.ng.bluemix.net
  7. 安装 IBM Bluemix Container Service 插件。 
    bx plugin install IBM-Containers -r Bluemix
    要验证是否已正确安装该插件，请运行以下命令：
    bx plugin list

    该插件会在结果中显示为 IBM-Containers。

您可随时登录并开始管理容器。

登录到 IBM Bluemix Container Service CLI 插件 (bx ic)

安装 CLI 后，需要登录才能使用。

开始之前，请完成以下任务。
  • 安装 IBM Bluemix Container Service (bx ic) 插件。
  • 决定在登录到 Cloud Foundry 时要使用的 Bluemix 组织和空间。 有关组织和空间的更多信息，请参阅组织和空间。
  • 有关可在 Cloud Foundry login 命令中指定哪些选项的更多信息，请参阅 Cloud Foundry 文档中的 Login 命令。
  1. 打开命令行实用程序。
    • 对于 OS X 和 Windows，双击 Docker 快速启动终端图标。此时将启动 Docker 守护程序，并为您设置环境变量。
    • 对于 Linux，打开任何命令行实用程序。
  2. 通过 Cloud Foundry CLI 登录到 Bluemix。 要指定特定区域，请使用 -a 选项来指定该区域的 API 端点。第一次使用 CLI 时，如果未指定该选项，那么系统会提示您输入 API 端点。指定 API 端点后，在后续登录时就无需输入 API 端点。 
    bx login [-a api.DomainName] [-sso]
    要登录到特定区域，请在登录到 Cloud Foundry CLI 时包括该区域的 API 端点。
    • 美国南部
      bx login -a api.ng.bluemix.net
    • 英国
      bx login -a api.eu-gb.bluemix.net
    当您使用联合标识登录时，需要单点登录参数 --sso。如果使用此选项，请打开 https://login.ng.bluemix.net/UAALoginServerWAR/passcode，以获取一次性密码。如果您未包括 --sso 选项，请完成以下子步骤。
    1. 对于电子邮件，请输入您用于 Bluemix 的 IBM 标识。
    2. 对于密码，请输入 IBM 标识的密码。 此时将检索您的 Bluemix 组织和空间。
    3. 输入代表您某个 Bluemix 组织的数字。
    4. 输入代表其中一个现有 Bluemix 空间的数字。
  3. 设置组织的名称空间。 名称空间是用于识别专用 Bluemix 映像注册表的唯一名称。创建容器时，必须通过包含具有映像名称的名称空间来指定映像的位置。
    注： 创建名称空间时，在创建后即不能进行更改，只能删除。 如果您 Bluemix 组织中的用户已经部署容器并设置名称空间，那么会显示消息，其中显示您组织的名称空间。 在此情况下，您可以继续执行下一步。
    bx ic namespace-set <my_namespace>
    提示： 请遵循名称空间的以下规则：
    • 名称只能包含小写字母、数字或下划线 (_)。
    • 名称可以为 4 到 30 个字符。如果计划从命令行管理容器，那么您可能倾向于使用能够快速输入的简短名称空间。
    • 其名称在 Bluemix 中必须是唯一的。
  4. 初始化 IBM Bluemix Container Service。 
    bx ic init
  5. 决定您是要运行 bx ic 还是本机 Docker 命令。 您可以在每次登录到插件时选择以下某个选项，也可以使用 shellinit 命令以始终选择“选项 2”。
    • 选项 1（缺省）

      IBM Bluemix Container Service 包装程序 Docker 的简单 shell。本地 Docker 守护程序不受 IBM Bluemix Container Service 的影响。bx ic 命令用于在 Bluemix 空间中管理容器，而 docker 命令用于在本地管理容器。但是，本地映像开发始终需要 docker 命令。

      例如，使用“选项 1”，bx ic run 命令会在 Bluemix 空间创建容器，而 docker run 会在本地创建容器。如果您对 Docker 和 IBM Bluemix Container Service 来说是新手，或者您想要在本地测试容器，那么您可能更愿意使用此选项。

    • 选项 2

      将当前 CLI 会话的本地 Docker 守护程序重定向到 IBM Bluemix Container Service，以便在 docker 命令可用于在 Bluemix 空间中管理容器时，替换 bx ic 命令。

      例如，使用“选项 2”，bx ic run 或 docker run 命令可用于在您的空间中创建容器。如果您是经验丰富的 Docker 用户，熟悉 docker 命令，或者您编写自动化脚本来使用 IBM Bluemix Container Service，那么您可能更愿意使用此选项。

      某些例外情况仍需要使用 bx ic 命令来执行特定于 IBM Bluemix Container Service 的任务。例如，要设置名称空间，您必须使用命令 bx ic namespace-set。docker namespace-set 不是有效的命令。要获取此配置支持的 docker 命令列表，请参阅IBM Bluemix Container Service 插件 (bx ic) 支持的 Docker 命令。

      要配置此选项，请复制并粘贴登录期间终端中提供的环境变量。
      • DOCKER_HOST 用于将 Docker 主机设置为 IBM Bluemix Container Service。
      • DOCKER_CERT_PATH 用于设置 IBM Bluemix Container Service Cloud Foundry 插件的认证密钥位置。
      • DOCKER_TLS_VERIFY 使用 TLS 并验证远程服务器。
      提示： 完成 IBM Bluemix Container Service 文档中的任务后，就可以在标记有星号 (*) 的所有步骤中运行 docker 命令。

bx ic attach [--no-stdin] [--sig-proxy] CONTAINER

用途
控制正在运行的容器或查看其输出。使用 CTRL+C 以退出并停止容器。有关更多信息，请参阅 Docker 帮助中的 attach 命令。
参数
--no-stdin
（可选）不包含标准输入。
--sig-proxy
（可选）缺省值为 true。
CONTAINER
（必需）容器名称或标识。
示例
以下示例是用于连接到容器 my_container 的请求。
bx ic attach my_container

bx ic build -t TAG [--no-cache] [--pull] [-q] DOCKERFILE_LOCATION

用途
在本地或在专用 Bluemix 映像注册表中构建 Docker 映像。有关更多信息，请参阅 Docker 帮助中的 build 命令。
参数
DOCKERFILE_LOCATION
（必需）本地主机上 Dockerfile 和上下文文件的路径。
--no-cache
（可选）首次构建映像时，会对整个组织的映像层进行高速缓存。构建映像的后续版本时，这些映像的高速缓存版本可用于加快构建过程。构建映像的新版本时，通过包含 --no-cache 选项，还可确保映像的每层都重新构建。
--pull
（可选）缺省情况下，仅当基本映像在组织的映像高速缓存中不存在时，才会从注册表中拉取基本映像。通过包含此选项，可以确保使用最新版本的基本映像来构建新映像。
-q, --quiet
（可选）禁止显示容器生成的详细输出。缺省值为 false。
-t TAG, --tag TAG
（必需）要应用到所创建映像的专用映像注册表的路径。
示例
以下示例是用于构建名为 myimage 的映像的请求。要在该构建中使用的 Dockerfile 和其他工件位于运行该命令的目录中。由于映像名称带有注册表和名称空间，因此会在组织的专用 Bluemix 映像注册表中构建映像。
提示： 运行 bx ic namespace-get 来检索您的名称空间，并将 <my_namespace> 替换为您的名称空间信息。
bx ic build -t registry.DomainName/<my_namespace>/myimage .

bx ic cp SRC_PATH CONTAINER:DEST_PATH | bx ic cp CONTAINER:SRC_PATH DEST_PATH

用途
将文件或文件夹从运行中或已停止的单个容器复制到本地计算机或反之。 有关更多信息，请参阅 Docker 帮助中的 cp 命令。
注： 仅支持此命令与单个容器一起使用。 当您想要从容器组复制文件或将文件复制到容器组，您必须从组中的每一个容器实例中复制文件或将文件复制到组中的每一个容器实例。
参数
DEST_PATH
（必需）要从源路径复制信息的本地计算机或容器上的文件或文件夹的路径。该文件或文件夹必须位于本地计算机或容器中，您才能开始复制。 如果您要复制到本地计算机上，那么您可以使用绝对或相对路径。 如果要复制到单个容器，那么您必须包括容器的名称、定界符 “:” 和目标文件或文件夹的路径。 路径必须相对于容器的根目录。
SRC_PATH
（必需）要复制的文件或文件夹的路径。如果文件或文件夹位于本地计算机上，那么您可以使用绝对或相对路径。 如果文件或文件夹位于单个容器中，那么您必须包括容器的名称、定界符 “:” 和要复制的文件或文件夹的路径。 路径必须相对于容器的根目录。
注： 您无法复制存储在容器 /dev、/etc、/proc 和 /sys 文件夹下的特定系统文件。
示例
此示例命令将名为 config.txt 的文件从本地计算机的当前目录复制到名为 my_container 的单个容器的 config 目录中。
bx ic cp ./config.txt my_container:/config

用途
访问 Docker Hub 映像或本地注册表中的映像，然后将其复制到专用 Bluemix 映像注册表。
注： 此命令是同时使用 docker pull 和 docker push 将映像添加到 Bluemix 映像注册表的替代方法。如果您更愿意使用 docker pull 和 docker push，请参阅通过命令行将本地映像推送到专用 Bluemix 注册表。
参数
DST
（必需）专用 Bluemix 映像注册表 URL，其中包含名称空间和目标映像名称。映像的标记是可选的。
SRC
（必需）源存储库和映像名称。
示例
bx ic cpi source_repository/source_image_name private_registry_URL/destination_image_name:tag

用途
在容器内执行命令。有关更多信息，请参阅 Docker 帮助中的 exec 命令。
参数
CMD
（必需）要在指定的一个或多个容器内执行的命令。
CONTAINER
（必需）容器名称或标识。
-d , --detach
（可选）在后台运行指定的命令。
-it
（可选）交互方式。使标准输入保持显示。输入 exit 以退出。
-u USER, --user USER
（可选）用户名
示例

bx ic exec -it my_container bash
此示例命令将在容器 my_container 中运行 date 命令。
bx ic exec my_container date

bx ic group-create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session-affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

用途
创建扩展组。缺省情况下，Bluemix 中的容器以拆离模式运行。
参数
--anti
（可选）使用反亲缘关系，使您的容器组具有更高的可用性。--anti 选项可强制组中的每一个容器实例置于不同的物理计算节点上，这可降低组中所有容器因硬件故障而发生崩溃的几率。 您可能无法对较大的组大小使用此选项，因为每一个 Bluemix 区域和组织可用于部署的计算节点集有限制。 如果部署不成功，请减少组中容器实例的数目，或者除去 --anti 选项。
--auto
（可选）针对容器组启用了自动恢复并且该组已运行了 10 分钟时，该组的负载均衡器会开始通过 HTTP 请求定期检查组中每个容器实例的运行状况。如果容器实例在 100 秒内未响应，即会将该实例标记为“不活动”。自动恢复会从组中除去不活动的容器实例并重新创建这些实例。自动恢复会尝试恢复组中的容器实例 3 次。在第三次尝试后，自动恢复在 60 分钟内不会再恢复组中的任何容器实例。60 分钟后，自动恢复进程将再次启动。有关运行状况检查的更多信息，请参阅 --http_monitor_enabled 选项。
CMD
（可选）该命令及相应自变量将传递给容器组来执行。此命令必须是长时间运行命令。请不要使用短时间运行命令（例如 /bin/date），因为此类命令可能会导致容器崩溃。
长时间运行命令的示例（使用 IBM Bluemix Container Service 提供的 ibmnode 映像）：
bx ic group-create --name my_container_group registry.DomainName/ibmnode ping localhost
bx ic group-create --name my_container_group registry.DomainName/ibmnode -- tail -f /dev/null
注： CMD 及其自变量必须位于 bx ic group-create 命令行的末尾。
-d DOMAIN, --domain DOMAIN
（可选）缺省系统域为 AppDomainName，此域已提供 SSL 证书，所以可以使用 HTTPS 来访问容器组，而无需任何额外配置。要使用定制域，必须在公共 DNS 服务器上注册定制域，在 Bluemix 中配置定制域，然后将定制域映射到公共 DNS 服务器上的 Bluemix 系统域。 定制域映射到 Bluemix 系统域后，对定制域的请求会路由到 Bluemix 中的应用程序。创建定制域时，不要在域名中包含下划线 (_)。有关定制域的更多信息，请参阅创建和使用定制域。为了保护定制域安全，请上传 SSL 证书，以便可以使用 HTTPS 访问容器组。

主机和域组合在一起构成了完整的公共路径 URL（例如 http://mycontainerhost.AppDomainName），并且该 URL 在 Bluemix 中必须唯一。使用 bx ic group-inspect 命令来查看容器组的详细信息时，主机和域会作为路径一起列出。

--desired DESIRED
（可选）组中的实例数。缺省值为 2。应用程序所需的容器组实例数取决于期望的工作负载和应用程序可以处理请求的时间。 例如，多任务或网络连接会影响应用程序所需的实例数。 要确定所需实例的数目，可在运行应用程序的 Bluemix 中部署单个容器，并在此容器上执行负载测试。例如，如果应用程序每秒可处理 100 个请求，但是您期望每秒平均工作负载为 300 个请求，那么容器组的所需实例数目为 3。 要使容器组具有高可用性且对故障的处理更富弹性，请考虑除最低要求的实例数外，额外包含更多实例，以用于处理预期的工作负载。额外的实例可在其他实例崩溃时处理工作负载。要确保自动恢复发生故障的实例，而不影响工作负载，请包含一个额外实例。要避免两个实例同时发生故障，请包含两个额外实例。此设置是 N+2 模式，其中 N 是处理请求的实例数，+2 是额外两个实例。
要点： 容器组应该至少包含 2 个容器实例。 如果组中仅包含 1 个实例，且该实例变得不可用，那么用户可能会遭遇应用程序停机时间。

有关更多信息，请参阅 Docker 帮助中的 top 命令。
参数
CONTAINER
（必需）容器名称或标识。
ps 选项
（可选）使用本机 Linux ps 命令提供的选项，以进一步详细列出或过滤 bx ic top 命令的输出。
示例
以下示例请求显示在名为 my_container 的容器中运行的所有当前进程的详细信息。
bx ic top my_container -aux

卸载 IBM Bluemix Container Service 插件 (bx ic)

如果不再需要 IBM Bluemix Container Service 插件，可以使用 bx 命令将其卸载。

注： 要卸载 IBM Bluemix Container Service Cloud Foundry 插件，请运行以下命令。
cf uninstall-plugin IBM-Containers
  1. 要为 Bluemix CLI 卸载 IBM Bluemix Container Service 插件，请运行以下命令。 
    bx plugin uninstall IBM-Containers
  2. 通过运行以下命令并检查已安装的插件的列表，验证该插件是否已卸载。 
    bx plugin list

Depuración de contenedores y revisión de registros

Revise las opciones que tiene para depurar sus contenedores y busque las causas raíz de los errores.

  1. Liste los contenedores y revise el estado actual.
    • Contenedor único
      bx ic ps -a
    • Grupo de contenedores
      bx ic groups
  2. Opcional: Consulte lo que significa cada estado y revise las sugerencias para evitar que falle un contenedor.
  3. Si el contenedor ha fallado o está bloqueado, inspeccione los contenedores para encontrar la causa raíz de la anomalía. El mensaje de error se puede encontrar en la sección Anomalía de su respuesta.
    • Contenedor único
      bx ic inspect CONTAINER
    • Grupo de contenedores
      1. Examine el grupo de contenedores.
        bx ic group-inspect GROUP
      2. Para obtener más información, liste las instancias de un grupo de contenedores e inspeccione cada instancia. 
        bx ic group-instances GROUP
        bx ic inspect GROUP_INSTANCE
  4. Si la app salió con un error, recupere los registros de la app.
    Nota: Este mandato solo se puede usar si la app escribe registros en las secuencias de salida STDOUT y STDERR estándar. La escritura en estos flujos de salida estándar le permiten ver registros de contenedor incluso cuando el contenedor se apaga o se bloquee.
    • Contenedor único
      bx ic logs CONTAINER
    • Grupo de contenedores
      1. Liste las instancias de contenedor de un grupo de contenedores.
        bx ic group-instances GROUP
      2. Recupere los registros para cada instancia. 
        bx ic logs GROUP_INSTANCE
  5. Si tiene que acceder a los registros de un contenedor que ya no está disponible, como por ejemplo una instancia de un grupo de contenedores que se ha vuelto a crear con la recuperación automática, utilice Kibana para ver sus registros.

    https://logmet.DomainName/

    Los registros están disponibles durante un máximo de 7 días en Kibana.

  6. Si los registros se han guardado en un archivo dentro del contenedor, deberá iniciar sesión en el contenedor para revisarlo.
    Importante: Solo puede iniciar sesión en un contenedor en ejecución. Si el contenedor está bloqueado y no se puede iniciar de nuevo, es posible que no pueda acceder a los registros de su contenedor. Considere la posibilidad de cambiar su app para escribir los flujos de salida STDOUT y STDERR estándar para que pueda acceder a los archivos de registro incluso cuando ya no se pueda acceder al contenedor.
    1. Si el contenedor está en ejecución, inicie el contenedor. 
      bx ic start CONTAINER
    2. Inicie sesión en el contenedor en ejecución. 
      bx ic exec -it CONTAINER bash
    3. Navegue hasta el archivo de registro y revise los registros.
  7. Si los registros se han guardado en un volumen y el contenedor ya no se inicia, cree un nuevo contenedor para acceder al volumen.
    1. Cree un nuevo contenedor que se monte en el volumen en el que estén almacenados los archivos de registro. 
      bx ic run --name logcontainer -m 64 --volume MOUNT_PATH_TO_VOLUME registry.DomainName/ibmliberty
    2. Inicie sesión en el contenedor en ejecución. 
      bx ic exec -it logcontainer bash
    3. Navegue hasta la vía de acceso de montaje del volumen y revise sus archivos de registro.
  8. Si necesita más soporte con depuración de contenedores, abra una incidencia de soporte de Bluemix.

Resolución de problemas de la configuración de la CLI

Error de método (rpc) durante init

Cuando intenta ejecutar bx ic init, recibe uno de los siguientes mensajes.
  • bx ic rpc: can't find method CliRpcCmd.IsLoggedIn
  • bx ic rpc: can't find method CliRpcCmd.IsMinCliVersion
La versión instalada de Cloud Foundry es una versión anterior a la admitida para IBM Bluemix Container Service.
  1. Ejecute cf -v para ver qué versión de la CLI de Cloud Foundry tiene instalada.
  2. Si la versión instalada es anterior a la 6.14.0, actualice la CLI de Cloud Foundry.

Error de autenticación de registro recibido durante bx ic init

Al ejecutar bx ic init, aparece uno de los siguientes mensajes de error.
  • No se ha podido autenticar con el registro de IBM Containers en DomainName
****exit status 1
****time="2015-07-31T08:58:32-04:00" level=fatal msg="Post http:///var/run/docker.sock/v1.18/auth: dial 
unix /var/run/docker.sock: este archivo o directorio no existe. ¿Está intentando conectarse a un daemon habilitado para TLS sin TLS?”
  • ** Recuperando certificados de cliente de IBM Containers
Anomalía en la autenticación. Compruebe su nombre de usuario y contraseña.
El formato de token no es válido. Genere un token nuevo.
O bien no se cumple uno de los requisitos previos de bx ic o IBM Bluemix Container Service no está disponible.
  1. Verifique que el daemon Docker esté en funcionamiento. Si no lo está, inícielo y vuelva a ejecutar bx ic init.
  2. Ejecute bx login para verificar que ha iniciado sesión en la CLI de Cloud Foundry. A continuación, vuelva a iniciar sesión en el plug-in ejecutando el mandato bx ic init.
  3. Verifique la disponibilidad de IBM Bluemix Container Service. Consulte la página de estado de Bluemix para obtener más información.

Sudo es necesario con mandatos bx ic

Cuando ejecute mandatos bx ic, se le solicitará que utilice el prefijo sudo con todos los mandatos.
De forma predeterminada, los mandatos bx ic deben ejecutarse con autenticación root y, por lo tanto, pueden necesitar el prefijo sudo, en función del sistema operativo.
Para ejecutar los mandatos docker y bx ic sin sudo, ejecute el siguiente mandato y, a continuación, si está en Linux debe cerrar la sesión y volver a iniciarla. 
sudo usermod -a -G docker <USUARIO_ACTUAL>

Resolución de problemas de cuotas

No se puede crear un contenedor porque no hay suficiente cuota de ORG

Al crear un contenedor, recibe un mensaje de error que indica que no tiene suficiente cuota en su espacio.
BXNUI0110E: No se han podido asignar recursos de IBM Containers. No hay cuota de ORG suficiente para asignar un nuevo espacio. No se pueden exceder las IP flotantes de la organización 'organización'. Utilice IBM
    Bluemix Manage Organizations para ajustar la asignación de la cuota.
Es posible que este mensaje se muestre al intentar crear contenedores en un espacio donde no tiene asignada suficiente cuota. De forma predeterminada, cada espacio precisa de 2 direcciones IP y 2 GB de memoria para asignar a dicho espacio para que funcione con el servicio de contenedor.
Para distribuir la memoria y la cuota de IP entre varios espacios:
  1. En los detalles de la cuenta, en Gestionar organizaciones, seleccione una organización.
  2. En la sección de cuota, vea los detalles para los contenedores.
  3. Pulse Editar y asigne la memoria y las direcciones IP entre los espacios de la organización.
  4. Cree un contenedor en cualquier espacio de la organización.

No se puede editar la cuota a pesar de que tiene una cuenta de pago

No es posible modificar su cuota de contenedor.
Es posible que exista alguna discrepancia entre el gestor de la organización y los datos de facturación.
El usuario que ha iniciado sesión debe ser el gestor de la organización. Para verificar el propietario, siga estos pasos.
  1. En los detalles de la cuenta, en Gestionar organizaciones, seleccione una organización.
  2. En la lista de usuarios, verifique que el gestor de la organización es el mismo que el ID con el que ha iniciado la sesión.
  3. Cierre la sesión de la cuenta e iníciela de nuevo.
Si lleva a cabo estos pasos y sigue sin poder editar la cuota del contenedor, póngase en contacto con el soporte de IBM.

Problemas con las imágenes de IBM WebSphere Application Server Liberty

Tiene problemas al utilizar una de las imágenes de IBM WebSphere Application Server Liberty en IBM Bluemix Container Service.

Puede que la imagen no se haya configurado correctamente.

Si tiene problemas relacionados concretamente con la imagen Dockerfile, utilice el Rastreador de problemas de GitHub. Para problemas generales relacionados con IBM WebSphere Application Server Liberty, puede obtener ayuda en la Comunidad WASdev.

Para obtener más información sobre las imágenes de IBM WebSphere Application Server Liberty en IBM Bluemix Container Service, consulte Imágenes de IBM WebSphere Application Server Liberty.

La imagen no puede extraerse de su registro de imágenes privado de Bluemix

Ha creado un contenedor de una imagen de contenedor en el registro privado de Bluemix, pero el contenedor se atasca en un estado BUILDING. Al ejecutar el mandato bx ic inspect. se muestra el siguiente error en la sección Env de la respuesta de CLI. 
"Env": [
            "metadata_exit_code=-1",
            "logging_password=",
            "space_id=0e65c436-9413-46e2-8b42-ff06aa74ac21",
            "logstash_target=logmet.opvis.bluemix.net:9091",
            "metadata_err_msg=Cannot pull image from registry: Pull failed with error message failed 
             to register layer: ApplyLayer exit status 1 stdout:  stderr: Container ID 1234567890 
             cannot be mapped to a host ID",
            "metrics_target=logmet.opvis.bluemix.net:9095",
            "metadata_running=1"
En el IBM Bluemix Container Service, la característica de espacio de nombres de usuario de Docker está habilitada en Docker Engine. La característica de espacio de nombres de usuario fue introducida en la versión 1.10 de Docker y proporciona la posibilidad de correlacionar un usuario root en el contenedor con un usuario no root fuera del contenedor. De esta forma, los permisos de un usuario que pueden llevar a cabo acciones de usuario root dentro del contenedor pueden restringirse fuera del contenedor, de modo que no se podrá acceder, alterar ni eliminar los archivos del host y de otros contenedores.
Nota: El espacio de nombres de usuarios de Docker es distinto del espacio de nombres configurado en Bluemix para identificar el registro de imágenes privado.

Al crear un contenedor a partir de una imagen de contenedor con el mandato bx ic run, la imagen se carga desde el registro de imágenes privado de Bluemix en IBM Cloud Docker Engine. A continuación, Docker Engine empezará a crear las capas de contenedor respectivas. En función de qué se especifique en la imagen del contenedor, los archivos de app de la máquina local podrían añadirse al contenedor. Todos estos archivos se crean por un ID de usuario (UID) y un ID de grupo (GID) en la máquina local. El UID y GID indican el permiso del usuario en la máquina local. Cuando desee añadir estos archivos al contenedor, los respectivos UID y GID se correlacionan con el UID y GID dentro del contenedor. El UID y GID correlacionado puede tener distintos permisos dentro del contenedor que la máquina local. Por ejemplo, el UID y el GID no root de la máquina local está correlacionado con un usuario root dentro del contenedor. De esta forma, el usuario puede actuar como un usuario root dentro del contenedor así como acceder a, alterar y eliminar archivos de contenedor. Como el contenedor se está ejecutando en un entorno aislado, el UID y el GID no puede acceder a ningún archivo en el host ya que tiene asignados permisos no root fuera del contenedor.

Para correlacionar un UID y GID de una máquina local con uno dentro del contenedor, la característica de espacio de nombres de usuario requiere que todos los archivos se creen con un UID y GID entre 0 y 65536.

El error mostrado se produce cuando se crea un archivo desde un UID o GID fuera de este rango y cuando este archivo está incluido en una de las capas de imágenes del contenedor. Al crear un contenedor desde la imagen, Docker Engine compila cada capa de la imagen. Cuando se encuentra un archivo con un UID o GID fuera del rango, Docker Engine falla porque este UID o GID no ha podido asignarse a uno válido del contenedor. Docker Engine también falla cuando se utiliza un archivo en una capa de imagen y se elimina en la siguiente capa de imagen.

Habilitar el espacio de nombres de usuario en su Docker Engine local
Cuando cree las imágenes del contenedor localmente, asegúrese de habilitar la característica de espacio de nombres de usuario en Docker Engine local. De esta forma, todos los UID y GID de la máquina local que están fuera del rango válido de espacios de nombres de usuario se correlacionan automáticamente con un ID válido dentro del rango. Pruebe la imagen del contenedor localmente antes de enviarla por push al registro de imágenes privado de Bluemix y utilícela con IBM Bluemix Container Service. Para obtener más información sobre cómo habilitar la característica de espacio de nombres de usuario, consulte Inicio del daemon con los espacios de nombres de usuario habilitados.
Comente la capa de imagen afectada y cambie la propiedad de los archivos afectados
Si no puede encontrar el archivo que causa el problema, intente comentar las capas de imágenes de su Dockerfile y descomente una capa tras otra. Compile su imagen y cree un contenedor para buscar la capa de imagen afectada.

Ejecute chown [-R] root:root <archivo_o_carpeta> para cambiar la propiedad del archivo afectado que esté guardado en su máquina local o añada este mandato a la capa de imagen afectada del Dockerfile.

Dockerfile de ejemplo

RUN curl -fsSL "https://www.kernel.org/pub/software/scm/git/git-${GIT_VERSION}.tar.gz" \
        | tar -xzC /tmp \
  && cd /tmp/git-${GIT_VERSION} \
  && ./configure \
  && make all \
  && make install \
  && chown -R root:root /tmp

La imagen no puede exportarse de su registro de imágenes privado de Bluemix

Ha iniciado sesión en el IBM Bluemix Container Service y desea extraer una imagen del registro privado de Bluemix a una máquina local mediante el mandato bx ic pull. El mandato no puede indicar que bx ic pull no está soportado.
El mandato bx ic pull no está soportado para extraer una imagen del registro privado de Bluemix a su máquina local.
Para exportar una imagen desde el registro privado de Bluemix, siga estos pasos.
  1. Inicie sesión en la IBM Bluemix Container Service. Si ya ha iniciado sesión, asegúrese de que las variables de entorno no están establecidas para utilizar los mandatos nativos de Docker en el IBM Bluemix Container Service.
  2. Liste las imágenes. 
    bx ic images
  3. Exporte una imagen desde el registro privado de Bluemix a la máquina local. Utilice el nombre o id de la imagen que se ha devuelto en el paso anterior. 
    docker pull <image_name_or_id>
  4. Verifique que la imagen se haya extraído a la máquina local. 
    docker images
  5. Guarde la imagen en un archivo .tar. 
    docker export <image_name_or_id> > <exported_image_name>.tar

El espacio de nombres para una organización no puede cambiarse

No puede encontrar un método para cambiar el espacio de nombres.
El espacio de nombres para una organización no puede cambiarse después de crearse.
Puede crear otra organización y establecer el espacio de nombres para dicha nueva organización en el nombre que desee.

Solicitud de inicio de sesión durante la ejecución de docker push

Cuando intenta ejecutar docker push para añadir una imagen al registro de imágenes privado de Bluemix, se le solicita que indique las credenciales de inicio de sesión. Al introducir las credenciales de Docker o Bluemix, se produce un error en la autenticación.
Es posible que la autenticación falle debido a los siguientes motivos.
  • La señal de transporte, que contiene información de autenticación de Cloud Foundry, ha caducado.
  • El servicio de autorización de Bluemix no está disponible.
  • Si utiliza el plug-in IBM Bluemix Container Service:
    1. Inicie sesión en la CLI de Cloud Foundry ejecutando el mandato para actualizar la señal de transporte del sistema.
      bx login [-a api.DomainName] [-sso]
      El parámetro de inicio de sesión único, --sso, se necesita cuando se inicia una sesión con un ID federado. Si se utiliza esta opción, abra https://login.ng.bluemix.net/UAALoginServerWAR/passcode para obtener una contraseña de uso único. Si no incluye la opción --sso, lleve a cabo estos subpasos.
      1. En Correo electrónico, especifique el ID de IBM que utiliza para iniciar sesión en Bluemix.
      2. En Contraseña, indique la contraseña del ID de IBM que utiliza para iniciar sesión en Bluemix. Al introducir el nombre de usuario y la contraseña, se recuperan su espacio y organización de Bluemix.
      3. Escriba el número que represente una de sus organizaciones de Bluemix.
      4. Escriba el número que representa uno de los espacios de Bluemix existentes.
    2. Inicialice la CLI de IBM Bluemix Container Service para volver a crear la señal de transporte.
      bx ic init

Si ha vuelto a iniciar sesión y se le sigue solicitando las credenciales, quiere decir que el servicio de autorización de Bluemix no está disponible. Vuelva a intentarlo más tarde.

Resolución de problemas de los contenedores

Error 502 Pasarela errónea al acceder a una app

Ha desplegado satisfactoriamente un grupo de contenedores, pero al abrir el URL para la app, se muestra el siguiente error.
502 Pasarela errónea: El punto final registrado no ha podido manejar la solicitud.
  1. Puede que el direccionamiento todavía se esté configurando.
  2. Puede que el contenedor no responda.
  3. Puede que haya un error con el direccionamiento aunque el grupo de contenedores se esté ejecutando satisfactoriamente.
  1. Espere unos minutos para dejar el tiempo suficiente para que el direccionamiento se complete. Puede haber una espera de 5 minutos después de la ejecución del proceso de contenedor antes de que el grupo de contenedores pueda conectarse con la ruta especificada.
  2. Si ha esperado 5 minutos y sospecha que se ha producido un error, consulte si el contenedor está respondiendo enlazando una dirección IP con el contenedor en ejecución y, a continuación, ejecute un mandato curl para dicha dirección IP y puerto.
    1. Ejecute el siguiente mandato para obtener el ID de una de las instancias de contenedor del grupo. Anote el ID de contenedor y el puerto.
      bx icps
    2. Solicite una dirección IP ejecutando el siguiente mandato.
      bx ic ip-request
    3. Enlace la dirección IP con la instancia de contenedor ejecutando el siguiente mandato.
      bx ic ip-bind IP_address container_ID
    4. En un navegador, escriba el URL para la aplicación con la dirección IP y el puerto. Por ejemplo, http://dirección_IP:puerto

      Si no puede acceder a la aplicación, significa que hay un problema con el grupo de contenedores y se debe volver a crear el grupo. Si puede acceder a la aplicación, el contenedor se está ejecutando satisfactoriamente, pero hay un error con el direccionamiento. Póngase en contacto con el soporte técnico de IBM Bluemix.

Las instancias del grupo de contenedores se vuelven a crear de forma repetida durante el arranque mediante la modalidad de recuperación automática de IBM Bluemix Container Service

Ha creado un grupo de contenedores y ha habilitado la modalidad de recuperación automática. Durante el arranque del grupo de contenedor, las instancias del grupo que no responden se eliminan y se vuelven a crear de forma repetida.

La app que se ejecuta en el grupo de contenedores tarda demasiado en iniciar la escucha en el puerto expuesto. Si ha habilitado la recuperación automática para el grupo de contenedores, IBM Bluemix Container Service comprueba el estado de cada instancia con una solicitud HTTP en el puerto TCP expuesto después de crear el grupo. Si una instancia no está a la escucha en el puerto expuesto durante 3 minutos, la instancia se marcará como que no responde y se eliminará, se volverá a crear y se reiniciará. Este proceso se repite hasta que se alcance el número máximo de instancias recreadas. A continuación, se inhabilitará la modalidad de recuperación automática.

Para ejecutar una app en un grupo de contenedores con la modalidad de recuperación automática habilitada, asegúrese de que la app se inicia y que esté a la escucha en el puerto expuesto en menos de 2 minutos. En función del tipo de app, existirán métodos distintos para asegurar un arranque rápido de la app.

El servicio de Bluemix no se enlaza directamente a un contenedor

El servicio de Bluemix no se enlaza directamente a un contenedor utilizando el enlace de servicio.
Algunos servicios de Bluemix no dan soporte al enlace de servicio directo a un contenedor y conducen a un error durante la creación del contenedor. Para estos servicios, el enlace de servicio no pasa la variable de entorno VCAP_SERVICES al contenedor. Como solución temporal para este problema, puede utilizar la CLI para crear una app de Cloud Foundry que actúe como puente para enviar VCAP_SERVICES del servicio a un contenedor.
  1. Seleccione un espacio en el que desplegar un contenedor, como por ejemplo dev.
  2. Identifique una app existente o cree una app de Cloud Foundry con la que enlazar el servicio de Bluemix. Esta app se utilizará para pasar la información de VCAP_SERVICES; por eso, se conoce como app de puente.
    Para crear una app de Cloud Foundry:
    Consejo: La app que cree realizando estos pasos es un marcador para almacenar los servicios de VCAP_SERVICES. Para incurrir en menos costes que una app en ejecución, esta app no se inicia ni ejecuta. No es necesario ejecutar la app para proporcionar VCAP_SERVICES al contenedor.
    1. Cree un directorio y un archivo de texto en el directorio de las apps de trabajo. Ejemplo: <directorio>/<empty_file.txt>. Este archivo sirve como archivo de origen para la app, aunque no es necesario que albergue contenido.
      • Ejemplo de Windows: C:\Users\User\containerbridge\empty_file.txt
      • Ejemplo de Linux: /directorio_inicio_usuario/containerbridge/empty_file.txt
      • Ejemplo de OS X: /Users/directorio_inicio_usuario/containerbridge/empty_file.txt
    2. Ejecute este mandato para crear una app de tiempo de ejecución cero sin ningún tipo de coste. La app puede tener cualquier nombre aunque containerbridge es un nombre común, ya que la app actúa como puente entre el contenedor y un servicio.

      cf push <nombreapp> -p <directorio> -i 1 -d AppDomainName -k 1M -m 64M --no-hostname --no-manifest --no-route --no-start
      Ejemplo: 
      cf push containerbridge -p containerbridge -i 1 -d AppDomainName -k 1M -m 64M --no-hostname --no-manifest --no-route --no-start
  3. Si el servicio de Bluemix que desea enlazar aún no se ha añadido, añádalo al espacio de Bluemix.

Cannot create child process con el mandato su

Cuando se utiliza el mandato de Linux para sustituir un usuario (su) con el objeto de editar los usuarios de un contenedor, recibe el mensaje su: cannot create child process: Resource temporarily unavailable, aun cuando el mandato su haya funcionado en el entorno Docker local. Puede experimentar este problema al iniciar la sesión en un contenedor o cuando se ejecuta el mandato su como parte de un script dentro del contenedor.

Este problema se produce a menudo en contenedores creados con imágenes de Centos 7, aunque también puede ocurrir con otros tipos de imágenes.

Para resolver el problema editando el Dockerfile:
  1. En el directorio local desde el que está creando la imagen, cree una carpeta etc si todavía no existe.
    mkdir etc

    cd etc
  3. En la carpeta etc, cree un directorio pam.d.
    mkdir pam.d
  4. Vaya al directorio pam.d.
    cd pam.d
  5. En el directorio pam.d, cree un archivo denominado su y ábralo con un editor de texto.

    RUN sed -i 's/session.*include/session        optional/' /etc/pam.d/su

  9. Vuelva a crear un único contenedor o grupo de contenedores. Para obtener más información, consulte Creación de un contenedor único o Creación de un grupo de contenedores.

El contenedor se detiene y se cierra tras su creación

Ha creado un contenedor, pero se ha cerrado inmediatamente.

Ha ejecutado un contenedor de forma local y se ha ejecutado correctamente, pero al ejecutar el contenedor en Bluemix, verá una de las siguientes situaciones.
  • Desde el panel de control de Bluemix, el estado del contenedor cambiará de Running a Stopped.
  • Desde la CLI, al ejecutar bx ic inspect container, verá que el estado es Shutdown.
También ha intentado ejecutar bx ic logs, pero no ha encontrado ningún problema con el contenedor.

Al ejecutar un contenedor en Bluemix, el contenedor siempre se ejecuta en modalidad desconectada, de forma similar a la ejecución de docker run -d cuando un contenedor se ejecuta de forma local. El resultado es que los contenedores impares que se inician de forma local no se ejecutan de forma continua en Bluemix.

Debe convertir al contenedor en una aplicación de larga ejecución editando el Dockerfile. Por ejemplo, si ha añadido un mandato para que aparezca la frase "hello world" cada diez segundos, dicha acción continua convierte al contenedor en una aplicación de larga ejecución.
ENTRYPOINT ["/bin/sh"]
CMD ["-c","while true; do echo hello world; sleep 10;done"]

El contenedor suprimido continúa mostrándose en la interfaz gráfica de usuario (GUI) de Bluemix

Desde la interfaz gráfica de usuario (GUI) de Bluemix, ha suprimido un contenedor, pero el contenedor continuará visualizándose en la interfaz gráfica de usuario (GUI) de Bluemix y utiliza cuota.

No se ha podido completar la supresión desde la interfaz gráfica de usuario (GUI) de Bluemix.

Suprima el contenedor de la línea de mandatos.

En el caso de contenedores individuales, siga estos pasos.
  1. Suprima el contenedor ejecutando una de las siguientes opciones de línea de mandatos:
    • Plug-in de IBM Bluemix Container Service con mandatos bx ic
      bx ic rm [-f] CONTAINER [CONTAINER]
    • Plug-in de IBM Bluemix Container Service con mandatos docker 
      docker rm [-f] CONTAINER
  2. Verifique que se ha eliminado el contenedor mediante la ejecución de uno de los siguientes mandatos para listar todos los contenedores en el espacio:
    • Plug-in de IBM Bluemix Container Service con mandatos bx ic
      bx ic ps -a
    • Plug-in de IBM Bluemix Container Service con mandatos docker 
      docker ps -a
En el caso de grupos de contenedores, siga estos pasos.
  1. Suprima el grupo de contenedores ejecutando el siguiente mandato.
    • Plug-in de IBM Bluemix Container Service
      bx ic group-remove [-f] GROUP [GROUP]
  2. Verifique que el contenedor se ha eliminado ejecutando el mandato para listar todos los contenedores en el espacio.
    • Plug-in de IBM Bluemix Container Service
      bx ic groups [-q]

Si ejecuta los mandatos y el contenedor sigue mostrándose en la interfaz gráfica de usuario (GUI) de Bluemix, espere para ver si se elimina más tarde.

El grupo de contenedores muestra el estado HAS_INACTIVE_INSTANCES

Cuando ejecuta bx ic group-inspect, el grupo de contenedores muestra el estado HAS_INACTIVE_INSTANCES.

Si no se puede configurar la conexión de red cuando se crea el grupo o no se pasa la comprobación de estado para una o varias instancias, el estado de la instancia se cambia por INACTIVE. Si alguna instancia de un grupo está inactiva, el estado del grupo cambia a HAS_INACTIVE_INSTANCES.

Si el problema se produce cuando falla la comprobación de estado para una instancia, se puede solucionar sin ninguna acción. Si el problema se produce cuando el grupo se crea por primera vez, intente volver a crear el grupo. Si el problema se vuelve a producir después de volver a crear el grupo, es posible que la red no se pueda configurar para la app cuando se despliega como grupo. En su lugar, intente desplegar la app como contenedor único.

Resolución de problemas de almacenamiento de datos

Error de permiso al cambiar el propietario de un directorio de volumen montado en un contenedor

Se obtiene un error de permiso al intentar cambiar la propiedad de un archivo o un directorio para un volumen que está montado en un contenedor.
Para IBM Bluemix Container Service, la característica de espacio de nombres de usuario está habilitada para Docker Engine. La raíz efectiva dentro del contenedor está correlacionada con un usuario no root en el host de cálculo. El origen del volumen montado es un sistema de archivos de red (NFS). Dado que NFS no reconoce los espacios de nombres de usuario, todas las comprobaciones se llevan a cabo para las operaciones en archivos y directorios montados dentro del contenedor.

  1. En Dockerfile para una imagen, añada las siguientes líneas para crear un usuario con un ID de usuario (UID) disponible. Por ejemplo, puede llamar al usuario editor_volumen.
    RUN groupadd --gid 1010 editor_volumen
RUN useradd --uid 1010 --gid 1010 -m --shell /bin/bash editor_volumen
  2. Compile y envíe por push la imagen al registro de imágenes privado de Bluemix.
  3. Cree un contenedor con la imagen donde se ha montado la vía de acceso de volumen.
    Consejo: Ejecute bx ic namespace-get para recuperar el espacio de nombres y sustituya <my_namespace> por la información del espacio de nombres. 
    bx ic run --volume volume_name:/path/to/files --name fixVolumePermissions registry.DomainName/<my_namespace>/image
  4. Inicie la sesión en el contenedor.
    bx ic exec -it container_name bash
  5. Defina una variable de entorno llamada MOUNTPATH para la vía de acceso de montaje.
    MOUNTPATH="/vía_acceso/a/archivos"
  6. Añada el usuario al grupo root.
    adduser editor_volumen root
  7. Establezca permisos para que todos los miembros del grupo tengan acceso de lectura y grabación al montaje de volumen. 
    chmod 775 $MOUNTPATH
  8. Cree una carpeta en el mountpath y establezca los permisos de lectura y grabación. En este ejemplo, la carpeta se denomina data.
    su -c "mkdir -p $MOUNTPATH/data" editor_volumen
    su -c "chmod 700 $MOUNTPATH/data" editor_volumen
  9. Verifique que el usuario editor_volumen puede acceder a los archivos y directorios en la vía de acceso de montaje.
    sudo -u editor_volumen ls -al $MOUNTPATH
  10. Opcional: Enlazar una carpeta en el contenedor a la carpeta data en el volumen. Este enlace asocia los permisos de grabación de $MOUNTPATH/data con cualquier carpeta en el contenedor.
    ln -sf $MOUNTPATH/data /folder/in/container/
  11. Si se pueden realizar modificaciones sin un error de permiso, elimine el usuario root del grupo editor_volumen.
    deluser editor_volumen root

Los registros de instancias de grupo de contenedores se sobrescriben en un volumen

Al crear un grupo de contenedores, ha montado un volumen y ha generado salidas de registros de cada instancia de grupo de contenedores al volumen. Cada instancia de grupo de contenedores graba en los mismos archivos y los archivos se sobrescriben por cada una de las instancias.
Ejemplo:
bx ic group-create –p 9080 –m 64 -v log_vol:/opt/ibm/logs -e LOG_LOCATIONS=/opt/ibm/logs/messages.log,/opt/ibm/logs/debug.log,/opt/ibm/logs/trace.log -–name my_container_group registry.domain_name/ibmliberty:latest
Cada instancia de contenedor graba un archivo en el volumen como si fuera el único grabador. Cada instancia tiene una memoria caché local en memoria. Los archivos en el volumen se sustituyen repetidamente por los datos almacenados en memoria caché desde cada instancia de contenedor.
Ajuste el nombre del archivo de registro para incluir un identificador para que los archivos sean distintos para cada instancia. Por ejemplo, incluya la variable $HOSTNAME en el nombre de archivo de registro generado por la app y que se ha especificado cuando se ha creado el grupo de contenedor.
Ejemplo:
bx ic group-create –p 9080 –m 64 -v log_vol:/opt/ibm/logs -e LOG_LOCATIONS=/opt/ibm/logs/messages.$HOSTNAME.log,/opt/ibm/logs/debug.$HOSTNAME.log,/opt/ibm/logs/trace.$HOSTNAME.log -–name my_container_group registry.domain_name/ibmliberty:latest

La transferencia de datos a un volumen o desde el mismo tarda más de lo esperado

La lectura de un volumen o la grabación en el mismo tarda más de lo esperado.
En IBM Bluemix Container Service, los volúmenes no tienen acceso al host local. Por lo tanto, todos los volúmenes se montan en una compartición de archivos NFS. El rendimiento del volumen lo dicta el tamaño y la IOPS (entrada/salida por segundo) de la compartición de archivos, junto con la carga de trabajo de la aplicación. La compartición de archivos donde está alojado el volumen puede que no sea adecuado para la carga de trabajo de los volúmenes y contenedores. Además, varios volúmenes de una compartición de archivos pueden competir por recursos de la compartición de archivos.

Para crear una nueva compartición de archivos, debe tener el rol de gestor de organización. Para ver información detallada, consulte Usuarios y roles.

  1. Determinar el ID de la compartición de archivos en donde se aloja el volumen.
    bx ic volume-inspect my_volume
    {
    "fsId": "fa277ff4-8a64-435b-9b75-0f11d59a3ae4", 
    "hostPath": "/vol/fa277ff4-8a64-435b-9b75-0f11d59a3ae4/my_volume", 
    "otherSpaceVisibility": [], 
    "spaceGuid": "fa277ff4-8a64-435b-9b75-0f11d59a3ae4", 
    "volName": "my_volume"
}
  2. Ver la compartición de archivos para determinar su tamaño e IOPS.
    bx ic volume-fs
    Name                                   Size   IOPS/GB   Created               State   
fa277ff4-8a64-435b-9b75-0f11d59a3ae4   20     4.00      2016-05-20 13:48:10   READY
  3. Ver los tamaños disponibles en GB para las comparticiones de archivos.
    bx ic volume fs-flavor
    [
    20, 
    40, 
    80, 
    100, 
    250, 
    500, 
    1000, 
    2000, 
    4000, 
    8000, 
    12000
]
  4. Determinar el tamaño y la IOPS que necesita para la carga de trabajo de la app. Los valores de IOPS válidos son 0,25, 2 o 4 IOPS por GB. El tamaño de la compartición de archivos y la IOPS por GB se multiplican para determinar la IOPS total de la compartición de archivos. Esta tabla incluye ejemplos de cómo se escalan distintos tamaños de comparticiones de archivos a 4 IOPS por GB.
    Tabla 1. Ejemplos de tamaños de comparticiones de archivos e IOPS por GB
    Tamaños de comparticiones de archivos IOPS por GB IOPS total
    20 GB 4 80
    100 GB 4 400
    1.000 GB 4 4.000
    4.000 GB 4 16.000
  5. Crear una nueva compartición de archivos. El primer número es el tamaño en GB. El segundo número es la IOPS de la compartición de archivos.
    bx ic volume-fs-create my_new_fs 1000 4
  6. Verifique la nueva compartición de archivos y espere a que el estado aparezca como listo.
    bx ic volume-fs
  7. Cree el volumen en la nueva compartición de archivos.
    bx ic volume-create my_new_volume my_new_fs
  8. Ejecute un contenedor o un grupo de contenedores desde la imagen con el nuevo volumen montado.
  9. Haga una copia de seguridad de los datos de my_volume y copie los datos en my_new_volume. Consulte Copia de seguridad y restauración de datos para obtener más información.

Error de análisis al crear un contenedor con un volumen desde la CLI

Al crear un contenedor único o un grupo de contenedores desde la CLI, recibirá un mensaje de error.
Ejemplo

Problemas conocidos

Información sobre los problemas conocidos que se pueden producir en IBM Containers.

Docker y IBM Bluemix Container Service

Algunos mandatos de Docker no se admiten en el plug-in de IBM Bluemix Container Service (bx ic)

Sólo determinados mandatos de Docker se admiten en (bx ic). Puede ver una lista completa de los mandatos admitidos en la Referencia de mandatos de Docker. Por ejemplo, tag no está disponible en bx ic, pero docker tag puede utilizarse localmente y, a continuación, la imagen actualizada se puede enviar por push al registro de Bluemix.

Problemas comunes entre Docker y Windows

Existen problemas comunes en la instalación y configuración de Docker en sistemas Windows. Para obtener más información sobre estos problemas comunes, revise Blog: Instalación de Docker para Windows: Correcciones para problemas comunes y Publicaciones de Stack Overflow etiquetadas como boot2docker.

Imágenes

Se produce un error al enviar por push y recuperar de imágenes de gran tamaño

Al ejecutar un mandato Docker para enviar por push o recuperar una imagen de gran tamaño del registro privado de Bluemix de su organización, la solicitud falla. Para determinar si hay algún problema con la imagen, primero verifique que la imagen puede enviarse por push a DockerHub. Si no hay ningún problema con la imagen, si es posible, reduzca el tamaño de la imagen.

El tamaño de una imagen en Bluemix no coincide con su tamaño localmente

Cuando envía por push una imagen al registro privado de Bluemix, el tamaño notificado para la imagen es inferior al tamaño de la misma imagen en Docker Engine local. La diferencia entre los tamaños no indican que se ha producido un problema cuando la imagen se envió por push. El tamaño comprimido de la imagen se notifica en IBM Bluemix Container Service.

Supervisión y registro

Caracteres adicionales de la salida para bx ic logs
Al ejecutar bx ic logs, los caracteres adicionales, como &, se muestran al principio de cada línea. Este problema se produce cuando los registros de docker entrantes se analizan mediante NGINX.
No se pueden utilizar expresiones para especificar las adiciones de archivos de registro

No es posible utilizar filtros ni expresiones comodín para especificar los archivos de registro que deben recopilarse. Por ejemplo, no es posible especificar /var/log/*.log. En su lugar, liste cada archivo de registro individualmente.

Contenedores únicos y grupos de contenedores

Error de conexión después de crear un contenedor

Debido a una limitación en la arquitectura de redes de Docker de IBM Bluemix Container Service, es posible que no funcione el acceso de red saliente al iniciar el contenedor. Si el contenedor intenta efectuar una conexión de red saliente para devolver un código de error o para generar una excepción si la red no está disponible, el contenedor se cierra y se muestra el estado "Bloqueado".

Para permitir que la red se inicialice y para habilitar que continúe el despliegue del contenedor, debe tener una sentencia de inactividad antes de iniciar cualquier proceso que requiera una conectividad saliente. Cree un Dockerfile personalizado para recalcular los mandatos de inicio de servidor existentes en el Dockerfile base. Para ver un ejemplo, consulte la aplicación de muestra de Bluemix Let's Chat.

La conectividad de entrada no es inmediata

La conectividad de entrada no es inmediata para los grupos de contenedores ni para los contenedores individuales, incluidas las rutas y las apps externas. Espere a que la conectividad esté disponible (pueden pasar hasta 5 minutos).

Distintas unidades de medida predeterminadas utilizadas en docker run -m y bx ic run -m
Si no se especifica ninguna unidad de medida en docker run -m, se utilizan los bytes como unidad de medida predeterminada. Si no se especifica ninguna unidad de medida con bx ic run -m, se utilizan MB como unidad de medida predeterminada.
El límite de datos de usuario impide que se cree un contenedor
Si tiene demasiados servicios enlazados a un contenedor, no podrá crearlo. Cada servicio inyecta datos al contenedor y se cuenta hacia un límite de 64 KB. Al crear un contenedor que supera este límite, recibe un mensaje Ha fallado el recurso CREATE... Datos de usuario demasiado grandes. Si ve este mensaje, elimine un servicio del contenedor.
El puerto 25 no está disponible para exponerlo en un contenedor
El uso del puerto 25 está bloqueado por SoftLayer, que forma parte de la arquitectura de IBM Bluemix Container Service. Este problema afecta al uso de algunos servicios SMTP en contenedores. Para obtener más información sobre la razón por la que este puerto está bloqueado, consulte el artículo Outbound Email on Port 25 en la documentación de SoftLayer.
La creación del contenedor o el proceso de red no se completa
Si un contenedor permanece en el estado de creación o de establecimiento de red durante demasiado tiempo, es posible que tenga que eliminar el contenedor y crear uno nuevo a partir de la misma imagen. Puede utilizar los siguientes mandatos bx ic de la CLI:
  1. Recuperar una lista de contenedores que se ejecutan en el espacio. De forma predeterminada, solo se muestran los contenedores en ejecución.Para obtener más información, revise la Referencia de bx ic ps de la CLI.
    bx ic ps -a
  2. Eliminar uno o varios contenedores.Para obtener más información, revise la Referencia de bx ic rm de la CLI. 
    bx ic rm -f ID_contenedor
  3. Iniciar un contenedor nuevo en el servicio en la nube de contenedores a partir del nombre de una imagen.Para obtener más información, revise la Referencia de bx ic run de la CLI. 
    bx icrun
La recuperación automática no es inmediata para grupos de contenedores

La recuperación automática de los grupos de contenedores puede tardar más de 15 minutos para que los nuevos sistemas se pongan en línea. Espere a que la recuperación automática esté disponible (pueden pasar más de 15 minutos).

El mandato cf delete-orphaned-routes elimina rutas de grupos de contenedores

Cloud Foundry no reconoce rutas correlacionadas con grupos de contenedores de la misma forma que reconoce rutas como vinculadas a aplicaciones. No ejecute el mandato cf delete-orphaned-routes cuando utilice grupos de contenedores.

Obtención de ayuda y soporte para IBM Bluemix Container Service

¿Dónde comienzo a resolver problemas de un contenedor?

  • Para ver si Bluemix está disponible, consulte la página de estado de Bluemix.
  • Revise los foros para ver si otros usuarios se han encontrado con el mismo problema. Cuando utiliza los foros para formular una pregunta, etiquete la pregunta para que la puedan ver los equipos de desarrollo de Bluemix.
    • Si tiene preguntas técnicas sobre cómo desarrollar o desplegar una app con IBM Bluemix Container Service, publique la pregunta en Stack Overflow y etiquete la pregunta con ibm-bluemix y containers.
    • Para las preguntas relativas a las instrucciones de inicio y el servicio, utilice el foro IBM developerWorks dW Answers. Incluya las etiquetas bluemix y containers.
    Consulte Obtención de ayuda para obtener más detalles sobre cómo utilizar los foros.
  • Póngase en contacto con el soporte de IBM. Para obtener información sobre cómo abrir una incidencia de soporte de IBM, o sobre los niveles de soporte y las gravedades de las incidencias, consulte Cómo contactar con el servicio de soporte.

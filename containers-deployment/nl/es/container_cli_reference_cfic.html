bx ic attach [--no-stdin] [--sig-proxy] CONTAINER

Finalidad
Controlar un contenedor en ejecución o ver su resultado. Utilice CTRL+C para salir y detener el contenedor. Para obtener más información, consulte el mandato attach en la ayuda de Docker.
Parámetros
--no-stdin
(Opcional) No incluir la entrada estándar.
--sig-proxy
(Opcional) El valor predeterminado es true.
CONTAINER
(Obligatorio) Un nombre de contenedor o ID.
Ejemplo
El ejemplo siguiente es una solicitud para agregarla al contenedor my_container.
bx ic attach my_container

bx ic build -t TAG [--no-cache] [--pull] [-q] DOCKERFILE_LOCATION

Finalidad
Crear una imagen de Docker localmente en el registro de imágenes de Bluemix privado. Para obtener más información, consulte el mandato build en la ayuda de Docker.
Parámetros
DOCKERFILE_LOCATION
(Necesario) La vía de acceso a Dockerfile y al contexto en el host local.
--no-cache
(Opcional) Cuando se crea una imagen por primera vez, las capas de la imagen se almacenan en la caché para toda la organización. Cuando se crean versiones posteriores de la imagen, estas versiones de la imagen almacenadas en la caché se utilizan para acelerar el proceso de creación. Si incluye la opción --no-cache al crear una nueva versión de la imagen, se asegurará de que todas las capas de la imagen también se creen de nuevo.
--pull
(Opcional) De forma predeterminada, la imagen base sólo se obtiene del registro si no se encuentra en la memoria caché de imágenes de la organización. Si incluye esta opción, se asegurará de que se utilice como mínimo la última versión para crear la nueva imagen.
-q, --quiet
(Opcional) Elimina la salida detallada que generan los contenedores. El valor predeterminado es false.
-t TAG, --tag TAG
(Necesario) La vía de acceso al registro de imágenes privado que se aplica a la imagen que se crea.
Ejemplo
El ejemplo siguiente es una solicitud para crear una imagen llamada myimage. El Dockerfile y otros artefactos que deben utilizarse en el proceso de creación están en el mismo directorio en el que se ejecuta el mandato. Dado que el registro y el espacio de nombres se incluyen con el nombre de la imagen, esta se crea en el registro de imágenes privado de la organización de Bluemix.
Consejo: Ejecute bx ic namespace-get para recuperar el espacio de nombres y sustituya <my_namespace> por la información del espacio de nombres. 
bx ic build -t registry.DomainName/<mi_espacio_nombres>/mi_imagen.

bx ic cp SRC_PATH CONTAINER:DEST_PATH | bx ic cp CONTAINER:SRC_PATH DEST_PATH

Finalidad
Copie un archivo o una carpeta de un único contenedor en ejecución o detenido a la máquina local y viceversa. Para obtener más información, consulte el mandato cp en la ayuda de Docker.
Nota: Este mandato únicamente se admite para utilizarlo con contenedores únicos. Cuando desee copiar archivos de o en un grupo de contenedores, debe copiar los archivos de o en cada instancia de contenedor del grupo.
Parámetros
DEST_PATH
(Necesario) La vía de acceso al archivo o carpeta en la máquina local o contenedor en el que desea copiar la información desde la vía de acceso de origen. El archivo o carpeta debe existir en su máquina local o en el contenedor antes de comenzar a copiar. Si desea copiar en su máquina local, puede utilizar la vía de acceso absoluta o relativa. Si desea copiar en su contenedor único, debe incluir el nombre del contenedor, el delimitador ":" y la vía de acceso en el archivo o la carpeta de destino. La vía de acceso debe ser relativa al directorio raíz del contenedor.
SRC_PATH
(Necesario) La vía de acceso al archivo o carpeta que desee copiar. Si el archivo o carpeta está en la máquina local, puede utilizar la vía de acceso absoluta o relativa. Si el archivo o la carpeta está en el contenedor único, debe incluir el nombre del contenedor, el delimitador ":" y la vía de acceso al archivo o a la carpeta que desee copiar. La vía de acceso debe ser relativa al directorio raíz del contenedor.
Nota: No puede copiar determinados archivos del sistema que se almacenan en la carpeta /dev, /etc, /proc y /sys del contenedor.
Ejemplo
Este mandato de ejemplo copia un archivo denominado config.txt del directorio actual de la máquina local en el directorio config de un solo contenedor denominado my_container.
bx ic cp ./config.txt my_container:/config

bx ic cpi SRC DST

Finalidad
Acceda a una imagen de Docker Hub o a una imagen del registro local y cópiela en el registro de imágenes privado de Bluemix.
Nota: Este mandato es una alternativa a utilizar docker pull y docker push conjuntamente para añadir una imagen al registro de imágenes de Bluemix. Si prefiere utilizar docker pull y docker push, consulte Envío por push de imágenes al registro de Bluemix privado desde la línea de mandatos.
Parámetros
DST
(Necesario) El URL de registro de imágenes privado de Bluemix, que incluye el espacio de nombres, y el nombre de la imagen de destino. La etiqueta de la imagen es opcional.
SRC
(Necesario) El repositorio de origen y el nombre de la imagen.
Ejemplo
bx ic cpi repositorio_origen/nombre_imagen_origen URL_registro_privado/nombre_imagen_destino:etiqueta
Consejo: Ejecute bx ic namespace-get para recuperar el espacio de nombres y sustituya <my_namespace> por la información del espacio de nombres. 
bx ic cpi training/sinatra registry.DomainName/<mi_espacio_nombres>/mysinatra:v1

bx ic exec [-d] [-it ] [-u USER] CONTAINER CMD

Finalidad
Ejecutar un mandato dentro de un contenedor. Para obtener más información, consulte el mandato exec en la ayuda de Docker.
Parámetros
CMD
(Necesario) El mandato a ejecutar en el contenedor o contenedores especificados.
CONTAINER
(Obligatorio) Un nombre de contenedor o ID.
-d , --detach
(Opcional) Ejecute el mandato especificado en segundo plano.
-it
(Opcional) Modalidad interactiva. Mantenga visualizada la entrada estándar. Escriba exit para salir.
-u USER, --user USER
(Opcional) Un nombre de usuario.
Ejemplos
Este mandato de ejemplo ejecuta un terminal bash interactivo en el contenedor my_container.
bx ic exec -it my_container bash
Este mandato de ejemplo ejecuta el mandato date en el contenedor my_container. 
bx ic exec my_container date

bx ic group-create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session-affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

Finalidad
Crear un grupo de escalado. De forma predeterminada, los contenedores en Bluemix se ejecutan en modalidad desconectada.
Parámetros
--anti
(Opcional) Utilice la antiafinidad para hacer que el grupo de contenedores esté más altamente disponible. La opción --anti fuerza que cada instancia de contenedor de un grupo se sustituya en un nodo de cálculo físico distinto, lo que reduce las probabilidades de que todos los contenedores de un grupo se bloqueen debido a una anomalía de hardware. Es posible que no pueda utilizar esta opción con tamaños de grupos más grandes porque cada organización y región de Bluemix tiene un conjunto limitado de nodos de cálculo disponibles para el despliegue. Si el despliegue no se ejecuta de forma satisfactoria, reduzca el número de instancias de contenedor en el grupo o elimine la opción --anti.
--auto
(Opcional) Cuando se habilite la recuperación automática para un grupo de contenedores y el grupo se haya estado ejecutando 10 minutos, el equilibrador de carga del grupo comenzará a comprobar de forma regular el estado de cada instancia de contenedor en el grupo mediante solicitudes HTTP. Si una instancia de contenedor no responde en el espacio de 100 segundos, se marcará como inactiva. Las instancias de contenedor inactivas se eliminarán del grupo y las volverá a crear la recuperación automática. La recuperación automática intentará recuperar instancias de contenedor en un grupo 3 veces. Tras el tercer intento, la recuperación automática no recuperará ninguna instancia de contenedor en el grupo durante 60 minutos. Tras 60 minutos, el proceso de recuperación automática empezará de nuevo.Para obtener más información sobre la comprobación de estado, consulte la opción --http_monitor_enabled.
CMD
(Opcional) El mandato y los argumentos se pasan al grupo de contenedores para su ejecución. Este mandato debe ser un mandato de larga ejecución. No utilice un mandato breve, como por ejemplo /bin/date, ya que ello podría hacer que el contenedor se bloqueara.
Mandatos de larga ejecución de ejemplo que utilizan la imagen ibmnode proporcionada por IBM Bluemix Container Service:
bx ic group-create --name my_container_group registry.DomainName/ibmnode ping localhost
bx ic group-create --name my_container_group registry.DomainName/ibmnode -- tail -f /dev/null
Nota: CMD y sus argumentos deben ir al final de la línea de mandatos bx ic group-create.
-d DOMAIN, --domain DOMAIN
(Opcional) El dominio de sistema predeterminado es AppDomainName y ya proporciona un certificado SSL, por lo que puede acceder a los grupos de contenedores con HTTPS sin ninguna configuración adicional. Para utilizar un dominio personalizado, debe registrar el dominio personalizado en un servidor de DNS público, configurar el dominio personalizado en Bluemix, y luego correlacionar el dominio personalizado con el dominio del sistema Bluemix en el servidor de DNS público. Después de correlacionar el dominio personalizado con el dominio del sistema Bluemix, las solicitudes correspondientes al dominio personalizado se direccionan a la aplicación en Bluemix. Cuando cree un dominio personalizado, no incluya caracteres de subrayado (_) en el nombre de dominio. Para obtener más información sobre dominios personalizados, consulte Creación y uso de un dominio personalizado. Para que el dominio personalizado sea seguro, cargue un certificado SSL para poder acceder al grupo de contenedores con HTTPS.

El host y el dominio combinados forman el URL completo de la vía de acceso pública, como por ejemplo, http://mycontainerhost.AppDomainName y debe ser único en Bluemix. Cuando revise los detalles de un grupo de contenedores con el mandato bx ic group-inspect, el host y el dominio aparecen juntos como la ruta.

--desired DESIRED
(Opcional) El número de instancias en el grupo. El valor predeterminado es 2. El número de instancias de grupos de contenedores que la app requiere dependerá de la carga de trabajo esperada y del tiempo en el que la app puede procesar una solicitud. Por ejemplo, la multitarea o la conectividad de red afecta a cuántas instancias pueden ser necesarias para la app. Para determinar el número de instancias deseadas, despliegue un contenedor único en Bluemix que se ejecuta en la app y lleve a cabo una prueba de carga en este contenedor. Si la app puede manejar, por ejemplo, 100 solicitudes por segundo, pero tiene previsto una carga de trabajo media de 300 solicitudes por segundo, el número deseado de instancias para el grupo de contenedor es 3. Para que el grupo de contenedores esté altamente disponible y más resistente a fallos, considere incluir más instancias que el mínimo para manejar la carga de trabajo esperada. Las instancias adicionales pueden manejar la carga de trabajo en el caso de que se bloque otra instancia. Para asegurarse la recuperación automática de una instancia fallida sin afectar la carga de trabajo, incluya una instancia adicional. Para la protección frente a dos anomalías simultáneas, incluya dos instancias adicionales. Esta configuración es un patrón N+2, donde N es el número de instancias para manejar las solicitudes y +2 son dos instancias adicionales.
Importante: Un grupo de contenedor debe incluir al menos 2 instancias de contenedor. Si sólo incluye 1 instancia en el grupo y esa instancia deja de estar disponible, los usuarios pueden experimentar un tiempo de inactividad para la app.
-e ENV, --env ENV
(Opcional) Establezca la variable de entorno donde ENV es un par key=value. Liste varias claves por separado y si incluye comillas, especifíquelas alrededor del nombre y del valor de la variable de entorno. Ejemplo: -e "key1=value1" -e "key2=value2a,value2b" -e "key3=value3"

--http_monitor_enabled
(Opcional) El valor predeterminado es true. Cuando se habilite la recuperación automática para un grupo de contenedores y el grupo se haya estado ejecutando 10 minutos, el equilibrador de carga del grupo comenzará a comprobar de forma regular el estado de cada instancia de contenedor en el grupo mediante solicitudes HTTP. Si una instancia de contenedor no responde en el espacio de 100 segundos, se marcará como inactiva. Las instancias de contenedor inactivas se eliminarán del grupo y las volverá a crear la recuperación automática. La recuperación automática intentará recuperar instancias de contenedor en un grupo 3 veces. Tras el tercer intento, la recuperación automática no recuperará ninguna instancia de contenedor en el grupo durante 60 minutos. Tras 60 minutos, el proceso de recuperación automática empezará de nuevo. Algunas apps podrían no requerir una comprobación de estado HTTP, como, por ejemplo, una app que utilice TCP para la comunicación en lugar de HTTP. Para inhabilitar el monitor HTTP, establezca la opción en false. Ejemplo: --http_monitor_enabled=false
--http_monitor_path
(Opcional) El valor predeterminado es /. Cuando está habilitada --http_monitor_enabled, puede usar esta opción para establecer una vía de acceso personalizada para la comprobación de estado, por ejemplo, --http_monitor_path="/health/path".
--http_monitor_rc_list
(Opcional) Cuando está habilitada --http_monitor_enabled, puede especificar la lista de códigos de respuesta para la comprobación de estado. El valor predeterminado incluye los códigos 200, 201, 202, 204, 300, 301, 302, 401, 403 y 404. Cualquier respuesta que no se encuentre en la lista de respuestas válidas cambia el estado de instancia del contenedor a INACTIVE mediante el equilibrador de carga. Un contenedor inactivo no recibe ninguna carga y el estado del grupo de contenedores cambia a HAS_INACTIVE_INSTANCES. Si ninguno de los contenedores está en estado ACTIVE, el estado del grupo de contenedores pasa a ser NO_ACTIVE_INSTANCES. Para obtener más información sobre los estados, consulte Estados de contenedor en IBM Bluemix Container Service.
IMAGE
(Necesario) La imagen que desea incluir en cada instancia de contenedor del grupo de contenedores. Puede listar mandatos después de la imagen, pero no coloque ninguna opción después de la imagen; incluya todas las opciones antes de especificar una imagen.

Si utiliza una imagen en el registro de imágenes privado de la organización de Bluemix, especifíquela con el siguiente formato: registry.DomainName/ESPACIO_NOMBRES/IMAGEN

Si utiliza una imagen proporcionada por IBM Bluemix Container Service, no incluya el espacio de nombres de su organización. Especifique la imagen con el siguiente formato: registry.DomainName/IMAGEN

--ip IP_ADDRESS
(Opcional) Si tiene una dirección IP disponible, enlace una dirección IP pública al grupo de contenedores. También debe exponer el puerto con el distintivo -p. Un grupo de contenedores puede tener una ruta pública o una dirección IP pública, pero no ambas. Para desenlazar la dirección IP y devolver la IP a la cuota del espacio, elimine el grupo de contenedores y vuelva a crear el grupo sin la dirección IP. Para solicitar una dirección IP antes de crear un grupo de contenedores, ejecute bx ic ip-request.
Nota: El enlace de una dirección IP pública no incluye un certificado SSL que cifra los datos que se envían a un grupo de contenedores o desde el mismo. Si la app requiere el cifrado SSL, puede implementar su propia solución SSL o correlacionar una ruta pública con el grupo de contenedores en lugar de enlazar una dirección IP pública. Las rutas públicas ya incluyen un certificado SSL, por lo que puede acceder al grupo de contenedores con HTTPS sin ninguna configuración adicional.
-m MEMORY, --memory MEMORY
(Opcional) Especifique un límite de memoria para el contenedor en MB. El límite de memoria forma parte del tamaño del contenedor que define la cantidad máxima de memoria y espacio de disco que un contenedor obtiene en el host de cálculo durante el tiempo de ejecución. Tras asignar un tamaño de contenedor, el valor no se puede cambiar. Los tamaños disponibles en IBM Bluemix Container Service son:
  • Pico (64 MB de memoria, 4 GB de espacio en disco)
  • Nano (128 MB de memoria, 8 GB de espacio en disco)
  • Micro (256 MB de memoria, 16 GB de espacio en disco)
  • Muy pequeño (512 MB de memoria, 32 GB de espacio en disco)
  • Pequeño (1024 MB de memoria, 64 GB de espacio en disco)
  • Medio (2048 MB de memoria, 128 GB de espacio en disco)
  • Grande (4096 MB de memoria, 256 GB de espacio en disco)
  • Extra grande (8192 MB de memoria, 512 GB de espacio en disco)
  • Extra grande 2X (16384 MB de memoria, 1 TB de espacio en disco)
Si no establece un tamaño para el contenedor, cada instancia de contenedor se crea con 256 MB.
Importante: Especifique la memoria del contenedor en MB sin la etiqueta de unidad. Por ejemplo, si desea crear un contenedor de tamaño pico, especifique -m 64.
--max MAX
(Opcional) Número máximo de instancias. El valor predeterminado es 2.
--min MIN
(Opcional) Número mínimo de instancias. El valor predeterminado es 1.
-n HOST, --hostname HOST
(Opcional) Nombre de host, como mycontainerhost. No incluya caracteres de subrayado (_) en el nombre de host. El host y el dominio combinados forman el URL completo de la ruta pública, como http://mycontainerhost.AppDomainName. Cuando revise los detalles de un grupo de contenedores con el mandato bx ic group-inspect, el host y el dominio aparecen juntos como la ruta.
--name NAME
(Necesario) Asigna un nombre al grupo. -n está en desuso.
Consejo: El nombre de contenedor debe empezar por una letra, y luego puede incluir letras en mayúsculas, letras en minúsculas, números, puntos (.), caracteres de subrayado (_) o guiones (-).
-p PORT, --publish PORT

(Obligatorio) Expone el puerto para el tráfico de red HTTP y privado. Si no expone un puerto, se ejecutará el grupo de contenedores, pero no se podrá acceder al grupo de contenedores a través del equilibrador de carga del grupo en la red pública o privada.

No es posible importar varios puertos para los grupos de contenedor. Si se enumeran varios puertos, sólo está expuesto el último puerto listado. El mandato omitirá el resto de los puertos.

Cuando enlaza una ruta, los contenedores del grupo deben estar a la escucha del tráfico HTTP en el puerto expuesto al grupo. Los puertos no HTTP no pueden exponerse públicamente. Cuando el tráfico HTTPS llega al puerto expuesto, (Go)Router completa la terminación de HTTPS. A continuación, el protocolo HTTP se utiliza en la red privada virtual entre (Go)Router y los contenedores. Cuando enlace una dirección IP, los contenedores del grupo pueden estar a la escucha del tráfico HTTP o HTTPs, pero puede seguir exponiendo sólo un puerto.
Si especifica un puerto en el archivo de Docker para la imagen que está utilizando, incluya dicho puerto.
Consejo:
  • Para la imagen de Liberty Server certificada por IBM o una versión modificada de esta imagen, escriba el puerto 9080.
  • Para la imagen de Node.js certificada por IBM o una versión modificada de esta imagen, escriba cualquier puerto, tales como el puerto 8000.
--session_affinity

(Opcional) El valor predeterminado es false. Cuando --http_monitor_enabled está establecida en true, puede establecer la opción --session_affinity en true para permitir que IBM Bluemix Container Service cree y gestione una cookie de sesión en nombre de la app. Esta opción no está disponible si --http_monitor_enabled está establecida en false o si utiliza TCP para la comunicación en lugar de HTTP.

--volume VOLUME:/DIRECTORY_PATH[:ro]
(Opcional) Adjunte un volumen a un contenedor especificando los detalles con el siguiente formato VOLUME:/DIRECTORY_PATH[:ro].

-m MEMORY, --memory MEMORY
(Opcional) Especifique un límite de memoria para el contenedor en MB. El límite de memoria forma parte del tamaño del contenedor que define la cantidad máxima de memoria y espacio de disco que un contenedor obtiene en el host de cálculo durante el tiempo de ejecución. Tras asignar un tamaño de contenedor, el valor no se puede cambiar. Los tamaños disponibles en IBM Bluemix Container Service son:
  • Pico (64 MB de memoria, 4 GB de espacio en disco)
  • Nano (128 MB de memoria, 8 GB de espacio en disco)
  • Micro (256 MB de memoria, 16 GB de espacio en disco)
  • Muy pequeño (512 MB de memoria, 32 GB de espacio en disco)
  • Pequeño (1024 MB de memoria, 64 GB de espacio en disco)
  • Medio (2048 MB de memoria, 128 GB de espacio en disco)
  • Grande (4096 MB de memoria, 256 GB de espacio en disco)
  • Extra grande (8192 MB de memoria, 512 GB de espacio en disco)
  • Extra grande 2X (16384 MB de memoria, 1 TB de espacio en disco)
Si no establece un tamaño para el contenedor, cada instancia de contenedor se crea con 256 MB.

(Necesario) Asigna un nombre al contenedor.

-p PORT, --publish PORT

(Opcional) Si desea que la app sea accesible desde Internet, debe exponer un puerto público. Cuando expone un puerto público, cree un Grupo de seguridad de red pública para el contenedor que le permita enviar y recibir datos públicos únicamente en el puerto expuesto. Los demás puertos públicos están cerrados y no pueden utilizarse para acceder a la app desde Internet. Puede incluir varios puertos con varias opciones -p. Los puertos no se pueden correlacionar ni reenviar. Si no expone un puerto, el contenedor solamente es accesible desde la red de contenedor privado. Puede utilizar la dirección IP privada asignada para comunicarse con el contenedor en la red privada.

Si especifica un puerto en el archivo de Docker para la imagen que está utilizando, incluya dicho puerto.
Consejo:
  • Para la imagen de Liberty Server certificada por IBM o una versión modificada de esta imagen, escriba el puerto 9080.
  • Para la imagen de Node.js certificada por IBM o una versión modificada de esta imagen, escriba cualquier puerto, tales como el puerto 8000.

Puede elegir entre UDP y TCP para indicar el protocolo IP que desea utilizar. Si no especifica ningún protocolo, el puerto se expone automáticamente para el tráfico TCP.

Ejemplos
  • -p 9080
  • -p 9080/udp
  • -p 9080/udp -o 8000/tcp

Si tiene una dirección IP en el espacio al que desea enlazar el contenedor, puede especificar la dirección IP con la información del puerto. Incluya la dirección IP, el puerto y la dirección IP y el número de puerto de contenedor, como por ejemplo <dirección_IP>:<puerto_en_IP>:<puerto_contenedor>. El puerto de la dirección IP debe coincidir con el puerto de contenedor expuesto en Dockerfile.

Ejemplo
-p 192.0.2.23:9080:9080
Nota: El enlace de una dirección IP pública no incluye un certificado SSL que cifra los datos que se envían a un contenedor único o desde el mismo. Si la app requiere el cifrado SSL, puede implementar su propia solución SSL o utilizar un grupo de contenedores en lugar de un contenedor único. Los grupos de contenedores están enlazados a una ruta pública que ya incluye un certificado SSL, por lo que puede acceder al grupo de contenedores con HTTPS sin ninguna configuración adicional. Para más información, consulte Creación de un grupo de contenedores con la interfaz gráfica de usuario de Bluemix.

Puede ver el estado actual del contenedor ejecutando el mandato bx ic inspect y localizando el campo Status. El estado del contenedor puede ofrecerle información adicional sobre la disponibilidad de las instancias del contenedor y los problemas que podrían haber surgido.

Finalidad
Crear un grupo de escalado. De forma predeterminada, los contenedores en Bluemix se ejecutan en modalidad desconectada.
Parámetros
--anti
(Opcional) Utilice la antiafinidad para hacer que el grupo de contenedores esté más altamente disponible. La opción --anti fuerza que cada instancia de contenedor de un grupo se sustituya en un nodo de cálculo físico distinto, lo que reduce las probabilidades de que todos los contenedores de un grupo se bloqueen debido a una anomalía de hardware. Es posible que no pueda utilizar esta opción con tamaños de grupos más grandes porque cada organización y región de Bluemix tiene un conjunto limitado de nodos de cálculo disponibles para el despliegue. Si el despliegue no se ejecuta de forma satisfactoria, reduzca el número de instancias de contenedor en el grupo o elimine la opción --anti.
--auto
(Opcional) Cuando se habilite la recuperación automática para un grupo de contenedores y el grupo se haya estado ejecutando 10 minutos, el equilibrador de carga del grupo comenzará a comprobar de forma regular el estado de cada instancia de contenedor en el grupo mediante solicitudes HTTP. Si una instancia de contenedor no responde en el espacio de 100 segundos, se marcará como inactiva. Las instancias de contenedor inactivas se eliminarán del grupo y las volverá a crear la recuperación automática. La recuperación automática intentará recuperar instancias de contenedor en un grupo 3 veces. Tras el tercer intento, la recuperación automática no recuperará ninguna instancia de contenedor en el grupo durante 60 minutos. Tras 60 minutos, el proceso de recuperación automática empezará de nuevo.Para obtener más información sobre la comprobación de estado, consulte la opción --http_monitor_enabled.
CMD
(Opcional) El mandato y los argumentos se pasan al grupo de contenedores para su ejecución. Este mandato debe ser un mandato de larga ejecución. No utilice un mandato breve, como por ejemplo /bin/date, ya que ello podría hacer que el contenedor se bloqueara.
Mandatos de larga ejecución de ejemplo que utilizan la imagen ibmnode proporcionada por IBM Bluemix Container Service:
cf ic group create --name my_container_group registry.DomainName/ibmnode ping localhost
cf ic group create --name my_container_group registry.DomainName/ibmnode -- tail -f /dev/null
Nota: CMD y sus argumentos deben ir al final de la línea de mandatos cf ic group create.
-d DOMAIN, --domain DOMAIN
(Opcional) El dominio de sistema predeterminado es AppDomainName y ya proporciona un certificado SSL, por lo que puede acceder a los grupos de contenedores con HTTPS sin ninguna configuración adicional. Para utilizar un dominio personalizado, debe registrar el dominio personalizado en un servidor de DNS público, configurar el dominio personalizado en Bluemix, y luego correlacionar el dominio personalizado con el dominio del sistema Bluemix en el servidor de DNS público. Después de correlacionar el dominio personalizado con el dominio del sistema Bluemix, las solicitudes correspondientes al dominio personalizado se direccionan a la aplicación en Bluemix. Cuando cree un dominio personalizado, no incluya caracteres de subrayado (_) en el nombre de dominio. Para obtener más información sobre dominios personalizados, consulte Creación y uso de un dominio personalizado. Para que el dominio personalizado sea seguro, cargue un certificado SSL para poder acceder al grupo de contenedores con HTTPS.

El host y el dominio combinados forman el URL completo de la vía de acceso pública, como por ejemplo, http://mycontainerhost.AppDomainName y debe ser único en Bluemix. Cuando revise los detalles de un grupo de contenedores con el mandato cf ic group inspect, el host y el dominio aparecen juntos como la ruta.

--desired DESIRED
(Opcional) El número de instancias en el grupo. El valor predeterminado es 2. El número de instancias de grupos de contenedores que la app requiere dependerá de la carga de trabajo esperada y del tiempo en el que la app puede procesar una solicitud. Por ejemplo, la multitarea o la conectividad de red afecta a cuántas instancias pueden ser necesarias para la app. Para determinar el número de instancias deseadas, despliegue un contenedor único en Bluemix que se ejecuta en la app y lleve a cabo una prueba de carga en este contenedor. Si la app puede manejar, por ejemplo, 100 solicitudes por segundo, pero tiene previsto una carga de trabajo media de 300 solicitudes por segundo, el número deseado de instancias para el grupo de contenedor es 3. Para que el grupo de contenedores esté altamente disponible y más resistente a fallos, considere incluir más instancias que el mínimo para manejar la carga de trabajo esperada. Las instancias adicionales pueden manejar la carga de trabajo en el caso de que se bloque otra instancia. Para asegurarse la recuperación automática de una instancia fallida sin afectar la carga de trabajo, incluya una instancia adicional. Para la protección frente a dos anomalías simultáneas, incluya dos instancias adicionales. Esta configuración es un patrón N+2, donde N es el número de instancias para manejar las solicitudes y +2 son dos instancias adicionales.
Importante: Un grupo de contenedor debe incluir al menos 2 instancias de contenedor. Si sólo incluye 1 instancia en el grupo y esa instancia deja de estar disponible, los usuarios pueden experimentar un tiempo de inactividad para la app.

Los puertos no se pueden correlacionar ni reenviar. Si no expone un puerto, el contenedor solamente es accesible desde la red de contenedor privado. Puede utilizar la dirección IP privada asignada para comunicarse con el contenedor en la red privada.

Si especifica un puerto en el archivo de Docker para la imagen que está utilizando, incluya dicho puerto.
Consejo:
  • Para la imagen de Liberty Server certificada por IBM o una versión modificada de esta imagen, escriba el puerto 9080.
  • Para la imagen de Node.js certificada por IBM o una versión modificada de esta imagen, escriba cualquier puerto, tales como el puerto 8000.

Puede elegir entre UDP y TCP para indicar el protocolo IP que desea utilizar. Si no especifica ningún protocolo, el puerto se expone automáticamente para el tráfico TCP.

Ejemplos
  • -p 9080
  • -p 9080/udp
  • -p 9080/udp -o 8000/tcp

Si tiene una dirección IP en el espacio al que desea enlazar el contenedor, puede especificar la dirección IP con la información del puerto. Incluya la dirección IP, el puerto y la dirección IP y el número de puerto de contenedor, como por ejemplo <dirección_IP>:<puerto_en_IP>:<puerto_contenedor>. El puerto de la dirección IP debe coincidir con el puerto de contenedor expuesto en Dockerfile.

Ejemplo
-p 192.0.2.23:9080:9080
Nota: El enlace de una dirección IP pública no incluye un certificado SSL que cifra los datos que se envían a un contenedor único o desde el mismo. Si la app requiere el cifrado SSL, puede implementar su propia solución SSL o utilizar un grupo de contenedores en lugar de un contenedor único. Los grupos de contenedores están enlazados a una ruta pública que ya incluye un certificado SSL, por lo que puede acceder al grupo de contenedores con HTTPS sin ninguna configuración adicional. Para más información, consulte Creación de un grupo de contenedores con la interfaz gráfica de usuario de Bluemix.
-P
(Opcional) Exponer automáticamente los puertos especificados en el archivo de Docker de la imagen para el tráfico HTTP.
